## 1. 다시 만나서 반가워요

여섯 번째 시간이에요. 다섯 번째 시간에 우리가 진짜 코드 한 줄을 짰죠. 어떤 분은 한 번에 됐을 거고, 어떤 분은 오타 한 번 났다가 고쳐서 됐을 거예요. 둘 다 잘하신 거예요. 첫 코드를 손으로 쳐 본 게 가장 큰 한 걸음이에요.

여섯 번째 시간은 분위기가 좀 달라요. 오늘은 응급 처치를 배워요. 컴퓨터가 갑자기 답답할 때, 갑자기 인터넷이 안 될 때, 갑자기 노트북이 뜨거워질 때, 어디부터 봐야 하나. 이걸 알면 "아, 망했다" 대신 "어, 이거 한 번 해볼까?"가 돼요.

## 2. 응급 처치의 기본 — 진단 먼저, 처방 나중

가장 먼저 약속할 게 있어요. **진단 먼저, 처방 나중.** 의사가 환자를 보면 청진기부터 대잖아요. 약 처방은 그 다음이에요. 컴퓨터도 똑같아요. 어디가 아픈지 모르고 약을 먹이면 더 나빠져요.

세 가지 단계예요.
1. **증상 듣기.** 답답하다고만 하지 말고, 어떤 작업이 답답한지, 언제 그러는지 정확히 보세요.
2. **진단 명령 치기.** top으로 누가 시피유를 잡고 있나, 누가 메모리를 잡고 있나 보세요.
3. **처방.** 그 친구를 멈추거나, 잠시 쉬게 하거나, 도구를 바꿔주세요.

이 세 단계를 머리에 넣고 한 시간을 가요.

## 3. 메모리가 부족할 때

증상부터요. 메모리가 부족하면 어떤 일이 벌어지나요. 노트북이 갑자기 답답해져요. 마우스가 움찔움찔해요. 앱을 클릭해도 한참 후에 떠요. 키보드가 한 박자씩 늦어요. 이게 메모리 부족 증상이에요.

진단 명령. macOS에는 메모리 압박을 보는 명령이 있어요.

> ▶ **같이 쳐보기** — 메모리 압박 보기
>
> ```bash
> vm_stat
> top -l 1 -s 0 | grep PhysMem
> ```

vm_stat은 가상 메모리(virtual memory) 통계예요. 한 번에 너무 많은 줄이 떠서 머리가 어지러워요. 두 번째 명령이 더 직관적이에요. `PhysMem: 12G used (3G wired), 4G unused.` 이런 식으로 나와요. used가 우리 도마에 올라간 양, unused가 빈 자리예요.

자, 처방. 메모리가 부족하면 두 가지를 해요. 첫째, **불필요한 앱 닫기.** 활성 상태 보기(시스템 모니터.app) 또는 top을 켜서 메모리 많이 쓰는 친구를 찾아요. 안 쓰면 닫아요. 둘째, **브라우저 탭 정리.** 사파리·크롬은 탭 한 개당 메모리를 한 모금씩 가져가요. 30개 탭 열어두면 30모금이에요. 두 모금만 남기세요.

> ▶ **같이 쳐보기** — 메모리 많이 쓰는 상위 5명
>
> ```bash
> ps aux | sort -nrk 4 | head -5
> ```

`-nrk 4`는 4번째 칸(메모리 사용률)을 숫자로(n) 큰 순으로(r) 정렬(k)이에요. 위 5명이 메모리 잡고 있는 친구들이에요.

## 4. 디스크가 꽉 찼을 때

증상. 새 파일을 못 저장해요. 앱이 갑자기 안 켜져요. "디스크가 거의 가득 찼습니다"라는 알림이 떠요.

진단. df로 전체 보기, du로 큰 폴더 찾기.

> ▶ **같이 쳐보기** — 디스크 진단
>
> ```bash
> df -h /
> du -sh ~/Downloads ~/Documents ~/Desktop ~/Movies ~/Pictures 2>/dev/null
> ```

첫 줄로 전체 사용률을 봐요. 90% 넘었으면 위험. 두 번째로 우리 큰 폴더 다섯 군데의 크기를 한 번에 봐요. 보통 다운로드, 영화, 사진이 가장 커요.

처방. 다섯 가지 순서로 가요.
1. **다운로드 폴더 비우기.** 90%는 안 봐도 될 파일이에요.
2. **휴지통 비우기.** 휴지통에 든 건 디스크를 차지해요.
3. **시스템 데이터 줄이기.** 사과 메뉴 > 이 Mac에 관하여 > 더 많은 정보 > 저장 공간으로 가서 카테고리별로 큰 것부터 정리.
4. **iOS 백업 정리.** Finder에서 iPhone 연결 후 백업 관리.
5. **앱 정리.** 안 쓰는 앱은 응용 프로그램 폴더에서 휴지통으로.

> ▶ **같이 쳐보기** — 시스템 캐시 폴더 크기
>
> ```bash
> du -sh ~/Library/Caches 2>/dev/null
> ```

캐시가 몇 기가씩 차지하면, Library/Caches 안의 폴더를 골라서 비울 수 있어요. 다만 시스템 폴더는 함부로 만지지 마세요. 잘못하면 앱이 다시 깔아야 해요. 작은 캐시 폴더부터 시작하세요.

## 5. 인터넷이 안 될 때

증상. 웹 페이지가 안 떠요. 카톡 메시지가 "전송 실패"로 떠요. 줌 회의 연결이 끊어져요.

진단. 단계적으로 가요. 가까운 곳부터 멀리.

> ▶ **같이 쳐보기** — 네트워크 진단 5단계
>
> ```bash
> # 1단계: 내 IP 있나
> ipconfig getifaddr en0
>
> # 2단계: 공유기에 닿나 (보통 192.168.0.1 또는 192.168.1.1)
> ping -c 3 192.168.0.1
>
> # 3단계: 인터넷 외부에 닿나 (구글 DNS)
> ping -c 3 8.8.8.8
>
> # 4단계: 도메인이 IP로 변환되나
> nslookup google.com
>
> # 5단계: 실제 웹 응답 받나
> curl -s -o /dev/null -w "HTTP %{http_code}\n" https://google.com
> ```

이게 네트워크 진단의 5단 사다리예요. 어디서 막히느냐에 따라 처방이 달라요.

- **1단계 실패** (IP 없음): 와이파이 꺼져 있거나 연결 안 됨. 와이파이 메뉴에서 다시 연결.
- **2단계 실패** (공유기 안 닿음): 와이파이는 잡혔는데 공유기와 통신 안 됨. 공유기 재시작.
- **3단계 실패** (외부 안 닿음): 인터넷 회선 문제. 통신사 점검.
- **4단계 실패** (도메인 변환 실패): DNS 문제. 시스템 설정 > 네트워크 > 고급 > DNS에 8.8.8.8 추가.
- **5단계 실패** (HTTPS만 안 됨): 인증서 문제, 시간 동기화 문제. 시계가 너무 어긋났는지 확인.

이 사다리는 두 해 동안 정말 자주 써요. 클라우드 배포할 때 "왜 안 되지?" 할 때마다 이 사다리를 처음부터 다시 올라가요.

## 6. CPU가 100%로 치솟을 때

증상. 노트북이 뜨거워져요. 팬 소리가 시끄러워요. 배터리가 빨리 닳아요.

진단. top으로 누가 시피유를 잡고 있나 봐요.

> ▶ **같이 쳐보기**
>
> ```bash
> top -o cpu -l 2 | grep -A 10 "PID"
> ```

`-o cpu`는 CPU 사용률 순으로 정렬, `-l 2`는 두 번 찍고 멈추기. 가장 위에 뜬 친구가 시피유 잡고 있는 친구예요.

처방.
1. **비정상이면 멈추기.** 알 수 없는 프로세스가 99% 잡고 있으면 의심. 활성 상태 보기에서 강제 종료.
2. **정상이면 기다리기.** 동영상 인코딩, 빌드, 백업처럼 CPU 많이 쓰는 게 정상인 작업이 있어요. 끝날 때까지 기다리세요.
3. **반복 발생하면 재부팅.** 가장 강력한 약. 의심스러운 백그라운드 프로세스가 사라져요.

재부팅은 농담 같지만 진짜 약이에요. "껐다 켜면 됩니다"는 IT 부서의 정직한 답이에요.

## 7. 종합 — 한 시간 후 다시 진단해보기

자, 한 시간 후에 자기 맥에 들어가서 다시 sysinfo.sh를 한 번 실행해 보세요. 그리고 H5에서 캡처한 사진과 비교해 보세요. RAM 사용률, 디스크 사용률이 어떻게 변했는지요. 변화가 거의 없으면 안정적인 거예요. 갑자기 디스크 사용률이 5% 늘었다면 뭔가 큰 파일이 들어온 거예요.

이 습관, **주기적 자기 진단**이 두 해 후에 우리가 만들 사이트(고양이 자경단)의 모니터링 시스템과 본질이 같아요. 컴퓨터 한 대를 진단하는 법을 알면, 클라우드 서버 100대를 진단하는 법도 같은 패턴이에요. 규모만 달라요. 이게 두 해 코스의 큰 그림이에요.

## 8. 마무리 — 다음 시간 예고

여섯 번째 시간 끝. 메모리·디스크·네트워크·CPU 네 친구의 응급 처치를 봤어요. 진단 먼저, 처방 나중. 이 한 줄만 가져가셔도 돼요.

다음 일곱 번째 시간은 좀 깊이 들어가요. 폰노이만 구조라는 단어를 한 번 만나고, 시피유 캐시라는 신비한 친구를 만나요. 왜 어떤 친구는 빠르고 어떤 친구는 느린가, 그 답이에요. 십 분 쉬고 와요.

## 👨‍💻 개발자 노트

> - "진단 먼저, 처방 나중" 원칙은 Ch 069 보안(공격 분류 전 처방 금지), Ch 116 분산 추적(원인 식별 전 핫픽스 금지), Ch 091 성능 튜닝(프로파일링 없이 최적화 금지)에서 거듭 회수됩니다.
> - 네트워크 5단 사다리는 OSI 7계층의 단순화. Ch 003에서 정식 OSI를 학습하지만, 본 챕터의 사다리는 학습자가 평생 쓸 트러블슈팅 휴리스틱.
> - `8.8.8.8`(Google Public DNS)을 진단 도구로 쓰는 패턴은 Ch 113(Route53), Ch 116(VPC 흐름 로그)에서 다시 등장.
> - 메모리 압박 명령 `vm_stat`은 macOS 전용. Linux는 `free -h`, `cat /proc/meminfo`, `vmstat`. Ch 002 OS 챕터에서 비교.
> - "껐다 켜면 됩니다"는 농담 같지만, Ch 110 Lambda 콜드 스타트, Ch 119 Bedrock Agent 세션 재시작에서 "상태 초기화의 힘"으로 진지하게 회수됩니다.
