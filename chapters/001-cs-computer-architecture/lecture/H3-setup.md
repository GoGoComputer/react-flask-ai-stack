## 1. 십 분 쉬셨어요

자, 세 번째 시간입니다. 두 시간 동안 비유랑 그림으로만 이야기했죠. 이제 손으로 진짜 명령을 칠 시간이에요. 손이 좀 떨리실 수도 있어요. 괜찮아요. 우리는 한 줄, 한 줄, 천천히 갈 거예요. 오타가 나도 괜찮아요. 한 번 더 치면 돼요. 두 번 쳐도 안 되면 손 들어 주세요. 같이 봐요.

## 2. 왜 신분증을 묻나요

질문 하나. 우리는 왜 우리 맥의 신분증을 봐야 하나요? 두 가지 이유가 있어요.

첫째, 인터넷에서 답을 찾을 때 필요해요. "내 맥이 안 돼요"는 답이 안 나와요. "macOS 14.4, M2 Pro, 16GB에서 도커가 안 돌아요"는 답이 줄줄이 나와요. 신분증을 안다는 건 검색할 수 있는 사람이 된다는 뜻이에요. 1교시에서 약속드렸죠.

둘째, 우리 코스가 두 해 동안 진행되거든요. 두 해 후에 여러분이 본인 맥을 한 번 갈 수도 있어요. 그때 "어, 옛날 맥은 어떤 거였더라" 기억이 안 나요. 지금 한 번 신분증을 출력해서 메모장에 붙여두시면, 두 해 후에 다시 볼 수 있어요. 이게 두 번째 이유예요.

방법은 다섯 가지를 보여드릴 거예요. 같은 정보를 다섯 가지 방법으로 보는 거예요. 왜 다섯 가지냐면, 어느 한 가지가 안 될 때 다른 방법을 쓸 줄 알아야 하니까요. 저는 다섯 가지 다 쓸 줄 알아야 진짜 안다고 생각해요.

## 3. 첫 번째 방법 — 사과 메뉴 "이 Mac에 관하여"

가장 쉬운 방법이에요. 명령어를 안 써요. 화면 왼쪽 위에 있는 사과 모양 아이콘을 눌러주세요. 메뉴가 죽 떠요. 그 중에 "이 Mac에 관하여" 또는 영문판이라면 "About This Mac"이 있어요. 누르세요. 작은 창이 떠요.

거기 다 있어요. 모델명 (예: MacBook Pro), 칩 (예: Apple M2 Pro), 메모리 (예: 16GB), 시리얼 번호, 그리고 macOS 버전. 캡처해 두세요. 정말로요. 캡처 단축키는 Shift + Command + 4를 누르고 영역 드래그예요. 사진이 바탕화면에 저장돼요. 이 사진 한 장이 두 해 후에 큰 도움이 돼요.

이 방법의 장점은 클릭 두 번이면 끝이라는 거예요. 단점은 "다른 컴퓨터에 들어가서 그 컴퓨터의 신분증을 알고 싶을 때" 못 써요. 예를 들어 회사 서버에 원격으로 접속했을 때 사과 아이콘을 누를 수가 없잖아요. 그래서 명령어로도 알아야 해요.

## 4. 두 번째 방법 — sw_vers와 system_profiler

명령어 두 개로 같은 정보를 봐요.

> ▶ **같이 쳐보기** — sw_vers
>
> ```bash
> sw_vers
> ```

세 줄이 떠요. ProductName, ProductVersion, BuildVersion. 첫 줄은 운영체제 이름 (macOS), 둘째 줄은 버전 (예: 14.5), 셋째 줄은 빌드 번호. 빌드 번호는 같은 14.5 안에서도 작은 패치 차이를 구분해요. 우리가 자주 볼 건 두 번째 줄이에요.

> ▶ **같이 쳐보기** — system_profiler 짧게
>
> ```bash
> system_profiler SPHardwareDataType
> ```

이건 하드웨어 정보를 다 보여줘요. 모델명, 칩, 코어 수, 메모리, 시리얼 넘버, UUID 등. 너무 많이 나와서 머리가 어지러워요. 그래서 우리는 보통 grep이라는 친구랑 같이 써요.

> ▶ **같이 쳐보기** — 모델명만
>
> ```bash
> system_profiler SPHardwareDataType | grep "Model Name"
> ```

깔끔하게 한 줄만 떠요. "Model Name: MacBook Pro" 같은 식으로요. 파이프 기호 |는 "이 명령어의 출력을 다음 명령어한테 입력으로 넘겨라"라는 뜻이에요. grep은 그 입력에서 우리가 찾는 글자가 든 줄만 골라서 보여주는 친구예요. 두 해 동안 정말 자주 쓸 거예요. 외우진 마세요. 자연스럽게 손에 붙어요.

## 5. 세 번째 방법 — uname 가족

uname (uname, "유네임")은 "유닉스 네임" 즉 운영체제의 이름을 묻는 명령이에요. 가족이 많아요. 옵션마다 다른 정보를 줘요.

> ▶ **같이 쳐보기** — uname 다섯 형제
>
> ```bash
> uname -s    # OS 이름 (Darwin)
> uname -m    # 머신 아키텍처 (arm64 또는 x86_64)
> uname -r    # 커널 릴리스 버전
> uname -n    # 호스트 이름 (이 맥의 별명)
> uname -a    # 위 모든 정보 한 줄
> ```

`uname -s`로 보면 macOS는 Darwin이라고 나와요. Darwin은 macOS의 진짜 안쪽 이름이에요. 우리가 사과 그림 운영체제라고 부르지만, 그 안의 진짜 코어는 Darwin이라는 이름의 유닉스 계열 시스템이에요. 두 해 후에 리눅스를 만나도 "아, Darwin이 macOS의 안쪽이듯, Linux는 우분투·아마존리눅스의 안쪽이구나"라고 자연스럽게 연결돼요.

`uname -m`이 1교시에 친 그 명령이에요. arm64인지 x86_64인지. 다시 한 번 자기 거 확인하세요.

`uname -n`은 우리 맥의 이름이에요. "OO의 MacBook Pro" 같이 떠요. 누가 지었어요? 처음 맥 켤 때 본인이 입력한 이름이에요. 기억 안 나시면 시스템 설정 > 일반 > 정보에서 바꿀 수 있어요.

## 6. 네 번째 방법 — sysctl 사전

sysctl (sysctl, "시스컨트롤")은 시스템 변수 사전이에요. 키 이름을 주면 값을 돌려줘요. 두 시간째에 우리가 한 명령들이 다 sysctl이었어요. 다시 정리할게요.

> ▶ **같이 쳐보기** — sysctl 단골 키들
>
> ```bash
> sysctl -n machdep.cpu.brand_string   # CPU 모델명
> sysctl -n hw.ncpu                    # 전체 코어 수
> sysctl -n hw.physicalcpu             # 물리 코어 수
> sysctl -n hw.logicalcpu              # 논리 코어 수 (하이퍼스레딩 포함)
> sysctl -n hw.memsize                 # 램 크기 (바이트)
> sysctl -n kern.osrelease             # 커널 릴리스
> sysctl -n kern.boottime              # 부팅 시각
> ```

키 이름이 외계어처럼 보이지만 규칙이 있어요. machdep는 머신 디펜던트(machine dependent), hw는 하드웨어(hardware), kern은 커널(kernel). 네임스페이스가 점으로 구분돼 있어요. 외울 필요는 없어요. 검색하면 다 나와요.

여기서 한 가지 짚고 가요. `hw.physicalcpu`와 `hw.logicalcpu`. 둘 다 코어 개수인데 다를 수 있어요. 인텔 맥은 보통 logical이 physical의 두 배예요. M 시리즈 애플 실리콘은 둘이 같아요. 왜 다르냐면, 인텔에는 하이퍼스레딩이라는 기능이 있어서 코어 한 개가 두 개인 척 일을 할 수 있어요. 자세한 건 7교시에 다뤄요.

## 7. 다섯 번째 방법 — 한 줄 종합 검진

자, 마지막 방법. 여러 명령을 한 줄로 묶어서 종합 신분증을 한 번에 출력해요.

> ▶ **같이 쳐보기** — 한 줄 종합
>
> ```bash
> echo "[OS] $(sw_vers -productName) $(sw_vers -productVersion)" && \
> echo "[Arch] $(uname -m)" && \
> echo "[CPU] $(sysctl -n machdep.cpu.brand_string)" && \
> echo "[Cores] $(sysctl -n hw.ncpu)" && \
> echo "[RAM] $(($(sysctl -n hw.memsize) / 1024 / 1024 / 1024)) GB"
> ```

다섯 줄짜리 명령이지만 백슬래시로 한 줄로 이어져 있어요. 다 치고 엔터. 이런 식으로 떠요.

```
[OS] macOS 14.5
[Arch] arm64
[CPU] Apple M2 Pro
[Cores] 12
[RAM] 16 GB
```

이게 여러분 맥의 한 줄 종합 신분증이에요. echo는 그냥 글자를 출력하라는 명령이고요, $()는 그 안의 명령을 실행해서 결과를 끼워 넣으라는 문법이에요. 마지막의 RAM 줄은 sysctl이 준 바이트 숫자를 1024로 세 번 나눠서 기가바이트로 바꾼 거예요. 5교시에 우리가 짤 sysinfo.sh의 핵심 부분이에요.

이 다섯 줄을 메모장에 한 번 붙여두세요. 두 해 후에도, 다른 사람의 맥을 봐줄 때도, 이 다섯 줄만 치면 그 맥의 자기소개서가 한 번에 떠요. 이게 진짜 자산이에요.

## 8. 다섯 가지 방법 비교표 — 언제 어느 걸 쓰나

자, 다섯 가지를 다 배웠어요. 한 번 정리하고 가요. 같은 정보를 다섯 가지 방법으로 보는 이유는, 상황마다 잘 맞는 방법이 달라서예요. 표로 한 번 봐요.

| 방법 | 속도 | 원격 가능 | 자동화 가능 | 사람 친화도 | 주 용도 |
|------|------|---------|-----------|-----------|--------|
| 사과 메뉴 | 즉시 | ❌ | ❌ | ⭐⭐⭐⭐⭐ | 처음 본 맥 빠르게 살펴보기 |
| sw_vers | 0.05초 | ✅ | ✅ | ⭐⭐⭐⭐ | OS 버전만 빨리 |
| system_profiler | 1~3초 | ✅ | ✅ | ⭐⭐⭐ | 자세한 하드웨어 보고 |
| uname | 0.01초 | ✅ | ✅ | ⭐⭐⭐ | 빠른 머신 분류 (arm64? Linux?) |
| sysctl | 0.01초 | ✅ | ✅ | ⭐⭐ | 특정 키 값 한 개만 |
| 한 줄 종합 | 0.5초 | ✅ | ✅ | ⭐⭐⭐⭐⭐ | 깔끔한 보고서 한 장 |

이 표에서 두 가지 패턴이 보이시죠. 첫째, **사람 친화적인 건 자동화가 어렵고, 자동화가 쉬운 건 사람한테 좀 불친절하다.** 사과 메뉴는 클릭이 필요해서 자동화가 안 되고, sysctl은 키 이름만 알면 1초 만에 파이프로 연결할 수 있지만 키 이름 자체가 외계어예요. 둘째, **원격 작업에서는 메뉴 방식이 통째로 빠진다.** 두 해 후 우리가 클라우드 서버에 SSH로 들어갈 때 사과 메뉴는 없잖아요. 명령어만 남아요. 그래서 명령어를 한 번씩 다 손에 익혀두는 게 두 해를 위한 투자예요.

저는 일할 때 어떤 걸 가장 자주 쓰냐고요? 솔직히 사과 메뉴랑 한 줄 종합이 8할이에요. 처음 보는 맥에 앉을 때는 사과 메뉴, 내 컴퓨터 검진은 한 줄 종합. 나머지는 가끔 필요할 때 골라 써요. 여러분도 우선 이 두 가지부터 손에 익히세요. 나머지 셋은 필요할 때 챕터로 다시 와도 늦지 않아요.

한 가지 팁. 한 줄 종합 명령은 사실 텍스트 파일에 저장해두면 더 좋아요. 명령 한 줄이 아니라 스크립트 한 개. 5교시에서 우리가 `start/sysinfo.sh`라는 이름으로 만들 거예요. 한 번 만들면 평생 써요. `bash start/sysinfo.sh` 한 줄이면 다섯 줄짜리 검진 결과가 떠요. 손가락이 편하면 자주 쓰게 되고, 자주 쓰면 컴퓨터의 안부를 자주 묻게 돼요. 좋은 사이클이에요.

## 9. 시리얼 번호와 UUID — 신분증의 진짜 ID 칸

신분증에는 이름, 사진, 주소도 있지만 **번호**가 가장 중요하잖아요. 컴퓨터의 신분증에도 번호가 있어요. 두 가지가 있어요.

첫째, **시리얼 번호(Serial Number)**. 이건 애플이 공장에서 찍어주는 진짜 일련번호예요. 전 세계에 단 하나뿐이에요. 노트북 뒷면에도 작게 새겨져 있고, 사과 메뉴 > 이 Mac에 관하여 > 자세히 정보에서 볼 수 있어요. 명령어로는 이렇게 봐요.

> ▶ **같이 쳐보기** — 시리얼 번호
>
> ```bash
> system_profiler SPHardwareDataType | grep "Serial Number"
> ```

이 번호는 왜 중요하냐면, 애플 케어 받을 때, 도난당했을 때, 회사에 자산 등록할 때 다 필요해요. 한 번 메모장에 적어두세요. 단, **인터넷에 공개적으로 올리지는 마세요.** 누군가 그 번호로 애플 케어 사칭을 하거나, 활성화 잠금 우회를 시도할 수 있어요. 시리얼 번호는 비밀번호는 아니지만 준비밀번호급으로 다루세요. 두 해 후 우리가 보안 챕터(약 Ch 064 부근)에서 "민감 정보의 등급" 이야기를 할 때 시리얼이 좋은 사례로 다시 나와요.

둘째, **UUID(Universally Unique IDentifier)**. 이건 OS가 부팅할 때 만드는 긴 임의 번호예요. 보통 이렇게 생겼어요: `12345678-1234-5678-1234-567812345678`. 32자리 16진수가 하이픈으로 끊겨 있어요. 명령어로는 이렇게 봐요.

> ▶ **같이 쳐보기** — UUID
>
> ```bash
> system_profiler SPHardwareDataType | grep "Hardware UUID"
> ```

UUID는 시리얼이랑 다른 점이 있어요. 시리얼은 공장에서 찍은 거라 평생 안 바뀌는데, UUID는 OS 재설치 같은 큰 변화에 바뀔 수도 있어요. 그리고 UUID는 두 해 동안 정말 자주 봐요. 데이터베이스에서 사용자 ID, 주문 ID, 게시글 ID로 쓰는 게 다 이 UUID예요. 우리 고양이 자경단의 모든 제보 한 건 한 건도 UUID로 식별돼요. **"전 우주에 단 하나뿐인 번호를 만드는 기술"** 이라는 게 컴퓨터 과학의 작은 마법 중 하나예요. 36자리짜리 번호인데 우주의 모든 컴퓨터가 동시에 만들어도 겹칠 확률이 사실상 0이에요. 어떻게 그게 가능한지는 두 해 후 분산 시스템 챕터(약 Ch 058)에서 다룹니다. 지금은 그냥 "이런 마법이 있다" 정도만요.

## 10. macOS 버전사 짧은 사이드바 — 우리가 어디서 왔나

`sw_vers`로 본 그 버전 번호. 한 번 짧게 역사를 보고 가요. 우리가 어디쯤 와 있는지 감을 잡으면 OS 업데이트 결정도 더 잘하게 돼요.

| 버전 | 코드네임 | 출시 | 특징 |
|------|--------|------|------|
| 10.0 | Cheetah | 2001 | 첫 macOS X. UI 혁명. |
| 10.6 | Snow Leopard | 2009 | "성능 다이어트" 버전. 아직도 그리워하는 사람 많음. |
| 10.10 | Yosemite | 2014 | 평면 디자인 도입. iOS와 통일감. |
| 10.15 | Catalina | 2019 | 32비트 앱 지원 종료. 많이 깨졌음. |
| 11 | Big Sur | 2020 | 첫 Apple Silicon 지원. M1 등장. |
| 12 | Monterey | 2021 | Universal Control. |
| 13 | Ventura | 2022 | Stage Manager. |
| 14 | Sonoma | 2023 | 위젯 데스크탑. |
| 15 | Sequoia | 2024 | iPhone 미러링. |
| 16 | Tahoe | 2025 | Apple Intelligence 본격화. |
| 17 | (예정) | 2026 | 우리 코스 진행 중에 출시 예정. |

자, 두 가지 패턴이 보이시죠. 첫째, **2020년에 11.0으로 한 번 크게 점프했어요.** 그 전엔 19년 동안 10.x였어요. 19년 동안 한 번도 11이 안 된 거예요. 그러다가 Apple Silicon이 등장하면서 11로 점프했어요. 큰 변화에 큰 번호. 그 다음부턴 매년 한 번씩 큰 번호가 올라가요. 둘째, **각 버전에 동물 이름 또는 캘리포니아 지명이 붙어요.** 처음엔 동물(Cheetah, Puma, Jaguar, Panther...)이었다가 2013년부터 캘리포니아 지명(Mavericks, Yosemite, ...)으로 바뀌었어요. 매장에서 "엘 캐피탄"이라고 들으시면 "아, macOS 10.11 코드네임이구나" 알아들으시면 멋있어요.

업그레이드 결정할 때 한 가지 팁. **메이저 버전(13→14, 14→15)은 나오자마자 바로 안 올려도 돼요.** 출시 후 두세 달은 자잘한 호환성 문제가 있어요. 도커가 안 돌거나, 특정 앱이 죽거나. 한 두세 번 패치(예: 14.1, 14.2)가 나온 후에 올리는 게 안전해요. 우리 코스에서는 강의가 진행 중일 때 새 메이저 버전이 나와도, 한 학기 끝날 때까지는 기다리는 걸 권장해요.

## 11. 잘 안 될 때 — 다섯 가지 함정과 응급처치

명령어를 쳤는데 잘 안 될 때가 있어요. 흔한 함정 다섯 가지 미리 보여드려요.

**함정 1 — `sysctl: unknown oid 'hw.foobar'`.**
원인: 키 이름 오타. 실제로는 `hw.memsize`인데 `hw.memorysize`로 쳤다든가. 응급처치: `sysctl hw | head -30` 으로 hw 네임스페이스의 모든 키를 한 번 쭉 보고 비슷한 이름을 찾으세요. `head -30`은 첫 30줄만 보여달라는 뜻. 다 보면 화면이 미친듯이 흘러가요.

**함정 2 — `system_profiler`가 너무 느려요.**
원인: SPHardwareDataType 빼고 그냥 `system_profiler`만 치면 모든 데이터타입을 다 모으느라 30초~1분 걸려요. 응급처치: 항상 데이터타입을 명시하세요. 자주 쓰는 데이터타입은 `SPHardwareDataType`(하드웨어), `SPSoftwareDataType`(OS·커널), `SPNetworkDataType`(네트워크), `SPStorageDataType`(디스크). 그리고 `system_profiler -listDataTypes` 한 번 쳐보시면 사용 가능한 전체 목록이 떠요.

**함정 3 — `sw_vers`의 -productVersion 같은 옵션을 쳤는데 빈 줄이 나와요.**
원인: 옵션 철자가 틀렸어요. macOS 버전마다 미세하게 다르기도 해요. 응급처치: 옵션 없이 그냥 `sw_vers`만 치면 다 떠요. 그 출력을 `grep` 또는 `awk`로 자르면 돼요. 옵션 외울 시간에 파이프 한 번이 더 빨라요.

**함정 4 — `uname -a`의 출력이 너무 길고 의미를 모르겠어요.**
원인: `uname -a`는 SVR4 표준대로 6~7개 필드를 한 줄로 붙여 줘요. 순서가 좀 헷갈려요. 응급처치: 처음에는 `-a` 대신 `-s`, `-m`, `-r`을 따로따로 치는 습관을 들이세요. 익숙해진 후에 `-a`로 통합. 사람 머리에 한 번에 너무 많이 들어오면 다 잊어요.

**함정 5 — 명령어 자체가 "command not found"로 거부돼요.**
원인: 거의 없어요. macOS 기본 셸이 zsh이고, 위 명령들은 다 기본 설치돼 있어요. 그런데 가끔 새 터미널 앱을 깐 후에 PATH 환경변수가 망가져 있을 수 있어요. 응급처치: `echo $PATH` 쳐 보세요. `/usr/bin`, `/bin`, `/usr/sbin`이 들어 있어야 해요. 빠져 있으면 `~/.zshrc` 파일 첫 줄에 `export PATH="/usr/bin:/bin:/usr/sbin:$PATH"` 한 줄 넣고 새 터미널 여세요. 이런 PATH 사고는 6교시 응급처치 시간에서 더 자세히 다뤄요.

이 다섯 가지 함정 중 가장 흔한 건 함정 1(키 오타)이에요. 두 번째로 흔한 건 함정 2(system_profiler 느림). 두 가지만 머리에 두셔도 어지간한 막힘은 풀려요.

## 12. 두 해 후 우리 사이트 — 서버에서 신분증 묻기

오늘 배운 명령들이 두 해 후에 어떻게 쓰일지 한 장면 미리 보여드릴게요.

장면. 두 해 후, 여러분은 고양이 자경단 사이트의 백엔드 서버 한 대에 SSH로 처음 접속해요. AWS의 EC2 인스턴스예요. 까만 화면이 떠요. `Last login: Sat Apr 25 ...` 한 줄이 보이고 커서가 깜빡여요. 가장 먼저 뭘 쳐야 할까요?

여러분의 손은 자동으로 이렇게 움직여야 해요.

```bash
uname -a              # 어떤 OS·커널·아키텍처지?
cat /etc/os-release   # 정확한 배포판 (Amazon Linux? Ubuntu?)
nproc                 # 코어 몇 개?
free -h               # 램 얼마?
df -h                 # 디스크 사용률?
uptime                # 얼마 동안 켜져 있었나? 부하는?
```

오늘 배운 macOS의 명령들과 짝이 맞죠. uname은 양쪽 다 똑같고, sw_vers 자리에 cat /etc/os-release, sysctl hw.ncpu 자리에 nproc, sysctl hw.memsize 자리에 free -h. **macOS에서 한 번 손에 익혀두면 리눅스에서도 0.5초 만에 짝을 찾을 수 있어요.** 이게 우리가 첫 챕터를 신분증 묻기로 시작한 진짜 이유예요. 두 해 후의 첫 SSH 접속을 위한 준비예요.

그리고 한 가지 더. 클라우드 서버는 우리가 직접 산 게 아니라 빌린 거예요. 시간 단위로 돈이 빠져요. 비싼 인스턴스(예: 메모리 64GB) 받아놓고 우리 사이트가 8GB만 쓰고 있으면 매달 수십만 원이 그냥 새는 거예요. 그래서 매주 한 번씩이라도 위 명령으로 "지금 자원이 남나, 모자라나"를 보는 습관이 큰돈을 절약해요. 두 해 후 클라우드 비용 챕터(약 Ch 109)에서 "관찰 = 절약"이라는 큰 주제가 다시 나와요. 첫 시간의 명령 한 줄이 미래의 절약입니다.

## 13. 자주 묻는 질문 다섯 — 환경 점검 편

**Q1. "내 노트북이 좀 오래된 모델인데, 코스 따라갈 수 있나요?"**
A. M1 이상의 애플 실리콘이면 충분해요. 인텔 맥(2019년 이전)도 따라갈 수 있는데, 일부 도커 이미지가 좀 느릴 수 있어요. 핵심은 OS 버전이에요. macOS 12(Monterey) 이상이면 거의 모든 챕터가 돌아요. macOS 11 이하면 일부 새 도구가 안 돌아요. `sw_vers`로 한 번 확인해 보세요.

**Q2. "사과 메뉴랑 명령어 결과가 미세하게 달라요. 어느 게 진짜예요?"**
A. 둘 다 진짜인데 보는 각도가 달라요. 사과 메뉴는 사람 친화적으로 가공한 값(예: "16 GB"), 명령어는 원시 바이트 값(예: 17179869184). 17179869184를 1024로 세 번 나누면 16이에요. 같은 값이에요. 혹시 사과 메뉴가 "16 GB"인데 sysctl이 8GB를 돌려주면 그건 진짜 문제예요. (램 한 칸이 죽었거나 OS가 인식 못함.) 그땐 애플 케어 가실 때예요.

**Q3. "터미널이 무서워요. 사과 메뉴만 쓰면 안 돼요?"**
A. 1~2 챕터 동안은 됩니다. 그 후엔 안 돼요. 두 해 코스의 90%가 터미널 위에서 일어나요. 무서움을 한 번에 없애는 방법은 한 가지밖에 없어요. **자주 켜기.** 매일 한 번씩, 5분만이라도 터미널을 켜고 아무 명령이나 쳐 보세요. 일주일이면 무서움이 반쯤 사라지고, 한 달이면 친구가 됩니다. 약속해요.

**Q4. "명령을 잘못 쳤는데 컴퓨터가 망가지지 않을까요?"**
A. 오늘 배운 명령들(sw_vers, uname, sysctl, system_profiler)은 다 **읽기 전용**이에요. 읽기만 하지 쓰지 않아요. 그래서 망가뜨릴 수 없어요. 마음 놓고 치세요. 위험한 명령은 보통 `sudo`로 시작하는 명령들이에요. 우리는 `sudo`를 칠 때마다 한 번 더 생각하는 습관을 두 해 동안 같이 들일 거예요.

**Q5. "왜 같은 정보를 다섯 가지 방법으로 가르치세요? 한 가지만 확실히 알면 안 돼요?"**
A. 한 가지만 알면 그 한 가지가 안 될 때 멈춰요. 두 해 후에 여러분이 만날 환경은 다양해요. 사과 메뉴가 없는 서버, system_profiler가 없는 컨테이너, sysctl 키가 다른 BSD 변종. 한 가지만 알면 다 막혀요. 다섯 가지를 다 알면 한 가지가 안 돼도 다른 걸로 우회해요. **여러 가지 방법을 안다는 게 진짜 힘이에요.** 코스 전체의 큰 철학이에요.

## 14. 환경 변수와 셸 — 신분증의 옆 칸

지금까지 본 신분증은 "하드웨어 + OS"의 신분증이었어요. 그런데 신분증의 옆에 작게 또 한 칸이 있어요. **셸(shell)과 환경 변수(environment variables)** 의 칸이에요. 이게 뭐냐면, "지금 내가 명령을 치고 있는 이 검은 창의 환경" 자체의 신분증이에요.

먼저 셸. 셸은 우리가 키보드로 친 글자를 받아서 컴퓨터한테 전달해 주는 통역사예요. macOS의 기본 셸은 zsh(지셸)예요. 옛날에는 bash(배쉬)였고, 그 전엔 sh(에스에이치). 종류가 여러 개예요.

> ▶ **같이 쳐보기** — 내 셸 신분증
>
> ```bash
> echo $SHELL
> echo $0
> ```

`echo $SHELL`은 우리가 로그인했을 때 자동으로 띄워지는 기본 셸. 보통 `/bin/zsh`가 떠요. `echo $0`은 지금 이 순간 우리가 실제로 쓰고 있는 셸. 새 터미널을 그냥 열었으면 둘이 같은 답이 나와요. 두 답이 다르면, 누군가 다른 셸을 그 안에서 또 띄운 상황이에요.

다음 환경 변수. 환경 변수는 셸이 외우고 있는 작은 메모장 같은 거예요. "내 이름은 mo, 내 집은 /Users/mo, PATH는 이러이러하다." 한 번 다 출력해 봐요.

> ▶ **같이 쳐보기** — 환경 변수 단골
>
> ```bash
> echo $USER     # 내 이름
> echo $HOME     # 내 홈 디렉토리
> echo $PATH     # 명령어를 찾는 경로 목록
> echo $LANG     # 내 언어 설정
> echo $TERM     # 터미널 종류
> ```

다섯 줄이 다 답을 줘요. 가장 중요한 건 `$PATH`예요. 이게 망가지면 거의 모든 명령이 안 돼요. PATH는 콜론(:)으로 구분된 폴더 목록인데, 셸은 우리가 명령을 치면 이 목록을 위에서 아래로 뒤져서 첫 번째로 발견되는 걸 실행해요. 두 해 후 우리가 새 도구(예: nvm, pyenv) 깔 때마다 PATH에 한 줄씩 추가될 거예요. 그때마다 "아, 이게 첫 챕터에서 본 그 PATH구나" 떠올려 주세요.

`$LANG`이 중요한 한 가지 이유. 만약 `ko_KR.UTF-8`이면 일부 명령의 출력이 한글로 나와요. `en_US.UTF-8`이면 영어로요. 인터넷에서 검색할 때는 보통 영어 에러 메시지가 결과가 더 잘 나와요. 그래서 한글 화면을 좋아하시더라도, **에러 메시지만큼은 영어로 보는 습관**이 두 해 후에 큰 차이를 만들어요. 임시로만 영어로 바꾸려면 `LANG=en_US.UTF-8 명령` 이렇게 명령 앞에 붙이면 돼요.

## 15. 타임존, 로케일, 키보드 — 잊기 쉬운 환경 3종

세 가지를 잠깐 짚고 가요. 다 쉽고, 다 두 해 후에 한 번씩 사고를 칠 수 있는 친구들이에요.

**타임존.** 우리 컴퓨터의 시각이 어느 지역의 시각인지. 보통 `Asia/Seoul`로 잡혀 있어요.

> ▶ **같이 쳐보기** — 타임존
>
> ```bash
> date                          # 현재 시각 + 타임존
> ls -l /etc/localtime          # 어디로 링크돼 있나
> ```

두 해 후 클라우드 서버 살 때 기본 타임존이 UTC예요. 한국 시각보다 9시간 늦어요. 로그가 9시간 어긋나서 새벽 3시 사고를 보고 "어, 어제 오후에 났네?" 하고 헷갈리는 사고가 정말 자주 나요. 서버 켤 때 가장 먼저 타임존부터 한국으로 바꾸는 습관을 들이세요. (아니면 로그를 UTC로 통일하는 게 더 좋다는 의견도 있어요. 두 해 후 운영 챕터에서 논의해요.)

**로케일.** `$LANG`에서 잠깐 봤죠. 숫자 표기, 날짜 표기, 정렬 순서를 결정해요. 1,000.50 (영어식)인지 1.000,50 (유럽식)인지가 로케일 따라 달라요. 데이터 처리할 때 이 차이가 사고를 부를 수 있어요.

**키보드 레이아웃.** macOS 시스템 설정 > 키보드 > 입력 소스. 한국에서 산 맥은 보통 두벌식 한글 + ABC가 들어 있어요. 외국에서 산 맥은 다른 레이아웃일 수 있어요. 한영 전환이 안 되면 거의 100% 입력 소스 설정이에요. 우리 코스에서 명령을 쳐야 하는데 한영 전환이 안 되면 일단 그것부터 해결하시고 진행하세요. 까만 화면에 한글이 들어가면 거의 모든 명령이 깨져요.

이 세 가지는 평소엔 안 보이다가 사고 났을 때 갑자기 등장하는 친구들이에요. 미리 한 번 머리에 두면 사고 났을 때 "아, 이게 그 친구구나" 알아챌 수 있어요. 알아채는 게 절반이에요.

## 16. 한 가지 진짜 고장 사례 — 신분증이 사람 살린 이야기

마지막으로 짧은 일화 하나. 작년에 제가 본 일이에요. 어느 후배가 "도커가 안 돌아요" 한 시간째 헤매고 있었어요. 슬랙에 도와달라고 도움 요청 올라왔어요. 제가 한 줄 보냈어요. "uname -m 결과 좀 알려줘." 후배가 답을 보냈어요. `arm64`. 그리고 같이 보던 도커 이미지가 x86_64 전용이었어요. 그 한 줄에서 원인이 잡혔어요. 한 시간을 헤맨 문제가 30초 만에 해결됐어요. 도커 이미지를 멀티 아키 빌드로 바꾸거나, `--platform linux/amd64` 옵션으로 우회하면 끝.

이 일화에서 한 가지 교훈. **에러를 받았을 때 가장 먼저 던질 질문은 "환경이 뭐냐"다.** 같은 코드, 같은 명령이라도 환경이 다르면 결과가 달라져요. 환경이 같은지부터 확인해야 시간을 안 낭비해요. 오늘 배운 다섯 가지 방법은 그 첫 질문을 1초 만에 던지게 해 주는 도구예요. 후배는 이제 슬랙에 도와달라고 올리기 전에 자동으로 한 줄 종합 신분증부터 붙여요. 그게 사람을 살리는 한 줄이에요.

## 17. 디스크와 네트워크 — 신분증의 마지막 두 칸

지금까지 OS, CPU, RAM, 셸, 환경 변수까지 봤어요. 신분증의 마지막 두 칸을 채우러 가요. 디스크와 네트워크.

먼저 디스크. 우리 맥의 디스크가 얼마나 차 있는지, 얼마나 남았는지 한 줄로 봐요.

> ▶ **같이 쳐보기** — 디스크 신분증
>
> ```bash
> df -h /
> ```

`df`는 디스크 프리(disk free)의 줄임말. `-h`는 human-readable, 즉 1024로 나눠서 GB·TB로 보여달라는 옵션. `/`는 루트 파일 시스템, 즉 가장 위의 디렉토리. 이 한 줄을 치면 표 한 줄이 떠요. `Filesystem Size Used Avail Capacity Mounted on`. 보통 Size 460GB, Used 200GB, Avail 250GB, Capacity 45% 같은 식으로요.

여기서 가장 중요한 건 **Capacity** 칸이에요. 이 숫자가 95% 넘으면 위험 신호. 100%가 되면 새 파일을 못 저장하고, 도커가 안 돌고, 슬랙도 안 켜질 수 있어요. 매주 한 번씩 `df -h /` 한 줄로 안부 묻는 습관을 들이세요. 5교시 sysinfo.sh가 이걸 자동으로 묶어줄 거예요.

다음 네트워크. 우리 맥의 IP 주소를 봐요.

> ▶ **같이 쳐보기** — 내 IP
>
> ```bash
> ipconfig getifaddr en0
> ```

`en0`는 보통 첫 번째 와이파이 카드의 이름이에요. (유선 랜이면 다른 이름.) 192.168.0.10 같은 사설 IP가 떠요. 이 주소는 우리 집 공유기 안에서만 의미가 있어요. 인터넷에 보이는 진짜 IP는 다음 명령으로 확인해요.

> ▶ **같이 쳐보기** — 인터넷에서 본 내 IP
>
> ```bash
> curl -s ifconfig.me
> echo
> ```

`curl`은 인터넷에서 한 줄 받아오는 명령. `ifconfig.me`는 "당신의 IP를 알려주는 무료 서비스". 이 두 IP는 보통 다른 숫자예요. 왜 다른지는 두 해 후 네트워크 챕터(약 Ch 056)에서 NAT(네트워크 주소 변환)이라는 개념으로 자세히 다뤄요. 지금은 "내 컴퓨터에는 IP가 두 개 있다 — 집 안에서 보이는 것과 인터넷에서 보이는 것 — " 정도만 머리에 두세요. 두 해 후 클라우드 보안 챕터에서 이 두 가지 IP의 차이가 핵심 주제로 다시 나와요.

이 두 줄(디스크 + 네트워크)이 신분증의 마지막 두 칸이에요. 이제 우리 맥의 신분증이 완성됐어요. OS, CPU, RAM, 셸, 환경, 디스크, 네트워크. 일곱 칸이 다 찼어요. 첫 한 시간만으로 우리는 컴퓨터의 일곱 칸짜리 종합 자기소개서를 출력할 줄 아는 사람이 됐어요. 이게 작아 보여도, 한 달 후 옆자리 동료가 못하는 일이 됩니다. 진짜로요.

## 18. 오늘의 미니 과제 — 챕터 끝나고 한 번만

쉬는 시간 전에 한 가지 작은 과제를 드릴게요. 강요가 아니에요. 부탁이에요.

1. 한 줄 종합 명령을 한 번 더 치세요. (§7의 그 다섯 줄)
2. 결과를 복사해서 본인 메모장 또는 슬랙 본인 메시지로 보내세요.
3. 그 위에 한 줄 적으세요: "OOOO년 OO월 OO일, 두 해 코스 첫 챕터에서 출력한 내 첫 신분증."

끝이에요. 1분짜리 과제예요. 이 한 장의 메모가 두 해 후 졸업할 때 보면 진짜 뭉클해져요. "아, 이때 내가 이 노트북으로 첫 명령을 쳤구나" 하고요. 우리 코스의 두 해 시간 자체가 자료가 돼요. 첫 자료가 오늘의 신분증이에요.

## 19. 명령어가 외워지지 않을 때 — 검색·메모·치트시트의 삼각형

여기까지 따라오시면서 한 가지 걱정이 들 거예요. "이 명령들이 안 외워지면 어떡하지?" 외우려고 하지 마세요. 외우려고 한 명령은 일주일이면 다 잊어요. 대신 세 가지 도구를 같이 쓰세요. 검색·메모·치트시트의 삼각형이에요.

**검색.** 모르면 그냥 검색하세요. "macOS check ram size command" 이렇게 영어로 검색하면 1초만에 답이 나와요. 검색은 부끄러운 일이 아니에요. 모든 개발자가 매일 검색해요. 저도 매일 검색해요. 진짜로요. 외운 사람은 단 한 명도 없어요. 다들 검색해서 풀고, 자주 쓰는 건 자연스럽게 손에 붙는 거예요.

**메모.** 챕터마다 한 페이지 메모장을 만들어 두세요. "Ch 001 핵심 명령" 이라는 제목으로요. 거기에 오늘 친 명령 다섯 개만 복사해 두세요. 두 해 후엔 메모장이 두꺼워져요. 그게 여러분의 진짜 자산이에요. 두 해 코스의 진짜 결과물은 강의 영상이 아니라 여러분의 메모장이에요.

**치트시트.** 한 페이지짜리 요약표. 우리 코스에서는 챕터 끝마다 cheat sheet 한 장을 같이 만들어요. 8교시 끝에 Ch 001의 cheat sheet 한 장이 완성될 거예요. 인쇄해서 책상 옆에 붙이세요. 보고 또 보세요. 한 달이면 손에 다 붙어요.

이 삼각형이 두 해를 가는 비결이에요. 외우려고 끙끙거리지 마시고, 세 도구를 즐겁게 쓰세요. 즐거우면 오래 가요. 즐거움이 진짜 비결이에요. 저는 두 해 동안 여러분의 즐거움을 지키는 게 제 가장 큰 목표예요.

## 20. 오늘 친 모든 명령 한 번에 — 치트시트 미리보기

자, 한 시간 동안 명령이 좀 많이 나왔어요. 한 번 다 모아서 정리해 드릴게요. 이 박스 한 장만 캡처해 두시면 오늘의 모든 명령이 한 손에 잡혀요.

```bash
# OS 신분증
sw_vers
sw_vers -productVersion
system_profiler SPHardwareDataType
system_profiler SPHardwareDataType | grep "Model Name"
system_profiler SPHardwareDataType | grep "Serial Number"
system_profiler SPHardwareDataType | grep "Hardware UUID"

# 머신 신분증
uname -s
uname -m
uname -r
uname -n
uname -a

# CPU·RAM 신분증
sysctl -n machdep.cpu.brand_string
sysctl -n hw.ncpu
sysctl -n hw.physicalcpu
sysctl -n hw.logicalcpu
sysctl -n hw.memsize

# 셸·환경
echo $SHELL $0
echo $USER $HOME $PATH $LANG $TERM

# 디스크·네트워크
df -h /
ipconfig getifaddr en0
curl -s ifconfig.me; echo

# 시간
date
```

스무 줄 조금 넘죠. 이 스무 줄이 두 해 동안 여러분이 가장 자주 칠 명령들이에요. 한 번에 다 외우려고 하지 마시고, 매 챕터 시작할 때 이 박스 한 번씩만 보세요. 한 달이면 다 손에 붙어요. 그리고 5교시에서 이 스무 줄을 sysinfo.sh라는 한 개의 스크립트로 묶어요. 그 후엔 `bash start/sysinfo.sh` 한 줄이면 끝이에요. 손가락이 편해져요. 손가락이 편하면 자주 쓰게 되고, 자주 쓰면 컴퓨터의 안부를 자주 묻는 사람이 돼요. 좋은 사이클이에요.

한 가지 약속 더. 이 박스를 보고 한 줄이라도 "어, 이건 안 외워지네" 싶은 게 있으면 챕터 노트의 그 자리에 별표를 쳐 놓으세요. 일주일 후에 다시 챕터 와서 그 별표 자리만 한 번 더 쳐 보세요. 이걸 두 번 정도 하시면 거의 다 손에 붙어요. **별표가 학습이에요.** 모르는 곳에 별표 친 사람이 알게 돼요. 모른다는 걸 표시 못 한 사람은 영영 모르고 가요. 두 해 코스의 작은 비밀이에요.

마지막 한 줄 더. 이 박스의 명령들은 다 **읽기 전용**이에요. 컴퓨터를 망가뜨릴 수 없어요. 그래서 백 번 천 번 마음 놓고 쳐 보세요. 오타 나도 괜찮고, 결과 못 알아봐도 괜찮아요. 그저 손가락이 자판 위에서 명령을 만드는 그 감각만 익히시면 충분해요. 두 해 후엔 손가락이 머리보다 먼저 명령을 만들어요. 그 단계가 진짜 즐거워요. 그때까지 같이 가요.

## 21. 리눅스 친구·윈도우 친구를 위한 메모

같은 일을 다른 운영체제에서 하는 법도 잠깐 보여드릴게요. 두 해 후에 클라우드 서버에 들어가면 거의 다 리눅스예요. 미리 한 번 보세요.

| 정보 | macOS | Linux | Windows (PowerShell) |
|------|-------|-------|---------------------|
| OS 버전 | `sw_vers` | `cat /etc/os-release` | `Get-ComputerInfo` |
| 아키텍처 | `uname -m` | `uname -m` | `$env:PROCESSOR_ARCHITECTURE` |
| CPU 정보 | `sysctl -n machdep.cpu.brand_string` | `cat /proc/cpuinfo \| grep "model name" \| head -1` | `Get-CimInstance Win32_Processor` |
| 코어 수 | `sysctl -n hw.ncpu` | `nproc` | `Get-CimInstance Win32_Processor` |
| 램 크기 | `sysctl -n hw.memsize` | `free -h` | `Get-ComputerInfo \| Select-Object TotalPhysicalMemory` |

복잡해 보이죠. 외우지 마세요. **"각 운영체제마다 같은 일에 다른 명령이 있다"** 만 기억하세요. 표는 두 해 동안 두고두고 참고하시라고 만들어 둔 거예요.

한 가지 더. 표의 모든 명령에 공통점이 하나 있어요. 보면 다 동사가 비슷해요. "보여줘", "출력해줘", "리스트 해줘". 운영체제가 달라도 사람이 컴퓨터한테 묻는 질문의 종류는 거의 같아요. "너 누구니, 어떤 칩 쓰니, 메모리 얼마니, 디스크 얼마 남았니." 이 다섯 질문은 50년 동안 안 변했어요. 단어만 OS마다 다를 뿐이에요. 그래서 한 OS에서 잘하는 사람은 다른 OS에서도 금방 잘해요. 사고 패턴이 같으니까요. 두 해 후 여러분이 macOS를 익히고 나서 처음 리눅스 서버에 접속할 때, 막막함이 거의 없을 거예요. **"아, 같은 질문을 다른 단어로 던지면 되겠네"** 그게 다예요.

## 22. 한 줄 정리 — 머리에 남길 한 가지

자, 길게 이야기했어요. 한 시간 강의의 핵심을 단 한 줄로.

> **컴퓨터한테 질문을 던질 줄 알아야 답을 들을 수 있다. 그 첫 질문이 "너 누구니" 다.**

이 한 줄이 두 해의 디버깅 능력의 첫 벽돌이에요. 모든 디버깅은 결국 "지금 환경이 뭐지?"부터 시작해요. 똑같은 코드가 내 노트북에선 돌고 동료 노트북에선 안 돌면, 100% 환경이 달라요. 그 다른 환경을 한 줄 명령으로 잡아낼 수 있느냐 없느냐가 1년의 디버깅 시간을 결정해요. 오늘 다섯 가지 방법은 그 첫 한 줄을 배운 거예요. 작아 보이지만 평생 자산이에요.

## 23. 흔한 실수 다섯 가지 + 안심 멘트 — 학습 편

§11에서 명령이 안 될 때의 함정 다섯 가지를 봤어요. 그건 컴퓨터 쪽 함정이었어요. 이번엔 본인 학습 자세 쪽 함정 다섯 가지를 짚고 가요. 학습 자세가 흔한 실수 안에 들어 있어요. 미리 보시면 본인이 빠질 때 빨리 알아챌 수 있어요.

첫 번째 실수, 다섯 가지 방법을 다 외우려고 한다. 한 시간에 sw_vers, system_profiler, uname, sysctl, 한 줄 종합. 다 다른 명령. 본인도 모르게 "이 다섯 개를 머리에 다 박아야지" 하고 끙끙대세요. 안심하세요. 다 외우실 필요 없어요. 한 줄 종합 한 가지만 손에 익히세요. 나머지 네 개는 "이런 게 있다" 정도만 머리에 두세요. 두 해 동안 매번 만나면서 자연스럽게 박혀요. 다 외우려고 한 사람은 한 달이면 다 잊어요. 한 가지에 집중한 사람은 한 달 후에 그 한 가지를 진짜로 손에 가졌어요. 한 가지가 다섯 가지보다 강해요. 첫 챕터에서는 가장 단순한 강함을 먼저 갖추세요.

두 번째 실수, 한 글자 오타에서 그 자리에 멈춘다. `sw_vers`를 `swvers`로 치셨어요. command not found가 떴어요. 본인이 멘붕이 와요. "내가 명령어를 외우지 못한 건가? 다시 외워야 하나?" 안심하세요. 한 글자 차이는 모든 사람이 매일 만나는 일이에요. 저도 매일 만나요. 위쪽 화살표 키 한 번 누르세요. 방금 친 명령이 다시 나타나요. 거기서 한 글자만 고쳐서 다시 엔터. 30초면 끝이에요. 한 글자 오타에 5분 이상 멈추는 건 본인 실력의 문제가 아니에요. 그저 위쪽 화살표 키를 모르셨던 것뿐이에요. 이제 아세요. 두 해 동안 위쪽 화살표가 본인의 가장 친한 친구가 될 거예요.

세 번째 실수, 영어 에러 메시지를 보고 닫는다. 화면에 영어가 한 줄 떴어요. "command not found" 또는 "permission denied" 또는 "no such file or directory". 영어니까 본인도 모르게 닫아버리세요. 안심하세요. 그 영어 한 줄이 정답으로 가는 가장 빠른 길이에요. 영어가 어렵다고 안 보고 닫으면 5시간 헤매요. 한 줄을 그대로 복사해서 검색창에 붙이고 검색하면 30초 만에 답이 나와요. 모르는 영어 단어는 한국어로 한 번 더 검색하면 풀려요. 에러 메시지는 본인의 적이 아니라 친구예요. 컴퓨터가 본인한테 도와달라고 손을 내미는 거예요. 그 손을 닫지 마세요. 한 번 잡아 보세요. 30초가 5시간을 살려요.

네 번째 실수, 옆 친구 속도랑 본인 속도를 비교한다. 옆 친구는 명령을 1초 만에 치는데 본인은 30초 걸려요. "나는 못한다" 라는 생각이 머리에 들어와요. 안심하세요. 빠른 친구가 잘 배우는 게 아니에요. **끝까지 가는 친구가 잘 배워요.** 1교시에 강의 약속 다섯 번째였죠. 자기 페이스 지키기. 옆 친구는 옆 친구의 길이고 본인은 본인의 길이에요. 30초 걸려도 첫 명령을 친 본인은 첫 명령을 안 친 사람의 100배 앞서 있어요. 빠르게 가는 게 아니라 가는 것 자체가 진보예요. 그러니까 옆 친구의 화면 보지 마세요. 자기 화면만 보세요. 자기 속도 그대로 가세요. 두 해 후엔 두 사람 다 도착해요.

다섯 번째 실수, "환경 점검은 한 번 하면 끝"이라고 생각한다. 첫 시간에 신분증을 한 번 출력하셨어요. 그러고 나서 두 해 동안 안 다시 보세요. 안심하세요, 실수의 정체부터 풀어드릴게요. 환경은 변해요. macOS가 업데이트되고, 도구를 새로 깔고, PATH에 한 줄이 추가돼요. 6개월 후엔 본인 노트북의 환경이 오늘과 달라요. 그래서 매주 한 번씩, 또는 큰 업데이트 후마다 신분증을 다시 출력하시는 습관이 좋아요. 사람이 매년 건강검진 받듯, 컴퓨터도 매주 받아 주세요. 5교시 sysinfo.sh가 그 검진을 1초로 줄여줄 거예요. 한 번 만든 도구를 매주 한 번씩만 돌리세요. 그게 진짜 환경 점검의 정신이에요.

다섯 가지 학습 함정, 다 들으셨어요. 외우진 마세요(또 외움 함정 만들어낼 뻔했죠). 그저 본인이 빠졌을 때 한 번씩 떠오르시면 돼요. 한 글자 오타에서 멈췄을 때 "아, 두 번째 함정이구나" 하고 위쪽 화살표 한 번 누르시면 돼요. 옆 친구랑 비교됐을 때 "아, 네 번째 함정이구나" 하고 본인 화면으로 시선 돌리시면 돼요. 떠올리는 것만으로 절반은 풀려요. 두 해 동안 이 다섯 함정이 본인의 진짜 적이 될 거예요. 미리 알아두면 미리 이겨요.

## 24. 마무리 — 다음 시간 예고

자, 세 시간째 끝. 오늘은 신분증 묻는 다섯 가지 방법을 봤어요. 사과 메뉴, sw_vers, uname, sysctl, 그리고 한 줄 종합. 다섯 가지를 한 번씩 다 쳐 보셨으면 충분해요. 다 외워야 하는 거 아니에요. 챕터 노트에 다 적혀 있어요. 필요할 때 다시 와서 복사하세요. 모든 개발자가 매일 그래요. 그리고 사이사이에 시리얼·UUID, 환경 변수, 타임존, 디스크, 네트워크까지 한 칸씩 더 채웠어요. 일곱 칸짜리 종합 신분증이 손에 들어왔어요. 첫 한 시간 치고는 큰 수확이에요.

그리고 한 가지 약속해 주세요. 오늘 챕터 끝에서 한 줄 종합 명령을 한 번 더 쳐 보시고, 그 결과를 메모장 또는 슬랙 본인에게만 보낸 메시지로 한 번 저장하세요. 두 해 후에 다시 보시면 신기해요. "아, 이 노트북으로 첫 챕터를 들었었지" 하는 작은 기록이에요. 우리 코스는 두 해라는 시간 자체를 자료로 만들어요. 첫 시간의 자기 신분증을 적어두는 게 그 시작이에요.

다음 네 번째 시간에는, 신분증 말고 시스템을 다루는 일상 명령어 열 개를 카탈로그처럼 한 번에 정리해요. 파일 보기, 디렉토리 옮기기, 프로세스 보기, 네트워크 보기. 사전을 한 번 죽 훑는 시간이에요. 외울 필요 없어요. "이런 게 있구나" 한 번 보시면 돼요. 그리고 5교시에서 그 명령들을 모아서 sysinfo.sh 한 개로 묶어요. 6교시는 응급처치, 7교시는 폰 노이만+캐시, 8교시는 의사결정 + 회고. 한 챕터의 절반쯤 왔어요. 잘 따라오고 계세요.

쉬는 시간에 물 한 잔, 어깨 한 번 돌리시고, 거울 한 번 보고 자기 칭찬 한 마디. 십 분 후에 만나요. 약속이에요. 우리 이미 세 시간을 같이 했어요. 두 해의 360분의 1이에요. 작아 보여도 첫 360분의 1이 가장 중요한 단계예요. 시작이 절반이에요. 절반을 이미 한 거예요. 박수 한 번 쳐 주세요.

## 👨‍💻 개발자 노트

> - sysctl 키 네임스페이스(`hw`, `kern`, `machdep`)는 BSD 전통. Linux의 `/proc`·`/sys`와 정신적 친척. Ch 002 OS 챕터에서 "파일 시스템처럼 보이는 시스템 정보"로 회수됩니다.
> - `hw.physicalcpu` vs `hw.logicalcpu`의 SMT/하이퍼스레딩 구분은 Ch 047(컨테이너 CPU 한도), Ch 087(병렬 처리), Ch 108(EC2 vCPU = thread)에서 직접 회수됩니다.
> - macOS의 Darwin 커널 명시는 Ch 002 OS 첫 시간에 "macOS는 Mach + BSD = Darwin, Linux는 별개 커널 가족"으로 회수됩니다.
> - bash `$(...)` 명령 치환과 `&&` 연쇄는 Ch 006 터미널 챕터에서 본격 정복. 이번 챕터에서는 "같이 쳐보고 결과만 확인" 수준으로 노출합니다.
> - 비교표의 PowerShell 명령은 정확하나, 본 코스의 메인 OS는 macOS이고 클라우드 호스트는 Linux. Windows 사용자에게는 WSL2 안내(`docs/STYLE-GUIDE.md` §9 권장 환경)가 필요합니다.
> - §11 함정(컴퓨터 쪽)과 §23 실수(학습 자세 쪽)의 분리는 의도적. LECTURE-TEMPLATE §N-1 표준 항목을 기술 진단(§11)과 메타 학습(§23) 두 축으로 풀어 비개발자/중급 개발자 양쪽 청자를 모두 충족. §23 두 번째 실수(한 글자 오타→위쪽 화살표)는 Ch 006 터미널 챕터에서 zsh history-search-backward 단축키(`Ctrl+R`)로 회수합니다.
> - §14 `$LANG` 영어 메시지 권장은 STYLE-GUIDE.md §3 "외래어 첫 등장 규칙"과 별개. 본문은 한국어 톤, 진단 시는 영어 에러 메시지로 검색 — 이 분리는 모든 챕터에서 일관 적용. Ch 069 OWASP에서 영어 에러 페이지의 정보 노출 위험과는 별개 주제.
> - §15 타임존(`/etc/localtime`)은 macOS와 Linux 공통 (BSD 유산). `date` 명령의 출력 포맷은 macOS(`Sat Apr 25 10:00:00 KST 2026`)과 GNU Linux(`Sat Apr 25 10:00:00 AM KST 2026`)에서 미세하게 다릅니다. 자동화 스크립트에서는 `date -u +%Y-%m-%dT%H:%M:%SZ` (ISO 8601 UTC) 사용 권장 — Ch 102 LLM Eval 챕터의 로그 타임스탬프 표준.
> - §16 일화(`uname -m` arm64 디버깅)는 Ch 047 Docker 챕터의 `--platform linux/amd64` 옵션 도입 시 직접 회수. arm64/x86_64 분기는 학기 1 ~ 8 전 구간 통과 주제이며, H3에서 `uname -m`을 외우게 만드는 것이 학기 전체 디버깅 비용 최대 절감 지점.
> - §17 `curl ifconfig.me` vs `ipconfig getifaddr`의 두 IP 차이는 Ch 003 H7 NAT, Ch 056 백엔드 환경, Ch 106 VPC 네트워킹에서 단계적으로 깊이 회수. 첫 챕터에서는 "두 IP가 다르다는 사실의 인식"만 목표.
