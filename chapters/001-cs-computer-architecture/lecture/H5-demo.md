## 2. 오늘 만들 도구 — 자기소개서 출력기

만들 도구의 이름은 `sysinfo.sh`예요. .sh 확장자는 "셸 스크립트(shell script)" 즉 터미널 명령을 묶어서 한 번에 실행하는 작은 프로그램이에요. 우리가 두 해 동안 자주 만나는 형식이에요.

도구의 목적: 한 번 실행하면 우리 맥의 다음 정보를 한 화면에 깔끔하게 보여줘요.

```

================================================================
[OS]      macOS 14.5 (Build 23F79)
[Arch]    arm64
[CPU]     Apple M2 Pro
[Cores]   12 (Physical: 12, Logical: 12)
[RAM]     16 GB
[Disk]    245G used / 466G total (53%)

[Uptime]  up 3 days, 14:22
```

이거예요. 이걸 한 번 실행하면 다 떠요. 자, 시작합시다.

`start/sysinfo.sh`의 내용은 이래요. 빈 껍데기예요.

```bash
#!/usr/bin/env bash
# sysinfo.sh — 내 맥의 자기소개서 한 장 출력
# TODO: H5에서 같이 채웁니다.

```

다섯 줄이에요. 첫 줄 `#!/usr/bin/env bash`는 셔뱅이라고 부르는데, 이 파일을 실행할 때 bash라는 셸로 실행하라는 표시예요. 셔뱅이 뭔지는 6번째 챕터에서 진짜 다뤄요. 오늘은 그냥 "맨 위에 한 줄 있어야 한다" 정도로 알아주세요. `set -e`는 안전 장치예요. 명령 중 하나라도 실패하면 그 자리에서 스크립트를 멈춰요.

자, 우리는 이 빈 껍데기를 다섯 단계로 채워나갈 거예요. `start/sysinfo.sh`를 그대로 두고, 챕터 폴더에 새로 `my-sysinfo.sh`를 하나 만들어서 거기에 따라 치셔도 되고, `start/sysinfo.sh`를 직접 고치셔도 돼요. 자기 편한 대로요.

## 4. 1단계 — 헤더와 운영체제 정보

먼저 헤더와 OS 정보를 추가해요.

```bash
#!/usr/bin/env bash
set -e

echo "================================================================"
echo "  ✨ My Mac — System Information ✨"
echo "================================================================"

OS_NAME=$(sw_vers -productName)
OS_VERSION=$(sw_vers -productVersion)
OS_BUILD=$(sw_vers -buildVersion)
echo "[OS]      ${OS_NAME} ${OS_VERSION} (Build ${OS_BUILD})"
```

새로 나온 게 있어요. `OS_NAME=$(sw_vers -productName)`. 이건 변수 할당이에요. 등호 양쪽에 공백이 없어야 해요. 한 가지 셸 문법의 함정이에요. `OS_NAME =$(...)`라고 띄어쓰면 안 돼요. 등호 붙여 쓰세요.

`$()`는 H3에서 본 적 있죠. 그 안의 명령을 실행해서 결과를 가져와요. `${OS_NAME}`은 변수 값을 끼워 넣을 때 쓰는 표시예요. 중괄호 안 써도 되지만 쓰는 게 더 안전해요.

> ▶ **여기까지 한 번 실행해보기**
>
> ```bash
> bash start/sysinfo.sh
> ```

세 줄짜리 헤더와 OS 한 줄이 떴으면 1단계 성공. 안 뜨면 손 들어 주세요.

## 5. 2단계 — CPU 정보 채우기

```bash
ARCH=$(uname -m)

echo "[Arch]    ${ARCH}"

```

H3에서 본 sysctl 키들이에요. 한 번 더 만나니까 친근하죠. 다시 실행해 보세요.

이렇게 떴으면 2단계 성공. 자기 맥의 정보가 다 다르게 떠요. 옆 친구 거랑 비교해 보세요.

## 6. 3단계 — RAM과 디스크

```bash
RAM_BYTES=$(sysctl -n hw.memsize)
RAM_GB=$((RAM_BYTES / 1024 / 1024 / 1024))

DISK_INFO=$(df -h / | tail -1)
DISK_TOTAL=$(echo "$DISK_INFO" | awk '{print $2}')
DISK_USED=$(echo "$DISK_INFO" | awk '{print $3}')
DISK_PCT=$(echo "$DISK_INFO" | awk '{print $5}')

echo "[RAM]     ${RAM_GB} GB"
echo "[Disk]    ${DISK_USED} used / ${DISK_TOTAL} total (${DISK_PCT})"
```

새 친구 두 명. `$(( ... ))`는 산술 연산이에요. bash가 안에 든 식을 계산해서 결과를 돌려줘요. RAM 바이트를 1024로 세 번 나눠서 GB로 바꿨죠. `awk` (awk, "오크")는 표 형식의 텍스트에서 특정 칸만 뽑아내는 도구예요. df -h /의 결과가 한 줄에 여러 칸이 있는데, 그 중 2번째 칸은 전체 용량, 3번째는 사용량, 5번째는 사용률이에요. awk가 그 칸을 뽑아줘요.

awk는 두 해 동안 자주 만나요. 오늘은 "특정 칸 뽑는 친구" 정도로 기억하세요.

## 7. 4단계 — 네트워크와 가동 시간

```bash
IP_ADDR=$(ipconfig getifaddr en0 2>/dev/null || echo "N/A")
UPTIME_INFO=$(uptime | sed 's/^ *//')

echo "[IP]      ${IP_ADDR}"
echo "[Uptime]  ${UPTIME_INFO}"
```

`ipconfig getifaddr en0` 다음의 `2>/dev/null || echo "N/A"`는 안전 장치예요. en0이 없거나 와이파이가 꺼져 있으면 N/A를 출력해요. 2>는 에러를 보내라는 표시고, /dev/null은 "버려라"는 뜻의 특수한 자리예요. ||는 "왼쪽이 실패하면 오른쪽을 해라"예요. 기억해 두세요. 두 해 동안 자주 봐요.

`sed 's/^ *//'`는 uptime 출력의 맨 앞 공백을 잘라내는 거예요. uptime이 보통 공백으로 시작하거든요. 모양 다듬기예요.

## 8. 5단계 — 색깔 입히기와 마무리

마지막은 헤더에 색깔을 좀 입히고, 푸터를 추가해요.

```bash
GREEN='\033[0;32m'
BOLD='\033[1m'
RESET='\033[0m'

echo -e "${GREEN}${BOLD}================================================================${RESET}"
echo -e "${BOLD}  ✨ My Mac — System Information ✨${RESET}"
echo -e "${GREEN}${BOLD}================================================================${RESET}"

# ... 위에 만든 OS, CPU, RAM, Disk, IP, Uptime 줄들 ...

echo -e "${GREEN}${BOLD}================================================================${RESET}"
```

`\033[0;32m`는 ANSI 색깔 코드예요. 터미널이 이 코드를 보면 그 다음 글자에 초록색을 입혀요. RESET은 색을 원래대로 돌리는 코드. echo의 `-e` 옵션은 이런 특수 코드를 해석하라는 옵션이에요. 안 붙이면 코드가 그대로 출력돼서 깨져 보여요.

전체 완성본은 이 챕터의 `finish/sysinfo.sh`에 들어 있어요. 다 따라치는 게 부담스러우면 `finish/`의 코드를 그대로 가져다 쓰셔도 돼요. 다만 한 번 손으로 쳐 보시는 게 진짜 좋아요. 처음 한 번이 가장 어렵고, 그 다음엔 쉬워져요.

## 10. 셔뱅과 set -e — 안전 장치 두 줄의 정체

스크립트의 맨 위 두 줄을 다시 봐요.

```bash
#!/usr/bin/env bash
set -e
```

첫 줄 `#!/usr/bin/env bash` 가 **셔뱅(shebang)** 이에요. 이상한 이름이죠. # = 샵(sharp), ! = 뱅(bang). 두 글자를 합쳐서 sha-bang. 이 두 글자가 파일의 첫 두 글자면, 운영체제가 이 파일을 실행할 때 그 다음에 적힌 프로그램으로 실행해요. 즉 "이 파일을 bash로 돌려 줘" 라는 표시예요.

`/usr/bin/env bash`는 좀 신중한 표현이에요. 그냥 `/bin/bash`라고 쓸 수도 있는데, env를 거치면 환경에서 첫 번째로 발견되는 bash를 써요. macOS는 /bin/bash가 옛날 버전(3.2)이고, 사용자가 Homebrew로 더 새 버전(예: 5.x)을 따로 깔 수 있어요. env를 거치면 그 새 버전을 자동으로 쓰게 돼요. 좀 더 똑똑하고 안전한 한 줄이에요.

두 번째 줄 `set -e`. 이게 안전 장치예요. 보통 셸 스크립트는 명령이 실패해도 그냥 다음 줄로 넘어가요. 그러면 가짜 성공 같은 사고가 나요. "어, 실행은 됐는데 결과가 이상해"라는 미스터리. set -e가 있으면 한 줄이라도 실패하면 거기서 즉시 멈춰요. 무리한 진행을 막아요. 비행기 안전벨트 같은 거예요.

set 가족이 좀 더 있어요. 셋이 같이 다녀요.

| 옵션 | 별명 | 하는 일 |
|------|------|---------|
| `set -e` | "에러나면 멈춰" | 명령 하나라도 실패하면 즉시 종료 |
| `set -u` | "미정의 변수면 멈춰" | $UNDEFINED 같은 거 못 쓰게 |
| `set -o pipefail` | "파이프 안에서도 에러 잡자" | `cmd1 | cmd2` 에서 cmd1 실패도 잡음 |

이 셋을 묶어서 `set -euo pipefail` 한 줄로 쓰는 게 시니어 개발자들의 표준이에요. 두 해 후 우리가 짤 모든 진지한 스크립트는 이 한 줄로 시작해요. 오늘은 첫 챕터라 부담을 줄이려고 `-e` 만 썼어요. 6번째 챕터쯤에서 셋 다 같이 만나요.

`set -e` 말고 또 한 가지 안전 장치를 같이 보세요. 우리 스크립트의 IP 줄에서 `2>/dev/null || echo "N/A"`. 이게 set -e의 짝이에요. set -e는 "실패하면 멈춰"인데, 이 ||는 "실패해도 N/A로 우회해" 예요. 안전 장치를 켜놓고도 우회 길을 만들어 둔 거예요. 두 안전 장치는 모순처럼 보이지만 진짜 잘 어울리는 짝이에요. **"기본은 멈추고, 일부만 우회한다."** 이게 좋은 셸 스크립트의 패턴이에요.

## 11. 변수와 명령 치환 — 셸의 두 손

스크립트에서 가장 많이 본 패턴이 이거예요.

```bash
OS_NAME=$(sw_vers -productName)
echo "[OS]      ${OS_NAME}"
```

두 줄에 두 가지 도구가 들어 있어요. 변수 할당과 명령 치환.

**변수 할당 `이름=값`.** 셸에서 변수에 값을 담는 문법. 등호 양쪽에 공백이 없어요. 진짜 중요해요. `OS_NAME =$(...)` 라고 띄어쓰면 셸이 OS_NAME을 명령으로 보고 "command not found"라고 화내요. 이 함정에 두 해 동안 한 번씩 빠질 수 있어요. 미리 한 번 머리에 두세요.

**명령 치환 `$()`.** 그 안의 명령을 실행해서 결과 텍스트로 바꿔치기. `$(sw_vers -productName)`는 실행되면 "macOS"라는 텍스트로 바뀌어요. 그래서 OS_NAME에 "macOS"가 담겨요. 옛날에는 백틱(`)으로 같은 일을 했는데, 백틱은 보기 헷갈려서 요즘은 $()를 써요. 두 해 코스 내내 우리는 $() 만 써요.

**변수 사용 `${OS_NAME}`.** 담아둔 값을 꺼내 쓰기. 중괄호 없이 `$OS_NAME` 도 되지만, 중괄호가 있으면 더 안전해요. 예를 들어 `$OS_NAME_HISTORY` 라고 쓰면 셸이 "OS_NAME_HISTORY 라는 변수의 값"으로 봐요. 우리 의도가 "$OS_NAME 의 값 + _HISTORY" 였다면, `${OS_NAME}_HISTORY` 라고 중괄호로 명확히 잘라줘야 해요. 작은 차이가 큰 사고를 막아요.

이 두 손(변수, 명령 치환)이 셸의 진짜 양손이에요. 두 해 동안 만나는 모든 셸 스크립트의 절반이 이 두 패턴의 반복이에요. 첫 챕터에서 한 번 익혀두시면 평생 자산이에요.

한 가지 작은 팁. **변수 이름은 대문자**가 셸 관습이에요. `OS_NAME`, `RAM_GB`, `IP_ADDR`. 다른 프로그래밍 언어(파이썬, JS)에서는 보통 소문자를 쓰지만 셸에서는 대문자가 표준이에요. 환경 변수와 헷갈리지 말라고 그래요. 두 해 후 파이썬 챕터에서 다시 만날 때 "어, 셸은 대문자였지" 하고 헷갈리지 않게 미리 알아두세요.

## 12. awk·sed·tail — 가공 도구 삼총사 한 번 더

H4에서 grep·find·which를 봤죠. 이번엔 가공 도구 셋. awk, sed, tail. 우리 스크립트에서 다 한 번씩 만났어요.

**awk** — 칸 뽑기. `df -h /` 결과는 표 한 줄이에요. 칸이 공백으로 나뉘어 있어요. `awk '{print $5}'` 는 다섯 번째 칸만 뽑아요. 우리 스크립트에서는 이걸로 디스크 사용률(45% 같은 거)을 뽑았어요. awk는 사실 작은 프로그래밍 언어예요. 두 해 동안 점점 깊이 들어가요. 오늘은 "$N으로 N번째 칸" 만 기억.

**sed** — 텍스트 치환. `sed 's/^ *//'` 는 줄 맨 앞의 공백을 다 잘라내요. s = substitute(치환), ^ = 줄 시작, ` *` = 공백 0개 이상, // = 빈 문자열로 바꿔라. 모양을 정리할 때 쓰는 친구예요. 이것도 작은 언어예요. 두 해 동안 점점 깊이.

**tail** — 끝부분 보기. `df -h / | tail -1` 은 마지막 한 줄만 봐요. df 결과는 보통 두 줄(헤더 + 데이터)인데, 우리는 데이터만 필요하니까 tail로 마지막 줄만 가져왔어요. 짝꿍은 head(앞부분 보기). H4에서 봤죠.

이 셋을 파이프로 이으면 진짜 강력해요. 우리 스크립트에서 디스크 정보 처리는 사실 `df -h / | tail -1 | awk '{print $5}'` 라는 한 줄이에요. df가 결과 뱉기 → tail이 마지막 줄만 추리기 → awk가 5번째 칸 뽑기. 세 명이 차례로 일하는 작은 공장이에요. 유닉스 철학(작은 도구 + 파이프) 그 자체예요. 두 해 후 우리가 만들 파이프라인의 모양이 다 이런 모양이에요. 첫 챕터에서 한 번 본 모양이 평생 가요.

여기서 살짝 보너스. 파이프로 단계가 많아지면 한 줄이 너무 길어져요. 그땐 변수에 중간 결과를 담는 게 읽기에 더 좋아요.

```bash
DISK_INFO=$(df -h / | tail -1)            # 한 번에 가져오기
DISK_PCT=$(echo "$DISK_INFO" | awk '{print $5}')   # 그 결과에서 칸 뽑기
```

이게 우리 스크립트가 한 방식이에요. 한 줄로 다 이으면 빠르지만, 두 줄로 나누면 디버깅이 쉬워요. 중간에 echo "$DISK_INFO" 한 번 넣어서 들여다 볼 수 있으니까요. 속도와 가독성의 트레이드오프예요. 두 해 동안 본인 스타일을 잡아 가시면 돼요.

## 13. ANSI 색깔 코드 — 까만 화면을 화려하게

마지막 단계에서 본 색깔 코드. 다시 보세요.

```bash
GREEN='\033[0;32m'
BOLD='\033[1m'
RESET='\033[0m'
```

`\033` 는 ESC 키 코드예요. 8진수로 33. 터미널이 ESC 다음에 `[숫자m` 형식이 오면 "이 다음 글자에 이런 효과를 입혀라" 라고 해석해요. 이게 **ANSI 이스케이프 시퀀스(ANSI escape sequence)** 예요. 1970년대에 만들어진 표준이에요. 셸 50년 역사의 또 다른 뿌리예요.

자주 쓰는 색깔 번호 몇 개 알려 드릴게요.

| 효과 | 코드 | 용도 |
|------|------|------|
| 검정 | `\033[0;30m` | 어두운 배경에는 안 보임. 주의 |
| 빨강 | `\033[0;31m` | 에러, 경고 |
| 초록 | `\033[0;32m` | 성공, 정상 |
| 노랑 | `\033[0;33m` | 주의, 진행 중 |
| 파랑 | `\033[0;34m` | 정보 |
| 진한 글씨 | `\033[1m` | 강조 |
| 밑줄 | `\033[4m` | 강조 |
| 리셋 | `\033[0m` | 효과 끝 |

마지막 RESET이 진짜 중요해요. 색깔 입히고 RESET 안 해 두면, 그 다음 줄도, 그 다음 명령도, 새 터미널 열 때까지 계속 그 색깔로 떠요. 사고 나요. 항상 색깔 켰으면 끄세요. 안전 수칙이에요.

`echo -e` 의 -e 옵션이 또 중요해요. echo는 기본적으로 \033 같은 코드를 그냥 글자로 출력해요. 사람이 보면 깨진 글자처럼 보여요. -e를 붙여야 코드를 진짜 효과로 해석해요. 한 글자 옵션이지만 결과가 천지 차이예요. 잊지 마세요.

두 해 후 우리가 만들 모든 로그 시스템이 이 색깔 코드를 써요. Flask 서버의 로그, Docker의 로그, AWS CloudWatch의 로그. 다 ANSI 시퀀스로 색칠돼 있어요. 첫 챕터에서 한 번 본 코드가 평생 가요. 신기하죠.

## 14. 확장 시나리오 — sysinfo.sh를 한 단계 키우기

오늘 만든 50줄짜리는 1단계 버전이에요. 두 해 동안 여러분이 본인 손으로 한 단계씩 키워가시면 좋아요. 아이디어 다섯 개 던져 드려요.

**확장 1 — JSON 출력 옵션.** `--json` 인자를 받으면 사람이 읽는 표 대신 JSON 한 덩어리로 출력. 두 해 후 클라우드 모니터링 시스템에 보낼 때 표보다 JSON이 더 좋아요. JSON은 18번째 챕터쯤에서 본격적으로 만나요.

**확장 2 — 임계값 알림.** 디스크 사용률이 90% 넘으면 빨간 글씨로, 95% 넘으면 깜빡이게(ANSI `\033[5m`). 시각적 경고를 자동으로요. 두 해 후 자동 알림 시스템의 가장 작은 버전이에요.

**확장 3 — 한 줄 요약 모드.** `--brief` 인자를 받으면 표 대신 한 줄 요약. 예: `[OK] M2 Pro · 16GB · 53% disk · 192.168.0.10 · up 3d`. 슬랙 봇이 한 줄로 보고할 때 유용해요.

**확장 4 — 히스토리 저장.** 매번 실행할 때마다 결과를 `~/.sysinfo-history.log` 에 한 줄씩 추가. 일주일 후에 보면 디스크가 어떻게 차오르는지, 가동 시간이 어떻게 늘어나는지 보여요. 두 해 후 시계열 모니터링의 가장 작은 버전이에요.

**확장 5 — 크로스 플랫폼.** macOS와 Linux 둘 다에서 돌아가게. `case "$(uname -s)" in Darwin) ... ;; Linux) ... ;; esac` 패턴. 두 해 후 우리가 만들 거의 모든 도구는 macOS 노트북과 Linux 서버 둘 다에서 돌아야 해요. 첫 단계예요.

이 다섯 가지를 두 해 동안 본인 페이스로 하나씩 추가해 보세요. 두 해 후엔 sysinfo.sh가 200줄짜리 본인만의 헬스체크 도구로 자라 있을 거예요. 그게 본인의 첫 작품이에요. GitHub에 올리시면 이력서에도 한 줄 쓸 수 있어요. 진짜로요. "Maintained personal Mac/Linux health check tool, 2 years." 작아 보이지만 꾸준함을 증명해요. 시니어 면접관이 가장 좋아하는 거예요.

## 15. 두 해 후 우리 사이트 — 서버 헬스체크의 첫 벽돌

오늘 만든 sysinfo.sh가 두 해 후 우리 고양이 자경단 서버에서 어떻게 자라는지 한 장면 보여 드릴게요.

장면. 두 해 후, 우리는 AWS EC2 인스턴스 다섯 대를 운영해요. 각각의 인스턴스가 우리 사이트의 일부를 처리해요. 한 대가 죽으면 사용자가 못 들어와요. 그래서 우리는 1분에 한 번씩 다섯 대 모두에 헬스체크 신호를 보내요. 그 헬스체크의 첫 버전이 오늘의 sysinfo.sh의 직계 후손이에요.

```bash
# 두 해 후 버전: server-healthcheck.sh
#!/usr/bin/env bash
set -euo pipefail

INSTANCE_ID=$(curl -s http://169.254.169.254/latest/meta-data/instance-id)
LOAD=$(uptime | awk -F'load average:' '{print $2}' | awk '{print $1}')
DISK_PCT=$(df -h / | tail -1 | awk '{print $5}' | tr -d '%')
MEM_PCT=$(free | grep Mem | awk '{print int($3/$2 * 100)}')

# JSON으로 모니터링 시스템에 보고
cat <<EOF
{
  "instance_id": "${INSTANCE_ID}",
  "load_avg_1m": ${LOAD},
  "disk_pct": ${DISK_PCT},
  "mem_pct": ${MEM_PCT},
  "ts": "$(date -u +%Y-%m-%dT%H:%M:%SZ)"
}
EOF
```

20줄짜리 미래 버전이에요. 오늘 본 패턴이 다 들어 있죠. 셔뱅, set 안전 장치, 변수, 명령 치환, awk 칸 뽑기, df, uptime. 차이는 `cat <<EOF ... EOF` 라는 새 친구(여러 줄 텍스트 출력)와 AWS 메타데이터 URL(서버 자기 정보 가져오기) 정도예요. 첫 챕터의 sysinfo.sh를 한 번 손에 익혀두시면, 두 해 후 이 server-healthcheck.sh의 90%가 친근해요.

그리고 이 헬스체크를 어떻게 1분에 한 번씩 자동으로 돌리느냐. **cron(크론)** 이라는 도구로요. 두 해 후 운영 챕터(약 Ch 110)에서 본격적으로 만나요. cron이 이 스크립트를 1분에 한 번 실행해서 결과를 모니터링 시스템(예: AWS CloudWatch)에 보내요. 사람이 안 자도 시스템이 24시간 우리 서버를 지켜봐요. 새벽 3시 사고도 자동으로 알림이 와요.

오늘 만든 50줄이 두 해 후 24시간 자동으로 돌면서 우리 사이트를 지키는 도구로 자라요. 작아 보이지만 진짜 큰 첫 벽돌이에요. 이게 우리가 5교시를 챕터의 하이라이트로 둔 이유예요. 보지 못하던 미래 한 조각을 손에 쥐고 가시는 거예요.

## 16. 실패 일곱 — 자주 만나는 에러와 응급처치

처음 셸 스크립트를 짜면 100% 한 번씩은 막혀요. 자주 만나는 에러 일곱 가지를 미리 보여 드릴게요.

**실패 1 — `OS_NAME =$(sw_vers -productName)` 했더니 `OS_NAME: command not found`.**
원인: 등호 양쪽에 공백. 응급처치: 공백 없애기. `OS_NAME=$(...)`.

**실패 2 — `bash: sysinfo.sh: command not found` 또는 `permission denied`.**
원인: 실행 권한 없음 또는 경로 안 맞음. 응급처치: `bash sysinfo.sh` (bash 앞에 붙이기) 또는 `chmod +x sysinfo.sh && ./sysinfo.sh`.

**실패 3 — 색깔 코드가 `\033[0;32m` 같이 글자 그대로 떠요.**
원인: `echo` 에 `-e` 옵션 빠짐. 응급처치: `echo -e "..."` 로 바꾸기. macOS의 기본 `/bin/echo` 는 -e 안 먹기도 해요. 그땐 `printf` 로 대체. `printf '%b\n' "${GREEN}hello${RESET}"`.

**실패 4 — `RAM_GB=$((RAM_BYTES / 1024 / 1024 / 1024))` 했는데 0이 떠요.**
원인: RAM_BYTES 변수가 비어 있음. 응급처치: 한 줄 위에 `echo "DEBUG: $RAM_BYTES"` 넣어서 진짜로 값이 들어 왔는지 확인. 보통 sysctl 명령이 실패해서 빈 값이 들어와요.

**실패 5 — `ipconfig: command not found`.**
원인: macOS 아닌 환경(Linux). 응급처치: `ip addr show | grep "inet "` 같은 리눅스 명령으로 우회. 또는 H8의 크로스 플랫폼 분기 패턴 참고.

**실패 6 — uptime의 출력이 너무 길고 뒤죽박죽이에요.**
원인: macOS 버전마다 uptime 형식이 미세하게 다름. 응급처치: `awk` 로 필요한 칸만 뽑기. 또는 그냥 `uptime` 통째로 출력하고 사람이 알아보게 두기. 첫 버전에선 그게 더 안전해요.

**실패 7 — 스크립트가 중간에 멈춘 후 그 후 명령이 실행 안 돼요.**
원인: `set -e` 가 중간 실패에서 즉시 종료시킴. 응급처치: 어느 줄에서 멈췄는지 보려면 `set -x` 를 추가하면 한 줄씩 출력해요. 또는 `bash -x sysinfo.sh` 로 실행. 디버깅의 진짜 친구예요.

이 일곱 가지가 첫 셸 스크립트 사고의 90%를 커버해요. 미리 한 번 머리에 두시면, 막혔을 때 "아, 이거 그 실패구나" 알아채실 수 있어요. 알아채는 게 절반이에요.

## 17. 자주 묻는 질문 다섯 — 데모 편

**Q1. "스크립트 vs 일회성 명령. 언제 어느 걸 써야 해요?"**
A. 두 번 이상 칠 명령이면 스크립트로. 한 번만 칠 거면 그냥 명령. 단순 규칙이에요. 그리고 일회성도 자주 친다 싶으면 alias로 (H4 §18.9). 셋의 위계가 있어요. **명령 → alias → 스크립트.** 자주 쓰는 정도에 따라 점점 정식화돼요.

**Q2. "bash 말고 zsh으로 짜도 돼요?"**
A. 네, 됩니다. macOS 기본 셸은 zsh이고, 우리 sysinfo.sh의 명령들은 bash·zsh 둘 다에서 동작해요. 다만 셔뱅을 `#!/usr/bin/env zsh` 로 바꿔야 해요. 두 해 코스에서는 `bash`로 통일해요. 이유는 리눅스 서버가 거의 다 bash 기본이라서요. 호환성을 위해서요.

**Q3. "Python으로 짜도 돼요?"**
A. 네, 됩니다. 사실 더 깔끔할 수도 있어요. 그런데 Python 스크립트는 Python이 깔려 있어야 동작해요. 셸 스크립트는 거의 모든 유닉스 환경에 기본으로 있어요. 그래서 시스템 도구는 보통 셸로 짜요. 큰 도구나 복잡한 로직은 Python으로 짜요. 두 해 동안 둘 다 만나요.

**Q4. "이 스크립트를 실수로 지우면 어떡하죠?"**
A. 그래서 **Git** 이라는 도구로 버전 관리해요. 두 해 코스의 11번째 챕터에서 본격적으로 만나요. 일단 오늘은 챕터 폴더의 finish/sysinfo.sh를 한 번 별도로 백업해 두시면 안전해요. cp finish/sysinfo.sh ~/sysinfo-backup.sh 한 줄.

**Q5. "다른 사람의 sysinfo.sh랑 내 거랑 결과가 달라요. 정상이에요?"**
A. 정상이에요. 우리 맥들이 다 다른 모델, 다른 OS 버전, 다른 IP를 가지고 있으니까요. 결과의 형식만 같으면 돼요. 옆 친구의 결과랑 비교해 보시면, "아, 같은 도구가 다 다른 자기소개서를 만드는구나" 하고 신기해요. 첫 챕터의 작은 즐거움이에요.

## 17.5. start vs finish — 폴더 두 개의 약속

이 챕터의 폴더 구조에 한 가지 약속이 있어요. `start/` 와 `finish/` 두 폴더예요. 이게 우리 코스 전체의 표준이에요. 한 번 짚고 가세요.

**start/** 는 빈 껍데기예요. 학습자가 따라 치기 시작할 출발선이에요. 셔뱅과 set -e와 TODO 한 줄만 들어 있어요. 손이 떨려도 시작할 수 있게 최소한의 발판만 놔둔 거예요.

**finish/** 는 완성본이에요. 챕터의 모든 단계가 끝나면 도달할 도착점이에요. 학습자가 막히면 finish/를 들여다보고 비교하면서 막힌 부분을 풀 수 있어요. 컨닝 페이퍼가 아니라 정답지예요. 정답지를 보는 건 부끄러운 일이 아니에요. 정답지가 없으면 막혔을 때 영영 못 풀잖아요.

이 두 폴더 약속이 두 해 코스의 모든 데모 챕터에 적용돼요. 약 18챕터분의 코드가 이런 구조로 진화해요. 챕터마다 start/는 빈 껍데기, finish/는 완성본. 학습자는 자기 페이스로 start/에서 finish/로 가요. 빨리 가도 좋고, 천천히 가도 좋아요. 정답이 어디 있는지만 알면 마음이 편해요.

한 가지 팁. start/만 보고 한 번 진짜로 막히면, finish/의 같은 줄을 한 번에 통째로 보지 마시고, 한 줄씩만 비교해 보세요. 어디서 본인이 다르게 갔는지가 보여요. 그 다른 한 줄이 본인의 학습 포인트예요. 그 한 줄에 별표 치고 다음에 한 번 더 쳐 보시면 진짜 본인 것이 돼요. H3에서 약속한 별표 학습이에요.

그리고 한 가지 더. 챕터를 다 끝내고 나면 본인이 짠 sysinfo.sh를 한 번 finish/와 비교해 보세요. 다르면 어느 쪽이 더 본인 마음에 드는지 결정하세요. 더 좋은 아이디어가 있으면 본인 거를 keep 하세요. **finish/는 정답이 아니라 한 가지 답일 뿐이에요.** 본인의 답이 더 좋을 수 있어요. 두 해 동안 본인의 답을 만들 줄 아는 사람이 되시면 충분해요.

## 17.7. 첫 코드 한 줄 — 30년 전 사람의 손가락

마지막으로 잠깐 옆길로 새서, 셸 스크립트의 짧은 역사를 짚고 갈게요. 다섯 번째 시간이라 한 번 음미할 만해요.

오늘 친 `#!/usr/bin/env bash` 의 셔뱅. 1980년에 데니스 리치(Dennis Ritchie, C 언어와 유닉스의 아버지)가 만들었어요. 두 글자(#!) 가 운영체제에게 "이 파일을 어느 프로그램으로 돌릴지" 알려주는 표시. 40년이 넘었지만 아직도 모든 유닉스 시스템에서 같은 약속으로 동작해요.

`bash` 자체는 1989년, 브라이언 폭스(Brian Fox)가 GNU 프로젝트의 일부로 만들었어요. Bash는 "Bourne Again SHell" 의 줄임말. 그 전의 sh(Bourne shell)를 이은 새 셸이라는 뜻인데, 영어 발음 장난이에요(born again, "다시 태어난"). 30년이 넘은 도구가 아직도 매일 수십억 번 호출돼요. 좋은 도구는 안 죽어요.

`awk` 는 1977년에 Aho, Weinberger, Kernighan 세 사람이 만들었어요. 이름이 그 셋의 머리글자예요. 49년 된 도구가 우리 sysinfo.sh의 한 줄에 살아 있어요. `sed` 는 1973년. 53년 된 도구. `df` 는 1971년. 55년 된 도구.

오늘 친 50줄 안에 50년 전 컴퓨터 과학자들의 손가락이 다 살아 있어요. 우리는 그들과 같은 동작을 하고 있어요. 1971년 벨 연구소의 어느 책상에서 처음으로 df 한 줄을 친 사람이 있었어요. 우리는 2026년 본인 책상에서 같은 한 줄을 쳐요. 같은 결과를 봐요. 시간을 가로지르는 한 가지 약속이에요. 어떻게 보면 작은 영광이에요.

이게 셸을 사랑하는 사람들의 마음이에요. 50년 전 도구가 아직 살아 있고, 그 도구로 오늘 우리가 진짜 일을 해요. 두 해 후 여러분이 만들 사이트도 50년 후에 같은 셸 명령들 위에서 돌고 있을 가능성이 높아요. 우리는 50년 짜리 사슬의 한 마디예요. 작아 보이지만 진짜 큰 한 마디예요.

## 17.9. 두 가지 디버깅 자세 — 한 줄씩 보기와 통째로 보기

스크립트가 안 돌면 어떻게 디버깅할까요. 두 가지 자세가 있어요. 둘 다 중요해요.

**자세 1 — 한 줄씩 보기 (Step debugging).** `bash -x sysinfo.sh` 라고 실행하세요. -x 옵션이 한 줄 한 줄 실행되는 모습을 보여줘요. 한 줄 실행 → 결과 → 다음 줄 실행. 어디서 멈췄는지, 어느 변수에 어떤 값이 들어갔는지가 한눈에 보여요. 작은 사고는 90%가 -x 한 번에 풀려요. 두 해 동안 가장 자주 쓸 디버깅 도구예요.

**자세 2 — 통째로 보기 (Bird's eye view).** 한 줄씩 보다가도 안 풀리면, 한 발 떨어져서 전체 흐름을 다시 봐요. "이 스크립트가 결국 뭘 하려고 하지?", "내가 의도한 결과는 뭐였지?". 가끔은 한 줄에 매몰되면 큰 그림을 놓쳐요. 종이 한 장 꺼내서 흐름을 손으로 그려 보세요. 1단계 → 2단계 → ... → 5단계. 각 단계의 입력과 출력을 적어 봐요. 어디서 어긋났는지 보여요.

이 두 자세가 두 해 동안 모든 디버깅의 양 날개예요. 자세 1로 안 풀리면 자세 2로, 자세 2로 안 풀리면 자세 1로. 왔다 갔다 하면서 막힘을 풀어요. 한 가지 자세만 고집하면 못 풀어요.

여기에 한 가지 더 비밀. **잠깐 자리를 떠나기.** 30분 막혔으면 일어나서 한 바퀴 돌고 오세요. 진짜로요. 머리가 너무 한 가지에만 갇혀 있으면 옆 답을 못 봐요. 한 바퀴 돌고 와서 다시 보면 "어, 이게 왜 안 보였지?" 하는 답이 보여요. 모든 시니어 개발자가 매일 쓰는 진짜 비밀이에요. 컴퓨터 앞에서 막힐수록 컴퓨터에서 떨어져 보세요. 두 해 동안 백 번 천 번 검증되는 비밀이에요.

그리고 한 가지 더. 막혔을 때 누군가에게 한 번 설명해 보세요. 동료, 가족, 아니면 책상 위의 인형한테라도요. 입으로 한 번 설명하다 보면 본인이 답을 찾는 경우가 진짜 많아요. 이 현상에 이름이 있어요. **러버 덕 디버깅(rubber duck debugging).** 1999년 책 The Pragmatic Programmer에서 처음 소개됐어요. 노란 고무 오리에게 코드를 한 줄씩 설명하면 답이 보인다는 농담 같은 진짜 진짜 효과적인 방법이에요. 두 해 후 책상에 노란 고무 오리 한 마리 두시는 분이 많을 거예요. 농담 아니에요. 진짜로 효과 있어요.

## 18. 한 줄 정리 — 머리에 남길 한 가지

오늘 한 시간을 한 줄로.

> **명령 여러 개를 묶어서 한 도구로 만든 그 순간이 진짜 개발자의 시작이다.**

오늘까지 우리는 명령을 한 줄씩 쳤어요. 5교시에서 처음으로 그 명령들을 묶어서 한 개의 도구를 만들었어요. 이게 큰 변화예요. 한 줄씩 치는 사람은 사용자, 묶어서 도구를 만드는 사람은 개발자예요. 첫 챕터의 5교시에서 여러분은 이미 그 경계를 한 번 넘었어요. 작아 보이지만 진짜 큰 한 걸음이에요. 두 해 동안 우리는 이 도구를 점점 키워서, 두 해 후엔 사이트 한 채를 만드는 큰 도구 모음으로 자라게 할 거예요. 첫 도구가 sysinfo.sh예요. 평생 기억하실 작품이에요.

한 줄 더. 두 해 후 여러분이 본인 회사에서 어떤 자리에 가 있을지 모르겠어요. 백엔드일 수도, 프론트일 수도, 데브옵스일 수도, AI 엔지니어일 수도 있어요. 어느 자리든 매일 한 번은 셸 명령을 묶어서 새 도구를 만드는 순간이 와요. 그 순간이 올 때마다 오늘의 sysinfo.sh가 살짝 떠오르길 바라요. "아, 첫 챕터에서 50줄짜리 자기소개서를 만든 적 있었지." 그 작은 기억이 본인의 손을 한 번 더 움직이게 만들어요. 두 해 코스의 시작이 거기에 있어요.

## 19. 마무리 — 다음 시간 예고

자, 다섯 번째 시간 끝. 오늘 우리는 진짜 코드 한 개를 같이 짰어요. 50줄짜리 작은 도구지만, 우리 맥의 자기소개서를 한 번에 출력하는 도구예요. 셸 스크립트의 기본 문법(셔뱅, set 안전 장치, 변수, 명령 치환, 산술 연산, awk·sed·tail 가공, ANSI 색깔)을 한 번에 다 만나봤어요. 두 해 동안 이 패턴을 100번은 더 봐요. 한 번 손에 익혀두시면 두 해의 거의 모든 자동화 도구가 친근해져요.

그리고 한 가지 약속. 오늘 만든 sysinfo.sh를 본인의 홈 디렉토리에 한 번 복사해 두세요. `cp finish/sysinfo.sh ~/sysinfo.sh`. 그리고 zshrc에 alias 한 줄 추가. `alias myinfo='bash ~/sysinfo.sh'`. 새 터미널 열고 `myinfo` 한 줄만 치면 자기소개서가 떠요. 매주 한 번씩 myinfo 쳐 보세요. 디스크 사용률이 어떻게 변하는지, 가동 시간이 얼마나 늘어나는지 자연스럽게 알게 돼요. 컴퓨터의 안부를 자주 묻는 사람이 되시면 큰 사고를 90% 막아요.

다음 여섯 번째 시간에는, 컴퓨터를 관리하는 법을 봐요. 메모리가 부족할 때, 디스크가 꽉 찼을 때, 인터넷이 안 될 때. 응급 처치 가이드예요. 5교시가 평소의 도구를 만드는 시간이었다면, 6교시는 사고 났을 때의 대응을 배우는 시간이에요. 둘이 짝이에요. 평소의 관찰이 사고를 줄이고, 사고가 났을 땐 빠른 응급처치가 피해를 줄여요. 둘 다 우리한테 있어야 해요. 십 분 쉬고 와요. 물 한 잔, 어깨 한 번 돌리시고요. 잘하고 계세요. 진짜로요.

그리고 한 가지 부탁. 쉬는 시간에 본인의 sysinfo.sh를 한 번 더 돌려 보세요. 한 번 더, 또 한 번 더. 같은 결과가 떠도 좋아요. 본인 손가락이 명령어를 외우게 만드는 시간이에요. 두 해 후 자다 일어나서도 손이 자동으로 ./sysinfo.sh를 칠 수 있게요. 머리가 아니라 손이 외워야 진짜 본인 도구가 돼요. 손은 머리보다 정직해요. 한 번 외워두면 사고 났을 때도 자동으로 움직여요. 진짜 큰 자산이에요. 손가락의 기억은 시험에서도, 면접에서도, 새벽 장애 대응에서도 흔들리지 않아요. 이따 6교시에 다시 만나요. 수고하셨어요.

## 👨‍💻 개발자 노트

> - 본 스크립트는 의도적으로 macOS 전용. 크로스 플랫폼 분기(`uname -s` case문)는 Ch 006 터미널 챕터에서 다루며, 본 챕터에서는 "환경 일치를 강조하고 분기는 미루는" 교육 설계.
> - `set -e`만 사용하고 `set -u`(미정의 변수 에러)는 의도적으로 미사용 — 첫 챕터의 학습자가 변수 오타로 막히는 사고 방지. Ch 006에서 `set -euo pipefail` 패턴으로 졸업.
> - `awk '{print $5}'`로 디스크 사용률 칸을 가져오는 건 GNU/BSD coreutils 출력 칸 순서 의존. macOS의 BSD df는 `Capacity`가 5번째 칸. Linux GNU df는 `Use%`가 5번째 칸. 다행히 둘 다 5번 — 우연한 호환.
> - ANSI 색상 코드 노출은 Ch 050 Flask 로깅, Ch 069 OWASP 로깅 마스킹, Ch 116 CloudWatch Logs 색상 표시에서 다시 회수.
> - `start/`와 `finish/`의 분리 패턴은 본 코스 전 챕터의 표준. 학습자가 빈 껍데기에서 시작해 같은 결과에 도달하는 "닫힌 실습 단위"로 설계됩니다(`docs/STYLE-GUIDE.md` §8).
