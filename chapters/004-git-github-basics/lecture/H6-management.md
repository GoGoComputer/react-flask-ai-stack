# Ch004 · H6 — 저장소 운영: Issue·PR·Project·Discussions·branch protection·CODEOWNERS

> **이 H에서 얻을 것**
> - GitHub의 6개 운영 도구(Issue·PR·Project·Discussions·Actions·Pages)를 자경단의 일주일에 어떻게 배치할지.
> - **PR 리뷰 문화** — 코멘트의 5가지 톤, "이건 nit이에요" 같은 작은 단어가 만드는 큰 차이.

> - 자경단 1주일 운영 시뮬레이션 — Issue 한 장이 PR로, PR이 deploy로.

## 회수: H5의 손가락이 H6의 운영으로

지난 H5에서 본인은 빈 폴더에서 `git init`부터 첫 push까지를 한 호흡으로 두드렸어요. conflict도 한 번 만났고, `reset --hard`로 commit을 날린 후 `reflog`로 살려도 봤어요. 손가락에 git이 박힌 시간. 이제 본인은 commit을 만들 줄 알고, 가지를 따고 합칠 줄 알고, 인터넷 너머 GitHub로 보낼 줄 알아요. **하지만 혼자 두드리는 git은 git의 50%**예요. 나머지 50%는 **여러 사람이 모인 운영**입니다.

GitHub은 사실 git만 호스팅하는 사이트가 아니에요. **git + 협업 도구 한 무리**가 묶인 플랫폼이에요. Issue로 일을 적고, PR로 코드 변경을 제안하고, Project로 일정을 보고, Discussions로 이야기하고, Actions로 자동화하고, Pages로 배포하고. **6개 도구가 본인의 일주일을 만들어요.** 본인이 이 도구들 없이 git만 쓰면 회사 저장소의 5%만 쓰는 거예요.

이번 H6는 본인이 자경단 저장소를 진짜 회사 저장소처럼 굴리는 시간이에요. **branch protection rule** 한 줄로 main에 직접 push를 막고, **CODEOWNERS** 한 장으로 특정 폴더의 자동 리뷰어를 지정하고, **Issue 템플릿** 한 장으로 신입이 보고를 잘하게 만들고, **PR 템플릿** 한 장으로 본인의 1차 자기 리뷰를 자동화. 이 네 장의 파일이 본인의 작은 자경단을 회사 수준의 운영으로 옮겨요. 외부에 PR을 받기 시작했을 때 본인이 매번 반복하던 코멘트가 템플릿 한 장으로 사라져요.

또 하나 — **PR 리뷰 문화**. 5년 차의 가장 중요한 자산은 본인이 동료의 PR에 다는 코멘트의 톤이에요. "이거 잘못됐어요"라는 한 줄과 "여기 한 가지 제안 있어요"라는 한 줄은 같은 의미지만 동료의 마음에 다르게 박혀요. 작은 단어가 큰 문화를 만들어요. 본인이 신입일 때부터 이 톤을 익히면, 본인이 5년 차에 팀 리더가 될 때 그 톤이 본인의 평판이 됩니다.

## 1. GitHub의 6개 도구를 한 표로

H4의 무리 짓기와 비슷해요. 6개 도구를 한 표에 누이고, 각각이 자경단의 일주일에 어떤 역할을 하는지 적어요.

| 도구 | 한 줄 정의 | 자경단의 자리 | 빈도 |
|------|-----------|--------------|------|
| **Issue** | 할 일·버그·아이디어를 한 장에 적기 | "지하철역 길고양이 신고가 들어왔다" | 매일 |
| **Pull Request** | 코드 변경을 제안하고 리뷰받기 | "신고 처리 페이지 한 장 만들었습니다, 봐 주세요" | 매일 |
| **Project** | 여러 Issue·PR을 보드로 시각화 | "이번 주 길고양이 5건의 진행도" | 주 1회 |
| **Discussions** | 자유 게시판·Q&A | "야간 신고 처리 어떻게 하는 게 좋을까요?" | 주 1~2회 |
| **Actions** | push·PR 시 자동 워크플로우 실행 | "PR마다 테스트 자동, main merge 시 자동 배포" | 매번(자동) |
| **Pages** | 저장소를 정적 웹사이트로 호스팅 | "자경단 README와 가이드 페이지" | 한 번 셋업 |

6개 도구 + 23개 git 명령어 + 7줄 config = 본인의 일상 운영 도구함이에요. 이 도구함을 알면 본인은 GitHub의 99%를 써요. 모르면 5%만 써요. **차이가 95%**입니다.

여기서 **회사 vs 자경단의 차이** 한 가지 — 회사는 보통 GitHub Enterprise를 쓰거나 GitLab·Bitbucket을 써요. 도구 이름은 살짝 달라도 6개 무리는 똑같아요. Issue는 어디나 Issue, PR은 GitLab에선 MR(Merge Request), Project는 GitLab·Bitbucket에선 Boards. **본인이 GitHub에서 익힌 6개 무리가 다른 플랫폼으로 95% 이전**돼요. 안전하게 한 곳에서 깊게 배우면 되는 이유.

## 2. Issue — 일의 한 장 한 장

자경단 사이트에 길고양이 신고가 들어왔어요. 본인이 그걸 처리하기 전에 가장 먼저 해야 할 일은 **Issue 한 장을 여는 거**예요.

```
제목: [신고] 강남구 역삼동 길고양이 부상 신고 (2026-04-27)
본문:
- 신고자: 익명 (전화번호: 010-XXXX-1234, 동의 받음)
- 위치: 역삼동 어딘가 골목
- 상태: 다리를 다친 듯, 절뚝거림
- 사진: (첨부)
- 긴급도: 중간 (즉시 위험은 아니지만 24시간 내 확인 필요)

## 처리 단계
- [ ] 1. 현장 확인자 배정
- [ ] 2. 동물병원 연락
- [ ] 3. 구조 일정 잡기
- [ ] 4. 처리 결과 기록

라벨: 신고, 강남구, 부상, 긴급도-중간
담당자: @volunteer-1
마일스톤: 4월 4주차
```

이 한 장이 자경단의 한 주를 구성하는 단위예요. **체크박스로 진행도 표시**(`- [ ]`을 `- [x]`로 바꾸면 진척률 자동 계산), **라벨**로 분류, **담당자(assignees)** 로 책임자, **마일스톤**으로 시기. 본인이 매주 화요일에 "이번 주 마일스톤" Issue 목록을 한 번 보면 자경단의 한 주가 한눈에 보여요.

**Issue 템플릿 한 장**을 만들어 두면 신입이 보고를 잘해요. `.github/ISSUE_TEMPLATE/cat-report.md`에 위 형식을 미리 박아 두면, 다른 사람이 New Issue 누를 때 자동으로 그 양식이 떠요. 신입의 보고가 첫날부터 표준 양식에 맞아요. **반복되는 양식을 자동화**하는 게 운영의 핵심.

GitHub은 Issue 템플릿을 두 가지 형식으로 받아요 — Markdown(`.md`)과 YAML(`.yml`, "Issue Forms"). YAML 쪽이 더 강력해요 — 드롭다운·체크박스·필수 입력 필드를 만들 수 있어요. 본인이 신고 양식을 진지하게 받기 시작하면 YAML로 옮기는 게 좋아요. 다만 첫 시도엔 Markdown 한 장이면 충분.

**또 하나 — 라벨 시스템**. 본인이 Issue를 100개 받으면 라벨 없이는 못 찾아요. 자경단의 표준 라벨 — **유형**(`신고`·`버그`·`기능`·`문서`·`질문`), **지역**(`강남구`·`서초구`·`송파구` 등), **긴급도**(`긴급도-높음`·`긴급도-중간`·`긴급도-낮음`), **상태**(`상태-대기`·`상태-진행중`·`상태-완료`·`상태-보류`), **종류**(`부상`·`구조`·`임시보호`·`입양`). 5개 무리. GitHub에서 색깔까지 칠하면 한눈에 분류돼요. **첫날에 라벨 한 번 정리**해 두면 평생 굴러갑니다.
**라벨 색깔 표준** — 빨강(긴급), 주황(중요), 노랑(주의), 초록(완료·OK), 파랑(정보·문서), 보라(아이디어·기능), 회색(중립·메타). 본인이 보드 위에서 한눈에 색깔만 봐도 위험도가 잡혀요. 자경단의 야간 신고는 빨강, 일반 신고는 노랑, 아이디어는 보라. **시각 신호가 인간의 뇌에 가장 빠른 정보**예요. H4의 위험도 신호등이 명령어용이었다면, 라벨 색깔은 Issue용 신호등. 같은 원리.

**또 하나의 강력한 패턴 — 마일스톤 + 프로젝트의 이중 묶기**. 마일스톤은 "언제까지"(시간 축), 프로젝트는 "어디까지"(상태 축). 한 Issue가 마일스톤 "4월 4주차"에 들어가면서 프로젝트의 "진행 중" 칸에도 동시에 들어가요. 두 축으로 한 Issue를 잡으면 본인의 한 주가 두 방향에서 보여요. 시간으로 한 번, 상태로 한 번. **두 번 본 일은 안 잊혀요.**

## 3. Pull Request — 코드 변경을 제안하기

PR은 본인이 두드린 commit을 동료가 보고 의견 주는 자리예요. 단순히 머지 버튼을 누르는 곳이 아니라 **코드의 한 페이지가 동료와의 대화로 변하는 자리**. 좋은 PR은 5분 만에 머지되고, 나쁜 PR은 일주일을 끌어요. 차이는 **PR 본문**과 **PR 크기**.

**좋은 PR의 5가지 특징** —

1. **작아요(작은 변경).** 50줄 미만이 이상적. 500줄 PR은 동료가 1시간 동안 봐도 다 못 봐요.
2. **하나의 일.** "버그 고침 + 리팩터링 + 새 기능"이 한 PR에 들어 있으면 안 돼요. 한 PR은 한 일.
3. **본문이 명확.** "왜 이 변경이 필요했나"·"무엇을 바꿨나"·"어떻게 테스트했나" 세 줄.
4. **테스트 포함.** 새 기능엔 테스트 한 줄, 버그 고침엔 회귀 테스트 한 줄.
5. **Issue와 연결.** PR 본문에 `Fixes #123` 한 줄 박으면 머지 시 그 Issue가 자동 닫혀요.

**PR 템플릿 한 장**(`.github/pull_request_template.md`)에 위 형식을 박아 두면 본인이 PR 만들 때마다 자동으로 그 양식이 떠요. 본인이 매번 적어야 할 5줄을 잊지 않게 만드는 손가락 안전장치. 회사 저장소엔 거의 다 이 템플릿이 있어요.

**PR 본문 표준 한 장** —

```markdown
## 무엇을
- 자경단 신고 페이지에 사진 업로드 기능 추가

## 왜
- Issue #42 — 신고자가 사진을 같이 보내고 싶다는 요청

## 어떻게 테스트
- [x] local에서 사진 업로드 후 미리보기 확인
- [x] 5MB 초과 시 에러 메시지 확인
- [x] jpg·png·webp 3가지 포맷 통과 확인
- [ ] (CI에서 자동 테스트 실행 중)

## 스크린샷
(업로드 직후 미리보기 화면)

Fixes #42
```

**한 장이지만 동료의 시간을 한 시간 절약**해요. 동료가 본인의 PR을 보자마자 "왜"를 알고, 어디를 봐야 하는지 알고, 테스트까지 확인할 수 있어요. **본인의 PR 본문이 본인의 평판**이에요.

**한 가지 더 — Conventional Commits 규칙**. PR 제목과 commit 메시지를 표준화하면 자동화가 쉬워져요. `feat: 사진 업로드 추가`(새 기능), `fix: 5MB 초과 시 에러 메시지`(버그 고침), `docs: README에 설치 가이드 추가`(문서), `chore: 의존성 업데이트`(잡일), `refactor: 신고 처리 함수 분리`(리팩터링), `test: 사진 업로드 회귀 테스트`(테스트), `style: 들여쓰기 정리`(코드 스타일). 7가지 prefix. 자동 changelog 생성·SemVer 버전 자동 결정·릴리스 노트 생성에 다 활용. 본인의 자경단 첫날부터 이 규칙으로 박아 두면 1년 후 v1.0.0 릴리스 노트가 자동으로 만들어져요. **표준이 자동화의 토양**이에요.

**Draft PR을 적극 활용** — "아직 완성 안 됐지만 일찍 보여드리고 싶은" PR은 Draft로 만들어요. 머지 버튼이 비활성화돼서 실수로 머지될 일이 없어요. 동료에게 일찍 피드백을 받아서 끝까지 다 만든 후 큰 방향이 틀렸다고 처음부터 다시 만드는 사고를 막아요. **"early feedback"이 5년 차의 비밀**이에요. 본인이 신입일 땐 더더욱 — 첫 30%만 만들고 Draft PR을 열어 "방향 맞나요?"를 동료에게 한 줄 묻기. 30분 후의 답이 본인의 1주일을 절약해요. Ready for review로 바뀌는 그 순간이 본인의 PR이 진짜로 시작되는 시점.

**또 하나 — 작은 PR의 비결: stack(스택) PR**. 한 큰 변경을 5개 작은 PR로 쪼갤 때, 첫 PR이 머지되기 전에 두 번째 PR을 첫 PR 위에 만들 수 있어요. 첫 PR의 가지에서 다시 가지를 따고, 두 번째 가지를 그 위에 쌓아요. PR을 만들 때 base를 main이 아니라 첫 PR의 가지로 지정. **스택을 쌓듯 PR을 쌓는 패턴**. Meta(Facebook)·Google이 표준으로 쓰는 방식. graphite·spr 같은 도구도 있지만 손으로도 가능. 자경단 단계에선 이 패턴을 알기만 해도 됨, 진짜 협업이 깊어지면 도구 도입.

## 4. PR 리뷰 — 코멘트의 5가지 톤

리뷰 코멘트는 본인의 5년 차 평판을 만드는 가장 비싼 텍스트예요. 같은 내용이라도 톤에 따라 동료의 마음이 달라져요. 5가지 톤을 알아 두세요.

**톤 1: 칭찬(👏)** — "여기 변수명 좋네요, 의미가 한눈에 들어와요." 좋은 PR엔 코멘트의 30%가 칭찬이어야 합니다. 안 좋은 부분만 짚으면 동료의 의욕이 꺾여요. 칭찬은 거짓말이 아니라 **본인이 진짜 좋다고 느낀 부분을 짚어 주는 것**.

**톤 2: 질문(❓)** — "여기 왜 try-catch를 빼셨나요? 의도가 궁금해요." 본인이 모르는 게 있을 때 비난 대신 질문. 동료가 답하면서 의도를 정리해요. 본인의 무지가 동료의 정리를 만들어요. 신입에게 가장 좋은 톤.

**톤 3: 제안(💡)** — "이 부분, 만약 이렇게 하면 어떨까요? `if (a && b)` 대신 `if (a) return; if (b) return;` 같은 early return." 명령이 아니라 제안. 동료가 본인의 의견을 받아들일지 결정해요. **결정권은 PR 작성자에게**.

**톤 4: 요청(🔧)** — "이 부분은 보안 취약점입니다. main에 들어가기 전에 꼭 고쳐야 해요." 객관적으로 틀린 부분, 안전 문제는 명확한 요청. 다만 **이유를 함께 적기** — "왜"가 빠지면 갑질로 느껴져요.

**톤 5: nit(🪶)** — "nit: 여기 공백 한 칸 더 들어갔어요. 안 고쳐도 머지하셔도 돼요." nit = nitpick(작은 트집). 사소하지만 본인 눈에 거슬리는 것. **"안 고쳐도 됨"을 명확히 표시**해서 동료가 부담 없이 결정. nit이 코멘트의 30~40%를 차지하는 PR이 건강한 PR.

**한 가지 더 — `LGTM`**(Looks Good To Me, "보기 좋아요"). 5년 차의 표준 승인 한 줄. 코멘트 0개에 LGTM이면 본인 PR이 완벽했다는 뜻. **LGTM 받은 날엔 본인이 동료의 신뢰를 한 칸 얻은 거**예요. 모이면 본인의 평판.

본인이 신입일 때 동료의 PR에 코멘트 다는 게 무서울 수 있어요. **그래도 다세요.** 신입의 한 코멘트가 5년 차의 한 코멘트보다 의미 있는 경우가 많아요 — 신입은 코드를 처음 보니까 "이 함수 이름이 헷갈려요" 같은 객관적 평가를 줘요. 5년 차는 코드에 익숙해서 그런 게 안 보여요. **본인의 신입성은 자산**이에요. 무서워하지 말고 코멘트.

**한 가지 더 — 코드 리뷰의 두 단계: 큰 그림 + 디테일**. 본인이 PR을 처음 열면 코드 한 줄 한 줄에 빠지지 마세요. 먼저 PR 본문 + 변경된 파일 목록 + 큰 흐름을 5분 봐요 — "이 PR의 의도가 맞나? 구조가 합리적인가? 이 위치에 이 변경이 맞나?" 큰 그림에서 문제가 있으면 디테일 코멘트는 무의미해요. 큰 그림 OK 후에야 줄 단위 디테일. **순서를 지키면 본인의 한 시간이 동료의 하루를 절약**해요. 회사의 시니어 리뷰어가 다 이 순서로 봐요. 본인이 신입일 때부터 이 순서를 흉내내면 본인의 리뷰가 5년 차보다 빠르게 신뢰를 받아요.

**또 하나 — "resolve conversation" 버튼의 의미**. 코멘트가 해결됐을 때 누가 이 버튼을 누르는가? 회사마다 컨벤션이 달라요. 어떤 회사는 PR 작성자가 누르고, 어떤 회사는 코멘트 단 사람이 누르고. **본인 회사의 컨벤션을 첫 PR 때 한 번 물어보세요.** 일관성이 협업의 마찰을 줄여요. 자경단에선 보통 작성자가 "고쳤습니다"라고 답글 달면 코멘트 단 사람이 확인 후 resolve. 양쪽 합의가 한 버튼에 담겨요.

**마지막으로 — 머지 버튼의 3가지 옵션 (Merge commit / Squash and merge / Rebase and merge)**. 회사마다 다르지만 자경단의 권장은 **Squash and merge** — PR의 모든 commit을 한 commit으로 합쳐 main에 박아요. main의 history가 PR 단위로 깔끔해요. 1 PR = 1 main commit. "이 기능이 어느 PR에서 들어왔나"를 한 줄로 추적. **회사의 80%가 squash merge**를 표준으로 써요. Rebase merge는 PR의 commit이 다 의미 있을 때(예: 큰 리팩터링), Merge commit은 거의 안 씀(history가 지저분해짐).

## 5. branch protection rule — main에 직접 push 막기

본인이 자경단 저장소의 owner라면 첫 운영 셋업으로 **main 가지 보호 규칙**을 박아 둬요. github.com/<you>/<repo>/settings/branches에서 "Add rule":

- **Branch name pattern**: `main`
- ☑ Require a pull request before merging
  - ☑ Require approvals: 1 (또는 팀 크기에 따라 2)
  - ☑ Dismiss stale pull request approvals when new commits are pushed
  - ☑ Require review from Code Owners(아래 CODEOWNERS와 연결)
- ☑ Require status checks to pass before merging
  - ☑ Require branches to be up to date before merging
  - 체크할 status check: `ci` (Actions에서 정의)
- ☑ Require linear history (rebase·squash만 허용, merge commit 금지)
- ☑ Include administrators (본인도 규칙 적용)
- ☑ Restrict pushes that create matching refs (force-push 금지)

이 한 페이지를 셋업하면 — 본인조차 main에 직접 push 못 해요. 모든 변경은 PR을 통해야 해요. **인간의 실수를 시스템이 막는** 거예요. 회사의 main 가지는 모두 이렇게 보호돼 있어요. 본인의 자경단도 첫날부터 회사처럼.

**"한 명짜리 자경단인데 PR 거치면 너무 번거로워요."** 처음엔 그렇게 느껴요. 그래도 본인이 본인 PR을 한 번 self-review하는 습관이 5년 차로 가는 가장 빠른 길. **혼자라도 PR**. 본인이 본인을 가장 엄격한 리뷰어로 만들면 동료가 가장 신뢰하는 동료가 돼요. 자경단이 한 명에서 다섯 명으로 커지는 날, 본인이 셋업한 protection 규칙이 그대로 굴러가요. 첫날의 5분 셋업이 5년 후의 안전.
**또 하나 — 단세 을이 너무 쓸다면 self-review만이라도 제대로**. PR 만들고 바로 머지하지 말고, GitHub UI에서 "Files changed" 탭 열어 한 줄 한 줄 다시 보기. 본인 스스로에게 "이 줄을 동료가 보면 이해할까?" 묻기. 5분이면 끝나요. 그 5분이 main에 들어간 후 버그를 입탐한 둘째날 아침 30분 디버깅을 대신해 줘요. **self-review도 코드 리뷰의 한 형태**예요. 주니어 면접에서 "혼자 일하면 코드 리뷰를 어떻게 하나요?" 질문의 답이 이 한 줄. "동료의 눈으로 다시 보는 self-review를 해요."

## 6. CODEOWNERS — 자동 리뷰어 지정

`.github/CODEOWNERS` 파일에 한 장 적으면 — 어떤 폴더가 바뀐 PR엔 자동으로 그 폴더의 owner가 리뷰어로 지정돼요.

```
# 자경단 저장소 CODEOWNERS

# 전체 기본 리뷰어
*                       @gogocomputer

# 백엔드 폴더
/backend/               @backend-team @gogocomputer

# 프런트엔드 폴더
/frontend/              @frontend-team

# 인프라 (DB·CI·k8s) — 더 신중한 리뷰
/infra/                 @gogocomputer @senior-devops
/.github/workflows/     @senior-devops

# 보안 관련 — 무조건 두 명
/auth/                  @gogocomputer @security-team
SECURITY.md             @security-team
```

이 한 장의 효과 — PR을 만들면 GitHub이 자동으로 리뷰어를 적어 줘요. 본인이 매번 "누구에게 리뷰 받아야 하지?"를 고민 안 해도 돼요. 회사에선 이게 표준. 신입이 첫날 PR 만들면 자동으로 시니어가 리뷰어로 박혀요. **사람이 아니라 폴더가 리뷰어를 결정**해요.

**조심할 점** — CODEOWNERS는 GitHub 사용자명·팀명을 직접 적어요. 사람이 회사를 떠나면 그 사람의 username을 빼야 해요. 안 그러면 그 PR이 영원히 리뷰 못 받아 머지가 안 돼요. 분기마다 한 번 CODEOWNERS를 점검하는 표준 운영. 본인의 자경단이 커지면 한 분기에 한 번 30분 투자.

**또 한 가지 함정 — 한 줄에 소유자가 여러 명이면 OR이 아니라 AND**. `/auth/  @gogocomputer @security-team`은 둘 다가 리뷰해야 머지 가능이 아니라 둘 중 한 명이면 충분(GitHub 기본). 둘 다가 승인해야 하는 정책은 branch protection의 "Required approvals: 2"로 별도 설정. **세부사항**이지만 한 번 읽어봐야 사고 안 나요.

**CODEOWNERS 파일의 위치 — 세 곳 중 하나**. `.github/CODEOWNERS`(권장), `CODEOWNERS`(루트), `docs/CODEOWNERS`. 보통은 `.github/CODEOWNERS`에 둡니다 — 다른 .github/ 파일들과 함께 모아두는 게 깔끔해요. 문자 GitHub 공식 문서가 다 `.github/CODEOWNERS`로 그려져 있어요.

**매칭** — .gitignore와 비슷한 글로브 패턴. `*.js`는 모든 .js 파일, `/auth/`는 루트의 auth 폴더, `**/test/*`는 아무 깊이의 test 폴더 내 모든 파일. 본인이 잘 모르면 테스트 저장소 하나 만들고 PR 한 번 올려서 매칭이 되는지 확인. CODEOWNERS는 높은 원리만큼 디테일에서도 함정이 많아요.

**한 가지 더** — CODEOWNERS와 branch protection의 "Require review from Code Owners" 체크박스를 같이 켜야 진짜 강제력이 생겨요. 한쪽만 있으면 권고 수준. 둘이 합쳐져야 시스템이 동료의 승인 없이 못 머지하게 막아요. **2단계 안전장치**.

## 7. Project — 일주일을 보드로 보기

GitHub Projects는 Issue·PR을 보드(Trello·Notion 같은)로 시각화해 줘요. 자경단의 한 주를 4칸 보드로 만들어 봐요.

```
┌──────────┬──────────────┬──────────┬─────────┐
│  대기    │   진행 중    │  리뷰    │  완료   │
├──────────┼──────────────┼──────────┼─────────┤
│ #102     │ #99 (강남)   │ #95 PR   │ #91     │
│ #103     │ #100 (송파)  │          │ #92     │
│ #104     │              │          │ #93     │
└──────────┴──────────────┴──────────┴─────────┘
```

월요일 아침 — 본인이 보드를 한 번 보면 이번 주의 5건이 어디 있는지 한눈에 보여요. 길고양이 신고 #99·#100이 진행 중, #95는 리뷰 단계, #91·#92·#93은 지난주에 완료. **한 페이지로 한 주가 보여요.**

**자동화 한 줄** — Project Settings에서 "Auto-add to project: Status: Open, Type: Issue or PR"을 켜면 새 Issue·PR이 자동으로 "대기" 칸에 들어가요. 본인이 손으로 끌어다 놓을 필요 없음. **사람이 놓치는 걸 시스템이 잡아요.**

**Status 자동 이동** — 워크플로우 한 줄로 "PR이 만들어지면 진행 중", "리뷰 요청되면 리뷰", "머지되면 완료" 자동 이동. Ch014에서 본격적으로 다뤄요. 지금은 손으로 끌어다 놓는 것도 충분.

**Project의 두 종류** — 옛 Projects(repo-scoped)와 새 Projects(beta·organization-scoped). 새 쪽이 더 강력하고 여러 저장소를 한 보드에 묶을 수 있어요. 자경단이 저장소 5개로 커지면 새 Projects로 통합 보드. 첫날엔 옛 Projects 한 장으로 시작해도 충분.

## 8. Discussions — Q&A·아이디어·자유 게시판

코드가 아닌 이야기는 Discussions에. github.com/<you>/<repo>/discussions에서 Categories를 만들어요 — `Q&A`(질문·답변), `Ideas`(아이디어), `Show and tell`(자랑·후기), `Polls`(투표), `Announcements`(공지). 자경단의 표준 5칸.

자경단 Discussions의 일주일 예 — Q&A에 "야간 신고는 어떻게 처리하나요?", Ideas에 "구조한 길고양이 입양 페이지 따로 만들어 볼까요?", Show and tell에 "이번 주 구조한 까미, 입양 갔어요 🐾", Announcements에 "다음 주 토요일 자경단 정기 모임".

**Issue vs Discussions의 구분** — 코드·작업이면 Issue, 이야기·질문이면 Discussions. 본인이 헷갈리면 일단 Discussions, 작업으로 변환할 가치가 있으면 GitHub에 "Convert to issue" 버튼이 있어요. 한 클릭으로 변환. **양쪽이 갈라져 있는 게 깨끗**해요. 회사 저장소도 그래요.

## 9. README·CONTRIBUTING·CODE_OF_CONDUCT·SECURITY·LICENSE — 5장의 문서

저장소를 외부에 공개할 때 5장의 표준 문서가 있어요. 본인이 자경단을 외부 봉사자에게 열기 시작하면 이 5장을 한 번에 깔아 두세요.

**README.md** — 첫인상. "이 프로젝트가 무엇인가, 왜 만들었나, 어떻게 쓰나" 세 단락. 5분 안에 외부 사람이 본인 프로젝트를 이해해야 함. 가장 중요한 한 장.

**CONTRIBUTING.md** — 외부 사람이 PR을 만들 때 따를 가이드. "버그 보고는 Issue로, 코드 변경은 PR로, PR엔 테스트 포함". 한 페이지로 충분.

**CODE_OF_CONDUCT.md** — 행동 강령. Contributor Covenant 표준 양식 그대로 쓰면 돼요. github.com이 자동 템플릿 제공. **외부 봉사자가 모인 자경단은 이 한 장이 안전망**.

**SECURITY.md** — 보안 취약점 신고 채널. "보안 문제는 issue로 올리지 말고 security@example.com으로 메일." 외부 보안 연구자가 본인 사이트의 취약점을 찾으면 그들이 가장 먼저 보는 페이지.

**LICENSE** — 라이선스. 자경단 같은 오픈소스 비영리 프로젝트는 보통 **MIT**(가장 간단·자유) 또는 **Apache 2.0**(특허 조항 포함) 또는 **AGPL-3.0**(서버 코드 수정 시 공개 의무). 본인이 영원히 자유롭게 두고 싶으면 MIT, 기업이 마음대로 가져가는 게 싫으면 AGPL. 자경단은 보통 MIT.

**5장이 다 있는 저장소는 외부 봉사자가 첫날부터 안전감을 느껴요.** 본인이 100명짜리 회사에 입사했을 때 회사가 입사 매뉴얼·복지 문서·비상 연락망을 안 보여 주면 불안하잖아요. 같은 원리. **문서 5장이 본인의 환영사**예요.

**추가로 두면 좋은 두 장** — **CHANGELOG.md**(어느 버전에 무엇이 들어갔나 — SemVer와 짝)·**ROADMAP.md**(앞으로 어디로 갈 건가 — 외부 봉사자·기여자에게 필요). 합쳐서 7장. 자경단이 100명의 외부 자원봉사자를 받기 시작한다면 7장을 표준으로. 이 7장을 고정 파일로 붙이는 게 **자경단을 "취미 익명 모임"에서 "조직"으로 올리는 계단**이에요. 외부의 눈으로 볼 때 자경단이 "이 적이죠"과 "이 조직이죠" 사이의 차이가 크죠.

## 10. Actions·Pages — 자동화와 호스팅

**GitHub Actions** — push·PR마다 자동으로 워크플로우 실행. `.github/workflows/ci.yml` 한 장이면 시작. push되면 테스트 자동 실행, PR이면 lint·typecheck 자동 실행, main에 머지되면 배포 자동 실행. 본 코스 **Ch014에서 본격적으로** 다뤄요. H6엔 한 줄만 — Actions가 본인의 24시간 자동 동료라는 사실. 본인이 자는 동안에도 PR이 자동 검사돼요. **조금 더 구체적으로** — GitHub Actions는 무료 티어 2,000분/월(public repo는 무제한)을 주고, marketplace에 수만 개의 올-롤-업 더했기 액션이 있어요. checkout, setup-node, setup-python, upload-artifact, codecov, slack-notify, dependabot, semantic-release 완 행주. 본인이 필요한 게 있으면 marketplace에서 먼저 찾고, 없으면 만드는 순서. **키르고 있는 생태계**예요.

**GitHub Pages** — 저장소를 정적 웹사이트로 호스팅. github.io 도메인. 자경단의 README와 가이드 페이지를 정적 사이트로 띄울 수 있어요. Settings → Pages에서 한 번 셋업이면 끝. **무료**예요. 본인이 사이드 프로젝트의 문서·랜딩 페이지를 띄울 때 가장 빠른 길. Vercel·Netlify와 비교하면 기능은 적지만 셋업이 1분. **Jekyll·MkDocs·Docusaurus·Astro 같은 정적 사이트 생성기**를 GitHub Actions와 연결하면 본인이 README와 docs/ 폴더에 markdown만 올려도 예쁜 문서 사이트가 자동 빌드돼요. 자경단의 사용 가이드·자원봉사자 매뉴얼·FAQ를 markdown으로만 적으면 사이트 한 장이 만들어지는 거예요. 본인이 첫 1년엔 README 한 장으로 충분, 콘텐츠가 늘어나면 docs/ 폴더 + Jekyll 한 장으로 진화. **단계적으로**. 한 번에 다 깔면 유지보수가 무거워져요.

이 두 도구는 본 H6의 마지막 절. Ch014·Ch062에서 깊게 다룰 예정이라 여기선 자리만 잡아요.

## 11. 자경단 1주일 운영 시뮬레이션

월~금까지 본인의 일주일을 시뮬레이션해 봐요.

**월요일 09:00** — Project 보드 열기. 지난 주 완료 5건, 이번 주 대기 3건. 진행 중 칸에 끌어다 놓고 본인 자신을 assignee로 지정. **월요일의 첫 5분이 일주일의 카오스 80%를 막아요.** 09:30 — Issue #99(역삼동 길고양이) 본문 다시 읽고 처리 단계 체크리스트 한 칸 ✅. `git switch -c feat/issue-99`로 가지 따기. 쿼 한 잔 안 마시고도 5분의 계획이 있을 수 있는 거예요.

**화요일 10:00** — Issue #99 처리 페이지 코드 작성. 5개 commit. push. PR 만들기 — 본문에 `Fixes #99` 한 줄. CODEOWNERS 덕에 자동으로 동료 한 명 리뷰어 지정. **PR 템플릿이 자동으로 떠서 5줄을 채움**. 동료에게 슬랙 한 줄 — "리뷰 부탁드립니다 🙇". 30분 후 동료가 "일점 텍스트 이상해요" 라고 코멘트 한 개. 더 국한동 한 곳 아니었고, 본인이 "아 잠시만요, 판괴부 설정이 잊혀어요" 대식 한 줄 commit 추가. 작은 사이클. 동료가 세세한 곳을 잡아 주어서 더 큰 사고가 막혀요. **코드 리뷰가 쓸모 있는 올환의 수간**이에요.

**수요일 14:00** — 동료가 PR에 코멘트 3개. 칭찬 1개, 제안 1개, nit 1개. 본인이 제안을 받아들여 한 commit 추가, nit는 "안 고치고 가도 될까요?"라고 답글. 동료 LGTM. 본인이 "Squash and merge" 버튼 눌름. main에 들어가고 GitHub Actions가 자동 배포. **Issue #99 자동 닫힘**(Fixes 덕). Project 보드에서 Issue가 자동으로 "완료"로 이동. 본인은 손가락 한 동작으로 4개 도구가 동시 동작하는 마법을 봤어요.

**목요일** — 새 Issue 두 장이 들어옷. Discussions에 신입 봉사자가 "처음인데 어떻게 시작하나요?" 질문. 본인이 답변(Q&A 카테고리). README의 시작하기 섹션을 가리킴. 한 줄의 답이 신입의 한 시간을 절약해 주고, **그 신입이 평생 자경단의 동료**가 돼요. 첫인상이 평생 인상이에요.

**금요일 17:00** — 한 주 정리. Project 보드를 보면 완료 칸에 5건. 이번 주 마일스톤이 100% 채워짐. Discussions의 Show and tell에 "이번 주 까미·노럑이 입양 갔어요 🐾" 한 장. 본인의 한 주가 한 페이지로 남음. 주말 일이 아니라 그냥 금요일의 17:00. **일의 끝이 재미있는 자경단의 리듬**이에요.

**한 주의 흐름이 6개 도구에 골고루 닿아요.** 본인이 어디 한 곳도 비워 두지 않으면 운영이 매끄러워요. 한 도구가 비면 본인이 카오스를 손으로 메우게 돼요. **6개를 다 켜 두는 게 가장 적은 일**.
**월요일 아침 09:00 5분 의식이 핵심**이에요. 월요일의 첫 5분을 Project 보드 앞에서 보내는 습관을 들이세요. 관리자의 시간이 아니라 본인의 시간. 이번 주에 부덱뜨릴 일이 몇 개인지, 지난 주 넘어온 건이 있는지, 동료의 PR 중 내 리뷰가 필요한 게 있는지 한눈에 보이면 한 주가 머릿속에서 정리돼요. 월요일의 이 5분이 일주일의 카오스를 80% 줄여요. **관리자·팀리더 재목으로 가는 신입의 공통점이 월요일 보드 의식**이라고 관찰되고 있어요.

**금요일 17:00 주간 정리**도 의식. 한 주간 끌난 일 정리, 다음 주로 넘어가는 일 정리, Discussions의 Show and tell에 한 장 "이번 주 성과" 글. 자경단의 한 주가 한 장의 글로 남아요. **1년 후 52장의 주간 정리**가 본인의 자경단 연사예요. 세미나·추석·자원봉사자 모집 때 그 52장이 본인의 가장 강력한 자료.

## 13. 흔한 오해 5가지

**오해 1: "Issue는 버그 보고용이에요."** — Issue는 **모든 일**의 한 장. 버그·기능·신고·문서·아이디어 다. 본인이 손가락으로 두드리는 모든 일이 Issue 한 장으로 시작. "이건 Issue 만들 일은 아니야" 같은 판단을 하지 마세요. 만들면 자취가 남고, 안 만들면 잊혀요. **회사에서 1년 차에 가장 자주 듣는 칭찬 중 하나가 "이 사람 Issue 잘 적네"** 예요. 본인이 머리에 떠오른 일을 1분 안에 명확한 한 장으로 옮기는 능력이 5년 차로 가는 빠른 길.

**오해 2: "PR은 머지되면 끝이에요."** — 머지 후에도 PR은 영원히 그 자리에 있어요. 1년 후 누군가가 "이 줄이 왜 이런가요?"를 추적할 때 그 PR을 보고 답을 찾아요. **PR은 결정의 영구 기록**. 본문을 정성껏.

**오해 3: "코멘트는 코드 문제만 짚어야 해요."** — 칭찬 30%·질문 30%·제안 20%·요청 10%·nit 10%가 건강한 비율. 칭찬이 0개인 코멘트는 동료를 지치게 해요. **본인이 좋다고 느낀 부분을 짚는 것도 리뷰**.

**오해 4: "branch protection은 회사용이에요, 개인 저장소엔 필요 없어요."** — 개인 저장소에 더 필요해요. 본인이 야근에 졸면서 main에 직접 push하는 사고를 시스템이 막아 줘요. 5분 셋업이 평생의 안전.

**오해 5: "GitHub Actions는 어려워요, 나중에 배워요."** — 초기 셋업은 5분. `.github/workflows/ci.yml`에 30줄짜리 표준 양식 한 장 넣으면 끝. 매뉴얼 테스트를 매번 두드리는 시간이 더 비싸요. **자동화는 빨리 깔수록 절약**. Ch014를 못 기다리면 GitHub의 Marketplace 액션 한 줄을 미리 깔아도 돼요.

## 14. FAQ 5가지

**Q1. Issue를 너무 많이 만들면 관리가 안 돼요.**
**소개의 먹이사슬 한 명을 더 — issue·PR이 닫혀도 GitHub에서 안 사라져요**(아카이브만 됨). "close" 버튼이 삭제가 아니라 아카이브. 난장판이 깔끔해지는 대신 자취는 있어요. 수년 후에도 #42는 #42이고, 본문과 코멘트가 끝까지 남아요. 그래서 **함부로 닫는 게 아굌을 파괴시키는 일이 아니**에요. "이건 더 이상 관련 없으니 닫이겠습니다" 한 줄 달고 close. 다시 떠오르면 reopen. **하나의 Issue가 세 번 떠오르는 경우도 있어요.** 그때결제그때의 자경단 상황에 따라.

**Q2. PR 리뷰 누구에게 부탁해야 할지 모르겠어요.**
A. CODEOWNERS 한 장이 답이에요. 폴더가 결정해 줘요. CODEOWNERS가 없으면 GitHub의 "Suggested reviewers" — 그 폴더를 가장 자주 만진 사람을 GitHub이 자동 추천. 본인이 손가락으로 고민 안 해도 돼요.

**Q3. PR 본문을 한국어로 쓸까요 영어로 쓸까요?**
A. 회사·프로젝트의 공식 언어로. 자경단이 한국 커뮤니티면 한국어. 외부에 영어권 봉사자를 받기 시작하면 영어. 한 저장소 안에서 두 언어를 섞지 마세요 — 일관성이 신뢰를 만들어요. **한 가지 톤**으로 통일.

**Q4. 작은 개인 프로젝트엔 모든 도구가 과해요.**
A. 6개 다 안 켜도 돼요. 최소 — Issue + PR + README. 셋만으로도 90% 운영. Project·Discussions는 협업자가 늘어날 때 추가. **시작은 가볍게, 자라면 켜기**.

**Q5. Discussions와 Issue의 차이가 헷갈려요.**
A. **답이 정해진 일**(코드 변경·버그 처리·작업 진행)은 Issue. **답이 안 정해진 이야기**(아이디어·질문·후기·투표)는 Discussions. 헷갈리면 일단 Discussions, 작업이 결정되면 Convert to issue 버튼 한 번. 본인이 매번 정확히 분류 안 해도 시스템이 도와줘요.

**보너스 Q: 외부에 PR 받기 시작하면 무엇이 달라지나요?**
A. CODEOWNERS·CONTRIBUTING·CODE_OF_CONDUCT 세 장이 진짜로 작동해요. 외부 봉사자가 본인의 표준을 본문에서 읽고 따라요. 본인이 매번 "이렇게 해 주세요"를 안 적어도 됨. **문서가 본인의 손가락을 절약**해요. 자경단이 한 명에서 다섯 명으로 갈 때 가장 먼저 와 닿는 변화.

## 15. 추신

추신 1. Issue는 일의 한 장. 일이 머리에 떠오르면 1분 안에 Issue로. 미루면 잊어요. **머리는 떠올리는 곳, Issue는 보관하는 곳**. 회사 5년 차의 비밀 중 하나가 "절대 머리로 일을 들고 있지 않기"예요. 떠오르면 즉시 Issue 또는 Todo 앱으로. 머리는 새로운 생각을 위해 비워 두기.

추신 2. PR은 작게. 50줄 미만이 가장 빠르게 머지돼요. 큰 변경은 5개 작은 PR로 쪼개기. 동료의 시간을 5번 5분씩 쓰는 게 1번 30분 쓰는 것보다 빨라요. **연구 결과** — Google이 자사 코드베이스를 5년간 분석한 보고서에 따르면 PR이 한 줄 늘 때마다 머지 시간이 비선형적으로 증가해요. 50줄까지는 평균 1시간, 500줄은 평균 1주, 5,000줄은 평균 한 달. **작게 쪼개는 것은 본인의 일주일을 절약**해요.

추신 3. 코멘트의 첫 단어 — "혹시", "한 가지", "여기 좋네요". 부드러운 첫 단어가 동료의 마음을 열어요. 5년 차의 비밀 한 줄.

추신 4. LGTM은 신뢰의 이름표. 동료에게 LGTM을 자주 주는 사람이 동료에게 LGTM을 자주 받아요. 한 줄짜리 평판 통화.

추신 5. branch protection은 5분 셋업. 첫날에 박아 두면 평생 굴러요. 미루지 마세요. 사고 한 건이 평생의 후회.

추신 6. CODEOWNERS는 폴더가 리뷰어를 결정. 본인이 매번 고민 안 해도 됨. 분기마다 한 번 점검.

추신 7. Project 보드는 매주 월요일 09:00에 한 번 보기. 5분 투자가 한 주의 카오스를 막아요.

추신 8. Discussions의 Q&A는 신입의 입장권. 첫 질문에 친절히 답하면 그 신입이 평생 본인 자경단에 남아요.

추신 9. README는 5분 자기소개서. 외부 사람이 5분 안에 본인 프로젝트를 이해해야 함. 본인이 자경단의 얼굴.

추신 10. Actions는 24시간 자동 동료. 본인이 자는 동안에도 일해요. Ch014에서 본격적으로, 그때까지 한 줄짜리 standard CI라도 깔아 두기.

추신 11. 자경단의 5장 문서(README·CONTRIBUTING·CODE_OF_CONDUCT·SECURITY·LICENSE)는 외부 봉사자의 환영사. 5장이 있는 저장소는 첫날부터 안전감을 줘요.

추신 13. 본인이 이 H6를 한 번 다 읽은 다음, 본인의 자경단 저장소 Settings 페이지를 30분만 열어보세요. branch protection 규칙 한 개, CODEOWNERS 한 장, Issue 템플릿 한 개, PR 템플릿 한 장 — 4개 셋업이 30분이면 끝나요. 그 이후의 자경단 운영은 완전히 다른 수준이 돼요. **투자 대비 효과가 가장 큰 30분**이 H6의 30분이에요.

추신 14. PR 리뷰에 "이건 머지하면 안 돼요"라고 사용하는 GitHub 기능이 있어요 — **Request changes**(approve와 comment과 함께 세 옵션 중 하나). 해결되지 않으면 머지가 안 돼요. 강한 안전망. 다만 면접에서 파악되는 "주니어 티어"의 특징 중 하나가 Request changes를 너무 쉬운 이유로 눌러 동료를 못 대워렌하는 패턴. **되돌릴 수 없는 문제**(예: 보안 트래 취약, main 업은 회귀)에만 Request changes. 그 외는 comment로 충분해요. 동료의 자존심을 아끼는 도구.

추신 15. 머지 후 **가지는 아카이브 용도로 남겨 두고 자동 삭제를 설정**해 둘 수 있어요. Settings → General → Pull Requests → "Automatically delete head branches" 철크. 동료·본인이 머지된 가지를 손으로 재니 안 소화해도 돼요. 시스템이 자동으로 카오스를 정리. **3주 후 가지 100개가 쌓이는 사고**를 철 철광으로 막아요.
