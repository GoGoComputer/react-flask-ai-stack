> - H5 데모(터미널 실연)로 넘어가기 전, 카탈로그를 머리에 한 페이지로 넣어 두기.

## 회수: H3에서 우리가 들고 온 환경

지난 시간엔 손가락이 닿을 키보드를 셋업했어요. macOS에서 Homebrew로 git 깔고, `git config --global`에 본인 신분증 다섯 줄 적고, ed25519 SSH 키를 GitHub에 등록하고, .gitignore 한 장 둬서 비밀 새는 사고를 막았죠. 이제 키보드는 준비됐고, 왼쪽 화면엔 빈 터미널이 깜빡거립니다. 이번 시간엔 그 터미널에 본인이 두드릴 **단어들의 카탈로그**를 만들어요. 본인이 1년 쓸 git의 80%는 23개 단어 안에 다 들어가요. 그 23개를 한 표에 누이고, 각 단어 옆에 신호등을 켜 둡니다.

**왜 표인가요?** 본인이 5년 차가 돼도 Git 명령어를 외워서 쓰진 않아요. 매일 쓰는 5~6개는 손가락에 박혀 있고, 가끔 쓰는 10개는 머릿속 색인을 살짝 더듬어 찾고, 1년에 한두 번 쓰는 7~8개는 매번 `git help`이나 검색으로 다시 봐요. 외우기 시도하지 마세요. 외울 필요가 없어요. 표를 한 번 만들어 두면, **그 표의 위치**만 머리에 박아도 본인은 5분 만에 어떤 명령어든 찾아낼 수 있어요. 본인 노트북 책장 위에 종이로 한 장 붙여 두는 분도 많아요. 본인 머릿속에도 그 종이 한 장을 그려 두자는 게 이 시간의 목표예요.

또 하나 — **위험도 신호등**. Git 명령어는 겉으론 다 평등해 보이지만 속을 보면 **위험도가 매우 다릅니다.** `git status`는 백 번 두드려도 아무 일도 안 일어나는 안전한 명령어고, `git log`도 마찬가지예요. 반대로 `git reset --hard`나 `git push --force`는 한 번 잘못 두드리면 본인이나 동료의 한 시간 작업을 덮어쓸 수 있어요. **본인이 위험한 명령어를 두드릴 땐, 두드리기 전에 한 번 더 보는 습관**을 만들어야 합니다. 그래서 표에 신호등을 칠해요. 초록은 읽기만(아무 부작용 없음), 노랑은 안전한 쓰기(되돌릴 수 있음), 빨강은 위험한 쓰기(되돌리기 어려움 또는 다른 사람 영향). 손가락이 빨강 위에 놓이면 잠깐 멈추고, 두 번 봅니다.

이 H가 끝나면 본인은 23개 명령어를 머릿속에 색깔로 배치할 수 있어요. 그게 H5의 데모를 차분히 보는 사전 준비입니다.

## 1. 큰 그림 — Git 명령어를 5개 무리로 묶기

23개를 그냥 쭉 늘어놓으면 외우기 힘들어요. 사람의 머리는 **무리(group)** 로 묶을 때 잘 기억해요. 이렇게 묶어 보세요.

| 무리 | 무엇을 하는가 | 명령어 |
|------|---------------|--------|
| 1. **만들기·확인** | 저장소 만들기, 지금 상태 보기 | `init`, `status`, `log`, `diff` |
| 2. **저장하기**(local) | 파일을 commit으로 굳히기 | `add`, `commit` |
| 3. **이동·갈아타기** | 시간·줄기 사이를 오가기 | `branch`, `checkout`, `switch`, `restore` |
| 4. **합치기·재배치** | 줄기를 합치거나 옮기기 | `merge`, `rebase`, `cherry-pick`, `tag` |
| 5. **원격(remote)** | 인터넷 너머 GitHub와 주고받기 | `remote`, `clone`, `push`, `pull`, `fetch` |
| 6. **임시·되돌리기** | 잠시 미루거나 과거로 가기 | `stash`, `reset`, `revert`, `reflog` |

23개가 6개 무리로 줄었어요. 손가락 하나에 무리 하나씩 — 양손 손가락 만큼이 채 안 됩니다. 본인이 매일 쓰는 건 무리 1·2·5에서 거의 다 나와요. 무리 3·4는 일주일에 한두 번. 무리 6은 일이 꼬였을 때만. 이렇게 빈도까지 머리에 그려 두면, 처음 보는 명령어가 무서워 보이지 않아요. 어느 무리에 속하는지만 알면 위험도까지 자동 추정이 돼요.

**한 가지 더 — Git 명령어는 영어 단어다.** `commit`은 굳히다, `branch`는 가지, `merge`는 합치다, `rebase`는 베이스(기반)를 다시 잡다, `revert`는 돌려놓다, `stash`는 숨겨 두다. 영어 단어 그대로의 뜻이 명령어가 하는 일과 일치해요. 한국어로 외우려고 하지 말고, 영단어 본래의 뜻으로 한 번만 새기면 평생 안 잊어요. 본인이 영어가 약하다면 더더욱 — Git 23개 단어가 본인 영어 어휘력의 좋은 출발점이 됩니다.

**또 한 가지 — 명령어의 무리는 사실 본인의 일과 시간대와 일치해요.** 아침 출근 직후엔 무리 1·2·5(상태 확인·동기화·로컬 작업), 점심 즈음엔 무리 3(가지 갈아타기), 오후엔 무리 4(합치기·PR), 퇴근 전엔 다시 무리 5(push). 본인의 하루가 무리 6개의 시계 방향 회전이에요. 일주일을 그렇게 돌면, 무리 자체가 시간 감각으로 박혀요. 어느 명령어를 두드리는 시간대가 본인 손가락에 자연스러운지가 곧 본인의 git 리듬이 됩니다. 회사에서도 똑같아요. 신입이 가장 빨리 적응하는 비결은 23개를 외우는 게 아니라 6개 무리의 일과 흐름을 따라가는 거예요.

## 2. 위험도 신호등 — 초록·노랑·빨강의 정의

본격 카탈로그 들어가기 전에 신호등 정의를 확실히 해 둡시다.

### 🟢 초록 — 읽기만(read-only)

저장소를 **읽기만** 합니다. 파일을 바꾸지 않고, commit을 만들지 않고, 다른 사람에게 보내지도 않아요. 백 번 두드려도 안전. 신입이 무서워서 못 두드릴 이유가 없어요. 오히려 자주 두드리는 게 약속이에요. `git status`는 명령어를 두드리기 전과 후에 한 번씩, 두 번 보는 게 표준 리듬입니다.

해당 명령어: `status`, `log`, `diff`, `reflog`(읽기 부분), `branch`(목록 보기), `remote`(목록 보기), `tag`(목록 보기).

### 🟡 노랑 — 안전한 쓰기(local, reversible)

저장소를 바꾸지만 **본인 노트북 안에서**만 바꿔요. 인터넷에 안 나가요. 그리고 **되돌리기 쉬워요**. 잘못 두드려도 `reflog`로 거의 다 살릴 수 있어요. 두려워하지 말고 두드리되, 두드리기 전에 `git status`로 한 번만 확인하는 습관.

해당 명령어: `add`, `commit`, `branch`(생성), `checkout`(파일 복원 제외), `switch`, `restore`, `merge`(local), `rebase`(local·미공유), `tag`(생성), `stash`, `cherry-pick`(local), `init`, `clone`.

### 🔴 빨강 — 위험한 쓰기(irreversible 또는 remote 영향)

**되돌리기 어렵거나, 다른 사람에게 영향**을 줍니다. 두드리기 전에 두 번 봐요. 동료에게 한 줄 알려도 좋아요("나 force-push 합니다"). 회사에서 빨강 명령어를 잘못 친 사고는 매년 발생해요. 본인이 신중하다고 안 일어나는 게 아니라 — 야근 끝에 졸면서 빨강을 두드리면 일어나요. 그래서 손가락 신호등이 필요한 거예요.

해당 명령어: `push`(특히 `--force`), `pull`(rebase 자동 시 충돌 위험), `reset --hard`, `rebase`(공유된 commit), `commit --amend`(공유 후), `branch -D`(미머지 삭제), `tag -d`(공유된 tag), `clean -fd`(추적 안 된 파일 영구 삭제).

신호등은 **명령어 자체**가 아니라 **명령어 + 옵션 + 상황**이 결정해요. `git reset` 자체는 노랑이지만 `--hard`가 붙으면 빨강. `git push`는 노랑인데 `--force`가 붙으면 빨강. 옵션을 보고 신호등을 다시 칠하는 눈이 필요합니다.

**상황도 신호등을 바꿔요.** 똑같은 `git rebase`도 본인 가지(미공유)에선 노랑, main(공유)에선 빨강. 똑같은 `git push`도 본인 feature 가지에선 노랑, main에 직접 push면 빨강(회사 정책에 따라 차단됨). 똑같은 `git branch -d`(소문자)는 머지된 가지를 안전 삭제하는 노랑이지만, `git branch -D`(대문자)로 미머지 강제 삭제하면 빨강. **명령어를 외우지 말고 옵션과 상황의 결합을 보는 눈을 키우세요.** 그게 본인을 5년 차로 끌어올리는 진짜 학습입니다. Git을 1년만 써도 본인은 자연스럽게 옵션·상황·신호등을 동시에 읽어요. 그땐 표가 없어도 손가락이 알아요.

## 3. 본 표 — 23개 명령어를 한 페이지에

자, 이제 본 표예요. 한 줄에 한 명령어, 위험도·한 줄 정의·자주 만나는 함정. 본인이 이 페이지를 한 번 인쇄해서 책상 위에 두면 1년치 git 학습의 절반이 끝납니다.

| # | 명령어 | 무리 | 신호등 | 한 줄 정의 | 자주 만나는 함정 |
|---|--------|------|--------|-----------|------------------|
| 1 | `git init` | 만들기 | 🟡 | 현재 폴더를 Git 저장소로 만든다 (`.git/` 생성) | 이미 init된 폴더 안에서 또 init하면 의도와 달리 nested 저장소 |
| 2 | `git status` | 확인 | 🟢 | 지금 working tree·staging의 상태를 본다 | 자주 두드리지 **않아서** 사고가 남, 더 자주 두드릴 것 |
| 3 | `git log` | 확인 | 🟢 | commit 히스토리를 본다 | 옵션 없이 치면 너무 길다, `--oneline --graph --all` 별칭 추천 |
| 4 | `git diff` | 확인 | 🟢 | 변경된 줄을 본다 | `git diff`(staging 전) vs `git diff --staged`(staging만) 구분 필요 |
| 5 | `git add` | 저장 | 🟡 | working → staging으로 파일을 올린다 | `git add .`로 비밀 파일까지 올리는 사고, .gitignore 먼저 |
| 6 | `git commit` | 저장 | 🟡 | staging → repository로 굳힌다 | `-m` 한 줄 메시지에 끝내지 말고 다음줄에 본문도 쓰는 습관 |
| 7 | `git branch` | 이동 | 🟢/🟡 | 가지를 보거나 만든다 | `branch -D <name>`은 미머지 삭제 — 빨강 |
| 8 | `git checkout` | 이동 | 🟡 | 옛날 명령어, 가지 갈아타기·파일 복원 둘 다 | 너무 많은 일을 해서 헷갈림 → `switch`/`restore`로 갈아탈 것 |
| 9 | `git switch` | 이동 | 🟡 | 가지만 갈아탄다(2019년 신규) | 새 가지 만들 땐 `-c` 옵션 필요 |
| 10 | `git restore` | 이동 | 🟡 | 파일을 옛 상태로 되돌린다(2019년 신규) | `--staged`로 staging 취소도 가능 |
| 11 | `git merge` | 합치기 | 🟡/🔴 | 다른 가지를 현재 가지로 합친다 | conflict 나면 **종료 안 함**, 해결 후 `git commit` 필요 |
| 12 | `git rebase` | 합치기 | 🟡/🔴 | 부모 commit을 옮겨 sha를 재계산한다 | 공유된 commit을 rebase하면 동료의 작업을 망가뜨림 |
| 13 | `git remote` | 원격 | 🟢/🟡 | 원격 저장소 별명을 관리한다 | 별명만 추가, 실제 통신은 push/pull/fetch가 함 |
| 14 | `git push` | 원격 | 🟡/🔴 | local commit을 원격으로 보낸다 | `--force`는 빨강, `--force-with-lease`로 안전화 |
| 15 | `git pull` | 원격 | 🟡/🔴 | 원격에서 가져와서 자동 합친다(fetch+merge) | conflict 자동 발생 가능, `pull.rebase=true` 권장 |
| 16 | `git fetch` | 원격 | 🟢 | 원격 변경을 가져오기만 한다(머지 안 함) | "보고 합치고 싶다"면 pull 대신 fetch + merge 분리 추천 |
| 17 | `git clone` | 원격 | 🟡 | 원격 저장소를 통째로 받아 폴더를 만든다 | 큰 저장소는 시간이 걸림, `--depth=1` shallow 옵션 |
| 18 | `git tag` | 합치기 | 🟢/🟡 | commit에 이름표(version)를 붙인다 | annotated(`-a`) vs lightweight, 배포는 annotated 권장 |
| 19 | `git stash` | 임시 | 🟡 | 작업 중인 변경을 잠시 숨긴다 | `stash pop` vs `stash apply` — pop은 적용+삭제, apply는 적용만 |
| 20 | `git reset` | 되돌리기 | 🟡/🔴 | HEAD를 옮기고 staging·working을 함께 옮길 수도 | `--hard`는 빨강, working tree까지 덮어씀 |
| 21 | `git revert` | 되돌리기 | 🟡 | 한 commit을 **취소하는 새 commit**을 만든다 | 히스토리를 안 지움 → 협업에 안전 |
| 22 | `git cherry-pick` | 합치기 | 🟡/🔴 | 다른 가지의 한 commit만 가져온다 | 같은 변경을 두 번 적용해서 충돌·중복 가능 |
| 23 | `git reflog` | 되돌리기 | 🟢 | HEAD가 지나간 자취를 본다(90일 보관) | "rebase 망쳤다, reset 망쳤다" 거의 모든 사고의 응급실 |

23줄. 이 한 페이지가 본인 git의 80%예요. 매일 쓰는 5~6개는 1·2·5번 무리, 즉 status·add·commit·push·pull·log. 이 6개가 본인 손가락에 박힙니다. 그 외엔 필요할 때 표를 다시 봅니다. 표를 다시 보는 건 부끄러운 일이 아니에요. 5년 차도 매일 표를 다시 봐요. 외우려는 시도가 오히려 시간 낭비입니다.

## 4. 매일 쓰는 6개 명령어의 손가락 리듬

23개 중에 매일 쓰는 6개를 따로 떼어 손가락 리듬으로 박아 둡시다. 본인이 자경단 사이트 한 줄을 고치고 push할 때까지 두드리는 명령어가 정확히 이 순서입니다.

> ▶ **같이 쳐보기** — 본인 push 의 6 박자 한 줄씩
>
> ```bash
> git status                                # 1. 지금 어디 있는지
> git diff                                  # 2. 무엇이 바뀌었는지
> git add cat-list.html                     # 3. staging
> git commit -m "feat: 길고양이 카드 한 장 추가"  # 4. 굳히기
> git pull --rebase                         # 5. 동료 변경 합치기
> git push                                  # 6. 보내기
> ```

여섯 줄이 하나의 호흡이에요. 회사에서 본인이 1년에 두드릴 git 명령어의 80%가 이 여섯 줄입니다. 손가락에 박혀서 생각 안 하고 두드리게 되는 게 목표예요. 하루에 30번씩 두드리면 한 달이면 박힙니다. 한 달 안에 자동화돼요.

**리듬에 한 가지 보너스 단계** — 1·2번 사이에 **`git log --oneline -5`** 한 줄 추가하면 본인이 마지막에 무슨 commit을 했는지 떠올릴 수 있어요. 어제 퇴근 전 마지막 commit이 뭐였더라? 한 줄로 답이 나옵니다. 5년 차의 손가락엔 이 한 줄도 무의식에 들어 있어요.

**또 하나의 보너스 — `git diff --staged`** 를 4번 commit 직전에 한 번. staging에 정확히 무엇이 올라가 있는지 마지막 검토. 본인이 `git add .`을 두드린 후 비밀 키나 디버그 print문 한 줄이 끼어 있지 않은지 확인하는 마지막 검문소. 회사에서 코드 리뷰가 본인이 commit한 후에야 일어나는데, `--staged` 한 번이 본인 스스로의 1차 리뷰예요. 1차 리뷰를 본인이 하면, 동료의 시간이 절약되고 PR 코멘트가 줄어들어요. **자기 자신을 코드 리뷰하는 습관**이 5년 차로 가는 가장 빠른 길이라고 어느 시니어가 적어 두기도 했어요.

그리고 매일 6번째 명령어 다음에, 사실 **7번째 단계**가 하나 더 있어요 — 브라우저로 GitHub의 본인 PR 페이지를 한 번 보는 것. CI(GitHub Actions)가 초록불이 켜졌는지, 동료가 코멘트를 달았는지. push 후 1분 안에 한 번 확인하는 게 표준 리듬. 야근하면서 push만 하고 자러 가면 다음 날 아침 빨간불이 본인을 기다려요. 1분의 확인이 다음 날 30분의 디버깅을 막아요.

## 5. 일주일에 한두 번 쓰는 7개 — branch·switch·merge·rebase·stash·tag·reflog

매일 6개에 익숙해지면, 그다음 익혀야 할 7개가 있어요.

**`git branch <name>` + `git switch <name>`** — 새 기능 작업할 때 본인 이름으로 가지를 따요. `git switch -c feat/cat-list`(branch+switch 한 줄). 자경단에선 길고양이 카드 기능마다 가지를 한 개씩 따요. 본인이 망쳐도 main은 안 망가집니다. **이 두 줄이 본인 첫 협업의 출발점**이에요.

**`git merge`** — 본인 가지에서 main을 가져오거나, main에서 본인 가지를 합쳐요. PR을 만들고 GitHub UI에서 "Merge pull request" 버튼으로 자동 merge 되는 경우가 99%지만, 본인 노트북에서 직접 merge할 일도 있어요. conflict이 나면 일단 침착, 충돌 표시(`<<<<<<<`, `=======`, `>>>>>>>`) 줄을 손으로 정리하고 `git add` + `git commit`.

**`git rebase`** — main의 최신 변경을 본인 가지 아래에 깔아요. PR을 만들기 전에 한 번 rebase해서 깔끔한 히스토리 만들기. **공유된 commit엔 절대 rebase하지 마세요.** 본인 가지에서만, 본인 commit에만. **rebase가 무서운 분에게** — 본인이 rebase 직전에 `git branch backup-before-rebase` 한 줄로 백업 가지를 따 두세요. 41바이트짜리 안전장치예요. rebase가 망하면 `git reset --hard backup-before-rebase` 한 줄로 시간이 되돌아옵니다. 5년 차의 비밀 한 가지가 이 한 줄. 두려움 없이 rebase하세요.

**`git stash`** — main에서 일하다가 갑자기 본인 가지로 갈아타야 할 때, "잠깐 멈춰" 버튼이에요. 작업 중인 변경을 stash로 숨기고, 가지 갈아타고, 일 끝내고, 다시 돌아와서 `git stash pop`. 손가락 한 번에 본인 작업이 다시 살아나요. **stash는 스택 구조**라서 여러 번 stash하면 차곡차곡 쌓이고, `git stash list`로 목록을 봐요. 한 가지 함정 — 동료에게 화면 공유로 물어보다가 stash한 걸 까먹고 다음 날 그 위에 또 stash하는 사고. `git stash list`를 가끔 두드려서 스택을 청소하는 습관을 두세요. **이름표를 붙이는 stash** — `git stash push -m "WIP: 카드 카테고리 필터"`로 메시지를 적어 두면 일주일 후에도 그 stash가 뭐였는지 알 수 있어요.

**`git tag v1.0.0`** — 배포할 때 commit에 버전 이름표 붙이기. 자경단 사이트가 첫 외부 공개되는 날 v1.0.0 태그를 박아 둡니다. 평생 그 시점이 그 태그 한 줄로 남아요. **annotated tag**(`-a -m`)는 작성자·날짜·메시지가 박힌 "진짜" 태그, **lightweight tag**는 그냥 sha 별명. 배포·외부 공개에는 annotated. 본인이 한 가지 더 알면 좋은 — **SemVer 컨벤션**(MAJOR.MINOR.PATCH). v1.0.0 = 첫 정식, v1.0.1 = 작은 버그 수정, v1.1.0 = 호환되는 기능 추가, v2.0.0 = 호환 깨지는 변경. 자경단 사이트가 안드로이드 앱이 생기는 날엔 v2.0.0 — API가 바뀌니까. 본인이 회사에서 만나는 거의 모든 라이브러리가 SemVer를 따라요. 본인의 첫 태그부터 SemVer로.

**`git reflog`** — 모든 사고의 응급실. "어, 내 commit이 사라졌어요"라고 외친 신입의 90%가 reflog 한 번 두드려서 살려냅니다. 90일 동안 본인 HEAD가 지나간 모든 자취가 남아 있어요. **reset --hard로 망친 작업도 reflog로 살릴 수 있어요.** 본인이 사고 친 다음 첫 번째로 떠올릴 단어가 이 단어예요. **있을 말하면** — reflog는 본인 노트북 `.git/logs/HEAD` 파일에 텍스트로 쌓여있어요. 한 줄이 "`<옆 sha> <이전 sha> <작성자> <시간> <액션명>: <메시지>`". 일주일 출근 후 reflog를 보면 일주일치 본인의 손가락 활동이 다 남아 있어요. 희말한 일지예요. **reflog는 원격으로 push 안 돼요.** 본인 노트북에만 남아 있어요. 그래서 노트북 자체를 잃으면 reflog도 사라져요. 본인의 노트북 백업이 reflog 백업이기도 한 점.

**한 가지 더 — `git switch -`** — 마이너스 한 글자가 "직전 가지"를 의미해요. main → feat/cat-list로 갔다가 다시 main으로 올 때 `git switch -` 한 번이면 끝. zsh의 `cd -`와 같은 패턴이에요. 본인의 일주일 협업에서 매번 만나는 작은 기쁨.

## 6. 1년에 한두 번 쓰는 8개 — 위험한 명령어들

1년에 한두 번만 쓰는 명령어 8개가 남아요. 이게 빨강(🔴) 무리에 모여 있어요. 본인이 평소엔 두드릴 일이 없지만, 두드릴 일이 생기면 두 번 보고 칩니다.

**`git reset --hard <commit>`** — HEAD와 working tree를 모두 그 commit으로 덮어써요. 본인이 잘못 만든 commit 다섯 개를 한 번에 버릴 때. **하지만 working tree의 안 commit한 변경도 사라져요.** `git status`로 commit 안 된 변경이 없는지 확인 후 두드릴 것. 그래도 잘못 두드렸으면 `git reflog`.

**`git push --force`** — 원격을 본인 local로 덮어써요. 보통은 rebase 후에 본인 가지에 한 번. **공유 가지(main)에 절대 force-push 금지.** 회사에서 main에 force-push해서 동료 10명의 commit을 날린 사고는 매년 어느 회사에선가 일어나요. **`--force-with-lease`** 옵션이 안전한 대체 — 본인이 마지막으로 본 원격 상태에서 누가 끼어들지 않았을 때만 force가 됩니다.

**`git revert <commit>`** — 한 commit을 취소하는 **새 commit**을 만들어요. 히스토리를 안 지우니까 협업에 안전해요. main에 잘못 들어간 commit을 빼낼 땐 reset 대신 revert. revert한 commit이 또 한 줄 history에 남는 게 정상이에요. **이게 reset과의 결정적 차이** — reset은 "없었던 일로", revert는 "있었지만 취소함". 협업에선 항상 후자가 안전. 동료가 이미 본인의 commit을 base로 작업했다면 reset은 동료의 작업까지 무너뜨리지만 revert는 동료의 작업을 보존해요. main 같은 공유 가지의 사고는 무조건 revert로.

**`git cherry-pick <commit>`** — 다른 가지의 한 commit만 가져와요. hotfix 가지의 한 commit만 main에 적용하고 싶을 때. 같은 commit을 두 가지에 두 번 적용하면 conflict이 나기 쉬우니 한 번만.

**`git commit --amend`** — 마지막 commit을 고쳐요. 메시지를 고치거나 빠뜨린 파일을 추가할 때. **이미 push한 commit엔 amend 금지** — sha가 바뀌어서 force-push가 필요해지고, 그러면 협업이 망가져요.

**`git rebase -i HEAD~5`** — 최근 5개 commit을 인터랙티브 rebase. 합치기(squash)·메시지 고치기(reword)·삭제(drop)를 표 편집기처럼 할 수 있어요. PR 만들기 전 본인 가지의 commit 7개를 의미 있는 3개로 정리할 때 쓰는 강력한 도구. **공유된 commit엔 금지** 규칙은 똑같아요.

**`git clean -fd`** — 추적 안 된 파일과 폴더를 영구 삭제. .gitignore에 안 들어간 새 파일까지 다 지워요. 일주일 만에 만든 빌드 산출물을 한 번에 정리할 때 유용하지만, 한 번 두드리면 휴지통도 안 거치고 사라집니다. 두드리기 전에 `git clean -nd`(dry run)로 미리 보기 필수.

**`git filter-branch` / `git filter-repo`** — 히스토리를 통째로 다시 쓰는 가장 위험한 도구. AWS 키가 5년 전 commit에 박혀 있어서 모든 commit에서 그 줄을 지워야 할 때. 회사에서 1년에 한 번 있을까 말까. **두드리기 전에 백업, 동료에게 알리기, 그리고 가능하면 git filter-repo(BFG Repo-Cleaner)** 같은 검증된 도구 사용.

이 8개가 본인 손가락 신호등의 빨간불이에요. 빨간불이 켜지면 손가락이 **자동으로 한 박자 멈춰야** 합니다. 그게 본인을 5년 차 사고에서 구해 주는 작은 습관이에요.

## 7. 자경단 적용 — 카드 한 장을 push하기까지의 정확한 손가락 흐름

자경단 사이트의 길고양이 카드 한 장을 추가한다고 칩시다. 본인이 두드릴 명령어를 시간순으로 적어 봅니다.

> ▶ **같이 쳐보기** — 첫 PR 의 13줄 — 월요일 오전부터 머지 후 정리까지
>
> ```bash
> # (월요일 오전 10시, main에서)
> git status                                # 🟢 깨끗한지 확인
> git pull                                  # 🟡 동료 변경 받기
> git switch -c feat/add-cat-mini           # 🟡 새 가지 따기
>
> # (10:30, cat-list.html 에 한 카드 추가)
> git status                                # 🟢 어떤 파일 바뀌었나
> git diff cat-list.html                    # 🟢 줄 단위 확인
> git add cat-list.html                     # 🟡 staging
> git commit -m "feat: 미니 길고양이 카드 한 장 추가"  # 🟡
>
> # (PR 만들기 직전, main 의 최신을 가져와 깔끔히)
> git fetch origin                          # 🟢 원격 최신 받기
> git rebase origin/main                    # 🟡 본인 가지를 main 위로
> git status                                # 🟢 충돌 없는지
>
> # (push)
> git push -u origin feat/add-cat-mini      # 🟡 첫 push, upstream 설정
>
> # (GitHub UI 에서 PR + 리뷰 + merge 후 다시 main 으로)
> git switch main                           # 🟡
> git pull                                  # 🟡 방금 merge 된 PR 가져오기
> git branch -d feat/add-cat-mini           # 🟡 끝난 가지 정리
> ```

13줄. 23개 카탈로그 중 9개를 썼네요(status, pull, switch, diff, add, commit, fetch, rebase, push, branch). 본인의 1년치 git 사용이 이 13줄 안에 거의 다 들어 있어요. 회사에 들어가도 똑같아요. 자경단에서 13줄을 100번 두드리면, 회사에서 첫 PR을 두드릴 때 손가락이 이미 알고 있어요.
이 13줄의 흥미로운 점 하나 더 — **컴퓨터가 13줄 중 몇 줄을 대신 해 주는지 핬아보세요.** GitHub의 "Merge pull request" 버튼이 8번째 이후의 merge를 대신해 주고, GitHub Actions가 push 되자마자 테스트를 돌려주고, dependabot이 의존성 업데이트를 자동 PR로 올려줬어요. **본인이 두드릴 13줄은 종률 키보드의 한적 일 부분**이고, 나머지는 자동화가 감당해요. 그래서 본인이 두드릴 명령어가 23개도 안 되는 거예요. 자동화 주변을 모르면 더 많은 일을 손으로 하게 돼요. **Ch014에서 그 자동화를 본격적으로 다룰 예정이에요.** H4의 23개 카탈로그가 Ch014의 GitHub Actions 철로 공사장으로 이어져요.
**한 가지 더 — 위험도 신호등을 따라가 보세요.** 13줄 중 빨강이 0개. 노랑이 8개, 초록이 5개. **본인의 일주일 99%는 노랑+초록입니다.** 빨강은 일이 꼬였을 때만 만나요. 본인의 평소 손가락은 무서울 게 없어요. 무서운 건 야근·졸음·서두름 때만 들어오는 빨강. 그래서 신호등이 필요한 거예요.

## 8. H3와 H4의 연결 — 셋업한 환경 위에 명령어를 얹다

H3에서 본인이 셋업한 환경(`git config --global` 7줄·SSH 키·credential helper)이 H4의 명령어가 매끄럽게 작동하는 토양이에요. 예를 들어 `git push`가 비밀번호를 안 묻는 건 SSH 키나 credential helper 때문이고, `git commit`이 작성자를 자동으로 본인 이름으로 박는 건 `user.name·user.email` 때문이고, `git pull`이 자동으로 rebase하는 건 `pull.rebase=true` 때문이고, `git push`가 새 가지에 자동으로 upstream을 잡는 건 `push.autoSetupRemote=true` 때문이에요. **H3의 7줄이 H4의 23줄을 매끈하게 만들어 준 거**입니다. 셋업의 가치가 카탈로그를 두드릴 때 비로소 손가락에 와 닿아요.

거꾸로도 — H4의 23개를 한 번 둘러보고 나면, H3에서 왜 그 7줄을 깔아 뒀는지 이해가 깊어져요. "아, `core.autocrlf`는 `git diff`가 거짓 줄바꿈을 안 보여 주게 하려는 거구나, `pull.rebase`는 `git pull`이 더러운 merge commit을 안 만들게 하려는 거구나, `push.autoSetupRemote`는 `git push -u origin <branch>`를 매번 안 치게 하려는 거구나." H3과 H4가 서로를 보강해요. 한 사이클을 한 번 돌면 두 시간이 같이 단단해집니다.

이게 본 코스의 회수 구조예요. 본인이 H4를 마치고 H3을 다시 펼치면 새로운 줄이 보여요. 한 번 더 펼치면 또 새로운 줄이 보여요. 한 챕터를 끝까지 안 외우려고 하는 이유가 이거예요. **반복이 외우기보다 강해요.**

## 9. H5 예고 — 카탈로그를 손가락으로 실연

H5에선 이 23개를 본인이 직접 터미널에서 두드려요. 본인 자경단 폴더에서 `git init`부터 첫 `git push`까지 한 호흡으로. 본인이 머리로 외운 카탈로그가 손가락에 박히는 순간이에요. H4가 머리의 시간이라면 H5는 손가락의 시간. 두 시간을 합치면 본인은 실전 git을 시작할 수 있어요.

특히 H5에선 **위험도 빨강을 일부러 한 번 두드려 봐요.** `git reset --hard`로 본인 commit을 한 번 날려 보고, `git reflog`로 다시 살려 봅니다. 무서운 명령어를 안전한 환경에서 한 번 두드려 보면, 회사에서 야근에 두드릴 때 손가락이 더 신중해져요. **두려움은 무지에서 와요.** 한 번 만나면 두려움이 줄어들고, 신중함만 남아요.

## 10. 큰 그림으로 한 발 — 다른 챕터에서 카탈로그가 어떻게 깊어지나

23개 명령어가 다른 챕터에서 더 깊어집니다.

- **Ch005 Git 협업 워크플로우** — `branch`·`merge`·`rebase`·`pull`·`push`를 팀 단위로 묶어서. PR 리뷰·conflict 해결·feature flag·trunk-based vs gitflow.
- **Ch006 GitHub 활용 심화** — `remote`·`tag`·`fork`·Issue·Project·Discussions를 GitHub 기능과 묶어서.
- **Ch014 자동화·CI 입문** — `push`가 GitHub Actions를 자동 트리거. 본인이 두드린 push 한 번이 빌드·테스트·배포로 이어져요.
- **Ch020 코드 리뷰** — `diff`·`blame`·`log`가 리뷰의 도구. 누가 왜 그 줄을 썼는지 5초 만에 추적.
- **Ch041 모노레포** — 한 저장소에 여러 프로젝트. `sparse-checkout`·`worktree` 같은 진보된 기능.
- **Ch062 백엔드 배포** — `tag` 기반 자동 배포. v1.2.3 태그 하나가 production 한 번을 만든다.
- **Ch070 Kubernetes** — git이 GitOps로 인프라까지 관리. `commit`이 클러스터 상태가 됨.
- **Ch080 보안** — `secret scanning`·`pre-commit hook`·`signed commit`으로 사고 예방.
- **Ch103 오픈소스 기여** — `fork`·`PR`·`upstream` 워크플로우. 본인이 자경단 외부에 첫 PR을 만들 때.
- **Ch118 팀 빌딩** — git 컨벤션을 팀 룰로. 신입 온보딩 1일 차 git 매뉴얼.
- **Ch120 회고와 다음 길** — 1년치 git 사용 데이터로 본인 손가락 습관 회고.

23개 단어가 11개 챕터에서 각도를 바꿔 가며 다시 등장해요. **본 코스의 모든 회수는 이미 H4의 표 한 장에 출발점이 박혀 있어요.** H4가 코스 전체의 기둥 한 개입니다.

## 11. 흔한 오해 5가지

**오해 1: "git 명령어가 너무 많아 외울 수 없어요."** — 외우지 마세요. 23개 표 위치만 머리에 두세요. 매일 쓰는 6개만 자동으로 박힙니다. 나머지 17개는 평생 표를 다시 봐도 됩니다. 5년 차도 그래요.

**오해 2: "위험한 명령어는 안 쓰면 안전해요."** — 반대예요. 안 써 봐서 무서움이 커지면, 정작 써야 할 때 더 큰 사고가 나요. 안전한 환경에서 한 번씩 써 봐야 손가락이 신중해집니다.

**오해 3: "GUI(GitHub Desktop·GitKraken·VS Code 패널)만 쓰면 카탈로그를 안 외워도 돼요."** — 절반은 맞아요. 매일 쓰는 6개는 GUI로 가능. 하지만 사고가 났을 때 GUI는 reflog·rebase --abort·force-with-lease를 못 보여 줘요. 응급실용 터미널 명령어는 손가락에 박아 둬야 합니다.

**오해 4: "checkout은 옛날 명령어니까 안 써도 돼요."** — 신규 명령어 `switch`/`restore`가 권장되지만, 회사 코드베이스의 git 책·동료의 입에서 checkout은 평생 나옵니다. 양쪽 다 알아 둬야 협업이 매끄러워요. 본인 손가락은 switch로, 동료의 입은 checkout으로 듣고.

**오해 5: "force-push는 절대 쓰면 안 돼요."** — 본인 가지에선 자주 써요. 본인이 rebase로 commit을 정리한 후엔 force-push가 필요. 단지 **공유 가지(main, develop)** 엔 절대 안 됩니다. 위험한 건 force-push 자체가 아니라 force-push되는 가지가 누구의 것이냐예요. **타인의 가지**에 force-push하는 것도 대재앙 — 동료가 그 가지에서 작업 중이면 그 commit을 짤렉 지워요. "본인의 가지에만 force-push한다"가 정확한 규칙. PR이 머지되면 그 가지는 움직이지 않으니 force-push할 이유도 없어져요.

**보너스 오해: "git pull이 가장 안전한 동기화예요."** — 안전해 보이지만 자동으로 merge·rebase를 해 버려서 충돌을 사용자에게 떠넘겨요. 본인이 신중하다면 `git fetch + git log origin/main..main + git merge` 세 줄로 분리하는 게 더 안전. 매일은 pull, 중요한 합치기엔 fetch + 분리.

## 12. FAQ 5가지

**Q1. 23개 외에 더 알아야 할 명령어가 있나요?**
A. 있어요. `git blame`(누가 그 줄을 썼나), `git show`(한 commit 자세히), `git submodule`(저장소 안의 저장소), `git worktree`(같은 저장소를 여러 폴더에서 동시에), `git bisect`(버그가 어느 commit에서 들어왔나 이등분 탐색). 매일은 안 쓰지만 1년에 한두 번 만나요. 본인이 카탈로그 23개에 익숙해진 후 천천히 추가하세요. 한 번에 다 외우려 하면 머리가 터져요.

**Q2. `git checkout` vs `git switch`/`git restore`, 어느 쪽을 본인이 쓰면 되나요?**
A. 새로 배우는 본인은 `switch`(가지)와 `restore`(파일)를 쓰세요. 명확해요. 한 명령어가 한 일을 합니다. `checkout`은 한 명령어가 두 일을 해서(가지 갈아타기 + 파일 복원) 신입에게 헷갈려요. 다만 회사 동료의 입에선 checkout이 자주 나오니 알아듣기는 해야 합니다. **쓰는 건 switch/restore, 듣는 건 양쪽.**

**Q3. `git pull`을 그냥 두드리면 위험한가요?**
A. 매우 위험하진 않지만, 자동 merge가 본인의 의지 없이 일어나요. 그래서 H3에서 `pull.rebase=true`를 깔았어요. 또 fetch + 보고 + merge 세 줄로 분리하는 패턴도 권장. 본인이 매일 main에서 pull만 하는 단순 작업자라면 그냥 pull로 충분, 협업이 깊어질수록 fetch 분리.

**Q4. `git reset`은 모드가 세 개(`--soft`, `--mixed`, `--hard`) 있다던데 어떻게 다른가요?**
A. `--soft`: HEAD만 옮김. staging·working은 그대로 — 가장 안전. `--mixed`(기본): HEAD+staging 옮김. working은 그대로 — 보통의 reset. `--hard`: HEAD+staging+working 모두 옮김 — **빨강**, 안 commit한 변경 사라짐. 본인이 commit을 한 개 빼고 싶다면 `git reset --soft HEAD~1`이 가장 안전. `--hard`는 정말 다 버릴 때만.

**Q5. 23개를 다 외우는 데 얼마나 걸리나요?**
A. 못 외워요. 외울 필요도 없어요. 매일 쓰는 6개는 한 달, 일주일에 한두 번 쓰는 7개는 3개월, 1년에 한두 번 쓰는 8개는 5년 — 그 정도 걸려요. 그래도 표 한 장만 책상 위에 두면 본인은 첫날부터 23개를 다 쓸 수 있어요. **외우는 게 아니라 찾는 능력**이 git 카탈로그의 본질입니다. 본인은 이미 첫날 그 능력을 가졌어요.

**보너스 Q: GUI(VS Code·GitHub Desktop·GitKraken)를 쓰면 안 되나요?**
A. 써도 돼요. 매일 쓰는 6개는 GUI가 더 빠를 때도 많아요. 다만 본 코스에선 터미널을 기본으로 가르쳐요. 이유 — (1) 사고 응급실에선 터미널만 모든 옵션을 줍니다, (2) 회사·서버·CI 환경에선 GUI가 없어요, (3) 본인이 5년 차가 되었을 때 동료를 도울 수 있는 손가락은 터미널 손가락이에요. GUI를 보조로 두고 터미널을 메인으로.

## 13. 자경단 챕터 컨텍스트로 한 줄 요약

본인이 자경단 길고양이 카드 한 장을 추가하고 GitHub에 보내는 손가락 흐름은 **23개 명령어 중 9개**(status·pull·switch·diff·add·commit·fetch·rebase·push·branch)로 완결됩니다. 그 9개가 본인의 손가락에 박히면 회사 첫 PR도 똑같은 호흡으로 두드릴 수 있어요. **자경단의 한 카드가 본인 회사의 첫 commit과 같은 손가락 모양**이에요.

## 14. 추신

본인이 신입일 땐 표를 인쇄해서 책상 위에 둬요. 부끄러운 일이 아니에요. 5년 차 동료의 책상 위에도 그 종이가 한 장 붙어 있어요. 매일 쓰는 6개는 한 달이면 박혀요. 그때부턴 표를 안 봐도 손가락이 알아요. 본인은 그 한 달을 일부러 자경단에서 만들어 보세요. 100번 두드리면 박힙니다. 위험한 명령어는 안전한 환경에서 일부러 한 번 써 봐요. 본인 노트북에 빈 폴더 만들고 `git init` → 더미 commit 다섯 개 → `git reset --hard HEAD~3` → `git reflog` → `git reset --hard <옛 sha>`. 한 번만 해 보면 평생 무섭지 않아요.

force-push는 본인 가지에선 자주, 공유 가지엔 절대. 이 한 줄이 본인을 회사 사고에서 구해 줍니다. `--force-with-lease`를 본인의 기본 force로 만들어 두세요(`alias fpush = push --force-with-lease`). `git status`를 두 번씩 두드리는 습관 — 명령어 두드리기 전 한 번, 두드린 후 한 번. 이 두 번이 본인의 사고를 90% 막아 줍니다. 그리고 17개 중 가장 자주 만나는 건 `git reflog`. 사고가 나면 첫 단어로 떠올리세요. 본인의 90%의 사고는 reflog가 살려 줍니다.

카탈로그는 외우는 종이가 아니라 찾는 종이예요. 명령어가 기억이 안 날 때 `git help <명령어>` 또는 `git <명령어> -h` — 구글보다 빠르고, 정확하고, 오프라인에서도 돌아가요. 회사 면접관이 "force-push는 언제?"·"reset과 revert의 차이?"·"merge와 rebase 중 뭐?" 세 질문을 자주 해요. 답은 이미 이 H4에 다 들어 있어요. 외우는 게 아니라 **본인의 일주일을 설명하는 답** — "지난주 길고양이 카드 PR에서 rebase로 commit 7개를 3개로 정리했고, force-with-lease로 안전하게 push했어요"라는 한 줄이 책 한 페이지보다 강해요. H5에서 손가락으로 실연합시다. 🐾
