## 1. 들어가며 — Ch003 회수와 Ch004 첫 약속

자, Ch004에 오신 걸 환영해요. 새 챕터입니다. 지난 Ch003 마지막에 제가 한 줄 약속을 드렸죠. "본인 노트북이 인터넷에 연결되는 정확한 순서를 7단계로 설명할 수 있어야 한다." DNS, TCP 핸드셰이크, TLS 핸드셰이크, HTTP 요청, 응답 헤더, 응답 본문, 브라우저 렌더. 이 7단계가 머리에 들어오셨다면 Ch003은 본인 거예요. 안 들어오셨어도 괜찮습니다. Ch004 안에서 한 번 더 만나요. `git push`가 사실 그 7단계를 매번 하는 일이거든요. 자경단 사이트에 코드 한 줄을 올릴 때마다, 본인 노트북은 GitHub 서버와 DNS 조회하고, TCP 핸드셰이크하고, TLS 핸드셰이크하고, HTTP/2 위에 git protocol을 얹어서 보내요. Ch003에서 그렸던 그 그림이 매일 100번씩 일어납니다. 그러니 Ch003은 Ch004의 바닥이라고 보시면 돼요. 자, Ch003 끝낸 본인을 한번 돌아봐 주세요. 두 호텔 사이의 복도, 봉투(패킷), 객실 번호(포트), 통화 회선(소켓), 봉인 도장(TLS). 다섯 비유가 머리에 깔렸어요. 그 위에 오늘부터 8시간 동안 새 비유를 한 장 더 깝니다. **사진앨범.** Git은 코드의 사진앨범이고, GitHub은 그 앨범을 구름에 백업해 주고 친구들과 같이 보는 서비스예요. 1교시 약속은 세 가지입니다. 하나, **Git이 왜 만들어졌고 왜 모든 개발자가 매일 쓰는지를 한 비유로 설명할 수 있게** 만들어 드립니다. 사진앨범 한 장이면 끝나요. 둘, **`git status` 와 `git log` 두 줄을 본인 노트북에서 직접 쳐 보고 출력을 읽을 수 있게** 만들어 드립니다. 셋, **Ch004가 끝났을 때 본인 손으로 미니 레포를 만들고 GitHub에 push하고 친구를 collaborator로 초대할 수 있게** 만들어 드립니다. 8시간 뒤에 그게 됩니다. 오늘은 그 8시간의 큰 그림이고요. 한 가지만 더 약속드릴게요. 오늘 명령어를 외우지 않으셔도 됩니다. 비유 한 장만 머리에 들어오면 1교시는 가져가신 거예요. 명령어는 H3·H4에서 손에 박힙니다. 1교시는 지도 펴는 시간이에요. 산을 오르는 시간이 아닙니다. 가벼운 마음으로 따라와 주세요.

## 2. Git은 코드의 사진앨범이다

자, 비유부터 갑니다. Git이라는 단어를 처음 들으면 머릿속에 뭐가 떠오르세요. 까만 터미널? 헷갈리는 명령어? 겁나는 force push? 그건 다 표면이에요. Git의 본질을 한 문장으로 줄이면 이거예요. **"Git은 코드 폴더의 사진을 시간 순서대로 찍어 두는 도구다."** 이게 다입니다. 진짜로요. 사진앨범 한번 떠올려 보세요. 어린 시절 사진, 졸업 사진, 첫 직장 사진. 사진은 그 순간의 모든 것이 다 담겨 있어요. 옷, 머리, 표정, 배경. 다음 사진은 또 그 순간의 모든 것. 사진과 사진 사이에 무엇이 바뀌었는지는 두 장을 나란히 놓고 보면 보입니다. Git이 하는 일이 정확히 그거예요. 본인이 `git commit` 한 번 칠 때마다 Git은 본인 폴더 전체의 사진을 한 장 찍어 둡니다. 1초 전 사진, 5분 전 사진, 한 시간 전 사진, 어제 사진, 작년 사진. 다 찍어 둬요. 그리고 두 장을 비교해서 "어, 이 줄이 바뀌었네"를 보여 줍니다. 그게 `git diff`예요. 그러면 의문이 들죠. "아니, 사진을 매번 다 찍으면 디스크가 폭발하지 않나요?" 좋은 질문입니다. Git은 아주 영리한 알고리즘이 있어서, 바뀐 부분만 새로 저장하고 안 바뀐 부분은 이전 사진 파일을 가리키는 포인터만 남겨요. 그래서 1만 개 commit이 있어도 디스크는 가벼워요. 본인 자경단 레포가 1년 뒤에 commit 5,000개 짜리 거대한 앨범이 되어도, 실제 디스크는 50MB 정도예요. 영리한 거죠. 자, 사진앨범 비유를 한 번 더 확장합시다. 사진앨범은 혼자 보면 슬프잖아요. 친구한테 보여 주고, 가족한테 보여 주고, 인스타에 올리죠. 그게 GitHub이에요. **GitHub은 본인 사진앨범을 구름(클라우드)에 백업해 주고, 친구들과 같이 보고 같이 편집할 수 있게 해 주는 서비스예요.** 본인 앨범의 백업본이 GitHub.com에 항상 있고, 친구를 초대하면 그 친구가 본인 앨범의 사본을 자기 노트북에 받아 가서 새 사진을 추가하고 다시 올릴 수 있어요. 본인이 잠자는 동안에도 친구가 시드니에서 새 commit을 올리면 다음 날 본인이 깨서 `git pull` 한 줄 치면 그 사진이 본인 노트북으로 내려와요. 지구 반대편 사람과 같은 폴더를 같이 편집하는 마법. 그게 Git + GitHub의 진짜 힘입니다. 한 가지 더 짚고 가요. Git은 2005년에 리누스 토르발스가 만들었어요. 리눅스 커널 개발자들이 쓰던 BitKeeper라는 도구가 라이선스 분쟁으로 갑자기 못 쓰게 되자, 리누스가 "그러면 내가 2주 만에 새로 짜겠다"고 말하고 정말로 2주 만에 첫 버전을 짰어요. 그게 Git입니다. 처음부터 "수천 명이 동시에 같은 폴더를 편집하는 시나리오"를 상정하고 만들어졌기 때문에, 분산이고 빠르고 충돌 처리가 정교해요. 20년이 지난 지금 지구상 거의 모든 코드가 Git 위에 있습니다. 다른 비유로 한 번 더 해 볼게요. Git은 **타임머신**이에요. `git log`로 과거를 보고, `git checkout <sha>`로 그 시점으로 돌아가고, 잘못된 commit을 `git revert`로 무효화하고, 새 가지(branch)를 만들어 평행 우주를 열어 놓고 실험하다가 마음에 안 들면 그 우주를 통째로 버려요. 사진앨범 + 타임머신 + 분산 백업 + 협업 도구. 이 네 가지가 한 도구 안에 다 들어가 있어요. 그래서 Git은 처음에 어렵게 느껴집니다. 한 도구가 네 일을 동시에 하니까요. Ch004 8시간이 사실 이 네 일을 하나씩 풀어 보는 시간이에요. 1교시는 사진앨범 그림 한 장만 머리에 박으시면 됩니다. 나머지는 뒤에 쌓입니다.

한 가지 더 짚고 갈 게 있어요. Git을 처음 배울 때 많은 분이 빠지는 함정이 있습니다. **"명령어를 외우면 Git을 안다"**는 착각이에요. 아닙니다. Git의 모델 — 사진앨범 + 가지 + 원격 백업 — 이 머리에 들어와 있어야 명령어가 의미를 가져요. 모델 없이 명령어만 외우면, `git rebase -i HEAD~3` 같은 한 줄에서 손이 멈추고 무서워집니다. 모델이 있으면 "아, 최근 사진 세 장을 한 장으로 합치는 거구나"가 바로 떠올라요. 그래서 1교시는 일부러 명령어 앞에 비유를 먼저 놓습니다. 사진앨범, 평행우주, 구름 백업. 이 세 그림이 머리에 박힌 다음에 명령어가 들어가면, 명령어는 그 그림 위의 짧은 단어가 됩니다. 무서울 일이 없어져요.

## 3. 한 commit의 0.05초 — 미리 보기

이제 미리보기를 한 번 보여 드릴게요. 외울 필요 없어요. "아, 이런 일이 일어나는구나"라는 감각만 잡으시면 됩니다. 본인이 `git commit -m "fix: 고양이 카드 정렬"` 한 줄을 칠 때, 0.05초 안에 일어나는 일을 7단계로 잘라 봅시다. **1단계 — 변경 사항 스테이징 확인.** Git은 먼저 본인이 이전에 `git add`로 스테이징해 둔 파일들이 뭐였는지를 확인해요. 스테이징은 "이번 사진에 이 파일들을 넣겠다"라는 예약입니다. **2단계 — 스냅샷 생성.** 스테이징된 파일들의 내용을 SHA-1 해시로 변환해서 `.git/objects/` 안에 저장합니다. 같은 내용이면 같은 해시가 나오고, 이미 있으면 새로 안 만들어요. 이게 Git이 디스크를 절약하는 비법이에요. **3단계 — 트리 객체 생성.** 폴더 구조를 표현하는 tree 객체가 만들어져요. "이 폴더 안에 이런 파일들이 이런 해시로 들어 있다"는 목록입니다. **4단계 — commit 객체 생성.** tree 객체 + 부모 commit + 본인 이름·이메일 + 시각 + 메시지가 합쳐져서 commit 객체 한 개가 만들어져요. 이 commit 객체도 SHA-1 해시 한 개로 식별됩니다. `a1b2c3d`처럼 짧게 보이는 그 7글자가 이거예요. **5단계 — HEAD 갱신.** 현재 브랜치(예: main)가 가리키는 곳을 새로 만든 commit으로 옮깁니다. 한 줄짜리 텍스트 파일 `.git/refs/heads/main` 안의 sha 값이 새로 쓰여요. **6단계 — reflog 기록.** "방금 본인이 commit 했다"는 기록이 `.git/logs/` 안에 한 줄 추가됩니다. 30일 동안 보관돼요. 이게 나중에 잘못한 일을 복구할 때 생명줄이 됩니다. **7단계 — 화면 출력.** "[main a1b2c3d] fix: 고양이 카드 정렬 / 1 file changed, 3 insertions(+), 1 deletion(-)"이 터미널에 뜹니다. 0.05초. 진심 빠르죠. 그리고 이 7단계가 다 본인 노트북 안에서 끝나요. 인터넷 안 씁니다. 이게 Git의 핵심 특징이에요. **분산이라서 오프라인에서도 다 됩니다.** 비행기 안, 산속, 와이파이가 끊긴 카페 — 어디서든 commit이 됩니다. GitHub에 올라가는 건 본인이 `git push`를 칠 때예요. 그제서야 Ch003에서 배운 7단계(DNS·TCP·TLS·HTTP·...)가 발동합니다. 그래서 Git과 GitHub은 분리해서 보셔야 해요. **Git = 본인 노트북 안의 사진앨범**, **GitHub = 그 앨범의 구름 백업**. 둘이 같은 거 아닙니다. Ch004 8시간이 사실 이 둘의 경계를 정확히 그리는 시간이에요. H1은 그림, H2는 개념 사전, H3은 환경 점검(설치·SSH 키), H4는 명령어 카탈로그, H5는 demo로 직접 미니 레포 만들기, H6은 트러블슈팅(충돌·detached HEAD·.gitignore 빠뜨림·force push 사고), H7은 Git 내부(object 모델·.git 디렉터리·packfile·refs), H8은 자경단 레포 운영 적용+회고. 산이 작지 않아 보이지만 한 시간 한 시간 가면 됩니다. 오늘은 지도만 펴면 돼요.

## 4. 핵심 단어 네 개 — Repository·Commit·Branch·Remote 첫 만남

이제 단어 네 개를 더 자세히 봅시다. Ch004 8시간 동안 100번도 더 들으실 단어예요. 첫 만남이니 한 단어당 한 문단씩만 합시다.

**Repository(레포지토리, 줄여서 repo).** 한 폴더 + 그 폴더 안의 `.git/` 디렉터리. 이 둘을 합쳐서 repo라고 부릅니다. 본인이 `git init`을 친 그 폴더가 repo가 돼요. `.git/` 안에 사진앨범 전체(모든 commit, 모든 branch, 모든 설정)가 다 들어 있어요. 그래서 `.git/` 폴더 하나만 통째로 USB에 복사해서 다른 노트북에 가져가도 본인 사진앨범이 다 살아 있습니다. 무섭게 강력해요. 그리고 repo는 두 종류로 나뉩니다. **로컬 repo**는 본인 노트북 안의 폴더, **원격 repo**는 GitHub.com 위의 사본이에요. 같은 repo가 여러 곳에 사본으로 존재합니다. 본인 노트북에 한 개, GitHub에 한 개, 동료 노트북에 한 개. 이 사본들이 서로 push/pull로 동기화돼요. 분산이라는 말이 이거예요. 중앙 서버가 죽어도 사본 하나만 살아 있으면 전체 역사가 복원됩니다.

**Commit(커밋).** 사진 한 장. 본인이 `git commit`을 한 번 치면 commit 객체 한 개가 생기고, 그 commit은 SHA-1 해시 40글자(보통 7글자로 줄여서 표시)로 식별돼요. 한 commit에는 다섯 가지가 들어 있어요. (1) 그 시점 폴더 전체의 스냅샷, (2) 부모 commit의 해시(첫 commit 빼고), (3) 작성자 이름·이메일, (4) 작성 시각, (5) 메시지("fix: 고양이 카드 정렬" 같은 한 줄). 부모-자식 관계로 commit이 줄줄이 이어지면 그게 history예요. `git log`로 그 줄을 봅니다. 그래서 commit 메시지는 신중히 쓰세요. 1년 뒤 본인이, 또는 동료가, 그 한 줄을 보고 "이 commit이 왜 있었지"를 추측해야 하니까요. **"한 commit = 한 의도"** — H4·H8에서 commit 메시지 컨벤션(Conventional Commits)을 깊게 봅니다. 지금은 "사진 한 장 = commit 한 개 = 의도 한 개"라는 그림만 가져가세요.

**Branch(브랜치).** 평행 우주. 본인이 메인 사진앨범에서 가지 하나를 쳐 내서 별도 우주를 열고, 거기서 새 시도를 해요. main 브랜치는 안전한 운영 우주, feature 브랜치는 실험 우주, hotfix 브랜치는 응급 수술 우주. 각 우주에서 commit을 쌓다가, 결과가 마음에 들면 main 우주로 합치고(merge), 마음에 안 들면 그 우주를 통째로 버립니다. 브랜치는 사실상 공짜예요. `git branch new-feature` 한 줄이 0.001초고, 디스크는 한 줄짜리 sha 파일 하나만 더 만들어집니다. 그래서 자주, 가볍게, 부담 없이 브랜치를 따세요. 한 가지 작업 = 한 브랜치. 이게 H4·H5에서 손에 박힙니다. 지금은 "브랜치 = 평행 우주, 공짜로 따고 공짜로 버린다"는 그림만 가져가세요.

**Remote(원격).** 본인 사진앨범의 사본이 사는 다른 장소. 보통 GitHub.com입니다. `git remote -v`를 치면 본인 repo가 어떤 원격을 알고 있는지 보여 줘요. 관습적으로 첫 원격의 이름을 `origin`이라고 붙입니다. `git push origin main`은 "내 main 브랜치를 origin이라는 원격의 main 브랜치로 올려라"라는 뜻이에요. `git pull origin main`은 그 반대고요. 원격은 GitHub만이 아닙니다. GitLab, Bitbucket, 자체 호스팅 Gitea, 또는 회사 내부 서버. 같은 repo가 여러 원격을 가질 수도 있어요. 자경단 사이트는 두 원격을 씁니다. GoGoComputer/react-flask-ai-stack(원본)과 catguard-team/react-flask-ai-stack(미러). 매 commit이 두 곳에 다 올라가요. H8에서 미러 워크플로를 봅니다.

자, 네 단어가 다 나왔어요. 외우려고 애쓰지 마세요. 8시간 동안 한 단어씩 손에 들어옵니다. 지금은 "이 네 개가 Git의 네 기둥이다"라는 그림만 가져가세요. 그리고 한 가지 더 짚고 넘어갈 게 있어요. 이 네 단어는 서로 겹쳐 있습니다. **Repo 안에 commit들이 있고, commit들이 branch로 묶이고, branch들이 remote로 동기화돼요.** 계층 구조입니다. 안쪽부터 commit → branch → repo → remote. 이게 Git의 4층 구조예요. H7에서 `.git/` 디렉터리를 직접 열어 보면서 이 4층이 디스크에 어떻게 누워 있는지를 봅니다. 지금은 "네 단어가 공중에 떠 있는 게 아니라 양파처럼 겹쳐 있다"는 그림만 머리에 둬도 충분합니다.

## 5. git status / git log — 첫 번째 손풀기 두 줄

이제 손을 풉시다. 노트북을 여세요. 터미널을 여세요. 본인이 이미 어떤 폴더든 git repo로 작업한 적이 있다면 그 폴더로 들어가시고, 없으시면 다음 절(6번)에서 같이 만들 거예요. 우선 다음 두 줄만 가볍게 봅시다.

> ▶ **같이 쳐보기** — 사진앨범 두 줄 진단: 지금 상태와 최근 사진 10장
>
> ```bash
> git status
> git log --oneline | head -10
> ```

`git status`는 "지금 사진앨범 상태가 어떤가"를 묻는 명령어예요. 출력은 보통 세 가지 정보예요. (1) 현재 어느 브랜치에 있는가 — "On branch main" 같은 한 줄. (2) 원격과 비교해서 앞서 있나 뒤처져 있나 — "Your branch is up to date with 'origin/main'". (3) 변경된 파일이 있는가 — "nothing to commit, working tree clean" 또는 "Changes not staged for commit". 빨간 글씨가 보이면 "스테이징되지 않은 변경"이고, 초록 글씨가 보이면 "스테이징된 변경"이에요. 색깔 두 개가 Git의 작업 흐름을 압축한 그림입니다. 빨강(작업)→초록(스테이징)→commit→push. 네 단계예요. `git status`는 본인이 지금 그 네 단계 어디에 있는지를 보여 줘요. 하루에 100번 치게 됩니다. 무서운 명령어가 아니에요. 그냥 "어디 있어?"라고 묻는 거예요.

`git log --oneline`은 "사진앨범에 어떤 사진이 들어 있나"를 보는 명령어예요. 한 줄에 한 commit이 보입니다.

```
a1b2c3d (HEAD -> main, origin/main) Ch003 H8: new 17k chars + Ch003 완료
f81d70c Ch003 H7: new 17k chars (서버측서버)
04100b9 Ch003 H6: new 17k chars (운영관리)
0d5dc8e Ch003 H5: new 17k chars (데모)
...
```

왼쪽이 commit의 짧은 sha(7글자), 오른쪽이 commit 메시지. 첫 줄의 `(HEAD -> main, origin/main)`은 "지금 본인이 main 브랜치 끝에 있고, 그 위치가 GitHub의 main과 일치한다"는 뜻이에요. 이 한 줄이 본인의 현재 좌표예요. 우편 주소처럼. `git log` 옵션은 정말 많아요. `--graph`를 붙이면 브랜치가 그림으로 보이고, `--all`을 붙이면 모든 브랜치가 같이 보이고, `--author=mo`를 붙이면 본인 commit만 필터됩니다. H4에서 한 시간 동안 `git log` 옵션만 봐요. 정말 강력합니다. 지금은 "git log = 사진앨범 목차"라는 한 줄만 가져가시면 돼요. 한 가지 포인트 — `git status`와 `git log` 둘 다 **읽기만** 하는 명령어예요. 무엇도 바꾸지 않습니다. 그래서 안전해요. Git이 무서워질 때 이 두 줄을 먼저 치세요. 아무것도 망가지지 않으니까요.

## 6. git init + git add + git commit — 두 번째 손풀기 (3분 데모)

이번에는 진짜로 새 사진앨범을 한 권 만들어 봅시다. 3분 걸려요. 다음 다섯 줄을 그대로 따라치세요.

> ▶ **같이 쳐보기** — 5줄로 첫 사진앨범 한 권: init → add → commit
>
> ```bash
> mkdir cat-hello && cd cat-hello
> git init
> echo "# 자경단 첫 커밋" > README.md
> git add README.md
> git commit -m "first: README"
> ```

한 줄씩 풀어 볼게요. **첫 줄.** `cat-hello`라는 새 폴더를 만들고 그 안으로 들어갑니다. 이름은 아무거나 좋아요. **둘째 줄.** `git init`. 이 한 줄이 마법이에요. 빈 폴더에 `.git/`이라는 숨김 폴더가 생기고, 그 폴더 안에 사진앨범의 뼈대가 깔립니다. `ls -la`를 쳐 보시면 `.git/` 폴더가 보일 거예요. 그 안에 `objects/`, `refs/`, `HEAD`, `config` 같은 파일들이 있고, 그게 본인 사진앨범의 골격입니다. 이 한 줄이 "이 폴더는 이제 Git이 관리한다"는 선언이에요. **셋째 줄.** `README.md`라는 파일에 "# 자경단 첫 커밋" 한 줄을 쓰는 일반 쉘 명령어예요. Git과 직접 관련은 없지만, 사진을 찍으려면 먼저 찍을 게 있어야 하잖아요. 첫 파일을 만든 거예요. **넷째 줄.** `git add README.md`. "다음 사진에 이 파일을 넣어 두세요"라는 예약입니다. 스테이징이라고 합니다. `git status`를 다시 쳐 보시면 README.md가 빨강에서 초록으로 변해 있을 거예요. **다섯째 줄.** `git commit -m "first: README"`. 사진을 찍습니다. `-m`은 메시지를 같이 쓰는 옵션이고, 큰따옴표 안의 글자가 commit 메시지예요. 엔터를 누르면 한 줄짜리 출력이 떠요.

```
[main (root-commit) abc1234] first: README
 1 file changed, 1 insertion(+)
 create mode 100644 README.md
```

**축하합니다, 본인의 첫 commit이 만들어졌습니다.** `(root-commit)`은 "이게 사진앨범의 첫 사진이라 부모 commit이 없다"는 뜻이에요. abc1234가 그 commit의 sha. 이제 `git log --oneline`을 쳐 보세요. 한 줄이 보일 거예요. 사진 한 장이 든 앨범이 한 권 생긴 겁니다. 본인 손으로요. 이 5줄이 Git의 가장 기본 흐름이에요. **새 폴더 → git init → 파일 만들기 → git add → git commit.** 이걸 여러 번 반복하면 사진이 쌓이고, history가 만들어지고, 그게 본인 프로젝트의 시간 축이 됩니다. H5에서 이 폴더를 더 키워서 GitHub에 push하고 브랜치를 따고 충돌도 일부러 만들어 봐요. 1교시는 "5줄로 첫 commit이 된다"는 자신감 한 개만 가져가시면 돼요. Git은 어렵지 않습니다. 5줄이 시작이에요.

## 7. 사진앨범 비유 확장 — snapshot·HEAD·branch=평행우주·remote=구름 백업

자, 비유를 더 깊이 갑시다. Ch003에서 패킷=봉투, 포트=객실, 소켓=통화 회선, TLS=봉인 도장이라는 4단어 비유가 8시간 내내 따라다녔잖아요. Ch004도 똑같이 5단어 비유 사전을 깔겠습니다. 노트 첫 페이지에 적어 두세요.

**(1) Snapshot = 사진 한 장.** Git이 매 commit마다 폴더 전체의 사진을 찍는다고 했죠. 그 사진이 snapshot이에요. 다른 버전 관리 도구(SVN, Mercurial, Perforce)는 "변경 사항만 저장"하는 모델이지만, Git은 "매번 전체 폴더 스냅샷"을 저장해요. 무겁지 않냐고요? 아까 말한 영리한 알고리즘 — 같은 내용의 파일은 같은 sha로 한 번만 저장 — 덕분에 가벼워요. 이 모델 덕분에 Git은 빠릅니다. SVN처럼 "이 파일의 1년 전 버전을 보여 줘"가 디스크 한 번 읽기로 끝나요.

**(2) HEAD = 본인이 지금 보고 있는 사진의 위치.** 사진앨범을 펴면 "지금 몇 페이지를 보고 있는가"가 있잖아요. 그게 HEAD예요. 보통 HEAD는 어떤 브랜치(main, develop 등)를 가리키고, 그 브랜치가 그 브랜치의 가장 최근 commit을 가리켜요. `git log`의 첫 줄에 `(HEAD -> main)`이 보이는 게 그거예요. HEAD를 직접 다른 commit으로 옮길 수도 있어요(`git checkout <sha>`). 그러면 "이 사진을 지금 보고 있다"는 표시가 그 sha로 옮겨집니다. 이걸 detached HEAD 상태라고 해요. H6에서 이 상태에서 빠져나오는 법을 깊게 봅니다. 처음 만나면 무서운 메시지지만 알면 별것 아니에요.

**(3) Branch = 평행 우주.** 같은 코드베이스를 여러 갈래로 동시에 진행할 수 있어요. main 브랜치에서는 "운영에 나가는 안정 코드", feature/cat-card 브랜치에서는 "고양이 카드 새 디자인 실험", hotfix/dns-bug 브랜치에서는 "긴급한 DNS 버그 수정". 세 우주가 동시에 살아 있어요. 한 우주에서 commit을 쌓다가 결과가 좋으면 main으로 합치고(merge 또는 rebase), 결과가 나쁘면 그 우주를 통째로 지웁니다(`git branch -D`). 우주는 1초에 만들어지고 1초에 사라져요. 그래서 부담 없이 따세요. "한 작업 = 한 브랜치"가 황금률입니다.

**(4) Remote = 구름 백업.** 본인 노트북이 화재로 타거나 디스크가 죽거나 노트북을 잃어버려도, 본인 사진앨범은 GitHub.com에 있어요. 구름 백업입니다. 게다가 친구들이 본인 앨범을 받아 가서 새 사진을 추가하고 다시 올려요. 친구가 시드니에서 새 commit을 push하면, 본인이 서울에서 `git pull` 한 줄로 그 사진을 받아옵니다. 이게 협업이에요. 그리고 remote는 한 개가 아니어도 됩니다. GitHub + GitLab + 자체 서버 — 세 곳에 동시에 백업할 수 있어요. 자경단 사이트가 정확히 그렇게 운영됩니다. GoGoComputer 본 저장소 + catguard-team 미러. 이중 백업이에요.

**(5) .git/ = 사진앨범 그 자체.** 본인 폴더 안에 숨겨진 `.git/` 디렉터리가 사실상 본인 사진앨범의 본체예요. 그 안에 모든 commit, 모든 branch, 모든 설정이 다 들어 있어요. `.git/` 폴더만 통째로 USB에 복사하면 본인 앨범이 다 살아 있고, 반대로 본인 작업 파일만 복사하고 `.git/`을 빠뜨리면 사진앨범은 잃어버립니다. **그래서 절대로 `.git/` 폴더를 손으로 만지지 마세요.** 그 안의 한 글자만 잘못 바꿔도 앨범 전체가 깨질 수 있어요. Git의 모든 명령어가 이 폴더를 안전하게 다루는 추상화입니다. H7에서 이 폴더를 같이 열어 봐요. 무섭지 않습니다. 그냥 "구조를 안다"가 목표예요.

다섯 단어 — snapshot, HEAD, branch, remote, .git — 가 Ch004의 5비유 사전입니다. Ch003의 5비유(봉투·객실·회선·도장·복도)와 같은 무게로 8시간 동안 따라다닐 거예요. 노트 첫 페이지에 적어 두시고, 어렵게 느껴질 때마다 돌아와 보세요.

## 8. Ch004 8시간 큰 그림

이제 8시간 지도를 펼칩니다. H1부터 H8까지 한 줄씩.

- **H1 — 오리엔테이션 (이 시간).** 사진앨범 비유, 4단어, 5비유 사전, 두 줄 손풀기, 5줄 미니 commit.
- **H2 — 핵심 개념 4 깊게.** Snapshot의 내부 구조, commit 객체의 다섯 필드, branch가 사실 한 줄짜리 텍스트 파일이라는 충격, HEAD의 두 모드(attached/detached), remote와 origin의 관계, fast-forward vs three-way merge, rebase vs merge 첫 만남.

## 9. 고양이 자경단 다리 — 코드 한 줄을 운영 서버까지 보내는 7단계

이제 비유와 8시간 지도를 자경단 사이트에 붙여 봅시다. 본인이 자경단 사이트의 "고양이 신고 폼"에서 버튼 색깔을 파랑에서 초록으로 바꾸는 작업을 한다고 칩시다. 한 줄짜리 변경이에요. CSS 한 줄. 그 한 줄이 본인 노트북에서 운영 서버까지 가는 길은 7단계예요.

**1단계 — 본인 노트북의 작업 트리.** 에디터로 `static/style.css` 열고 `background: #00aaff;` 를 `background: #00cc66;` 으로 바꿉니다. 파일이 디스크에 저장돼요. 이때 Git은 "어, 이 파일 바뀐 게 보이네"라고 감지해요. `git status`로 빨간 글씨가 떠요.

**2단계 — 스테이징.** `git add static/style.css`. 빨강이 초록으로 변해요. 이번 사진에 이 변경을 넣겠다는 예약이에요.

**3단계 — Local commit.** `git commit -m "feat: 신고 버튼 컬러 초록으로"`. 본인 노트북 안에 사진 한 장이 찍혀요. `.git/objects/`에 새 파일이 생기고, `.git/refs/heads/feature-button-color`(또는 main)이 가리키는 sha가 갱신됩니다. 0.05초.

**4단계 — Push.** `git push origin feature-button-color`. 이때부터가 Ch003의 무대예요. 본인 노트북이 GitHub.com에 DNS 조회하고, TCP 핸드셰이크하고, TLS 핸드셰이크하고, HTTP/2 위에 git protocol을 얹어 새 commit 객체를 보냅니다. 200밀리초쯤. GitHub은 받은 객체를 자기 디스크에 저장하고 "OK, 받았어요"를 답해요.

**5단계 — Pull Request 생성.** GitHub.com에 가서 "Compare & pull request" 버튼을 눌러요. 본인의 feature-button-color 브랜치를 main으로 합쳐 달라는 요청이 만들어집니다. PR 본문에 "왜 바꿨는지, 어떻게 테스트했는지"를 적어요. 두 줄이면 충분해요.

**6단계 — 코드 리뷰 + CI.** 동료가 PR을 열어 보고 "버튼 hover 색깔도 같이 바꿔야 일관돼"라는 코멘트를 답니다. 동시에 GitHub Actions가 자동으로 돌아서 lint, test, build가 다 통과하는지 확인해요. 빨간 X가 뜨면 수정, 초록 체크가 뜨면 다음 단계.

**7단계 — Merge + 자동 배포.** 리뷰 OK, CI OK가 되면 "Squash and merge" 버튼을 눌러요. 본인 commit이 main 브랜치로 합쳐지고, 그 순간 GitHub Actions의 deploy job이 발동해서 운영 서버로 새 코드가 올라갑니다. 5분 뒤, 자경단 사이트 방문자의 신고 버튼이 초록으로 바뀌어 있어요.

**한 줄짜리 CSS 변경이 7단계를 거쳐서 운영에 나갑니다.** 처음엔 이게 너무 길어 보여요. "그냥 서버에 ssh 들어가서 한 줄 바꾸면 안 되나"라는 마음이 듭니다. 안 됩니다. 그러면 사고가 납니다. 7단계가 있는 이유는 (1) 본인이 잘못 바꾼 걸 동료가 잡고, (2) 자동 테스트가 회귀를 잡고, (3) 사고가 나도 commit 한 개 단위로 되돌릴 수 있고, (4) 1년 뒤 누구라도 "왜 이 색깔이 됐지"를 history로 추적할 수 있게 하기 위해서예요. **이 7단계가 사실상 모든 회사 개발팀의 일과예요.** Git을 배우는 진짜 이유가 이거예요. 한 줄 바꾸기 위해 7단계를 칠 줄 안다 = 회사가 본인을 신입으로 뽑을 자격이 됐다. Ch004 8시간이 사실 이 7단계를 본인 손에 익히는 시간이에요. H8 끝에 본인이 자경단 레포에 첫 PR을 보내고 머지가 되면, 본인은 이미 백엔드 개발자의 한 발자국을 디딘 거예요.

## 11. macOS ↔ Linux 변환표(Git 명령은 사실상 동일)

Ch001~Ch003에서는 OS별 명령어가 다 달랐어요. `ifconfig` vs `ip addr`, `lsof` vs `ss`, `brew` vs `apt`. Git은 다릅니다. **Git은 모든 OS에서 명령어가 똑같아요.** 이게 Git의 아름다움 중 하나예요. 본인이 macOS에서 배운 한 줄이 Linux 서버에서 그대로 돌고, Windows의 Git Bash에서도 그대로 돌고, 클라우드 셸에서도 그대로 돕니다. 다른 거라곤 설치 방법뿐이에요.

| 작업 | macOS | Linux | Windows |
|---|---|---|---|
| 설치 | `xcode-select --install` 또는 `brew install git` | `sudo apt install git` 또는 `sudo yum install git` | git-scm.com에서 설치 후 Git Bash 사용 |
| 설정 위치 | `~/.gitconfig` | `~/.gitconfig` | `C:\Users\본인\.gitconfig` |
| SSH 키 위치 | `~/.ssh/id_ed25519` | `~/.ssh/id_ed25519` | `C:\Users\본인\.ssh\id_ed25519` |
| 에디터 열기 | `open ~/.gitconfig` | `xdg-open ~/.gitconfig` | `notepad %USERPROFILE%\.gitconfig` |
| 모든 git 명령어 | 100% 동일 | 100% 동일 | 100% 동일 |

이게 Git이 지구상 어떤 환경에서도 일관되게 돌아가는 이유예요. 본인이 한번 배우면 평생 어디서나 같은 도구. 인생이 편해집니다.

## 12. 흔한 오해 다섯 개

1교시 끝에 흔한 오해를 다섯 개 부숩니다. 노트 한 페이지로 옮겨 두세요.

**오해 1 — "Git과 GitHub은 같은 거다."** 아닙니다. **Git은 본인 노트북 안의 도구**, **GitHub은 GitHub Inc(현재 Microsoft)가 운영하는 웹사이트**예요. Git 없이 GitHub은 못 쓰지만, GitHub 없이 Git은 잘 씁니다. 인터넷 끊긴 비행기 안에서도 Git commit은 됩니다. GitHub이 다운돼도(가끔 다운돼요) 본인 로컬 작업은 안 멈춰요. 둘을 헷갈리면 트러블슈팅이 막혀요. "GitHub이 안 되니까 commit이 안 돼"는 거짓말입니다. 그냥 push만 안 되는 거예요.

**오해 2 — "force push는 절대 쓰면 안 된다."** 절반만 맞아요. main 같은 공유 브랜치에 force push하면 동료의 작업이 사라질 수 있어서 위험합니다. 하지만 본인 개인 브랜치(feature/내이름)에서는 rebase 후 force push가 일상이에요. 위험한 명령어와 정상 명령어를 구분하는 법은 H4·H6에서 배워요. "절대 안 됨"이 아니라 "어디서 어떻게 쓰면 안 되는가"입니다.

**오해 3 — "commit 메시지는 'fix' 한 단어면 충분하다."** 거짓이에요. 1년 뒤 본인이, 또는 동료가 그 한 단어 보고 "이게 뭘 고쳤는지" 추측 못 합니다. **"fix: 신고 버튼 hover 색상 누락 (#234)"** 이 정도가 표준이에요. 동사+한 줄 요약+이슈 번호. H8에서 Conventional Commits 컨벤션을 깊게 봅니다.

**오해 4 — "브랜치는 무겁고 비싸다."** 정반대예요. **브랜치는 한 줄짜리 텍스트 파일**입니다(40글자 sha 한 개). 1초에 만들어지고 1초에 사라져요. 부담 없이 자주 따세요. 한 작업 = 한 브랜치. 이게 안 되면 main에 작업을 쌓다가 사고가 납니다.

**오해 5 — ".git 폴더를 손으로 열어 보면 안 된다."** **읽는 건 안전합니다.** 손으로 수정하는 게 위험한 거예요. H7에서 같이 열어 봅니다. `cat .git/HEAD`, `cat .git/refs/heads/main`, `ls .git/objects/`. 읽기는 100% 안전. 수정만 안 하면 됩니다. 안을 한 번 보면 Git이 마법에서 정직한 데이터베이스로 변해요.

**오해 보너스 — "git pull은 항상 안전하다."** 아니에요. `git pull`은 사실 `git fetch + git merge` 두 단계의 합쳐진 명령어예요. 본인 로컬에 commit이 쌓여 있는데 pull을 치면 자동 merge가 시도되고, 충돌이 나면 그 자리에서 본인 작업이 어중간한 상태가 됩니다. 그래서 안전한 패턴은 `git fetch` 먼저, `git log origin/main`으로 뭐가 새로 왔는지 확인, 그다음 `git merge` 또는 `git rebase`예요. "그냥 pull 치면 다 알아서 해 주겠지"는 사고의 시작이에요.

## 13. FAQ 다섯 개

**Q1. Git이 너무 어려운데 어디서 시작해야 하나요?** 1교시에 친 5줄로 시작하셨어요. `mkdir` → `git init` → `echo` → `git add` → `git commit`. 이 5줄이 Git의 75%를 보여 줍니다. 거기서 한 줄씩 깊어지시면 됩니다. 하루 한 줄이 한 달이면 30줄이고, 그 30줄이면 운영 사고 90%가 머리에 들어와요. Git은 산이지만 한 칸씩 오르면 됩니다.

**Q2. GUI 도구(SourceTree, GitKraken, GitHub Desktop)를 쓰면 안 되나요?** 써도 됩니다. 다만 처음에는 터미널에서 한 번 손에 박아 두세요. GUI는 결국 한 줄짜리 git 명령어를 그림으로 보여 주는 거예요. 명령어가 머리에 있으면 어떤 GUI든 5분이면 익어요. 명령어 없이 GUI만 외우면 GUI가 바뀔 때마다 처음부터 다시 배워야 해요. **터미널 → 명령어 → GUI** 순서가 정답입니다.

**Q3. Git 책 한 권 추천해 주세요.** 두 권. 첫 권은 *Pro Git*(Scott Chacon, 무료 온라인). git-scm.com/book에 있어요. 두 번째 권은 본인의 .git 디렉터리예요. H7 시간에 그 폴더를 같이 열어 보면 책 한 권보다 더 큰 게 들어옵니다. 책 사기 전에 본인 노트북에 이미 책 한 권이 깔려 있어요.

**Q4. 회사에 들어가면 Git 다르게 쓰나요?** 큰 틀은 같아요. 다만 "한 PR = 한 작업, 리뷰 2명 이상, CI 통과 필수, squash merge" 같은 회사별 컨벤션이 붙어요. 그래서 새 회사 첫 주에는 그 회사의 CONTRIBUTING.md를 읽고 한 번씩 try-and-error로 배우게 됩니다. 본인이 Ch004에서 깐 기초가 있으면 그 컨벤션 학습이 1주가 아니라 1일로 줄어요.

**Q5. GitHub 아닌 다른 서비스도 배워야 하나요?** 일은 같아요. GitLab, Bitbucket, Gitea, AWS CodeCommit — 다 같은 git protocol입니다. UI만 살짝 달라요. GitHub 한 번 익히면 다른 서비스는 30분이면 적응합니다. 이 코스에서는 GitHub에 집중해요. GitHub이 가장 큰 생태계고 자경단 사이트의 본거지니까요.

**Q보너스. 따곳이 안 잡히는데요?** 정상이에요. 1교시는 비유 주입 시간이라 "아직 손에 안 잡혀"가 자연스럽습니다. H4에서 명령어 23개를 손에 쥐고 H5에서 미니 레포 30단계를 본인 손으로 따라치면 "아, 이게 이건가"가 한꺼번에 터져요. 1교시에 100% 가져가는 건 사진앨범 비유 한 장과 단어 네 개뿐입니다. 나머지는 뒤에 쌓입니다. 조급해하지 마세요.

## 14. 본인 페이스 가이드 + 다음 시간 예고

Ch004는 Ch003보다 약간 쉽게 느껴질 거예요. Git은 본인 노트북 안에서만 도는 도구라 변수가 적습니다. 회사 와이파이, 카페 와이파이, 모바일 핫스팟 — Ch003에서 골치였던 환경 차이가 Ch004엔 거의 없어요. 그래서 페이스가 빨라요. 평균 4~6주에 끝내시는 분이 많아요. 빠른 분은 2주, 천천히 가시는 분은 8주. 본인 페이스대로 가시면 됩니다. 한 가지 차이는 Ch004는 "꾸준함"이 학습의 일부예요. 하루 한 commit. 그게 한 달이면 30 commit이고, 한 commit씩 본인 손에 박힙니다. Git은 외우는 게 아니라 손에 박는 도구예요. 다음 시간 예고. **H2 = 핵심 개념 4 깊게**입니다. Snapshot의 내부 구조, commit 객체의 다섯 필드, branch가 한 줄짜리 텍스트 파일이라는 충격, HEAD의 두 모드, fast-forward vs three-way merge, rebase vs merge 첫 만남. H2가 끝나면 `git log --graph`의 그림이 80% 풀려요. 본인 페이스에 맞게 천천히 가시면 됩니다. 한 가지 약속을 드리면 — 이번 시간과 다음 시간 사이에 5줄짜리 미니 commit을 한 번만 더 쳐 보세요. 같은 5줄이지만 두 번째 칠 때는 처음과 다르게 보이실 거예요. 그게 학습이에요.

## 15. 한 줄 정리 + 두 해 코스 위치

한 줄 정리. **Git은 코드 폴더의 사진앨범이고, GitHub은 그 앨범의 구름 백업이며, 우리는 1교시에 5줄로 첫 commit을 찍고 4단어(repo·commit·branch·remote)와 5비유 사전(snapshot·HEAD·branch·remote·.git)을 손에 쥐었다.** 두 해 코스 우리 위치는 25/960 = 2.60%입니다. Ch003 끝에서 2.50%였고, 한 시간을 더 했으니 2.60%. 한 시간이 0.1%를 더해요. 작아 보이지만 한 달이 4~5%, 두 해가 100%. 매주 1%씩 자라는 복리예요. 1년 뒤엔 다른 사람이 됩니다. 자, Ch004 H1은 여기서 닫습니다. 다음 H2 — 핵심 개념 4 깊게 — 에서 만나요. 그동안 5줄을 한 번 더 쳐 두세요. `mkdir test-repo && cd test-repo && git init && echo "test" > a.txt && git add a.txt && git commit -m "first"`. 8시간 같이 가실 분, 환영합니다. 수고 많으셨습니다.

## 👨‍💻 개발자 노트 (참고 — 비개발자는 그냥 넘기셔도 됩니다)

> - Git 모델 = content-addressable filesystem. SHA-1(이후 SHA-256으로 마이그 진행) 해시로 모든 객체 식별.
> - 4종 객체: blob(파일 내용), tree(폴더 목록), commit(스냅샷+메타), tag(태그 메타). H7에서 `git cat-file -p <sha>`로 직접 봄.
> - branch = 한 줄짜리 텍스트 파일 `.git/refs/heads/<name>` 안의 sha 한 개. 무겁다는 인상은 다른 VCS 잔재.
> - HEAD = `.git/HEAD` 안의 한 줄(`ref: refs/heads/main` 또는 직접 sha = detached).
> - reflog = 30일 보관(`gc.reflogExpire` 조절 가능). force push/reset --hard 후 생명줄.
> - GitHub의 git protocol은 HTTP/2 위에 smart protocol. 자세히는 H7 내부 + Ch118 CI/CD에서 다시.
> - 자경단 사이트 미러 워크플로(Ch003 중에 등장)가 Ch004 H8에 정식 등장.
> - 12회수지도 챕터 번호는 잠정. CHAPTER-MATRIX 변경 시 동기화.

## 추신

5줄 미니 commit이 1교시의 진짜 결과물이에요. 명령어를 손에 한 번 쥐었느냐가 비유 한 페이지보다 큽니다. `git status`와 `git log` 두 줄은 무엇도 망가뜨리지 않으니 무서울 때 가장 먼저 치세요. 사진앨범 비유는 H7에서 .git/objects/를 직접 열어 보면 한 번 더 깊어져요. Branch는 평행 우주 — 1초에 만들고 1초에 버립니다. 부담 없이 따세요. 한 작업 = 한 브랜치.

이번 시간이 어려웠다면 본인 잘못이 아니에요. Git은 한 도구가 네 일(저장·시간·분산·협업)을 동시에 해서 처음 만나면 무겁게 느껴집니다. H2부터 H7까지 한 시간씩 한 일씩 떼어 보면 가벼워져요. Ch004는 두 해 코스에서 가장 자주 다시 만나는 도구예요. Ch005 협업, Ch062 REST API PR, Ch070 자경단 백엔드 통합, Ch080 고양이 신고 통합, Ch118 CI/CD까지 다 Git 위에 얹혀요. 한 번 깊게 깔아 두면 평생 갑니다. 다음 시간에 만나요.
