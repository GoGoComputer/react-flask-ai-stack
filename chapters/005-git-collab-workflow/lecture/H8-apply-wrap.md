## 3. 다섯 원리 한 페이지

본 챕터의 모든 H를 5개 원리로 압축.

### 원리 1: 분산 — 5명 노트북이 다 합법

git의 본질. 한 서버 죽어도 5명 노트북에 다 있음. branch는 본인 노트북에서 자유롭게. 합치는 순간 합의 도구(branch protection·CODEOWNERS) 발동. **분산이 자유, 합치기가 합의**.

### 원리 2: 합의 → 도구

5명 매번 슬랙으로 합의하면 30분/PR. branch protection·CODEOWNERS·husky·commitlint가 합의를 평생 기억. **사람 합의 → 도구 강제**의 진화. 도구가 합의를 안 까먹어요.

### 원리 3: 자동화 → 곱셈

dependabot·release-please·stale·branch 자동삭제·docs Pages 5종 자동화. 셋업 30분 × 절약 80h/년 × 5년 = **400시간의 자산**. 곱셈이 자경단의 산소.

### 원리 4: 진화 의식

매주 점검·매월 통계·매분기 도구 점검·매반기 마이그·매년 1시간 회고. 5의식이 자경단의 시계. **의식 없는 협업은 죽은 협업**. 회고가 진화의 신호.

### 원리 5: 원리가 시니어

도구 손가락은 1년이면 충분. 도구의 원리(branch protection 강제·CODEOWNERS 알고리즘·conflict 4종 알고리즘·refspec·Actions·gh CLI)는 사고 시 시니어. **사고 5분 진단 vs 5시간 검색의 차이**가 H7의 가치.

## 9. 첫 PR 자신감 — 자경단 5명의 첫 화요일

본인이 자경단의 첫 PR을 만드는 그 화요일 14:00. 5분짜리 자신감 한 페이지.

```
14:00  $ git switch main && git pull --rebase
14:01  $ git switch -c feat/first-cat-card
14:02  (코드 한 줄 추가)
14:03  $ git add . && git diff --staged
14:04  $ git commit -m 'feat(cat-card): 첫 카드 컴포넌트'
14:05  husky 발동 — pre-commit·commit-msg ✓
14:06  $ git push -u origin HEAD
14:07  $ gh pr create --draft
14:08  PR #1 생성 — CI 시작
14:10  CI ✓ — 본인 승인
14:11  $ gh pr merge 1 --squash --auto
14:12  머지 완료. main에 본인의 첫 commit.
```

**12분의 첫 PR**. 셋업 30분 + 첫 PR 12분 = 42분이 자경단의 첫 1시간. 그 1시간이 5년의 자신감.

**왜 첫 PR이 자신감인가** — 본 챕터의 모든 도구가 첫 PR에 다 작동해요. branch protection·CODEOWNERS·CI·husky·gh CLI가 한 PR에 한 번씩 발동. 본인이 한 번 봤으면 5년 자동으로 작동.

## 10. 흔한 오해 7가지

**오해 1: "Ch005를 다 외워야 자경단 시작 가능."** — 30분 셋업 10단계만 알면 시작. 나머지는 사고 시 사전. **시작이 학습의 시작**.

**오해 2: "5명 자경단은 도구 다 안 써도 돼요."** — 5종 (organization·team·branch protection·CODEOWNERS·husky)가 핵심. 작은 팀일수록 ROI 큼.

**오해 3: "1년 안에 다섯 원리 다 익혀야."** — 손가락 1년, 도구 1년, 원리 5년. 평생 학습. 다섯 원리는 시간이 필요.

**오해 4: "협업은 5명 이상부터."** — 2명도 협업. 본인 + 까미 한 명만 있어도 organization·branch protection 효과. 인원이 적을수록 셋업 ROI 큼.

**오해 5: "Conventional Commits는 양식 강박."** — 양식이 자동화의 입력. release-please·CHANGELOG·log 검색이 한 양식에서. **강박이 아니라 자산**.

**오해 6: "브랜치 protection을 풀어 두는 게 빨라요."** — 거꾸로. 자물쇠 없으면 사고 한 번 = 5명 × 1시간. 자물쇠 30분 셋업이 사고 5번 방지.

**오해 7: "본 챕터의 도구는 GitHub만 가능."** — GitLab·Bitbucket·Gitea도 80% 같음. branch protection·CODEOWNERS 비슷한 도구. **도구가 아닌 원리**.

**Q2. private repo가 필요할 때 무료 가능한가요?**

**Q3. 본 챕터의 셋업이 다른 회사 입사 시 도움?**

**Q4. 본 챕터를 안 보고 협업하면?**

**Q5. Ch005를 1년 후 다시 읽으면?**
A. 같은 H가 다른 깊이로 읽힘. 1년의 경험이 30%를 새로 보여 줘요. 매년 한 번씩 다시 읽기 권장.

**Q6. 자경단의 5명 중 누가 책임자?**
A. 본인 (메인테이너 / Admin). 첫 1년은 1명 책임자가 단단함. 1년 후 7명 + 책임자 2명으로 분리 (frontend lead·backend lead). 5년 후 SRE팀이 운영 책임.

**Q7. Ch006으로 넘어가도 되나요?**
A. 본 H 끝까지 읽고 30분 셋업 한 번 시도 후. 셋업 안 해도 Ch006 진행 가능, 다만 셋업 한 번이 매 챕터의 협업 직관.

## 14. 자경단 첫 PR의 짜릿함 — 12분의 자산

본인이 자경단의 첫 PR을 머지하는 그 12분. 본 챕터의 결론.

**짜릿함 5가지**:
1. **자동화의 발동** — husky·CI·CODEOWNERS가 본인 작업에 한 번씩 작동. 셋업이 살아 움직임.
2. **5명의 알람** — 본인 PR 알람이 까미·노랭이·미니·깜장이에게. 5명이 본인의 변경을 봄.
3. **CI 초록불** — 1분 후 lint·test·type-check 다 ✓. 본인 코드의 첫 인정.
4. **승인 + Merge** — 동료 한 명이 LGTM. Merge 버튼 클릭. main에 본인의 commit.
5. **CHANGELOG 자동 한 줄** — 다음 release에 본인의 PR이 한 줄. 5년 후에도 그 한 줄이 git history에 남음.

**5짜릿함 × 5명 × 5년 = 자경단의 평생**. 본인의 첫 12분이 5년의 자산.

## 15. 자경단 5명 vs 큰 회사 500명 — 도구 비교

자경단의 5년 후가 큰 회사. 둘이 80% 같고 20% 다름. 다른 20%가 어디.

| 영역 | 자경단 5명 | 큰 회사 500명 | 자경단 진화 시점 |
|------|----------|-------------|--------------|
| 패턴 | GitHub Flow | Trunk-based | 25명+ |
| branch | feature 1주일 | feature 1일 | 50명+ |
| release | 매주 금요일 | 매일 (다양한 환경) | 100명+ |
| deploy | 1대 서버 | 100대 + canary | 50명+ |
| feature flag | 없음 (Ch090 이후) | LaunchDarkly 500개 | 1년+ |
| CI 시간 | 1~3분 | 5~30분 (matrix) | 50명+ |
| monorepo 도구 | npm workspaces | Nx·Turborepo·Bazel | 15명+ |
| on-call | 본인 1명 | SRE팀 1주 rotation | 1년+ |
| observability | GitHub Insights | Datadog·New Relic | 1년+ |
| governance | Slack 한 줄 | RFC·ADR 절차 | 25명+ |

**80% 같음 — 도구의 본질**. **20% 다름 — 인원·인프라**. 자경단의 5년 후엔 본 챕터의 도구 80% + 새 도구 20%.
