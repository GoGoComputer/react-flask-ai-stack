## 회수: Ch004의 git 한 사람에서 Ch005의 git 함께로

지난 챕터 Ch004에서 본인이 git의 일곱 얼굴을 다 만났어요. 표면(명령어 23개) → 운영(도구 6개) → 내부(객체 4종). 30분 종합 셋업으로 본인의 자경단 저장소가 회사 저장소처럼 단장됐어요. **본인 혼자**였어요. 본인이 commit하고, 본인이 push하고, 본인이 머지했어요. 본인의 머릿속이 본인의 history. 본인의 손가락이 본인의 워크플로우. **혼자 git의 시간**.

이번 챕터 Ch005는 **함께 git의 시간**이에요. 자경단에 동료가 4명 추가됐다고 가정해 봐요 — 까미(백엔드), 노랭이(프론트), 미니(인프라), 깜장이(디자이너). 5명이 한 저장소에 commit·push·머지를 동시에 해요. **무엇이 달라지나?** 모든 게 달라져요. 본인 혼자였다면 main에 직접 commit해도 됐어요. 5명이면? 한 사람이 main에 commit한 직후 나머지 4명의 working tree가 다 어긋나요. 본인이 force-push 한 번 잘못 치면 4명의 한 시간 작업이 날아가요. 본인이 `git pull`만 두드려도 자동 merge가 동료의 의도와 다르게 합쳐서 사고. **5배의 사람 = 25배의 사고 가능성**(조합론). 그래서 워크플로우가 필요해요.

**워크플로우가 뭔가요?** 한 줄로 — **"5명이 한 저장소에서 사고 없이 같이 일하는 합의된 절차"**. 합의된, 이게 핵심이에요. 본인 혼자라면 합의가 필요 없어요. 두 명만 돼도 합의가 필요해요. "main에 직접 push 안 한다", "PR로만 머지한다", "리뷰 1명 이상 받고 머지한다" — 이런 한 줄짜리 합의가 모이면 워크플로우가 돼요. 도구(GitHub Actions·branch protection)가 합의를 강제하면 더 단단해져요. **사람의 합의 + 도구의 강제**, 이 둘이 모인 게 워크플로우예요.

또 한 가지 — **워크플로우는 회사마다 달라요.** Meta·Google은 Trunk-based(매일 main에 머지). Microsoft·Adobe는 Git Flow(release branch). 스타트업·오픈소스는 GitHub Flow(가장 단순). 본인이 어느 회사를 가도 기본 셋 중 하나(또는 변형)예요. **세 패턴을 다 알면 어느 회사를 가도 1주일 안에 적응**해요. Ch005가 본인의 적응 비용을 1달에서 1주로 줄여 줘요.

## 1. 왜 워크플로우를 배워야 하나 — 7가지 이유

본인이 신입 개발자로 회사에 입사했다고 가정. 첫 주 월요일 아침 9시. 본인의 노트북에 회사 저장소를 clone해요. 그 다음 본인이 무엇을 두드릴지 — Ch005가 답이에요.

**이유 1: 본인의 첫 PR을 5명이 봐요.** 본인이 main에 직접 push하면 5명이 동시에 "어?" 해요. 본인이 PR을 잘 만들면 5명이 차례대로 리뷰. 첫 PR의 제목·본문·크기가 본인의 첫 인상이에요. 신입의 첫 한 달 이미지는 첫 다섯 PR로 결정돼요. **첫 PR이 첫 한 달**.

**이유 2: 본인이 회사에서 쓰는 시간의 30%는 PR·리뷰.** 5년 차 개발자도 하루의 30~40%를 PR 본문 쓰기·리뷰 답글·conflict 해결에 써요. 코드 짜는 시간보다 PR 시간이 길어요. **30%를 효율로 만들면 30%의 본인 시간이 살아나요.** 본 챕터가 그 30%의 효율을 가르쳐요.

**이유 3: 사고 한 번이 회사 시간을 날려요.** 본인이 force-push로 동료 4명의 한 시간 작업을 날리면 — 4명 × 1시간 = 4시간 + 복구 시간 1시간 + 동료의 신뢰 회복 시간 N. 본인의 1초 손가락이 회사의 5+ 시간. **워크플로우의 보호장치**(branch protection·CODEOWNERS·CI)가 본인 1초의 사고를 막아 줘요. 본인의 미래를 본인 현재가 보호하는 한 칸. 그리고 보호장치는 한 번 셋업하면 평생. 셋업 30분이 5년 사고 0회를 만들어요. ROI가 무한대.



**이유 4: 코드는 텍스트지만 협업은 사람.** 본인이 동료의 PR에 "이거 틀렸어요"라고 쓰면 동료는 한 시간 동안 답글을 어떻게 쓸지 고민. 본인이 "이 부분 의도가 ~인가요? 저는 ~로 이해했는데 어떻게 생각하세요?"라고 쓰면 동료가 5분 안에 답해요. **같은 정보, 다른 톤, 다른 결과**. 리뷰 톤이 코드 품질만큼 중요. Ch005에서 리뷰 5톤(칭찬·질문·제안·요청·nit)을 깊이 다뤄요. 톤은 타고나는 게 아니라 의식적으로 익히는 기술. 본인이 5번만 의식적으로 써 보면 손가락이 자동으로. 그 후엔 평생 자산. **톤 한 단어가 동료 한 시간을 절약**.

**이유 5: 본인이 5년 차가 되면 워크플로우를 만드는 사람이 돼요.** 신입 때 워크플로우를 따라가다가, 3년 차에 익숙해지고, 5년 차에 본인이 회사 워크플로우를 디자인해요. "우리 팀은 GitHub Flow + Conventional Commits + 자동 deploy preview로 가요" 같은 한 문장을 본인이 결정하는 자리. 그 자리에 가려면 셋 패턴의 장단을 다 알아야. **신입 때 배운 게 5년 후 본인의 결정 근거**. 본인이 한 번 디자인한 워크플로우는 그 회사에 1년 이상 남아요. 본인이 떠나도 동료들이 따라요. **본인 결정이 본인 영향력**. 신입의 한 결정이 5년 차의 한 결정보다 가벼워 보이지만, 신입의 결정이 5년 후 본인의 무게를 만들어요.

**이유 6: 오픈소스 기여의 첫 걸음이 워크플로우.** 본인이 React·Vue·FastAPI 같은 큰 오픈소스에 PR을 보내려면 그 프로젝트의 워크플로우를 이해해야. CONTRIBUTING.md를 읽고, 그 패턴에 맞게 PR을 만들어야 머지돼요. 워크플로우를 모르면 본인의 좋은 코드도 거부당해요. **워크플로우가 기여의 입장권**. 본인이 신입일 때 이름 있는 오픈소스에 PR 한 개 머지되면 GitHub 프로필에 평생 자산. 면접에서 "React에 commit 있어요"라는 한 줄이 면접관의 1초 신뢰. 그 1초가 입사 결정에 영향을 줘요.

**이유 7: 면접에서 자주 묻는 질문.** "Git Flow와 Trunk-based의 차이는?", "force-push는 언제 쓰나요?", "merge conflict가 났을 때 어떻게 해결하나요?", "PR 리뷰에서 강하게 반대해야 할 때 어떻게 말하나요?" — 다 협업 워크플로우 질문. 코드 짜기 질문보다 협업 질문이 신입 면접에서 더 중요. **회사가 원하는 건 코드를 짜는 사람이 아니라 같이 일할 수 있는 사람**. 본인이 신입 면접 5번을 보면 5번 다 협업 질문을 받아요. "가장 힘들었던 협업 경험은?", "의견 차이가 났을 때 어떻게 풀었나요?", "본인의 PR이 거부됐을 때 기분은 어땠나요?" — 다 사회적 충돌(깊이 3) 질문. 본인이 자경단에서 5명과 워크플로우를 한 번 굴려 보면 면접 답이 자연스럽게 나와요. **자경단이 본인의 면접 시뮬레이션**.

7가지 이유를 한 줄로 — **혼자 git을 잘 쓰는 사람은 신입, 함께 git을 잘 쓰는 사람은 시니어**. 본인이 시니어로 가는 다리가 Ch005예요.

**한 가지 더 — 본인이 일을 시작하면 가장 충격받는 게 "코드 짜는 시간이 의외로 짧다"는 사실.** 신입 때 상상으론 8시간 코딩일 줄 알지만, 실제론 2시간 코딩 + 3시간 PR·리뷰·미팅 + 2시간 디버깅·문서 + 1시간 그냥 동료와 대화. 코드 시간 25%. 이 비율은 5년 차도 비슷해요. 그래서 PR·리뷰의 효율이 본인 일하는 시간 전체의 효율을 결정해요. **본 챕터는 본인 시간의 30%를 더 풍요롭게 만들어 주는 챕터**예요. 코드 짜는 25%만큼 PR·리뷰 30%가 중요해요.

## 2. 세 가지 표준 워크플로우 — 첫인상

본 챕터의 H2에서 깊게 다루지만, 첫인상으로 셋을 보여 드릴게요. 어떤 회사 가도 이 셋 중 하나(또는 변형).

### 2-1. GitHub Flow (가장 단순)

**한 줄 요약**: main 한 개 + feature branch 여러 개 + PR 한 번 + 머지 후 즉시 deploy.

```
main ────●────●────●────●──────●────●──→
              │              │
              ●─●─● feature/x  ●─●─● feature/y
                    PR머지         PR머지
```

**누가 쓰나**: 스타트업, 오픈소스, 작은 회사, 자경단 같은 사이드 프로젝트. **GitHub 본인이 만들어서 쓰는 표준**. SaaS 같은 continuous deployment 환경에 자연스러워요. 2011년 GitHub의 Scott Chacon이 블로그 한 편으로 정리한 후 사실상 업계 기본이 됐어요. 단순함이 무기.

**장점**: 단순. branch가 main + feature 두 종류뿐. 새 멤버가 1시간 안에 이해. release 개념이 따로 없어서 "지금 main이 곧 production"이라는 단순한 멘탈 모델. 본인이 신입으로 입사한 첫 주 월요일에 동료에게 "우리 GitHub Flow예요"를 들으면 1시간 안에 첫 PR을 만들 수 있어요.

**단점**: main이 곧 production이라 main에 빨리 머지하기 어려운 큰 기능에 부적합. **release 일정**이 있는 제품(매년 v1·v2 식 메이저 release)엔 안 맞아요. 그리고 trunk-based만큼 빠르지도 않아요 — feature branch를 1주일 정도는 두는 패턴이라 그만큼 conflict 가능성도 있음.

**자경단의 정답**: GitHub Flow. 5명 미만의 팀에 가장 자연스러워요. 본인의 자경단은 이걸 채택.

### 2-2. Git Flow (가장 복잡)

**한 줄 요약**: main + develop + feature + release + hotfix 다섯 종류 branch.

```
main      ────────●────●─tag-v1.0────●──tag-v1.1──→
                  ↑release            ↑release
develop   ──●──●──●──●──●──●──●──●──●──●──→
            │     │  │           │
            ●─●─● │  ●─●─● feat   │
                  │                │
                  hotfix/y         feat/z
```

**누가 쓰나**: Adobe·Microsoft·게임 회사·은행·SaaS 큰 회사. **release 일정**이 있는 제품. v1.0·v2.0이 명확한 마일스톤인 곳. 2010년 Vincent Driessen의 블로그 글로 처음 정리. 한때 거의 모든 책에서 표준으로 다뤘다가, 지금은 큰 회사·전통 산업에서만 주로 써요. 작가 본인도 2020년에 "이젠 모든 팀에 권하지는 않는다"고 글을 갱신.

**장점**: release 관리가 정교. main은 항상 production 안정 상태. develop은 진행 중. release branch에서 QA·테스트 후 main으로. hotfix는 main에서 직접 따서 다시 main + develop으로 합쳐요. 분기마다 큰 release가 있는 회사에 자연스러워요. 게임 회사가 분기마다 패치 노트를 내는 리듬.

**단점**: 복잡. 새 멤버가 1주일 동안 헤매요. "지금 어디에 commit해야 하지?"가 매번 질문. release 사이클이 짧은(매일 deploy) 환경엔 과한 무게. **branch가 5종류라는 건 합의가 5종류 필요하다는 뜻**. 작은 팀엔 부담.

**자경단의 위치**: 자경단엔 안 맞아요. 다만 본인이 큰 회사 입사 시 만나요. 알고 있어야 적응 빠름.

### 2-3. Trunk-based (가장 빠른)

**한 줄 요약**: main 한 개에 모든 사람이 매일 매시간 머지. branch는 짧게(1일 미만). feature flag로 미완성 기능을 숨겨요.

```
main ──●─●─●─●─●─●─●─●─●─●─●─●─●─●─●─→
       (50명이 매시간 머지)
       ↑ feature flag로 미완성 기능 숨김
```

**누가 쓰나**: Google·Meta·Netflix·Spotify. **CI/CD가 완벽**한 회사. 자동 테스트 90%+, 자동 deploy, feature flag 인프라가 있는 곳. Google은 사실상 "한 거대한 monorepo + trunk-based"가 회사 기본. Meta도 비슷. Netflix는 Spinnaker라는 자체 deploy 도구로 자동화. 인프라 비용이 들지만 한 번 갖춰지면 50명이 1년 동안 거의 사고 없이 매시간 머지.

**장점**: integration hell이 없어요. branch가 길어지면 conflict가 쌓여요. trunk-based는 매시간 main에 머지하니 conflict가 작게 자주. 그리고 main이 항상 머지 가능 상태라 release가 자유로워요. "오늘 deploy", "내일 deploy", "일주일에 100번 deploy" 다 가능.

**단점**: 인프라 비용. CI/CD·feature flag·테스트 자동화·monitoring이 다 갖춰져 있어야. 이게 없는 작은 팀이 trunk-based를 흉내 내면 매일 production 사고. 그리고 feature flag를 잘못 관리하면 코드베이스가 if·else 미궁이 돼요. flag 정리(cleanup)가 별도 일.

**자경단의 위치**: Ch090(feature flag) + Ch118(GitOps) 이후에 자경단도 trunk-based로 넘어갈 수 있어요. 처음엔 GitHub Flow.

### 셋의 한 표

| 패턴 | branch 종류 | 적합 팀 | release 빈도 | 학습 곡선 |
|------|-----------|--------|------------|---------|
| GitHub Flow | main + feature | 5명 미만, 스타트업 | 수시 | 1시간 |
| Git Flow | main + develop + feat + release + hotfix | 50명+, 큰 회사 | 분기·반기 | 1주일 |
| Trunk-based | main(+ 짧은 feat) | 100명+, 인프라 완비 | 매일 100회 | 1달 |

**한 가지 결정 기준** — 팀 크기가 5명 미만이면 GitHub Flow, 5~50명이면 보통 GitHub Flow의 변형, 50명+ 이면 회사가 지정한 패턴 따라요. 본인이 회사 옮길 때마다 어떤 패턴인지 첫 주에 확인. CONTRIBUTING.md에 적혀 있어요. 안 적혀 있으면 동료에게 물어요. **첫 주의 한 질문이 1달의 적응 비용을 절약**. 그리고 한 가지 더 — 패턴 이름은 다르지만 본질은 비슷해요. 셋 다 "main 보호 + branch에서 작업 + PR로 리뷰 + 머지로 통합". 변수는 branch 종류 수와 release 주기뿐. 본질을 잡으면 표면이 바뀌어도 흔들리지 않아요.

**또 한 가지 — 패턴은 진화해요.** 같은 회사도 5명일 땐 GitHub Flow → 50명 되면 Git Flow 변형 → 500명 되면 Trunk-based로 옮길 수 있어요. 단순함 → 복잡함 → 자동화로 펼쳐진 단순함. 본인의 자경단도 5년 후엔 셋 중 다른 패턴으로 갈 수도. **워크플로우는 살아 있는 계약**이에요. 진화의 방향은 항상 "사람의 합의가 자동화로 옮겨 가는 길". 처음엔 사람이 매번 결정, 시간이 지나며 도구가 결정을 대신. 그게 성숙의 신호.

## 3. 충돌의 세 깊이 — 도구로 푸는 건 1개뿐

협업 워크플로우의 핵심 단어 하나가 **충돌(conflict)**이에요. 본인이 "충돌"이라고 들으면 보통 `git merge` 후의 빨간 `<<<<<<<` 표시를 떠올려요. 그건 충돌의 한 종류일 뿐이에요. 충돌엔 세 깊이가 있어요.

### 깊이 1: 코드 충돌 (Code conflict)

**같은 파일의 같은 줄을 두 사람이 다르게 수정.** git이 자동 merge를 못 해서 `<<<<<<<·=======·>>>>>>>` 마커를 박아요. Ch004 H5에서 본 그것. 가장 표면.

**해결 시간**: 5분 ~ 1시간. 본인이 두 변경의 의도를 보고 둘 다 살리거나, 한쪽을 선택. 도구가 도와줘요(VS Code 3-way merge, GitLens, mergetool). **도구로 풀 수 있는 유일한 충돌**.

**예방**: branch 짧게(1일 미만), 작은 PR(<300줄), pull --rebase 자주. 그러면 충돌 발생 빈도가 90% 줄어요. 본인이 매일 morning에 `git fetch && git rebase origin/main`만 한 줄 두드리면 conflict가 작게 자주 만나져요. **자주 작게**가 conflict 관리의 황금 규칙.

### 깊이 2: 의도 충돌 (Intent conflict)

**같은 기능을 두 사람이 다른 방향으로 구현.** 본인이 "고양이 사진 업로드"를 S3로 짜고, 동료가 같은 시간에 Cloudflare R2로 짠다면 — 코드는 충돌이 안 나요. 두 사람이 다른 파일을 건드릴 수도 있으니까. 하지만 머지 후 둘 다 main에 들어가면 시스템이 두 방향으로 갈라져요.

**해결 시간**: 며칠 ~ 몇 주. 누가 옳은지 동료들과 토론. 둘 다 폐기하고 새로 짤 수도. **도구로 못 풀어요**. 사람이 회의로 풀어야.

**예방**: PR 만들기 전에 **설계 문서**(RFC·ADR) 한 장. "이 기능을 S3로 갈 거예요"를 미리 적어 두면 동료가 R2를 짜지 않아요. 5년 차의 PM 능력. Ch006에서 깊이.

### 깊이 3: 사회적 충돌 (Social conflict)

**리뷰 톤·머지 권한·코드 스타일에 대한 사람 사이의 갈등.** 본인이 동료에게 "이거 잘못됐어요"라고 강하게 적어서 동료가 상처. 또는 본인이 시니어와 의견 차이가 났을 때 본인이 굽혀야 하나 vs 본인이 옳다고 끝까지 주장하나. **도구로 못 풀고, 회의로도 한 번에 못 풀어요**. 신뢰 관계로 시간 들여 풀어요.

**해결 시간**: 몇 주 ~ 몇 달. 또는 영원히 안 풀려서 한 사람이 회사를 떠남. 가장 비싼 충돌.

**예방**: 리뷰 5톤(칭찬·질문·제안·요청·nit) 의식적 사용. PR 본문에 "이 변경의 의도는 ~"을 명확히. 1대1 대화 채널 유지. **사람을 사람으로 대하는 태도**. Ch005·Ch006에서 깊이.

### 세 깊이 한 표

| 깊이 | 무엇 | 해결 시간 | 도구로? | 사람으로? |
|------|------|---------|--------|---------|
| 1 코드 충돌 | 같은 줄 다르게 | 5분 ~ 1h | ✅ | △ |
| 2 의도 충돌 | 같은 기능 다른 방향 | 며칠 ~ 몇 주 | ❌ | ✅ (회의·RFC) |
| 3 사회적 충돌 | 리뷰 톤·신뢰 | 몇 주 ~ 영원 | ❌ | ✅ (시간·태도) |

**가장 비싼 게 깊이 3, 다음이 깊이 2, 가장 싼 게 깊이 1**. 그런데 본인이 신입 때 가장 두려워하는 게 깊이 1(빨간 마커). 시간이 지나며 깊이 2·3이 더 무서워져요. **5년 차의 일은 깊이 1이 5%, 깊이 2가 30%, 깊이 3이 65%**. 본인이 시니어로 갈수록 git 도구보다 사람이 어려워요. Ch005가 그 입구.

**왜 신입은 깊이 1을 두려워하나** — 빨간 마커가 시각적으로 무섭게 보여요. "<<<<<<<"·"======="·">>>>>>>" 기호 자체가 위협적. 그리고 git이 실패한 듯한 메시지("Merge conflict in...")가 무서워요. 다만 한 번 풀어 보면 5분 작업. 본인이 자경단에서 일부러 충돌을 5번만 만들어 보면 손가락이 익혀요. **연습이 두려움을 풀어요**. Ch005 H5에서 일부러 만든 충돌 시나리오를 손으로 풀어 봅니다.

**왜 시니어는 깊이 3을 두려워하나** — 사람의 신뢰는 한 번 깨지면 회복이 어려워요. 본인이 동료에게 "이거 잘못됐어요"를 한 번 강하게 적으면, 그 동료는 본인의 다음 PR을 6개월 동안 색안경 끼고 봐요. 본인이 1년 동안 복구해도 처음 같은 신뢰는 안 돌아와요. **신뢰는 쌓는 데 1년, 깨는 데 1초**. 그래서 시니어는 리뷰 톤을 의식적으로 챙겨요.

**한 가지 더 — 깊이 0이 있어요. "발생 안 한 충돌"**. 본인이 좋은 워크플로우 + 좋은 커뮤니케이션을 가지면 깊이 1·2·3 다 발생 빈도가 90% 줄어요. **가장 좋은 충돌 해결은 충돌이 안 나게 만드는 것**. 그게 워크플로우의 진짜 가치예요. 사고 후 복구가 아니라 사고 전 예방. Ch005가 깊이 0을 만드는 챕터.

## 5. 자경단 5명의 첫인상 — 한 시나리오 미리보기

본 챕터에서 만날 5명을 미리 소개:

- **본인** — 풀스택. 자경단 메인테이너. main에 직접 머지 권한 있음(다만 안 함). 모든 PR 리뷰의 1차 책임. 새 멤버 onboarding·릴리즈 노트·아키텍처 결정의 최종 결재자.
- **까미** — 백엔드. Python·FastAPI 담당. CODEOWNERS의 `/backend/` 소유. DB schema·API contract·인증 로직의 1차 리뷰어. 까미가 approve 안 하면 backend PR 머지 불가.
- **노랭이** — 프론트. React·TypeScript 담당. CODEOWNERS의 `/frontend/` 소유. 컴포넌트·상태관리·라우팅의 1차 리뷰어. UX detail에 가장 예민한 사람. 픽셀 1px도 놓치지 않아요.
- **미니** — 인프라. AWS·Terraform·CI/CD 담당. CODEOWNERS의 `/infra/`·`.github/` 소유. deploy·monitoring·security의 1차 리뷰어. 새벽 3시 production 사고 시 본인 다음 호출.
- **깜장이** — 디자이너 + QA. Figma → 컴포넌트 + 수동 테스트. CODEOWNERS의 `/design/`·`/qa/` 소유. PR의 시각적·UX 검토 + release 전 한 사이클 수동 테스트. 사용자 시점의 마지막 방어선.

**5명의 한 주 흐름** — 월요일 09:00 standup 5분(slack 메시지로). 화요일 까미가 "고양이 등록 API" PR 만듦, 노랭이가 동시에 "고양이 카드 컴포넌트" PR 만듦. 수요일 미니 리뷰 + 깜장이 디자인 검토 + 본인 final approve. 목요일 머지 + deploy preview 자동 생성. 금요일 17:00 release v0.5.0 + Slack 공지. 5명 1주일 = 5~10개 PR 머지 + 0~1 hotfix. 월~금의 한 바퀴가 한 spirit. 다음 주 월요일이 또 돌아와요. **한 주의 리듬이 1년의 생산성**. 리듬이 흔들리면 한 달의 생산성이 흔들려요. 본인이 신입으로 입사 후 첫 한 달의 첫 미션 — 이 리듬을 본인 손가락에 박는 것.

**자경단은 GitHub Flow** — branch protection 7체크 + CODEOWNERS + Conventional Commits + CI 5단계(lint·test·build·preview·prod). Ch004 H8의 30분 셋업 그대로. 그 위에 5명의 합의(워크플로우)를 얹어요. **셋업이 도구, 워크플로우는 사람의 약속**. 둘이 같이 가야 작동. 본인이 신입으로 다른 회사에 가도 90% 비슷한 셋업을 봐요. 회사 5년 차들이 5년 동안 시행착오로 만든 결과를 본인은 첫 주에 받아요. 그 결과가 본인 자경단에도 그대로 박혀 있다는 건 본인이 회사 5년 차의 시행착오를 미리 흡수했다는 뜻. **자경단이 회사 입사 시뮬레이션**. 본인이 자경단에서 한 번 만진 셋업은 회사 첫 주 onboarding을 1시간으로 줄여 줘요.

**한 가지 미리보기 — H5의 한 장면.** 화요일 14:00, 까미와 노랭이가 같은 파일(`models/cat.py`)을 동시에 건드려요. 까미는 `name` 필드 추가, 노랭이는 `color` 필드 추가. 둘 다 PR 만들어서 머지. 까미가 먼저 머지되면, 노랭이가 PR을 갱신할 때 conflict 발생 — `git fetch && git rebase origin/main`으로 본인 변경을 까미 위에 다시 올리고 conflict 해결(이 경우 둘 다 살림). 1분 작업. 이게 깊이 1 코드 충돌의 일상. **무섭지 않아요**. 자주 만나면 본인의 손가락이 5분 안에 푸는 일이 돼요. 그리고 이 한 장면은 자경단의 일주일에 평균 2~3번 나와요. 5명이 같은 코드베이스를 동시에 만지면 자연스러운 빈도. 자주 작게 만나면 매번 1분, 한 번 크게 만나면 한 시간. **빈도가 비용을 결정**해요. 본인이 매일 morning에 `git fetch && git rebase`만 한 줄 두드리면 빈도 자동 조절. 한 줄의 손가락 습관이 한 주의 평화.

## 6. 본 H의 핵심 비유 — 합주

본 챕터를 한 비유로 — **협업 워크플로우는 5명의 합주**예요. 본인 혼자라면 솔로 연주. 솔로는 본인이 박자·키·템포를 다 정해요. 합주는 5명이 같은 박자·같은 키·같은 템포를 맞춰요. **악보(워크플로우)**가 있어야. 지휘자(메인테이너)가 있어야. 리허설(PR 리뷰)이 있어야. 무대(production deploy)가 있어야.

**솔로와 합주의 차이** — 솔로 잘하는 사람이 합주를 잘 하는 게 아니에요. 합주엔 **남의 박자를 듣는 능력**이 더 중요해요. 본인이 git 명령어를 23개 다 외워도 동료의 PR을 안 듣고 본인 PR만 강행하면 협업이 안 돼요. **듣기가 치기보다 중요**한 게 합주의 본질. 협업 워크플로우의 본질이기도.

**악보가 왜 필요한가** — 5명이 다 다른 악보를 보면 합주가 안 돼요. 한 사람이 1마디 앞서 가고 한 사람이 1마디 뒤처지면 음이 깨져요. 워크플로우가 같은 악보예요. "main에 직접 push 금지", "PR 1명 이상 approve 후 머지", "Conventional Commits prefix 필수" — 이런 한 줄들이 5명의 같은 악보. 본인이 신입일 때 악보를 받아 따라가다가, 시니어가 되면 악보를 쓰는 사람이 돼요.

**지휘자의 역할** — 메인테이너(또는 tech lead)가 지휘자. 의도 충돌이 났을 때 "이쪽으로 가요"를 결정. 사회적 충돌이 났을 때 중재. 5년 차의 자리. 본인이 신입 때 그 자리를 지켜보면서 5년 후 본인이 그 자리에 가요. **지휘자 없는 합주는 사고**, 지휘자 있는 합주는 음악. 그리고 지휘자는 악기를 직접 연주하지 않아요 — 다른 세션의 소리를 듣고 박자를 맞춰 주는 일. 시니어의 일의 60%는 코드 짜기가 아니라 동료를 조율하기예요. 코드 짜는 사람이 시니어가 아니라 동료를 조율하는 사람이 시니어. 본인이 신입 때 "와, 시니어는 코드를 잘 짜겠지"라고 상상하지만, 막상 시니어의 하루를 보면 코드 짜는 시간은 1~2시간. 나머지는 다 리뷰·회의·조율·onboarding·아키텍처 결정.

**한 가지 더 — 즉흥 합주(jam)**. 모든 합주가 악보대로 가지 않아요. 사고가 나면 즉흥. hotfix는 즉흥 합주. 본인이 새벽 3시에 production 사고를 만나면 워크플로우의 빠른 길(emergency hotfix process)을 통해 main에 직접 머지하기도 해요. 다만 그건 예외. 평소엔 악보. **악보가 있어야 즉흥도 가능**. 악보 없는 즉흥은 그냥 시끄러움. 그리고 즉흥 다음 날 아침엔 반드시 회고. "왜 새벽 3시 사고가 났나? 어떻게 예방하나? 누가 다음 새벽 3시를 받나?" 회고 없는 즉흥은 같은 사고를 한 달에 한 번 반복. 회고 있는 즉흥은 같은 사고를 평생 한 번. **즉흥은 한 번, 회고는 평생**. 본 챕터 H8에서 회고 양식을 다뤄요.

추신 0. 본 챕터를 끝낸 본인의 한 줄 능력 — "5명이 한 저장소에서 사고 없이 같이 일하는 절차를 셋업하고 굴릴 수 있다." 이 한 줄이 신입과 시니어를 가르는 다리. 본 챕터 8H가 그 다리.

## 7. 자경단 적용 + H2 예고

본인이 이번 챕터에서 자경단 저장소에 다음을 박을 거예요(미리 한 장 보기):

1. **워크플로우 한 장** — `.github/WORKFLOW.md` 새 파일에 "자경단은 GitHub Flow를 따른다 + 다음 5규칙" 한 페이지.
2. **CONTRIBUTING.md 보강** — Ch004 H8에서 만든 CONTRIBUTING.md에 "PR 절차 7단계"를 추가.
3. **commitlint + Conventional Commits 강제** — Ch004 H8에서 한 셋업이 5명에서 작동하도록 미세조정.
4. **자동 deploy preview** — PR 만들 때마다 `pr-123.cat-vigilante.org` 같은 URL 자동 생성. 깜장이가 디자인 검토하기 쉽게.
5. **release 자동화** — `git tag v0.5.0 && git push origin v0.5.0`만 두드리면 GitHub Release + CHANGELOG 자동 생성.

5개를 챕터 8H에 걸쳐 하나씩 추가. H8까지 가면 자경단이 5명 함께 매끈히 일하는 저장소 v1.0 완성. 5개의 추가가 합쳐지면 그 효과가 단순 합산이 아니라 곱셈이에요. 워크플로우 한 장 + Conventional Commits + commitlint + deploy preview + release 자동화 → 다섯이 서로 물려서 한 사람이 새 commit 한 줄을 push하는 순간 자동으로 lint + test + preview URL + 다음 release 노트 초안까지 생성. **다섯의 곱셈이 한 명의 새벽 잠을 지켜요**. 본인의 자경단이 새벽 3시 사고 0회의 1년을 만나려면 이 다섯이 다 박혀야.

**5개의 추가 비용은 다 합쳐서 셋업 1시간 + 문서 작성 1시간 = 2시간**. 5명이 1년 동안 사고를 안 만나는 데 드는 비용이 단 2시간. 본인이 신입으로 입사한 첫 주에 회사 저장소 셋업을 보면 거의 비슷한 5개가 있어요. 회사 5년 차들이 5년 동안 시행착오로 만든 결과. 본인이 그 결과를 자경단에 8시간 챕터로 옮기는 거예요. **5년의 시행착오를 8시간으로 압축**.

**그리고 한 가지 — 5개의 셋업이 워크플로우의 전부가 아니에요.** 진짜 워크플로우는 5명의 일주일 손가락 리듬에 있어요. 도구는 합의를 강제할 뿐, 합의 자체는 사람이 만들어요. "우리는 매일 09:00 standup", "우리는 PR을 24시간 안에 리뷰", "우리는 금요일 17:00에 release" — 이런 한 줄들이 진짜 워크플로우. 도구가 한 절반, 사람이 다른 절반. 둘 다 있어야 작동.

**다음 H2는 핵심개념** — 세 패턴(GitHub Flow·Git Flow·Trunk-based)을 깊이 다뤄요. 각 패턴의 branch 모델, release vs deploy 차이, hotfix 처리, 환경 분리(dev·staging·prod). 본 H1이 첫인상이라면 H2는 이론. **첫인상 → 이론 → 셋업 → 카탈로그 → 데모 → 운영 → 원리 → 적용**의 8H 흐름은 Ch004와 동일. 본인이 이미 익숙한 리듬. H2의 한 핵심 — release와 deploy가 다르다는 사실. release는 "사용자에게 새 버전을 알린다"의 사회적 사건, deploy는 "서버에 새 코드를 올린다"의 기술적 사건. 둘이 같은 날일 수도 다를 수도 있어요. Trunk-based는 deploy를 매일 100번 하지만 release는 분기에 한 번. 이 분리가 큰 회사가 매일 deploy하면서도 사용자엔 안정적인 버전을 보여 주는 비밀. H2에서 이 분리를 손가락에 박아요.

## 8. 흔한 오해 6가지

**오해 1: "워크플로우는 큰 회사용이에요. 자경단 5명엔 과해요."** — 5명이 워크플로우 없이 가면 첫 주에 사고. branch protection 한 줄, PR 절차 한 줄만 있어도 워크플로우. 워크플로우 = 합의된 절차. 5명이 한 저장소를 쓰는 순간 합의가 필요해요. **합의 없는 협업은 없어요**.

**오해 2: "Trunk-based가 가장 좋은 패턴이에요. 자경단도 trunk-based로 가요."** — 트렌드는 trunk-based지만, 인프라가 없는 5명이 trunk-based를 흉내 내면 매일 production 사고. CI/CD·feature flag·자동 monitoring이 다 갖춰져야 작동해요. 본인의 자경단은 GitHub Flow가 정답. trunk-based는 Ch090(feature flag) 이후에 검토. **트렌드보다 본인 팀의 인프라가 우선**.

**오해 3: "코드 충돌이 협업의 가장 큰 문제예요."** — 가장 작은 문제예요. 5분~1시간이면 풀려요. 의도 충돌·사회적 충돌이 며칠~영원이에요. 본인이 신입 때 빨간 마커가 무서워 보여도, 5년 차엔 그게 가장 쉬운 일이에요. **무서운 건 사람**.

**오해 4: "PR을 작게 자주 만드는 것보다 한 번에 큰 PR이 효율이에요."** — 반대예요. 큰 PR은 리뷰가 막혀요. 1000줄 PR을 1명 리뷰어가 1시간에 다 못 봐요. 며칠 묵혀요. 그동안 main이 갱신돼서 conflict 누적. **작은 PR(<300줄)이 평균 머지 시간 1일, 큰 PR(>1000줄)이 평균 5일**. 작은 게 빠르고, 빠른 게 안전.

**오해 5: "리뷰는 시니어가 주니어를 가르치는 자리예요."** — 양방향이에요. 주니어가 시니어 코드에 "이 부분 의도가 ~인가요?"라고 물으면 시니어도 자기 코드를 다시 봐요. 본인이 신입이라도 리뷰에서 좋은 질문 하나가 시니어를 살려요. **리뷰는 모든 사람이 모든 사람을 가르치는 자리**.

**오해 6: "Git Flow가 더 정교하니까 더 안전해요."** — 정교 ≠ 안전. branch가 5종류 있으면 합의가 5종류 필요해요. 잘못된 branch에 commit하는 사고도 5배. **단순함이 안전**. GitHub Flow의 단순함이 자경단의 안전.

**오해 7: "한 번 정한 워크플로우는 1년 동안 안 바뀌어요."** — 워크플로우는 살아 있는 계약. 팀이 5명에서 10명이 되면 바뀌어야. 새 도구가 나오면 흡수해야. 분기마다 한 번 정도 "이 워크플로우가 우리에게 맞나?"를 다 같이 회고. 안 바뀌는 워크플로우는 죽은 워크플로우. **계약은 갱신될 때 살아 있어요**. 본인이 자경단을 1년 굴리면 4번 회고 → 4번 갱신. 한 번에 큰 변경 말고 매번 작은 갱신. 작은 갱신이 누적되면 1년 후 워크플로우 v2가 자연스럽게 만들어져요. **갱신의 누적이 진화**.

**오해 8: "워크플로우는 도구가 다 해 줘요. 사람의 합의는 필요 없어요."** — 도구는 강제할 뿐 합의를 만들지 않아요. branch protection이 main에 직접 push를 막아도 "왜 막나?"의 합의가 없으면 동료들이 우회 방법을 찾아요. PR 템플릿이 있어도 "왜 본문을 채우나?"의 합의가 없으면 빈 PR이 올라와요. **도구는 합의의 표현, 합의는 사람의 약속**. 둘 다 필요해요. 본 챕터가 둘 다 다뤄요 — H3는 도구, H6는 사람.

## 9. FAQ 5가지

**Q1. 본 챕터 시작 전에 무엇을 준비해야 하나요?**
A. Ch004 H8에서 박은 자경단 저장소 셋업이 있어야 해요. branch protection·CODEOWNERS·Conventional Commits까지. 그 위에 Ch005가 워크플로우 한 장을 얹어요. Ch004를 안 듣고 Ch005를 들으면 도구가 없어서 워크플로우 합의가 떠도는 약속이 돼요. **셋업 → 워크플로우 → 운영**의 순서 중요.

**Q2. 자경단에 동료가 1명만 있어요. 5명 시나리오가 안 와닿아요.**
A. 1명이라도 워크플로우는 필요. 본인 자신과 1명의 합의도 합의예요. 그리고 미래의 5명을 미리 그려 두는 것도 의미 있어요. 1명일 때 워크플로우 셋업하면, 5명이 됐을 때 추가 셋업이 0이에요. **미리 셋업한 1명의 시간이 미래 5명의 시간을 절약**.

**Q3. 본인이 입사할 회사가 GitHub Flow가 아니면 본 챕터가 헛수고인가요?**
A. 헛수고 아니에요. 셋 패턴은 변형·조합으로 만난 적 다 있어요. GitHub Flow를 깊이 알면 다른 패턴도 1주 안에 적응. 그리고 워크플로우 사고 패턴(충돌 3깊이·리뷰 5톤·PR 사이즈)은 모든 패턴에 공통. **표면이 달라도 본질은 같아요**.

**Q4. 회사가 SVN이나 Mercurial을 쓰면 본 챕터가 의미 있나요?**
A. 90% 의미 있어요. 워크플로우의 사고는 도구 무관. branch·머지·리뷰 개념이 다 있어요. 다만 명령어가 달라요. 본인이 git을 깊이 알면 SVN·Mercurial도 1주에 적응. 다만 2026년 현재 SVN·Mercurial을 쓰는 회사는 매우 드물어요(전통 산업·게임 회사 일부). **git이 사실상 표준**.

**Q5. 본 챕터를 다 끝나고 나면 본인이 시니어가 되나요?**
A. 안 돼요. 시니어는 본 챕터의 내용을 5년 동안 적용해 보면서 만들어져요. 본 챕터는 시니어가 되는 길의 첫 걸음. 본인이 신입 때 워크플로우의 사고를 익혀 두면 5년 동안 같은 사고를 안 만나요. **5년의 사고를 8시간으로 압축한 챕터**. 그리고 시니어는 한 번에 되는 게 아니라 매일 한 칸씩 만들어져요. 본인의 매일 한 PR이 본인의 5년 후 무게. 매일이 누적이고 누적이 무게.

**Q6. 본인의 회사가 git을 안 쓰고 SVN/Perforce/Mercurial을 쓰면요?**
A. 90% 이전 가능해요. 워크플로우 사고(branch·머지·리뷰·release)는 도구 무관. 명령어만 다를 뿐. 본인이 git을 깊이 알면 SVN의 trunk·branches·tags 구조도 1주 안에 적응. 게임 회사가 Perforce를 쓰는 경우가 가장 흔한데 그것도 워크플로우 골격은 동일. **개념이 본질, 명령어가 표면**.

**Q7. 5명 자경단보다 더 큰 팀(20명, 50명) 워크플로우는 본 챕터로 충분한가요?**
A. 50명까지는 충분. 50명+ 큰 팀은 monorepo 도구(Bazel·Nx·Turborepo)·CI 인프라(self-hosted runner)·release engineering(SRE 팀)이 추가돼요. 그건 본 챕터가 아니라 Ch080(아키텍처)·Ch103(DevOps)에서. 본 챕터의 골격(GitHub Flow + 리뷰 + CODEOWNERS)이 50명 팀에서도 그대로 작동해요. **5명에서 50명까지는 같은 패턴, 50명+에서 인프라가 다층화**.

## 10. 추신

추신 1. 협업 워크플로우의 본질 한 줄 — **혼자 git을 잘 쓰는 사람은 신입, 함께 git을 잘 쓰는 사람은 시니어**. 본 챕터가 본인의 시니어 다리. 신입에서 시니어로의 거리는 코드 100만 줄이 아니라 협업 합의 100마디.

추신 2. 세 패턴(GitHub Flow·Git Flow·Trunk-based)은 변할 수 있어요. 본인의 자경단도 5년 후엔 다른 패턴으로 갈 수도. **워크플로우는 살아 있는 계약**. 팀이 변하면 워크플로우도 변해요. 패턴 이름에 매이지 말고 본질을 잡으세요.

추신 3. 충돌의 세 깊이 — 코드 < 의도 < 사회적. 도구로 푸는 건 깊이 1뿐. 깊이 2·3은 사람으로. **5년 차의 일은 깊이 2·3이 95%**.

추신 4. 가장 좋은 충돌 해결은 충돌이 안 나게 만드는 것. **깊이 0**. 좋은 워크플로우의 진짜 가치예요. 예방이 복구의 100배 싸요.

추신 5. PR 사이즈는 작게 자주. <300줄 PR이 머지 1일, >1000줄 PR이 머지 5일. **작은 게 빠르고, 빠른 게 안전**. 작은 PR이 황금 규칙.

추신 6. 리뷰 5톤(칭찬·질문·제안·요청·nit). 본인이 의식적으로 쓰면 동료의 답글이 5분 안에 와요. 안 쓰면 한 시간. **톤이 시간을 절약**.

추신 7. 자경단 5명 시나리오 — 본인·까미·노랭이·미니·깜장이. H5에서 1주일을 시뮬레이션. 본 챕터의 심장.

추신 8. 합주 비유 — 솔로 잘하는 사람이 합주를 잘 하는 게 아니에요. 듣기가 치기보다 중요. **협업은 듣기**. 듣기는 평생 자산.

추신 9. 본인이 회사 입사 첫 주 월요일에 동료에게 한 질문 — "우리는 어떤 워크플로우를 쓰나요?" CONTRIBUTING.md에 답이 있어요. 안 적혀 있으면 동료에게. 1주의 적응을 그 한 질문이 만들어요.

추신 10. 워크플로우는 계약이지만 강제는 도구가. 사람의 합의 + 도구의 강제. 둘 다 있어야 작동. branch protection 한 칸이 본인 합의 100마디보다 단단해요.

추신 11. 본 챕터의 H5(데모)가 가장 깊은 시간. 본인이 5명의 한 주를 머릿속에 그려 보면 협업이 손에 잡혀요. 글 100번보다 시뮬레이션 1번. **체험이 학습**.

추신 12. 다음 H2에서 만나요. 셋 패턴을 깊이 다룹니다. 본인의 자경단이 어느 패턴이 맞는지, 회사 입사 시 어느 패턴을 만날지, 5년 후 본인이 어느 패턴을 디자인할지의 토대. 🐾

추신 13. 본 H를 다 읽은 본인이 한 가지 실험을 해 보세요 — 본인 GitHub 프로필을 열어 가장 최근 PR 한 개를 다시 읽어요. 본인의 톤이 어떤가요? 본문이 충분했나요? 사이즈가 적절했나요? 리뷰어 입장에서 5분 안에 이해됐을까요? **자기 PR 회고가 가장 빠른 학습**. 본인의 어제 PR이 본인의 내일 교과서. 본인이 최근 PR 5개를 다시 보면 본인의 패턴이 보여요.

추신 14. 협업 워크플로우 책 두 권을 추천 — Scott Chacon의 *Pro Git*(공짜·온라인) 5장·6장이 GitHub Flow 원형. Vincent Driessen의 *A successful Git branching model* 블로그(2010, 2020 갱신)가 Git Flow 원형. 둘 다 30분 안에 읽혀요. 원전을 한 번 읽으면 본 챕터의 깊이가 2배. 한국어 번역도 다 있어요.

추신 15. 마지막 — 본 챕터를 끝낸 본인이 자경단 첫 모임을 잡아 보세요. 5명이 모여 "우리는 GitHub Flow를 쓴다"를 한 줄로 합의. 그 한 줄이 본인 자경단의 첫 워크플로우 계약. 합의의 첫 한 줄이 1년의 협업 기반. **첫 한 줄이 가장 무겁고 가장 가벼워요** — 무겁기에 1년을 가져요, 가볍기에 30초면 적어요. 30초가 1년을 만드는 한 칸. 🐾🐾
