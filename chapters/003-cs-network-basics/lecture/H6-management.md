# Ch003 H6 — 트러블슈팅: 네트워크 7다리 진단

> **이 H의 한 줄 약속**: "인터넷이 안 돼요"라는 한마디 비명 앞에서, **7개 다리**를 순서대로 한 다리씩 건너며 어디가 끊어졌는지 1분 안에 짚어내는 진단 프로토콜을 몸에 새긴다.

5교시(H5)에서 우리는 클릭 한 번이 0.0초에서 0.3초까지 30단계를 어떻게 흘러가는지 한 화면에 펼쳤다. 그 30단계는 평소엔 매끄럽게 흘러가지만, 어느 한 단계라도 막히면 사용자는 똑같은 한 문장 — "안 돼요" — 만 외친다. 6교시(H6)는 그 한 문장을 **위치 좌표가 있는 진단 보고서**로 바꾸는 시간이다. 우리가 만들 무기는 단 하나, **7다리 체크리스트**다. 다리 1부터 다리 7까지 순서대로 건너기만 하면, 어느 다리가 끊어져 있는지 — 즉 사용자의 비명이 정확히 어느 OSI 층, 어느 도구, 어느 명령에서 멈추는지 — 답이 자동으로 나온다.

이 H는 5교시의 30단계 흐름을 현실의 사고 시나리오로 압축해 보여주는 "사고 대응 매뉴얼" 성격을 띤다. 우리는 다리 1(인터페이스)부터 다리 7(응답)까지 차례로 건너며 각 다리에서 사용할 명령 한두 개, 정상 출력 한 줄, 실패 시그니처 한 줄을 카드로 적어 둘 것이다. 7장의 카드를 다 모으면 — 머릿속에 — 책상 위 어떤 사고가 들이닥쳐도 "지금 우리는 다리 N에 서 있다"라고 한 단어로 위치를 부를 수 있다. 그 한 단어가 곧 해결의 절반이다.

## 0교시 — 5교시 회수: 30단계를 7다리로 접기

5교시 30단계는 자세히 펼쳐 놓으면 한 화면을 가득 채우지만, 진단의 관점에서는 **딱 7개 관문**으로 접힌다. 30개를 매번 외우려고 하면 사고 한가운데서 머리가 하얘진다. 7개로 접어 두면 손가락 일곱 개로 셀 수 있고, 셀 수 있는 것은 빠뜨리지 않을 수 있다. 5교시가 "왜 이렇게 빠른가"를 보여줬다면, 6교시는 "왜 이렇게 안 되나"를 짚는다. 같은 길을 거꾸로 걷는 셈이다.

| 다리 | 이름 | 5교시 어느 단계? | 한 줄 질문 | 한 줄 도구 |
|---|---|---|---|---|
| 1 | 인터페이스 | 단계 0 (장비) | 내 컴퓨터 랜선·와이파이 살아 있나? | `ifconfig` / `ip a` |
| 2 | 게이트웨이 | 단계 0 (라우터) | 우리 집 공유기까지 닿나? | `ping <gateway>` |
| 3 | 라우팅 | 단계 1 (외부망 진입) | 인터넷 출구가 열렸나? | `route -n get default` / `ip r` |
| 4 | DNS | 단계 5~8 (이름 해석) | 도메인이 IP로 풀리나? | `dig` / `nslookup` |
| 5 | IP 도달 | 단계 9~10 (TCP 직전) | 그 IP까지 패킷이 가나? | `ping <ip>` / `traceroute` |
| 6 | 포트 | 단계 11~13 (TCP 핸드셰이크) | 그 포트가 열려 있나? | `nc -vz` / `telnet` |
| 7 | 응답 | 단계 14~30 (TLS·HTTP·렌더) | 정상 응답이 돌아오나? | `curl -I` / `curl -w` |

**핵심 원칙 셋**:
1. **순서대로 건너라**. 다리 1이 죽었는데 다리 7부터 보면, 다리 7의 실패는 거짓 단서다. 항상 1→2→3→4→5→6→7 순서로 의심한다.
2. **한 다리당 1분**. 7다리 × 1분 = 7분 안에 위치를 못 짚으면, 도구가 부족한 게 아니라 가설이 잘못된 것이다. 가설을 다시 세운다.
3. **정상 출력을 외워라**. 정상이 무엇인지 모르면 비정상도 못 본다. 각 다리의 "정상 한 줄"을 카드 뒤에 적어 둔다.

이 셋만 지키면 6교시의 70% 이상은 끝났다. 나머지 30%는 다리별 손맛이다.

한 가지 더 — **다리 7에서 사고가 보인다고 다리 7만 의심하지 말 것**. 사용자에게 보이는 증상은 늘 마지막 다리(응답)에서 터지지만, 진짜 원인은 거의 항상 더 앞쪽 다리에 숨어 있다. 502를 보고 "백엔드 앱이 죽었나"라고 곧장 점프하는 것은 베테랑이 가장 자주 빠지는 함정이다. 베테랑일수록 처음 1분은 무조건 다리 1부터 차례로 두드린다. 이 1분의 자제가 평균 진단 시간을 절반으로 줄인다. 사고 한가운데서 가장 비싼 자원은 두뇌의 집중력이고, 7다리 순서는 그 집중력을 가장 효율적으로 분배하는 알고리즘이다.

## 1교시 — 다리 1: 인터페이스 (Interface) 살아 있나

### 비유: 집 현관문

다리 1은 일곱 다리 중 가장 가까이에 있고, 가장 먼저 의심해야 하며, 동시에 가장 자주 "설마 여기서?"라며 건너뛰는 다리다. 베테랑일수록 "내 노트북 와이파이는 멀쩡할 게 뻔하다"라며 다리 1을 생략하다가 30분을 잃는다. 1분만 투자해 다리 1을 확인하면 그 30분은 통째로 절약된다. 집을 나가려면 일단 현관문이 열려야 한다. 현관문이 잠겨 있거나 떨어져 있다면, 게이트웨이도 라우팅도 DNS도 다 무의미하다. **인터페이스(NIC, Network Interface Card)** 가 OS에 인식돼 있고 IP가 박혀 있어야 비로소 "네트워크가 시작됐다"라고 말할 수 있다.

### 명령

> ▶ **같이 쳐보기** — 다리 1: 내 NIC 가 살아 있고 IP 가 박혀 있는지
>
> ```bash
> # macOS
> ifconfig en0
> # Linux
> # ip a show wlan0
> # 한 줄 요약
> ifconfig en0 | awk '/inet /{print "IP:", $2}'
> ```

### 정상 출력 한 줄

```
en0: flags=8863<UP,BROADCAST,SMART,RUNNING,SIMPLEX,MULTICAST> mtu 1500
        inet 192.168.0.42 netmask 0xffffff00 broadcast 192.168.0.255
        status: active
```

여기서 우리가 봐야 할 단어 셋: **UP**(링크 활성), **RUNNING**(케이블/와이파이 연결), **inet 192.168.0.x**(IP 부여 받음). 셋 다 있으면 다리 1 합격, 다리 2로 간다.

### 실패 시그니처 셋

| 시그니처 | 진단 | 응급 처치 |
|---|---|---|
| `status: inactive` | 와이파이/랜선 물리 단절 | 케이블 다시 꽂기 / 와이파이 재연결 |
| `inet` 줄이 없음 | DHCP 실패, IP 미부여 | `sudo ipconfig set en0 DHCP` (mac), `sudo dhclient -r && sudo dhclient` (Linux) |
| `inet 169.254.x.x` | DHCP 실패 후 자동 할당(APIPA) | 공유기 재부팅 + DHCP 재요청 |

### 사고 시나리오

"VPN 켰는데 갑자기 인터넷 안 돼요" → `ifconfig`로 보면 `utun0`이 살아 있고 `en0`은 죽어 있는 게 아니라, 라우팅 테이블이 `utun0`으로 다 빠져 있다. 다리 1은 합격, 진짜 범인은 다리 3(라우팅). 다리 1에서 멈추지 말고 다음으로 넘어가라.

### 추가 손맛

`ifconfig`가 너무 길면 `ifconfig | grep -E "^[a-z]|inet "`로 인터페이스 + IP만 한눈에. macOS에서는 `networksetup -listallhardwareports`로 "와이파이 = en0, 유선 = en1" 같은 매핑을 한 번 외워 두면 평생 편하다. Linux에서는 `nmcli device status` 한 줄이 같은 역할을 한다. 노트북·데스크톱·서버를 오갈 때 "내 와이파이 인터페이스 이름"부터 헷갈리는 일이 잦은데, 이 한 줄을 셸 히스토리 맨 위에 올려 두면 5초 안에 답이 나온다. 인터페이스 이름은 OS·드라이버·USB 동글에 따라 `en0`·`wlan0`·`wlp3s0`·`enp0s3`·`utun0`처럼 다 다르므로, 다리 1에서 가장 먼저 외워야 할 것은 명령이 아니라 **내 장비의 진짜 인터페이스 이름**이다.

## 2교시 — 다리 2: 게이트웨이 (Gateway) 까지 닿나

### 비유: 동네 골목 입구

다리 2는 "내가 우리 LAN의 끝까지는 닿는가"를 묻는 다리다. 이 다리가 죽어 있다면 인터넷 진단은 의미가 없다 — 우리는 아직 우리 동네 밖으로 나가지도 못한 셈이기 때문이다. 다리 2는 진단 시간이 가장 짧고(ping 4번 = 4초) 결과가 가장 명료한 다리이므로, 다리 1이 합격하면 즉시 다리 2를 두드리는 게 손에 익어야 한다. 현관문은 열렸다. 이제 골목 입구의 가로등(공유기)까지 걸어가 봐야 한다. 게이트웨이는 우리 LAN의 출구이고, 여기까지 ping이 안 가면 인터넷은 시작도 못 한다. 다리 1은 "내 다리가 있는가"였다면, 다리 2는 "옆집 대문 앞까지 다리가 닿는가"다.

### 명령

> ▶ **같이 쳐보기** — 다리 2: 우리 동네 골목 입구(공유기) 까지 닿나
>
> ```bash
> # 게이트웨이 IP 알아내기 — macOS
> route -n get default | awk '/gateway/{print $2}'
> # Linux 라면: ip route | awk '/default/{print $3}'
>
> # ping 4번
> ping -c 4 192.168.0.1
> ```

### 정상 출력 한 줄

```
64 bytes from 192.168.0.1: icmp_seq=0 ttl=64 time=2.341 ms
--- 192.168.0.1 ping statistics ---
4 packets transmitted, 4 received, 0.0% packet loss
```

`0.0% packet loss` + `time < 10ms`. LAN 안의 게이트웨이에 5ms 이상 걸리면 와이파이가 약하거나 공유기가 과부하다.

### 실패 시그니처 둘

- `Request timeout` 100% — 공유기가 죽었거나 ICMP 차단. 공유기 LED 확인, 재부팅.
- `0.0% loss`인데 `time = 800ms` — 와이파이 신호 약함. 채널 변경, 5GHz 대역 사용.

### 사고 시나리오

"카페 와이파이 잡히는데 인터넷 안 돼요" → 다리 1은 ✅(IP 받았음), 다리 2 ping 시도 → `Destination Host Unreachable`. 카페 라우터가 캡티브 포털(captive portal — 동의 페이지) 통과 전이라 게이트웨이까지의 트래픽을 막아 둔 상태. 브라우저로 아무 HTTP 사이트 한 번 열어 동의 페이지를 띄우면 해결.

### 추가 손맛

게이트웨이까지의 ping 평균 지연이 평소 2ms이던 와이파이가 갑자기 80ms로 튄다면 — 다리 2의 합격/불합격은 합격이지만 — 그 이후 모든 다리에 0.078초의 추가 지연이 누적된다. 즉 다리 2는 "예/아니오"만 보는 게 아니라 "얼마나 빠른가"도 함께 본다. `ping -c 100 192.168.0.1 | tail -3`으로 100번 ping 평균을 한 번 떠 두면, 평소 평균이 머릿속에 자리잡아 다음 사고 때 "오늘은 평소보다 5배 느리다"라는 미세한 신호도 즉시 잡을 수 있다. 와이파이 사고의 절반은 이 평균값 변동으로 먼저 들킨다.

## 3교시 — 다리 3: 라우팅 (Routing) 테이블

### 비유: 동네에서 시내로 나가는 길

다리 3은 보이지 않게 사고를 일으키는 가장 음흉한 다리다. 인터페이스도 정상, 게이트웨이도 정상, 그러나 라우팅 테이블이 엉뚱한 곳을 가리키면 패킷은 조용히 사라지고 사용자는 "왜인지 모르겠지만 안 돼요"만 외친다. VPN·도커·가상머신·코어 네트워크 도구가 늘어날수록 다리 3 사고는 비례해서 늘어난다. 그래서 다리 3 진단은 "한 번에 알아보는 마법 명령" 하나를 손가락에 새겨 두는 게 평생의 자산이다. 골목 입구까지는 갔다. 이제 시내(인터넷)로 나가는 길이 어떻게 그려져 있는지 지도를 봐야 한다. **라우팅 테이블**이 그 지도다. `default` 행이 누락됐거나 엉뚱한 인터페이스로 빠져 있으면, 게이트웨이가 살아 있어도 패킷은 다른 길로 새서 영영 돌아오지 않는다.

### 명령

> ▶ **같이 쳐보기** — 다리 3: 시내로 나가는 지도(라우팅 테이블) 한 번 펴기
>
> ```bash
> # macOS
> netstat -rn | head -20
> route -n get default
> # Linux 라면 두 줄로 끝:
> # ip route
> # ip route get 8.8.8.8     # 마법 한 줄: 지금 8.8.8.8 로 가는 첫 홉이 누구냐
> ```

### 정상 출력 한 줄

```
Destination        Gateway            Flags        Netif
default            192.168.0.1        UGScg          en0
```

`default`가 한 줄, 게이트웨이는 다리 2에서 본 그 IP, Netif는 다리 1에서 본 인터페이스. 셋이 일치하면 합격.

### 실패 시그니처 셋

| 시그니처 | 진단 |
|---|---|
| `default` 행 자체가 없음 | DHCP가 게이트웨이 정보 못 받음 |
| `default`가 두 개 (예: en0, utun0) | VPN 켜진 상태, 우선순위 충돌 |
| `default`가 `utun0`인데 VPN은 죽어 있음 | VPN 끊긴 후 라우팅 청소 안 됨 — 클라이언트 재시작 |

### 사고 시나리오

"회사 VPN 끄니까 인터넷 안 돼요" → 다리 3의 default가 아직 `utun0`로 박혀 있음. VPN 클라이언트가 끊길 때 라우팅을 원복하지 않는 버그. `sudo route delete default && sudo route add default 192.168.0.1`로 응급 처치, 재발 방지엔 VPN 클라이언트 업그레이드.

### 추가 손맛

`ip route get 8.8.8.8`(Linux) 또는 `route get 8.8.8.8`(macOS) 한 줄은 다리 3의 **결정적 도구**다. 이 명령은 "지금 이 시점에 8.8.8.8로 가는 첫 홉이 누구냐"를 OS의 실제 라우팅 결정 알고리즘에 물어보고 답한다. 라우팅 테이블을 사람 눈으로 읽는 것보다 빠르고 정확하다. 두 개의 default가 있을 때 어느 쪽이 실제 선택되는지도 이 한 줄로 즉시 드러난다. 기억할 명령은 `ip route get 8.8.8.8` 단 하나, 그것이 다리 3 카드 뒷면에 적힐 마법의 문장이다.

## 4교시 — 다리 4: DNS — 이름이 풀리나

### 비유: 전화번호부

다리 4 — DNS — 는 일곱 다리 중 가장 자주 죽고, 가장 헷갈리게 죽고, 가장 "내 책임 아닌데 내 책임처럼 보이는" 다리다. 캐시가 5층(브라우저·OS·시스템 리졸버·회사 캐시·ISP·권위 서버)으로 쌓여 있어 어느 층이 거짓말 중인지 한눈에 안 보이고, 동료의 결과와 내 결과가 다르고, 5분 뒤 알아서 풀리기도 한다. 그래서 베테랑들 사이에서는 "확실하지 않으면 일단 DNS 의심"이라는 농담 격언이 있다. 농담이 아니라 통계적으로 맞다. 라우팅까지 정상이라면 시내로 나갈 길은 있다. 그러나 우리가 가려는 곳은 "구글" 같은 이름이지 IP 숫자가 아니다. **DNS**는 이름을 IP로 바꿔주는 전화번호부다. 전화번호부가 닫혀 있으면, 길은 알아도 누굴 부를지 모른다. 다리 4는 가장 자주 죽는 다리이고, 가장 헷갈리게 죽는 다리다.

### 명령

> ▶ **같이 쳐보기** — 다리 4: 이름 → IP 전화번호부가 살아 있나
>
> ```bash
> dig google.com +short
> dig +trace google.com | head -30      # 루트 → .com → google.com 경로 추적
> nslookup google.com 8.8.8.8           # 강제로 구글 DNS 에 직접
> scutil --dns | head -20               # macOS: 현재 사용 중인 DNS 서버
> # Linux 라면: cat /etc/resolv.conf
> ```

### 정상 출력 한 줄

```
$ dig google.com +short
142.250.196.142
```

IP 한 줄. 끝.

### 실패 시그니처 넷 (다리 4가 죽는 4가지 방식)

1. **DNS 서버 자체가 죽음** — `dig`가 30초 timeout 후 `;; connection timed out`. → `nslookup google.com 8.8.8.8`로 강제 우회 시도.
2. **DNS 서버는 살아 있는데 내 캐시가 썩음** — 어제 IP를 들고 있어 새 서버로 못 감. → `sudo dscacheutil -flushcache`(mac) / `sudo systemd-resolve --flush-caches`(Linux).
3. **/etc/hosts에 잘못된 줄** — 모든 DNS보다 우선. → `cat /etc/hosts | grep -v "^#"`로 의심 줄 확인.
4. **회사망에서 사내 DNS만 풀고 외부 DNS 못 풀거나 그 반대** — VPN 분할 터널 설정 문제. → `dig @사내DNS google.com`과 `dig @8.8.8.8 사내도메인` 둘 다 시도.

### 사고 시나리오

"방금 도메인 산 사이트가 안 떠요. 동료는 잘 떠요." → 동료는 회사 캐시 DNS가 새 레코드를 이미 받았고, 나는 옛 NXDOMAIN을 캐시 중. 캐시 플러시 + `dig +trace`로 권위 서버까지 직접 물어 새 레코드를 강제 갱신.

### 추가 손맛

DNS는 5교시에서 배운 5절(루트→.com→google.com→ www→ A) 캐시가 곳곳에 있다는 사실 때문에 진단이 어렵다. **"내 캐시 → 시스템 리졸버 → 회사 캐시 → ISP 캐시 → 권위 서버"** 다섯 층 중 어디가 썩었는지 한 번에 보려면 `dig +trace`가 정답이다. `+trace`는 캐시를 모두 우회하고 루트부터 차례로 묻기 때문에, 어느 층에서 거짓말이 시작되는지를 직접 본다. 진단할 때마다 매번 친다고 손에 익혀 둘 것. 다리 4의 카드 뒷면 마법 문장은 `dig +trace 도메인 | head -30` 한 줄이면 충분하다.

## 5교시 — 다리 5: IP 도달 (Reachability)

### 비유: 집 주소를 알아도 차가 못 들어가는 골목

다리 5는 "패킷이 정말 거기까지 가는가"를 묻는 다리다. 4교시(DNS)에서 IP 주소를 얻었다고 해서 그 IP가 도달 가능하다는 보장은 없다. 그 IP의 데이터센터가 다운됐을 수도, 중간 ISP가 흔들릴 수도, 해저 케이블이 끊겼을 수도 있다. 다리 5는 "내 책임 영역"의 끝, "외부 세계의 책임 영역"의 시작 지점에 있는 결정적 경계선이다. 이 다리에서 사고가 나면 우리가 할 수 있는 일은 보통 "우회 경로 찾기"와 "기록 남기기" 둘뿐이다. DNS가 IP를 줬다. 이제 그 IP까지 패킷이 실제로 도달하는지 본다. 도메인은 풀렸어도 그 서버가 멀리 있는 데이터센터에서 죽어 있을 수도, 중간 라우터가 끊어졌을 수도 있다. 다리 5는 "패킷이 거기까지 가는가"를 직접 묻는다.

### 명령

> ▶ **같이 쳐보기** — 다리 5: 그 IP 까지 패킷이 진짜로 가는지
>
> ```bash
> ping -c 4 142.250.196.142          # 도달성 확인
> traceroute -n 142.250.196.142      # 어느 중간 홉에서 죽나
> mtr -n -c 30 142.250.196.142       # ping + traceroute 합본 (있으면 최고)
> ```

### 정상 출력 한 줄

```
$ ping -c 4 142.250.196.142
64 bytes from 142.250.196.142: icmp_seq=0 ttl=115 time=23.4 ms
4 packets transmitted, 4 received, 0.0% packet loss
```

`0.0% packet loss`. 손실 5% 넘으면 경로 어딘가가 흔들린다.

### 실패 시그니처 셋

| 시그니처 | 진단 |
|---|---|
| `ping`은 가는데 `traceroute`가 중간에서 `* * *` | 중간 라우터가 ICMP TTL 응답만 막음. 보통 정상. |
| `ping`이 100% 손실 | 서버 측 ICMP 차단 또는 진짜 다운 |
| `ping`이 50% 손실 | 경로 상의 패킷 드롭, 보통 와이파이 또는 ISP 장애 |
| `traceroute` 마지막 홉에서 `!H` | 호스트 도달 불가, 라우팅 블랙홀 |

### 사고 시나리오

"우리 서비스가 일본 사용자만 느려요" → mtr로 한국→일본 경로 30초 측정. 7번째 홉(해저 케이블 진입점)에서 패킷 손실 12%. 우리 책임 아님, ISP에 티켓. CDN 일본 엣지 활성화로 우회.

### 추가 손맛

다리 5에서 가장 흔한 함정은 "ICMP 차단 = 서버 죽음" 오해다. 많은 클라우드 보안 그룹이 ICMP를 기본 차단하고 TCP만 연다. 즉 `ping`이 100% 손실이라도 다리 5의 진짜 답은 "ICMP만 막혔을 뿐 TCP는 살아 있을 수 있다"이다. 이때는 즉시 다리 6으로 넘어가 `nc -vz <ip> 443`을 시도해 본다. TCP가 응답하면 다리 5는 (실용적으로) 합격이고, 진짜 사고는 응답 단계에 있다. 그래서 다리 5와 다리 6은 늘 짝꿍이다 — 한 쪽이 침묵해도 다른 쪽이 말하면 그쪽 답을 믿는다.

## 6교시 — 다리 6: 포트 (Port) 가 열렸나

### 비유: 호텔 객실 번호

다리 6은 다리 5와 짝꿍이다. 다리 5가 "호텔 건물까지 갔는가"라면 다리 6은 "우리가 묵을 객실 번호가 열려 있는가"이다. 클라우드 시대에 가장 자주 사고가 나는 다리이기도 하다 — Security Group, NACL, 회사 방화벽, OS 방화벽, 컨테이너 네트워크 정책, 서비스 메시 정책, K8s NetworkPolicy 등 "포트를 막는 장치"가 7~8겹으로 쌓여 있기 때문이다. 어느 한 층만 닫혀 있어도 다리 6은 무너진다. 서버까지 패킷이 갔다. 그러나 어느 객실(포트)이 열려 있는지는 따로 묻는다. 80은 닫혀 있고 443만 열려 있을 수도, 22(SSH)는 열려 있는데 5432(Postgres)는 막혀 있을 수도 있다. 다리 6은 5교시 30단계 중 "TCP 3-way handshake가 SYN-ACK로 응답하느냐"를 직접 짚는다.

### 명령

> ▶ **같이 쳐보기** — 다리 6: 그 객실(포트)이 열려 있는지 노크
>
> ```bash
> nc -vz google.com 443                # 한 포트 빠른 확인
> nc -vz google.com 80 443 8080        # 여러 포트 한 번에
> # nmap -p 80,443,5432 example.com    # 포트 스캔 — 자기 서버에만!
> ```

### 정상 출력 한 줄

```
Connection to google.com port 443 [tcp/https] succeeded!
```

`succeeded`. 이 한 단어가 SYN-ACK가 돌아왔다는 증거다.

### 실패 시그니처 셋

| 시그니처 | 진단 |
|---|---|
| `Connection refused` | 서버는 살아 있는데 그 포트에 프로세스가 없음. systemd가 죽인 상태. |
| `Operation timed out` | 방화벽이 SYN을 삼킴. AWS Security Group 또는 회사 방화벽. |
| `succeeded`인데 다리 7에서 응답 없음 | TCP는 받지만 애플리케이션이 멈춤. 다리 7로. |

세 시그니처의 차이는 결정적이다. **refused = 프로세스 없음**, **timeout = 방화벽**, **succeeded = 다리 7로**. 이 셋만 외워도 사고의 절반이 풀린다.

### 사고 시나리오

"방금 배포한 API가 502를 뱉어요" → 로드 밸런서 → 앱 서버로 `nc -vz app.internal 8080` → `Connection refused`. 앱 프로세스가 부팅 중 크래시했음을 다리 6이 즉시 알려준다. `journalctl -u myapp` 로그 확인 → 환경변수 누락, 롤백.

### 추가 손맛

`nc -vz`는 "한 패킷도 안 보내고 SYN만 던져 응답을 본다"는 점에서 가장 가벼운 도구다. 1초 안에 답이 나오고 서버에 거의 흔적을 남기지 않는다. 그래서 운영 사고 한가운데서 가장 자주 쓴다. 그러나 한 가지 함정 — 어떤 보안 솔루션은 SYN만 던지는 행동을 "포트 스캔"으로 분류해 IP를 차단한다. 회사망에서는 자기 서버에만, 바깥 서버에는 한두 번만. `nmap`은 더더욱 — 자기 권한 있는 서버에만. 권한 없는 서버에 `nmap`을 돌리는 건 나라마다 형사 처벌 대상이다. 다리 6의 카드 뒷면에 빨간 글씨로 적어 둘 것: **"내 권한 있는 서버에만"**.

## 7교시 — 다리 7: 응답 (Response) 이 정상인가

### 비유: 호텔 객실 문 열고 응대받기

다리 7은 가장 다채로운 다리이고, 사용자가 직접 보는 유일한 다리이며, 따라서 사고 신고가 가장 많은 다리다. 그러나 앞서 강조했듯 다리 7의 사고 보고는 종종 거짓 단서다 — 다리 1~6 어딘가가 진짜 범인이고 다리 7은 단지 그 결과를 사용자에게 보여줄 뿐인 경우가 절반을 넘는다. 다리 7 진단의 핵심 기술은 "한 화면에 다섯 숫자를 동시에 보는 것"이다. 그래야 어느 다리에서 시간이 새는지를 한눈에 안다. 객실은 열려 있다. 노크해서 종업원이 정상적으로 답하는지 본다. 다리 7은 5교시 30단계의 14~30단계 — TLS, HTTP 요청·응답, 헤더, 본문 — 전체가 정상으로 끝나는지를 한 줄 명령으로 묻는다.

### 명령

```sh
# 헤더만 빠르게
curl -I https://example.com

# 다섯 숫자(5교시 H5에서 본 그 명령!)
curl -w "@-" -o /dev/null -s https://example.com <<'EOF'
namelookup:    %{time_namelookup}s
connect:       %{time_connect}s
appconnect:    %{time_appconnect}s
starttransfer: %{time_starttransfer}s
total:         %{time_total}s
http_code:     %{http_code}
EOF

# 자세한 핸드셰이크
curl -v https://example.com 2>&1 | head -30

# TLS 인증서만
openssl s_client -connect example.com:443 -servername example.com </dev/null 2>/dev/null | openssl x509 -noout -dates -subject
```

### 정상 출력 한 줄

```
HTTP/2 200
content-type: text/html
```

`200` 또는 `301`/`302`(리다이렉트). `4xx`/`5xx`면 다리 7 불합격, 서버 측 사고.

### 실패 시그니처 다섯

| 코드 | 진단 |
|---|---|
| `000` (curl이 응답 못 받음) | 다리 1~6 어딘가 거짓 정상 — 다시 1로 |
| `301` `302` | 정상, 단 무한 루프면 앱 설정 |
| `403` | 권한·CORS·WAF 차단 |
| `502` `503` `504` | 게이트웨이 ↔ 앱 사이 사고. 다리 6 다시 |
| `200`인데 본문이 비어 있거나 깨짐 | 콘텐츠 인코딩(gzip) 오류, 캐시 풀링 |

### 사고 시나리오

"사이트가 반쯤 떠요. CSS만 깨져요." → 다리 7 `curl -I https://cdn.example.com/styles.css` → `200`이지만 `content-type: text/html`. CDN이 404를 200으로 위장한 fallback HTML을 보냄. 캐시 무효화 → 정상 복구.

### TLS 함정 셋

1. **인증서 만료** — `openssl x509 -dates`로 `notAfter` 확인. Let's Encrypt 자동 갱신 멈춤.
2. **SNI 미설정** — `curl --resolve` 또는 `openssl s_client -servername`으로 명시.
3. **체인 누락** — 중간 인증서를 서버가 안 보냄. `openssl s_client` 출력에 chain이 1개뿐이면 누락.

### 추가 손맛

다리 7의 `curl -w` 다섯 숫자는 5교시 H5의 30단계와 정확히 매핑된다. `time_namelookup`은 다리 4(DNS), `time_connect`는 다리 6(TCP), `time_appconnect`는 TLS, `time_starttransfer`는 서버 처리(TTFB), `time_total`은 전체. 이 다섯 숫자를 보면 7다리 중 어느 다리에서 시간이 새는지 한 줄로 보인다. 다리 진단은 "예/아니오"이지만 다섯 숫자는 "어느 다리가 느린가"까지 답한다. 그래서 다리 7의 카드 뒷면 마법 문장은 `curl -w` heredoc 한 토막이고, 이 한 토막을 셸 함수 `httptime`으로 등록해 두면 사고 90%가 1초 안에 위치 좌표를 얻는다.

## 8교시 — 통합 진단 함수: `nettest()` 한 방

7다리는 강력하지만 사고 한가운데서 7개의 다른 명령을 차례로 치는 일은 의외로 손에 부담이 된다. 패닉 상태에서는 `route -n get default | awk` 같은 한 줄도 오타가 난다. 그래서 베테랑은 거의 예외 없이 자기만의 "한 줄 진단 함수"를 갖고 있다. 이 함수에 도메인과 포트만 넘기면 7다리가 차례로 보고하고, ❌가 처음 뜨는 다리가 사고의 위치 좌표가 된다. 7다리를 매번 7개 명령으로 두드리면 손이 느리다. zsh 함수 한 개로 묶어 `nettest google.com 443`이라고 한 줄 치면 7다리가 차례로 보고하도록 만들어 두자.

```sh
# ~/.zshrc 에 추가
nettest() {
  local host="${1:-google.com}"
  local port="${2:-443}"
  local gw

  echo "════ 7다리 진단: $host:$port ════"

  echo "── 다리 1: 인터페이스 ──"
  ifconfig en0 2>/dev/null | awk '/inet /{print "  IP:", $2}' \
    || echo "  ❌ en0 없음"

  echo "── 다리 2: 게이트웨이 ──"
  gw=$(route -n get default 2>/dev/null | awk '/gateway/{print $2}')
  [[ -n "$gw" ]] && ping -c 1 -t 2 "$gw" >/dev/null 2>&1 \
    && echo "  ✅ gateway $gw 응답" \
    || echo "  ❌ gateway $gw 무응답"

  echo "── 다리 3: 라우팅 ──"
  route get 8.8.8.8 2>/dev/null | awk '/interface|gateway/{print "  ", $0}'

  echo "── 다리 4: DNS ──"
  local ip=$(dig +short +time=2 +tries=1 "$host" | head -1)
  [[ -n "$ip" ]] && echo "  ✅ $host → $ip" \
    || { echo "  ❌ DNS 실패"; return 1; }

  echo "── 다리 5: IP 도달 ──"
  ping -c 2 -t 3 "$ip" >/dev/null 2>&1 \
    && echo "  ✅ $ip 도달" \
    || echo "  ⚠️  ICMP 무응답 (다리 6으로)"

  echo "── 다리 6: 포트 ──"
  nc -vz -G 3 "$ip" "$port" 2>&1 | sed 's/^/  /'

  echo "── 다리 7: 응답 ──"
  curl -o /dev/null -s -w "  http=%{http_code} dns=%{time_namelookup}s tcp=%{time_connect}s tls=%{time_appconnect}s ttfb=%{time_starttransfer}s total=%{time_total}s\n" \
    "https://$host:$port/" 2>&1 \
    || echo "  ❌ curl 실패"

  echo "═════════════════════════════"
}
```

사용:

```sh

## 9교시 — 운영 사고 시나리오 셋

### 시나리오 1: "DNS 사고" — 인증 로그인 실패

**현상**: 화요일 오후 3시, 갑자기 사용자들이 "로그인하려는데 빙글빙글 돌기만 해요"라고 신고. 우리 사이트 메인 페이지는 잘 뜬다.

**진단**:
```sh
$ nettest auth.example.com 443
── 다리 4: DNS ──
  ❌ DNS 실패
```

다리 4에서 막힘. 그러나 `dig auth.example.com @8.8.8.8`은 답이 옴. 즉 회사 사내 DNS만 죽었다. `dig auth.example.com`(시스템 기본) → timeout. `scutil --dns`로 보니 사내 DNS가 두 개 등록되어 있고 첫 번째가 응답 없음.

**해결**: 인프라팀 호출, 사내 DNS primary 재시작. 5분 뒤 정상. 사용자에게는 별도 공지 없이 종료.

**교훈**: "우리 사이트는 멀쩡한데 외부 OAuth만 안 되면 다리 4 의심." DNS는 늘 둘 이상 두고, 헬스체크에 한 줄 추가.

### 시나리오 2: "인증서 사고" — 어제까진 됐어요

**현상**: 수요일 새벽 1시, 모니터링이 "https://api.example.com 502 빈도 폭증" 알람. 사람들 다 자고 있다.

**진단**:
```sh
$ nettest api.example.com 443
── 다리 7: 응답 ──
  http=000 ...

$ openssl s_client -connect api.example.com:443 -servername api.example.com </dev/null 2>&1 | grep -i "verify\|notAfter"
verify error:num=10:certificate has expired
notAfter=Apr 26 23:59:59 2026 GMT
```

다리 7 `http=000` + openssl이 expired 보고. 인증서가 어제 자정에 만료. Let's Encrypt 자동 갱신 cron이 90일 만에 한 번 도는데, 지난 갱신에서 cron 잡 자체가 죽어 있었음.

**해결**: 새벽 1시 30분에 수동으로 `certbot renew --force-renewal && systemctl reload nginx`. 정상 복구. 다음날 cron 모니터링 추가, 만료 30일 전 알람 추가.

**교훈**: 인증서 만료 알람은 30일·14일·7일·1일 4단계로. 자동 갱신은 반드시 수동 검증과 함께.

### 시나리오 3: "포트 충돌" — 배포 후 502

**현상**: 목요일 배포 직후, 새 버전이 502. 롤백할까 망설이는 중.

**진단**:
```sh
$ nettest api.internal 8080
── 다리 6: 포트 ──
  nc: connectx to 10.0.1.5 port 8080 (tcp) failed: Connection refused
```

`refused`. 즉 서버는 살아 있고 8080에 프로세스가 없음. `ssh app-1 'lsof -nP -iTCP:8080 -sTCP:LISTEN'` → 빈 줄. `journalctl -u myapp -n 50` → "Address already in use, port 8080" 후 즉시 죽음. 이전 버전 프로세스가 graceful shutdown 실패로 좀비 상태.

**해결**: `pkill -9 myapp` 후 `systemctl restart myapp`. 5초 뒤 정상. 배포 스크립트에 `pkill && wait` 추가, graceful shutdown 타임아웃 30초로 상향.

**교훈**: 다리 6의 `refused`는 "프로세스가 없다"의 결정적 증거. 즉시 `lsof`와 `journalctl`로 점프할 것.

## 10교시 — 흔한 함정 7가지

1. **순서 무시** — 다리 7부터 보고 "API가 죽었다"고 결론 내린 후 알고 보니 다리 4(DNS) 사고. 항상 1→7.
2. **ICMP만 믿기** — `ping`이 100% 손실이라고 서버가 죽은 게 아니다. ICMP 차단일 수 있다. 즉시 다리 6으로.
3. **자기 캐시 무시** — `dig`는 시스템 캐시를 본다. `dig +trace`나 `dig @8.8.8.8`로 우회해 봐야 진짜 답이 보인다.
4. **"내 컴퓨터에선 됐는데"** — 동료의 다리 4 캐시는 다를 수 있다. 동료에게도 `nettest`를 시켜라.
5. **방화벽 셀프 차단** — `nc`로 너무 많이 두드리면 회사 보안 솔루션이 내 IP를 차단한다. 한 사고당 최소 횟수만.
6. **VPN 좀비 라우팅** — VPN 끊긴 후 default가 `utun0`에 박혀 있으면 다리 3에서 잡힌다. 의심하면 `route get 8.8.8.8`.
7. **인증서 만료를 5xx로 오인** — `http=000`이면 즉시 openssl로 인증서 점검. 5xx는 앱 사고이지만 000은 핸드셰이크 사고.

## 11교시 — FAQ 일곱

**Q1. `ping`이 안 가면 무조건 서버가 죽은 건가요?**
A. 아닙니다. ICMP만 차단됐을 가능성이 큽니다. 즉시 다리 6 `nc -vz`로 TCP를 두드려 보세요. TCP가 응답하면 서버는 살아 있습니다.

**Q2. `traceroute`가 중간에 `* * *`로 막히면 그 라우터가 고장난 건가요?**
A. 대부분 아닙니다. 많은 라우터가 ICMP TTL 응답을 보안상 차단합니다. 마지막 홉만 도달하면 정상입니다. `* * *`는 "안 보여줄 뿐"이지 "죽음"이 아닙니다.

**Q3. `dig`와 `nslookup` 중 뭘 써야 하나요?**
A. `dig`. `nslookup`은 비대화형 모드에서 약간 옛날 도구이고, `dig`는 +trace, +short, @서버 같은 강력한 옵션이 직관적입니다.

**Q4. `nc -vz`와 `nmap` 중 뭘 써야 하나요?**
A. 한두 포트면 `nc -vz`, 수십 포트 한 번에면 `nmap`. 단 `nmap`은 권한 있는 서버에만.

**Q5. 503과 504 차이는?**
A. 503은 "서버는 살아 있는데 처리할 여력 없음"(과부하·점검 모드), 504는 "내(게이트웨이)가 뒷단(앱)을 기다리다 timeout". 503은 앱 자체, 504는 게이트웨이~앱 사이.

**Q6. HTTPS 사이트에서 `curl: (60) SSL certificate problem`은?**
A. 1) 인증서 만료 → openssl로 dates 확인. 2) 자체 서명 인증서 → `curl -k`로 임시 우회 후 정식 인증서로 교체. 3) 시스템 root CA 번들이 오래됨 → OS 업데이트.

**Q7. `nettest` 함수가 너무 길어요. 더 짧게 안 되나요?**
A. 가능합니다. 다리 1·2·3은 보통 한 번 보면 며칠은 안 죽으니, 운영 사고 시엔 다리 4·5·6·7만 묶은 `quicknet()`을 따로 만들어 둬도 좋습니다. 매일 쓰는 건 짧게, 새벽 사고 대응은 풀버전으로.

## 12교시 — 7교시(H7) 예고

다음 H7은 "서버 측 동작 깊게 — keepalive·HTTP/3·로드 밸런싱 내부"입니다. 이번 H6의 다리 6·7에서 우리가 본 "TCP 연결"과 "HTTP 응답"은 사실 서버 측에서 훨씬 더 많은 일을 합니다 — 같은 TCP를 여러 요청에 재사용하는 keepalive, UDP 위에서 도는 HTTP/3, 백엔드 5대를 가르는 로드 밸런서 알고리즘(round robin·least conn·consistent hash). H7이 끝나면 여러분의 다리 7 진단은 "200 OK"에서 멈추지 않고 "어느 백엔드, 어느 연결, 어느 프로토콜로 200을 받았는가"까지 보게 됩니다.

## 추신 1~24

1. 7다리는 OSI 7계층과 1:1 매핑은 아니지만 진단의 관점에서는 가장 잘 작동하는 추상이다. OSI는 학문, 7다리는 실전.
2. 사고 한가운데서 가장 큰 적은 패닉이다. 7다리 카드는 패닉을 막는 부적이다.
3. 정상 출력을 모르면 비정상도 못 본다. 한가할 때 `nettest google.com 443`을 한 번 떠 두라.
4. macOS의 `ifconfig`는 deprecation 경고가 뜨지만 아직 멀쩡히 작동한다. `ip a`(iproute2)는 Linux 표준이지만 macOS엔 없다.
5. `ping -c` 옵션은 macOS·Linux 동일, `-W`(타임아웃)는 macOS는 `-t`로 다르니 주의.
6. `traceroute`는 UDP 기반(macOS 기본), `traceroute -I`는 ICMP 기반. 회사 방화벽이 UDP를 막으면 `-I` 시도.
7. `mtr`는 강력하지만 macOS 기본 미설치. `brew install mtr` 후 `sudo mtr ...`로 사용.
8. `nc`는 BSD nc(macOS)와 GNU netcat(Linux)이 옵션이 다르다. macOS는 `-G`(연결 timeout), Linux는 `-w`. 셸 함수 만들 때 주의.
9. `dig +trace`는 권위 서버 부하를 살짝 준다. 운영 중인 도메인에 너무 자주 치지 말 것. 진단 시 한 번이면 충분.
10. `/etc/hosts`는 모든 DNS보다 우선. 개발자가 디버깅하다 남긴 줄이 사고의 원흉인 경우가 의외로 많다. 사고 시 한 번 들여다볼 것.
11. `curl -v`의 `*`는 디버그 정보, `>`는 보낸 헤더, `<`는 받은 헤더. 이 세 기호만 외우면 `-v` 출력이 시처럼 읽힌다.
12. `openssl s_client -showcerts`는 인증서 체인 전체를 보여준다. 체인 누락은 모바일에서만 터지는 사고이므로 반드시 체인 길이를 확인.
13. SNI(Server Name Indication)는 한 IP에 여러 도메인이 올라간 시대의 필수품. `curl --resolve`와 `openssl -servername`이 SNI 친구.
14. `nslookup`의 대화형 모드는 옛 유닉스 시절 흔적. 지금은 `dig`를 쓰는 게 모든 면에서 낫다.
15. Linux의 `systemd-resolved`는 `127.0.0.53:53`에서 캐시한다. 디버깅 시 이 캐시도 의심 대상.
16. macOS의 `mDNSResponder`는 .local 도메인과 시스템 캐시를 동시에 담당한다. `sudo killall -HUP mDNSResponder`가 만능 리프레시.
17. AWS·GCP·Azure의 보안 그룹은 기본적으로 ICMP 차단. EC2 인스턴스가 `ping` 안 가도 놀라지 말 것.
18. CloudFront·Cloudflare 같은 CDN은 사용자 위치에 따라 다른 엣지로 라우팅한다. 다리 5의 mtr 결과가 사람마다 다르다.
19. HTTP/3(QUIC)는 UDP 443에서 돈다. 다리 6의 `nc -vz`는 TCP 443만 본다. HTTP/3 진단은 `curl --http3`로.
20. WebSocket 사고는 다리 7에서 101 Switching Protocols 코드 확인. 200을 기대하면 안 됨.
21. CORS 사고는 사이트가 멀쩡한데 브라우저 콘솔에만 빨간 줄이 뜬다. 다리 7 `curl -I` 헤더에 `Access-Control-Allow-Origin` 보일 것.
22. CDN 캐시 사고는 `curl -I`의 `x-cache: HIT/MISS` 헤더로 잡힌다. HIT인데 옛날 콘텐츠면 캐시 무효화.
23. 7다리는 서버 운영뿐 아니라 본인 PC 셋업, 회사 입사 첫날 "왜 사내 사이트가 안 떠요" 사고에도 그대로 통한다. 평생 무기.
24. 6교시는 30단계 이론(H5)을 7개의 손가락으로 압축했다. 이제 사고가 들이닥쳐도 손가락 하나씩 짚어 가며 답을 찾을 수 있다. 7교시(H7)에서는 다리 7의 "200 OK" 뒤에 숨은 서버 측 깊은 사연으로 들어간다.
