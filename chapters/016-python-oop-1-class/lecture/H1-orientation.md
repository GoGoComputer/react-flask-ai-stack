## 2. 이게 뭔지 아시나요 — 진짜 자경단 다섯 마리

이게 뭔지 아시나요? 본인이 1년 동안 들어온 단어. 자경단 다섯 마리. 까미, 노랭이, 미니, 깜장이, 본인. 1년 내내 list 또는 dict로 표현했어요.

```python
cats = ["까미", "노랭이", "미니", "깜장이", "본인"]
ages = {"까미": 3, "노랭이": 2}
```

문제는요, 이 표현이 너무 빈약해요. 까미라는 cat에는 이름만 있는 게 아니에요. 나이가 있고, 색깔이 있고, 좋아하는 음식이 있고, 자기만의 특기가 있고. 본인이 이 모든 정보를 표현하려면 dict가 dict 안에 들어가는 그런 복잡한 구조가 돼요.

```python
cats = {
    "까미": {"age": 3, "color": "black", "skill": "스파이"},
    "노랭이": {"age": 2, "color": "yellow", "skill": "보디가드"},
    ...
}
```

OOP가 등장하는 순간이에요. 까미라는 cat이 그냥 dict의 한 항목이 아니라 **하나의 객체**가 돼요. 자기만의 정보 (속성)를 가지고, 자기만의 행동 (메서드)을 가지는. 본인이 까미.greet() 한 줄 치면 까미가 자기 인사를 해요. 까미.feed("참치") 하면 까미가 참치를 먹어요. 살아있는 객체.

이게 OOP의 첫 인상이에요. 자료를 묶고, 행동을 묶고, 같이 다닌다. 그래서 객체.

## 3. 옛날 이야기 — 첫 클래스를 짠 그 날

옛날 이야기 하나 할게요. 제가 처음 클래스를 짠 날. 13년 전쯤이에요.

회사에서 사용자 관리 시스템을 만들고 있었어요. 처음엔 dict로 다 했어요. user = {"name": ..., "email": ...}. 함수 50개가 그 dict를 받아서 처리. 한 달 후엔 코드가 미로가 됐어요. 어떤 함수가 어떤 키를 만지는지 알 수가 없었어요.

사수 형이 와서 "User 클래스로 묶어" 한 줄. 저는 처음 class 키워드를 쳤어요. dict였던 50줄이 클래스 한 개와 메서드 다섯 개로 바뀌었어요. 코드가 절반으로 줄었어요. 그리고 신기하게도 동료가 코드를 읽기 시작했어요. 전엔 "user dict가 뭐예요?" 물어보던 동료가 "User 클래스에 send_email 메서드 추가해 주세요" 같은 말을 해요. 같은 코드, 다른 표현. 다른 협업.

그날 저는 OOP의 진짜 가치를 깨달았어요. **OOP는 같이 일하는 사람들을 위한 도구**. 본인 혼자 코드 짜면 함수로 충분해요. 다섯 명이 같은 코드를 만지면 OOP가 강력해져요. 자경단 다섯 명이 한 저장소를 만지는 그 순간부터 OOP가 빛나요.

본인도 8시간 후 똑같이 깨달아요. 약속드려요.

## 4. 왜 OOP인가 — 일곱 가지 이유

본인이 OOP를 배워야 하는 이유 일곱 개. 한 개씩 짚어 갈게요.

**첫째, 자료와 행동을 같이 묶어요**. dict + 함수의 조합보다 자료와 그 자료를 만지는 행동이 한 곳에 있는 게 가독성이 좋아요. 까미의 정보와 까미의 행동이 까미 객체 안에 다 있어요.

**둘째, 협업의 어휘가 명확해져요**. "User 클래스에 send_email 추가해 주세요" 같은 한 줄로 동료와 합의가 돼요. dict 시절엔 안 됐어요.

**셋째, 코드 재사용**. 한 클래스를 만들면 인스턴스 100개 만들 수 있어요. 자경단 다섯 마리가 같은 Cat 클래스로 표현돼요. 코드 한 번, 데이터 백 번.

**넷째, 상속으로 확장**. SpecialCat이 Cat을 상속하면 Cat의 모든 기능 + SpecialCat의 추가 기능. Ch017에서 자세히.

**다섯째, FastAPI, SQLAlchemy, Pydantic이 다 OOP**. 본인이 백엔드 짜려면 이 세 도구를 매일 만나요. 모두 클래스 기반.

**여섯째, AI 시대의 OOP**. AI 도구한테 "User 클래스에 admin 메서드 추가" 부탁하면 5초에 해 줘요. 클래스 단위로 일해요.

**일곱째, 면접 단골**. "클래스와 인스턴스 차이는?" "상속과 합성 중 무엇이 좋나?" 신입 면접에 단골이에요.

일곱 이유. 본인의 2년차가 OOP 위에서 굴러가요.

## 6. 네 친구 — class·instance·attribute·method

OOP 안에 친구가 네 명 살아요.

**첫 친구는 class**. 객체의 설계도. Cat이라는 class는 cat이 어떤 모양인지 정의. 진짜 cat은 아직 안 만들어졌어요. 그냥 도면이에요.

**두 번째 친구는 instance**. 그 도면으로 만든 진짜 객체. `Cat("까미", 3)`이 인스턴스. 까미라는 cat이 메모리에 진짜로 살아 있어요.

**세 번째 친구는 attribute (속성)**. 인스턴스가 가지는 데이터. `까미.name`, `까미.age`. dict의 key 비슷.

**네 번째 친구는 method**. 인스턴스가 할 수 있는 행동. `까미.greet()`. 함수인데 객체에 붙어 있어요.

네 친구. 외우려 마세요. 8시간 동안 한 명씩 다시 만나요. 지금은 "OOP에 친구가 네 명 있구나"만 머리에.

## 7. 0.001초의 여행 — 인스턴스 생성 5단계

본인이 `Cat("까미", 3)` 한 줄 칠 때 0.001초 안에.

**1단계**. `__new__`가 호출돼서 메모리에 빈 객체 공간 할당.

**2단계**. `__init__`이 호출돼서 그 빈 객체를 초기화. self가 그 객체. self.name = "까미" 같은 줄이 실행.

**3단계**. 객체가 reference count 1로 시작. 본인이 까미 변수에 할당하면 카운트 유지.

**4단계**. 본인이 까미.name 칠 때 attribute lookup. 인스턴스에 없으면 클래스에 있나 확인.

**5단계**. 까미 변수가 사라지면 reference count 0. 가비지 컬렉터가 메모리 회수.

5단계. 0.001초. CPython의 객체 생성 흐름. H7에서 깊이 다뤄요. 오늘은 그림.

## 8. 자경단 다섯 마리를 객체로

자, 1년차에 list로 표현했던 자경단 다섯 마리를 클래스로 다시 짜 봐요.

```python
class Cat:
    def __init__(self, name, age, color, skill):
        self.name = name
        self.age = age
        self.color = color
        self.skill = skill
        self.energy = 100
    
    def greet(self):
        return f"안녕, 나는 {self.name}이에요. {self.age}살이고 특기는 {self.skill}이에요."
    
    def feed(self, food):
        self.energy = min(self.energy + 20, 100)
        return f"{self.name}가 {food}를 맛있게 먹었어요."
    
    def patrol(self):
        if self.energy < 20:
            return f"{self.name}는 너무 피곤해요."
        self.energy -= 30
        return f"{self.name}가 자경단 순찰을 다녀왔어요. 에너지: {self.energy}"


까미 = Cat("까미", 3, "검정", "스파이")
노랭이 = Cat("노랭이", 2, "노랑", "보디가드")
미니 = Cat("미니", 4, "회색", "야간 잠복")
깜장이 = Cat("깜장이", 5, "턱시도", "QA")
본인 = Cat("본인", 1, "흰색", "메인테이너")

vigilante = [까미, 노랭이, 미니, 깜장이, 본인]
for cat in vigilante:
    print(cat.greet())
    print(cat.feed("참치"))
    print(cat.patrol())
```

자, 이게 자경단 다섯 마리의 OOP 표현이에요. 각 cat이 자기 정보를 가지고, 자기 행동 (greet, feed, patrol)을 가지는. 본인이 patrol 한 번 돌리면 energy가 줄어요. 살아 있는 객체.

1년차에는 dict로 짰던 게 이제 클래스로. 표현이 풍부해졌어요.

2014년. Python 3.5. type hints + class 결합.

2018년. Python 3.7. dataclass. boilerplate 절감.
