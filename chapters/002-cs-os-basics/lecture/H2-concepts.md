## 3. 프로세스 (1/4) — 실행 중인 프로그램 한 개

**프로세스(process).** 영어 그대로의 한 단어. 한국어 번역 잘 안 써요. 그냥 "프로세스" 라고 해요.

한 줄 정의 다시. **"실행 중인 프로그램 한 개."** 이 정의의 핵심은 "실행 중" 이라는 세 글자예요. 본인 디스크에 sysinfo.sh 파일이 있어요. 그건 그냥 파일이에요, 프로세스가 아니에요. 본인이 `./sysinfo.sh` 라고 친 그 순간, OS가 그 파일을 메모리에 올리고 CPU에게 "이 명령들 실행해" 라고 시켜요. 그 순간부터 프로세스 한 개가 태어나요. 실행이 끝나면 죽어요. 디스크의 파일은 그대로 있어요. 프로그램(파일)과 프로세스(실행 중)는 다른 거예요. 외워두세요.

본인 맥에 지금 프로세스 200~400개가 있다고 했어요. 한 번 들여다 봐요.

> ▶ **같이 쳐보기** — 손님 명부 첫 10줄 한 번 보기
>
> ```bash
> ps aux | head -10
> ```

USER, PID, %CPU, %MEM, VSZ, RSS, TTY, STAT, START, TIME, COMMAND. 11개 컬럼이 나와요. 그 중에서 오늘은 5개만 봐요.

| 컬럼 | 의미 |
|------|------|
| **USER** | 누구의 프로세스인가 (root, 본인 ID, _appstore 등) |
| **PID** | 이 프로세스의 번호 (1번부터) |
| **%CPU** | 지금 CPU를 얼마나 쓰는가 |
| **%MEM** | RAM을 얼마나 쓰는가 |
| **COMMAND** | 어떤 명령인가 |

본인이 지금 출력에서 "어 크롬이 CPU 30% 쓰네" 하고 한 번 보세요. 또는 "아 이 프로세스는 root 가 도네, 시스템 거구나." 이렇게 보는 습관 한 번. ps 한 줄이 한 명의 손님 카드예요. 호텔 매니저의 손님 명부 한 장.

그리고 한 가지 큰 사실. **모든 프로세스는 메모리(RAM)에서 자기 자리를 가져요.** 그 자리를 다른 프로세스가 못 들여다봐요. 호텔의 방마다 자물쇠가 걸려 있어요. 크롬 프로세스가 본인 비밀번호 기억하고 있다 해도, VS Code 프로세스가 그걸 못 봐요. 메모리 격리. OS의 가장 큰 발명품 중 하나. 이게 없으면 본인 컴퓨터가 매일 해킹당해요. 진짜로요.

## 4. 프로세스의 일생 — 태어나서 죽기까지

프로세스 한 명의 인생을 한 번 따라가 봐요. 7단계예요.

1. **태어남(birth).** 본인이 명령을 치거나, 다른 프로세스가 새 프로세스를 만들어요. PID 부여.
2. **준비(ready).** "나 이제 일할 준비 됐어!" 매니저(스케줄러)에게 줄 섰어요.
3. **실행(running).** 매니저가 차례를 줘요. CPU 시간 0.001초 ~ 0.1초 받아요. 일 함.
4. **대기(waiting/blocked).** 디스크 읽기 같은 느린 일을 시작. 결과 기다리는 동안 잠깐 자요.
5. **다시 준비.** 결과 왔어요. 다시 줄 섬.
6. **반복.** 이 줄서기·실행·대기를 1초에 수만 번 반복.
7. **죽음(termination).** 일이 끝났거나, 누군가 kill 명령으로 죽임. 메모리 회수, PID 반납.

이 7단계가 모든 프로세스의 표준 인생. 매니저(스케줄러)가 1초에 수만 번 "이 손님 → 저 손님 → 저 손님" 차례를 돌려요. 시분할(time-sharing). 본인 눈에는 200개 손님이 동시에 일을 하는 것 같지만, 사실은 매니저가 너무 빨리 번갈아서 그렇게 보이는 거예요. CPU 코어 한 개는 한 번에 한 프로세스밖에 못 돌려요. 본인 맥이 8코어면 진짜 동시 실행은 8개. 나머지 200~400개는 매니저가 빠르게 돌리는 거예요.

본인이 H4에서 본 `top` 명령이 이 7단계 중 어느 단계에 있는지 보여줘요. STAT 컬럼이 그거예요. R(Running), S(Sleeping/대기), T(Stopped/멈춤), Z(Zombie/좀비), I(Idle). 이 5글자는 H6 응급 처치에서 진짜 자주 만나요. 좀비(Z)는 특히. 죽었는데 시체가 안 치워진 프로세스. 그 시체가 PID를 점유. 너무 많이 쌓이면 새 프로세스 못 만들어요. H6에서 자세히.

## 5. PID·PPID·부모 자식 트리

프로세스에는 PID만 있는 게 아니에요. **PPID(Parent PID)** 도 있어요. 모든 프로세스는 자기를 만든 부모가 있어요. 부모 프로세스의 PID 가 PPID. 그래서 본인 컴퓨터의 모든 프로세스는 한 그루의 큰 가족 나무를 만들어요.

루트(뿌리)는 PID 1번. macOS는 launchd, Linux는 init 또는 systemd. 본인 맥이 부팅된 그 순간 OS가 가장 먼저 만든 프로세스. 그 후 모든 프로세스가 launchd의 자식 또는 자식의 자식의 자식... 으로 만들어져요.

본인 셸의 가족 한 번 봐요.

> ▶ **같이 쳐보기** — 내 zsh 와 그 부모 PID 보기
>
> ```bash
> echo $$        # 현재 zsh의 PID. 예: 12345
> echo $PPID     # 그 zsh를 만든 부모. 보통 터미널 앱 (Terminal.app, iTerm2)
> ```

그리고 부모의 부모를 거슬러 올라가면 결국 launchd(PID 1)에 도달. 본인 맥의 모든 프로세스가 한 명의 조상에서 갈라진 자식들이에요. 진짜 한 가족이에요.

자식 프로세스는 부모가 죽으면 어떻게 되냐. **고아(orphan)** 가 돼요. 그러면 PID 1번이 양부모로 자동 입양해요. 그래서 launchd 가 "고아원 원장" 같은 역할도 해요. 모든 부모 잃은 프로세스를 받아줘요. 매니저의 또 다른 큰 일.

이 부모-자식 관계가 두 해 후 본인이 만나는 두 가지 큰 단어로 이어져요. **데몬(daemon)** = 부모가 일찍 죽고 launchd 의 자식이 된 백그라운드 프로세스. **세션(session)** = 한 부모 아래의 자식 무리. nginx·postgresql·redis 같은 서버 프로세스가 다 데몬이에요. 본인이 두 해 후 클라우드 서버 운영할 때 이 단어들이 매일 등장. 오늘 한 번 들어두세요.

## 6. fork() 와 exec() — 프로세스가 만들어지는 두 동작

프로세스가 어떻게 만들어지냐. 유닉스의 진짜 우아한 발명품 한 가지를 봐요. **fork() + exec()** 의 두 동작.

**fork().** 한 줄 정의. **"부모가 자기를 똑같이 복사해서 자식 한 명을 만든다."** 그래요, 진짜로 복사예요. 부모와 똑같은 메모리, 똑같은 코드, 똑같은 변수. PID만 달라요. 부모는 자식의 PID를 받고, 자식은 0을 받아요(자기는 자식이라는 표시). 그 뒤로 부모와 자식이 각자 길을 가요.

**exec().** 한 줄 정의. **"내 자신을 다른 프로그램으로 갈아끼운다."** fork 로 만든 자식은 부모랑 똑같은 코드만 도는데, exec 를 부르면 그 자식이 다른 프로그램(예: ls)으로 자기 자신을 통째로 바꿔요. 메모리 다 비우고 ls 코드 로딩. PID는 그대로.

그래서 본인이 zsh 에서 `ls` 를 칠 때 일어나는 일.

1. zsh 가 fork(). 본인 zsh 의 자식 zsh 한 명이 새로 태어남.
2. 그 자식 zsh 가 즉시 exec("/bin/ls"). 자기를 ls로 갈아끼움.
3. ls 가 실행. 결과를 화면에 출력.
4. ls 가 끝남. 자식 죽음.
5. 부모 zsh 가 다음 명령 받을 준비.

이게 유닉스에서 모든 명령 실행의 표준 절차예요. 본인이 매번 명령을 칠 때마다 이 5단계가 일어나요. 1초에 한 번. 본인이 모르는 사이에. **유닉스의 진짜 핵심 발명품**이에요. 다른 OS(예: 윈도우)는 이 분리가 없어요. 윈도우는 CreateProcess() 한 함수가 다 해요. 유닉스의 fork+exec 분리가 더 우아하고 더 유연해요. 그래서 셸 스크립트가 그렇게 강력해요.

이 단어들은 H7 (커널 vs 유저 공간)에서 더 깊이 봐요. 오늘은 단어 두 개만. **fork() = 복사, exec() = 갈아끼우기.** 머리에 두세요.

## 7. 프로세스 상태 5가지 — R·S·T·Z·I

H6 에서 자세히 만날 단어 5개를 미리. ps의 STAT 컬럼.

| 코드 | 영어 | 한 줄 의미 |
|------|------|------------|
| **R** | Running | 지금 CPU 쓰고 있음 |
| **S** | Sleeping | 자고 있음 (이벤트 기다리는 중) |
| **T** | Stopped | 멈춤 (Ctrl+Z 같은 신호로) |
| **Z** | Zombie | 죽었는데 시체 안 치워짐 |
| **I** | Idle | 거의 안 자는 자는 상태 (커널) |

대부분의 프로세스는 거의 항상 S 상태예요. 자고 있어요. 일이 들어오면 깨어나서 0.001초 정도 R 상태로 일하고 다시 S로 돌아가요. 본인 맥의 200개 프로세스 중에서 진짜 R 상태인 건 보통 5개 미만. 본인 ps aux 한 번 더 쳐서 STAT 컬럼 한 번 보세요. R 가 거의 없을 거예요. S 가 대부분.

Z(Zombie) 는 사고예요. 정상이 아니에요. 부모가 자식의 죽음을 확인 안 해줘서 시체가 PID 만 점유하고 있는 상태. 너무 많아지면 PID 가 바닥나서 새 프로세스 못 만들어요. H6에서 좀비 청소법 자세히 봐요. 오늘은 단어 한 개만. **Z = 위험 신호.**

## 8. 스레드 (2/4) — 한 프로세스 안의 일꾼들

두 번째 손길. **스레드(thread).** 한국어로는 "실" 이에요. 옷감 짤 때 그 실. 영어 thread 도 "실, 한 가닥" 이라는 뜻에서 왔어요. 한 줄짜리 실행 흐름이에요.

한 줄 정의 다시. **"한 프로세스 안의 실행 흐름 한 가닥."** 비유 다시. 프로세스가 호텔 방 한 칸이라면, 스레드는 그 방 안에서 일하는 사람 한 명. 한 방에 사람 여러 명 있을 수 있어요. 각자 다른 일.

크롬 한 프로세스의 예. 스레드 30~50개 있을 수 있어요.

- 스레드 1: 화면 그리기 (렌더링)
- 스레드 2: 자바스크립트 실행 (V8)
- 스레드 3: 네트워크 다운로드
- 스레드 4: 마우스/키보드 입력 받기
- 스레드 5~30: 각 탭의 일들
- 스레드 31~50: GPU·오디오·미디어·확장 등

이렇게 여러 일을 동시에 하려고 스레드 여러 명. 만약 스레드가 한 명만 있으면 다운로드 받는 동안 화면이 멈춰요. 본인 옛날 프로그램에서 가끔 화면이 회색으로 변하면서 "응답 없음" 뜨는 거 있죠. 그게 스레드 한 명이 무거운 일에 막혀서 다른 일을 못 하는 상태. 요즘 앱은 다 스레드 여러 개 써서 그 일이 잘 안 일어나요.

스레드와 프로세스의 진짜 큰 차이 한 줄. **"같은 방을 쓰느냐 다른 방을 쓰느냐."**

| 항목 | 프로세스 | 스레드 |
|------|----------|--------|
| 메모리 | 따로따로 (격리) | 같이 씀 (한 방 안) |
| 통신 비용 | 비쌈 (방 사이) | 쌈 (한 방 안) |
| 만들기 비용 | 비쌈 (수 ms) | 쌈 (수 us) |
| 안정성 | 한 명 죽어도 다른 명 안 죽음 | 한 명 죽으면 같은 방 다 죽음 |
| 비유 | 호텔 방 | 방 안의 사람 |

**같은 방을 쓴다는 게 진짜 큰 의미예요.** 한 방의 사람들이 같은 가방, 같은 메모, 같은 화이트보드를 공유. 그래서 통신이 빠른데, 그래서 충돌이 잦아요. 두 사람이 같은 메모를 동시에 쓰면 글자가 섞여요. 이걸 **경쟁 상태(race condition)** 라고 해요. 다음 절에서 한 번 봐요.

본인 맥의 스레드 한 번 보세요.

> ▶ **같이 쳐보기** — 내 맥의 총 스레드 수 한 줄로 보기
>
> ```bash
> top -l 1 | head -3
> # 둘째 줄에 "Threads: 3xxx total" 같은 게 나와요.
> ```

본인 맥에 보통 3,000~5,000 개의 스레드가 있어요. 프로세스 200~400 의 약 10~20배. 한 프로세스마다 평균 10~20 스레드. 이 숫자가 OS 가 진짜로 관리하는 일꾼의 숫자.

## 9. 동시성 vs 병렬성 — 헷갈리는 두 단어

스레드 이야기 나왔으니까 한 번 정리해야 하는 두 단어가 있어요. 면접에서 자주 나오는 질문이에요.

**동시성(concurrency).** 한 줄 정의. **"여러 일을 번갈아가며 한다."** 한 사람이 양손에 빵 굽기와 면 끓이기를 동시에 한다는 느낌. 진짜 동시는 아니에요. 한 손이 빵 보면서 다른 손이 면 보고, 빠르게 번갈아. 컴퓨터에서는 CPU 한 코어가 여러 프로세스/스레드를 빠르게 번갈아 처리하는 게 동시성.

**병렬성(parallelism).** 한 줄 정의. **"여러 일을 진짜 동시에 한다."** 두 사람이 한 명은 빵 굽고, 다른 한 명이 면 끓여요. 같은 시간에 두 일이 진짜로 동시에 진행. 컴퓨터에서는 CPU 코어 여러 개가 각자 다른 프로세스/스레드를 진짜로 동시에 처리하는 게 병렬성.

본인 맥이 8코어면, **8개까지는 병렬, 그 이상은 동시.** CPU 코어 8개가 각자 한 일씩 진짜 병렬. 그 위의 200~400개 프로세스는 8코어가 빠르게 번갈아가며 처리하는 동시성.

이 한 표를 외워두세요. 두 해 후 면접 한 번에 직격탄으로 나올 수 있어요.

| 단어 | 한 줄 | 비유 | 예 |
|------|-------|------|-----|
| 동시성 | 번갈아 한다 | 한 명이 양손 일 | 1코어 + 멀티 스레드 |
| 병렬성 | 진짜 동시 | 두 명이 각자 일 | 멀티 코어 + 멀티 스레드 |

흔한 답: "동시성은 일을 다루는 방식, 병렬성은 일을 실행하는 방식." 한 줄로 답하면 면접관이 만족해요.

## 10. 경쟁 상태(race condition) — 한 단어 미리보기

한 방의 사람 두 명이 같은 메모를 동시에 고친다고 했어요. 글자가 섞여요. 이 사고를 **경쟁 상태(race condition)** 라고 해요. 한 자원에 여러 일꾼이 경쟁적으로 접근해서 결과가 꼬이는 상태.

예를 들어 본인이 카운터 변수 한 개에 두 스레드가 +1 을 동시에 하려고 해요.

```
스레드 A: counter (= 5) 읽음 → 5
스레드 B: counter (= 5) 읽음 → 5
스레드 A: 5 + 1 = 6, counter = 6 저장
스레드 B: 5 + 1 = 6, counter = 6 저장
결과: counter = 6.  (기대 = 7)
```

두 번 +1 했는데 한 번만 됐어요. 사라진 +1. 본인이 보는 모든 동시성 버그의 원형이에요. 두 해 후 백엔드에서 본인이 진짜 만나는 사고. 사용자 두 명이 동시에 같은 좋아요 버튼 누르면 좋아요 수가 한 번만 올라가는 사고. 이게 race condition.

해결법은 **잠금(lock, mutex)** 이에요. 한 번에 한 명만 들어갈 수 있는 화장실 문 잠금처럼. 한 스레드가 카운터 만지는 동안 다른 스레드는 기다려요. 두 해 후 자세히. 오늘은 단어만 두 개. **race condition = 경쟁 상태, lock = 잠금.** 머리에 두세요.

## 11. 파일 (3/4) — 디스크 위 이름 붙은 데이터

세 번째 손길. **파일(file).** 한 줄 정의. **"디스크 위 이름 붙은 데이터 한 덩어리."** 본인이 만든 sysinfo.sh 가 파일 한 개. 사진 한 장도 파일. PDF 하나도 파일.

파일에는 두 가지가 있어요. **데이터 본체** (실제 0과 1)와 **메타데이터(metadata)** (그 데이터에 대한 정보 — 이름·크기·만든 시간·소유자·권한). 메타데이터는 파일 시스템이 디스크 어딘가에 따로 저장. macOS의 APFS, Linux의 ext4 등 파일 시스템이 이 일을 해요.

`ls -la` 한 줄을 한 번 분해해 봐요.

```
-rw-r--r--  1  mo  staff  1234  Apr 26 14:30  sysinfo.sh
```

| 부분 | 의미 |
|------|------|
| `-rw-r--r--` | 권한 (다음 절에서) |
| `1` | 하드 링크 수 |
| `mo` | 소유자 |
| `staff` | 그룹 |
| `1234` | 크기 (바이트) |
| `Apr 26 14:30` | 마지막 수정 시간 |
| `sysinfo.sh` | 이름 |

이 한 줄이 파일의 신분증이에요. 본인이 매일 `ls -la` 칠 때마다 이 7가지 정보를 받는 거예요. 단어 한두 개만 외워두세요. 권한·소유자·크기. 나머지는 자주 보면 익숙해져요.

그리고 파일에는 **inode 번호** 라는 게 있어요. 디스크 위 파일의 진짜 ID. 이름이 같아도 inode가 다르면 다른 파일. 이름이 달라도 inode가 같으면 같은 파일(하드 링크). `ls -li` 로 볼 수 있어요. 두 해 후 자세히.

## 12. 파일 디스크립터(file descriptor) — fd 한 단어

이건 두 해 코스에서 진짜 자주 만나는 단어. **파일 디스크립터(file descriptor, fd).** 한 줄 정의. **"한 프로세스가 연 파일에 OS 가 붙여 주는 작은 번호."**

본인이 셸 스크립트 안에서 파일 한 개를 열면, OS 가 "너가 연 그 파일은 3번이야" 라고 알려 줘요. 그 3번이 fd. 그 후에 본인은 "3번에 데이터 써" 라고 부탁하면 OS 가 그 파일에 써 줘요. 이름 대신 번호로 다루는 거예요. 더 빨라요.

기본으로 모든 프로세스가 태어날 때 fd 3개가 자동으로 열려 있어요. **0·1·2.**

| fd | 이름 | 한 줄 |
|----|------|-------|
| **0** | stdin (표준 입력) | 키보드에서 들어오는 입력 |
| **1** | stdout (표준 출력) | 화면에 나가는 출력 |
| **2** | stderr (표준 에러) | 화면에 나가는 에러 메시지 |

본인이 H4·H5 에서 본 `2>&1` 이라는 이상한 기호 기억나세요. "에러(2)를 출력(1)에 합쳐라" 라는 뜻이에요. 이 두 숫자가 정확히 fd 2와 fd 1. 본인이 모르고 썼던 그 숫자가 사실 fd 였어요. 오늘 한 번 정확히 봤어요.

본인 셸의 fd 한 번 보세요.

> ▶ **같이 쳐보기** — 내 셸이 지금 열고 있는 파일들 한 번에 보기
>
> ```bash
> lsof -p $$ | head -10
> ```

FD 컬럼에 0u, 1u, 2u 같은 게 있어요. 그게 fd 0·1·2. 다음 줄 3·4·5는 본인 셸이 추가로 연 파일들. zsh 가 보통 5~10개의 파일을 열어 둬요.

## 13. 표준 입출력 — stdin·stdout·stderr 세 채널

fd 0·1·2 를 좀 더 봐요. 유닉스의 진짜 우아한 발명품 중 하나. 모든 프로세스가 이 세 채널을 가져요.

- **stdin (fd 0):** 입력 받는 곳. 보통 키보드.
- **stdout (fd 1):** 정상 출력 보내는 곳. 보통 화면.
- **stderr (fd 2):** 에러 메시지 보내는 곳. 보통 화면.

왜 출력 채널이 둘이냐. **에러와 정상 출력을 분리하기 위해서예요.** 본인이 명령의 정상 결과만 파일에 저장하고 싶고 에러는 화면에 띄우고 싶을 때, 두 채널이 따로 있어서 가능해요.

```bash
ls /존재안함 > out.txt           # stdout만 out.txt로. stderr는 화면에.
ls /존재안함 > out.txt 2>&1      # 둘 다 out.txt로.
ls /존재안함 2>/dev/null         # 에러는 버리고, 정상 출력만 화면에.
```

이 세 줄이 두 해 코스에서 본인이 매일 100번 쓰는 패턴이에요. 오늘 한 번 정리하고 가요. **>** 는 stdout 리다이렉트, **2>** 는 stderr 리다이렉트, **2>&1** 은 stderr를 stdout과 합치기, **/dev/null** 은 모든 데이터의 무덤(쓰면 사라짐).

그리고 파이프(|)는 stdout 만 연결해요. 한 명령의 stdout 을 다음 명령의 stdin 으로. `ps aux | grep chrome` 에서 ps 의 stdout 이 grep 의 stdin 이 되는 거예요. 이게 유닉스 철학의 진짜 구현. fd 0과 fd 1을 파이프로 연결해서 작은 도구들을 큰 일로 묶어요. 50년 전 두 사람이 만든 기적.

## 14. 파일 권한 — rwx 9비트의 비밀

파일의 권한 부분을 한 번 풀어 봐요. `-rw-r--r--`. 이 10글자를 분해해요.

**첫 글자.** 파일 종류. `-` 는 일반 파일, `d` 는 디렉터리, `l` 은 심볼릭 링크. 그 외에도 c·b·s 등 있는데 일반 사용자는 거의 안 봐요.

**나머지 9글자.** 3개씩 묶음 3개. 각 묶음이 r(read)·w(write)·x(execute) 의 3비트.

```
-  rw-  r--  r--
↑   ↑    ↑    ↑
종류 소유자 그룹 기타
```

- **소유자(owner):** 파일 만든 사람.
- **그룹(group):** 그 사람이 속한 그룹의 멤버들.
- **기타(other):** 그 외 모든 사람.

각 그룹마다 r·w·x 3비트. 총 9비트.

| 글자 | 의미 (파일) | 의미 (디렉터리) |
|------|-------------|------------------|
| **r** | 읽기 | 디렉터리 안의 파일 목록 보기 |
| **w** | 쓰기 | 안에 파일 만들고 지우기 |
| **x** | 실행 | 디렉터리 들어가기 (cd) |

`-rw-r--r--` 풀이. 일반 파일. 소유자: 읽고 쓸 수 있음(rw-). 그룹: 읽기만(r--). 기타: 읽기만(r--). 본인이 만드는 보통 파일의 기본 권한.

`chmod` 명령으로 권한 바꿔요. `chmod +x sysinfo.sh` 가 실행 권한 추가. 본인이 H5에서 친 명령 기억나시죠. 그게 정확히 이거.

이 9비트가 OS 의 가장 기본 보안. 본인 맥의 모든 파일에 이 9비트가 붙어 있어요. 매니저가 매번 파일 접근할 때 이 9비트로 자격 확인. 두 해 후 본인이 클라우드 서버에서 매일 만나는 단어. 오늘 한 번 정리.

## 15. 시스템 콜 (4/4) — 앱이 OS에게 부탁하는 한 마디

마지막 손길. **시스템 콜(system call, syscall).** 한 줄 정의. **"앱이 OS(커널)에게 부탁하는 한 마디."**

H1 에서 단어만 봤어요. 오늘 좀 더 봐요. 시스템 콜은 본인 앱과 커널 사이의 약속된 함수예요. 본인 앱이 직접 디스크를 만질 수 없어요(user space). 커널만 만질 수 있어요(kernel space). 그래서 본인 앱은 커널에게 "이거 해 주세요" 라고 부탁해요. 그 부탁의 한 마디가 시스템 콜.

대표 시스템 콜 8개 (H1에서 본 표 다시).

| syscall | 한 줄 |
|---------|-------|
| `open()` | 파일 열고 fd 받기 |
| `read(fd)` | fd에서 바이트 읽기 |
| `write(fd)` | fd에 바이트 쓰기 |
| `close(fd)` | fd 닫기 |
| `fork()` | 프로세스 복사 |
| `exec()` | 프로세스 갈아끼우기 |
| `socket()` | 네트워크 소켓 만들기 |
| `mmap()` | 파일을 메모리에 매핑 |

이 8개가 유닉스의 진짜 척추예요. 본인이 만든 sysinfo.sh 가 실행되는 동안 이 시스템 콜들이 수백 번 호출돼요. 본인이 echo 한 번 칠 때마다 write(1, "...", n) 한 번이 일어나요. 본인이 cat 한 번 칠 때마다 open() → read() → write() → close() 가 일어나요. 모든 명령이 시스템 콜 시퀀스예요.

본인 맥에서 직접 한 번 보고 싶으면 dtruss 명령(권한 필요).

> ▶ **같이 쳐보기** — ls 한 번에 syscall 몇 번 부르는지 세기
>
> ```bash
> sudo dtruss -c ls 2>&1 | tail -20
> ```

ls 가 호출하는 시스템 콜 카운트가 나와요. 보통 ls 한 번에 수십 개. 본인이 ls 한 번 칠 때 매니저(커널)에게 수십 마디 부탁을 하는 거예요.

## 16. syscall 흐름 — 7단계 그림

시스템 콜이 일어나는 그 한순간의 7단계를 그려요. H7에서 자세히. 오늘은 큰 그림.

1. **앱이 함수 호출.** 본인 코드가 `read(fd, buf, size)` 호출.
2. **라이브러리가 받음.** libc(C 라이브러리) 가 그 호출을 받아서 정리.
3. **트랩 명령(trap).** CPU 에 특별한 명령(syscall, int 0x80, svc 등)을 보내서 "지금 커널 모드로 점프!"
4. **커널이 깨어남.** CPU 가 user mode → kernel mode로 전환. 커널이 받음.
5. **커널이 일.** 디스크에서 진짜로 데이터를 읽어옴. 또는 네트워크로 보냄. 또는 메모리 할당.
6. **결과 반환.** 결과를 정해진 자리에 놓고 user mode 로 다시 돌아옴.
7. **앱이 결과 받음.** 본인 코드가 read() 다음 줄로 진행.

이 7단계가 마이크로초 단위로 일어나요. 1초에 본인 컴퓨터 안에서 수백만 번. user ↔ kernel 점프가 매번 일어나요. 비용이 좀 들어요. 그래서 시스템 콜을 너무 자주 부르면 느려요. 그래서 read() 한 번 부를 때 1바이트씩 부르지 말고, 4096바이트씩 부르세요. 한 번 점프로 큰 일을 하기. 두 해 후 성능 튜닝 챕터에서 다시.

이 7단계 그림이 H7 의 척추예요. 오늘은 단어 두 개만. **trap = 트랩 (커널 점프), context switch = 컨텍스트 스위치 (모드 바꾸기).** 머리에 두세요.

## 17. 4가지 개념의 짝짓기 — 한 표 정리

자, 한 시간 끝났어요. 4가지 손길을 다 만났어요. 한 표로 정리.

| | 프로세스 | 스레드 | 파일 | 시스템 콜 |
|---|---------|--------|------|-----------|
| 정의 | 실행 중인 프로그램 | 프로세스 안 일꾼 | 디스크 위 데이터 | 앱 → OS 부탁 |
| 비유 | 호텔 방 | 방 안의 사람 | 창고 짐 | 인터폰 |
| ID | PID | TID | inode | (없음) |
| 만들기 | fork() + exec() | pthread_create() | open() | (자기 자신) |
| 죽이기 | kill PID | pthread_exit() | close(fd) | (자기 자신) |
| 명령 | ps, top | top -H | ls, lsof | dtruss, strace |

이 표를 본인 노트에 한 번 옮겨 적어 두세요. 두 해 코스 동안 30번 이상 회수돼요. 첫 챕터에서 한 표를 잘 외워두면 두 해가 가벼워져요.

그리고 4개의 진짜 큰 관계 한 줄. **"프로세스가 스레드를 가지고, 스레드가 파일을 시스템 콜로 다룬다."** 이 한 줄이 4단어를 한 줄로 묶어 줘요. 프로세스 안에 스레드가 있고, 그 스레드가 일을 하려면 파일을 다루어야 하고, 파일을 다루려면 시스템 콜로 OS에게 부탁해야 해요. 4단어가 한 흐름.

## 18. 고양이 자경단의 한 요청 — 4개념이 다 등장

H1 에서 봤던 고양이 자경단 7단계를 다시. 이번엔 4개념을 다 꽂아요.

자원봉사자 김씨가 신고 한 건을 등록하는 그 7초.

1. **HTTP 요청 도착.** 매니저가 네트워크 fd에서 socket() syscall 로 받음. → **fd, syscall.**
2. **본인 Flask 앱(프로세스)이 받음.** 매니저가 이 프로세스를 깨움. → **프로세스.**
3. **요청 파싱은 메인 스레드, 사진 저장은 워커 스레드.** → **스레드.**
4. **워커 스레드가 사진을 디스크에 write() syscall.** → **파일, syscall.**
5. **PostgreSQL 프로세스(다른 프로세스)에 socket() syscall로 INSERT 보냄.** → **프로세스, syscall.**
6. **PostgreSQL이 fsync() syscall로 디스크에 박음.** → **파일, syscall.**
7. **응답을 socket fd 에 write().** → **fd, syscall.**

7단계 모두 4개념이 등장해요. 본인이 두 해 후 짜는 모든 백엔드 코드의 진짜 모양이에요. **본인 코드 100줄 뒤에 시스템 콜 1만 줄, 그 1만 줄을 받는 매니저가 한 명.** 오늘 H2 끝의 마지막 그림.

## 19. 자주 묻는 질문 5

**Q1. "프로세스와 스레드의 차이를 한 줄로 어떻게 답해요?"**
A. "프로세스는 메모리가 격리된 실행 단위, 스레드는 같은 메모리를 공유하는 실행 흐름이에요." 면접 정답이에요.

**Q2. "왜 OS는 모든 앱을 user space에 두나요? 그냥 다 kernel space에서 돌리면 더 빠르지 않나요?"**
A. 빠를 수 있어요. 그런데 한 앱의 버그가 OS 전체를 죽일 수 있어요. 1990년대 윈도우 95의 블루 스크린이 그 일이었어요. 보안과 안정성을 위해서 분리해요. 0.001ms 의 syscall 비용이 1년에 한 번 시스템 죽는 사고를 막아요.

**Q3. "fd 가 한 프로세스에 몇 개까지 가질 수 있나요?"**
A. 기본 macOS 256개. Linux 1024개. `ulimit -n` 으로 확인. 서버 운영할 때는 65,536개로 늘려요. nginx 같은 게 동시에 만 명 받으려면 fd 가 만 개 필요해요. 두 해 후 자세히.

**Q4. "race condition 어떻게 막아요?"**
A. 잠금(mutex/lock). 한 번에 한 스레드만 자원 만지게. 또는 atomic 연산(분리할 수 없는 한 동작으로 묶기). 또는 싱글스레드 모델(Node.js 처럼). 세 가지 길이 있어요. 두 해 후 한 챕터.

**Q5. "오늘 4개를 다 외워야 해요?"**
A. 한 표 한 번 보세요. 한 단어씩 친근해지면 돼요. 외우려 마시고 자주 만나려 해 보세요. H4·H5에서 ps·top·lsof 한 번씩 쳐 보면 4단어가 다 손에 들어와요. 한 시간이 끝일 뿐 4단어가 끝은 아니에요. 8시간이 끝일 때 4단어가 본인 거예요.

## 20. 4개념의 비용 비교 — 만들고 죽이는 데 얼마

성능 감각 한 번 짚고 가요. 4개의 비용을 표로.

| 동작 | 비용 (대략) | 1초에 몇 번? |
|------|------------|--------------|
| 프로세스 만들기 (fork+exec) | 1~10 ms | 100~1,000회 |
| 스레드 만들기 | 10~100 us | 10,000~100,000회 |
| 파일 열기 (open) | 1~100 us | 10,000~1M회 |
| 시스템 콜 한 번 | 0.1~1 us | 1M~10M회 |

이 차이가 진짜 큰 의미예요. **프로세스 만들기는 스레드 만들기의 100배 비싸요.** 그래서 웹 서버는 보통 스레드를 많이 쓰지 프로세스를 많이 안 만들어요. 1초에 만 명 받는 서버에서 매번 fork() 하면 1초가 부족해요. 그래서 미리 스레드 풀(thread pool) 을 만들어 두고 재사용. 두 해 후 백엔드 챕터에서.

그리고 **시스템 콜 한 번 = 0.1~1 us.** 빠르지만 무료가 아니에요. 본인 코드가 read() 를 1바이트씩 1만 번 호출하면 1만 번 syscall = 1~10ms. 4096바이트씩 한 번 호출하면 1번 syscall = 1us. 만 배 빨라요. **버퍼링(buffering)** 이라는 단어. 미리 큰 덩어리로 모아서 한 번에 syscall. 두 해 후 자세히. 오늘은 단어만.

이 비용 표가 두 해 코스의 진짜 큰 자산이에요. 본인이 코드 짤 때 "어 여기서 fork 너무 자주 하네, 스레드로 바꿔야 하나" 하는 직감이 이 표에서 나와요. 첫 챕터 H7의 100만 배 속도 표가 부품 4명의 비용이었다면, 오늘의 이 표는 매니저의 4가지 일의 비용. 두 표가 한 묶음으로 본인의 성능 감각의 척추를 만들어 줘요.

## 21. 프로세스 가족의 진짜 큰 사실 — pid 1 의 책임

호텔 매니저(OS)의 가장 큰 비밀 한 가지. PID 1 의 책임.

PID 1은 부팅의 첫 프로세스. macOS의 launchd, Linux의 systemd 또는 init. 이 한 명이 안 죽으면 컴퓨터가 안 죽어요. 이 한 명이 죽으면 OS 전체가 패닉. **PID 1의 책임 3가지.**

**1. 모든 시스템 데몬을 켜고 관리.** SSH 서버, 웹 서버, 데이터베이스, 로그 서비스 등. 본인이 맥북 켜자마자 자동으로 도는 그 모든 것이 PID 1의 자식.

**2. 고아 입양.** 부모가 일찍 죽은 자식을 자동 입양. 시체 처리(좀비 정리)도 PID 1의 일.

**3. 신호(signal) 분배.** 본인이 Ctrl+C 누르면 그 신호가 PID 1을 거쳐 분배될 수도 있어요(셸이 처리하기 전).

이 한 명이 진짜 중요한 사람이에요. systemd 가 너무 중요해서 한때 Linux 커뮤니티에서 큰 논쟁이 있었어요. "systemd 가 너무 많은 일을 하면 안 된다, 유닉스 철학(작은 도구)에 어긋난다." 라는 비판. 그래서 일부 배포판은 systemd 안 쓰고 다른 init 시스템을 써요. 두 해 후 클라우드 챕터에서 한 번 더.

오늘은 단어 하나. **PID 1 = 시스템의 시조.** 머리에 두세요. 본인이 두 해 후 `systemctl status nginx` 같은 명령 칠 때, 그 systemctl 이 PID 1(systemd)에게 부탁하는 명령이에요. 50년 전 유닉스에서 init 한 명이 모든 걸 시작했고, 지금도 그 한 명이 시작해요.

## 23. 흔한 오해 5 — 4개념 편

본인이 두 해 동안 만날 흔한 오해 5개를 미리.

**오해 1. "스레드가 항상 더 빠르다."** 아니에요. 만들기 비용은 스레드가 싸요. 하지만 race condition 디버깅 비용은 스레드가 100배 비싸요. 동시성 버그가 진짜 잡기 어려워요. 단순한 작업은 그냥 프로세스 여러 개가 더 안전해요. 적재적소.

**오해 2. "프로세스는 메모리를 무조건 따로 쓴다."** 거의 그래요. 다만 **공유 메모리(shared memory)** 라는 시스템 콜로 두 프로세스가 메모리 일부를 같이 쓸 수 있어요. PostgreSQL 같은 DB 가 자주 써요. 두 해 후 자세히.

**오해 3. "파일 = 디스크 위 데이터."** 거의 그래요. 다만 유닉스의 "모든 게 파일" 철학상 키보드·네트워크 카드·메모리·프린터까지 다 파일로 보여요. 본인이 lsof 한 번 더 쳐서 fd 보면 파일이 아닌 게 더 많을 수도 있어요. socket·pipe·device 등.

**오해 4. "syscall 은 함수와 같다."** 비슷하지만 달라요. 일반 함수는 user space 안에서 점프. syscall 은 kernel 까지 갔다와요. 비용이 100~1000배. 그래서 syscall 적게 부르는 코드가 빠른 코드.

**오해 5. "PID가 한 번 정해지면 평생 그 프로그램의 번호다."** 아니에요. 프로그램이 죽으면 PID가 회수돼요. 다음 프로그램이 그 PID를 받을 수 있어요. PID 는 프로세스의 한 인생 동안만 유효한 번호. 본인이 ps 출력 보고 PID 12345 가 어제는 크롬이었는데 오늘은 VS Code 일 수 있어요.

이 5가지 오해를 한 번 정리해 두면 두 해 동안 본인의 큰 자산. 면접에서 한 번 등장할 수 있어요. 면접관이 의도적으로 던지는 함정 질문이에요. 오늘 한 번 정리.

## 24. 한 줄 비유 모음 — 8개념을 8줄로

오늘 H1 + H2에서 만난 핵심 8개념을 한 줄 비유 8개로 정리할게요. 본인 노트에 옮겨 적어 두세요.

1. **OS = 호텔 매니저.** 24시간 안 자고 부품 4명을 지휘.
2. **커널 = 매니저의 핵심 사무실.** 모든 권한 가진 곳.

## 25. 작은 실험 — 본인 셸의 부모를 끝까지 따라가기

H3 가기 전 작은 실험 하나. 본인 셸의 부모를 PID 1까지 거슬러 올라가요. 가족 나무 끝까지.

> ▶ **같이 쳐보기** — 내 zsh 의 가족 나무 한 줄로 거슬러 올라가기
>
> ```bash
> pid=$$
> while [ "$pid" != "0" ] && [ -n "$pid" ]; do
>   ps -o pid=,ppid=,comm= -p "$pid" 2>/dev/null
>   pid=$(ps -o ppid= -p "$pid" 2>/dev/null | tr -d ' ')
> done
> ```

이 5줄이 본인 zsh → 터미널 앱 → launchd(PID 1) 까지 가족 나무 한 줄을 출력. 보통 3~4단계예요.

```
12345 12340 zsh
12340 12335 -zsh
12335 12330 login
12330 1 Terminal
1 0 launchd
```

이 출력이 본인 호텔 매니저(launchd, PID 1) 가 본인 zsh 의 증조할아버지라는 사실을 보여줘요. 본인이 친 모든 명령은 launchd 의 4대손이에요. 작은 실험이지만 큰 깨달음. 본인의 컴퓨터가 한 그루의 가족 나무라는 사실. 그 나무의 뿌리가 PID 1. 그 한 명이 본인 컴퓨터의 시조. 이 실험을 두 주 동안 한 번 더 해 보세요. 가족 나무가 본인 머리에 자리 잡아요.

## 26. 흔한 실수 다섯 가지 + 안심 멘트 — 4개념 학습 편

§23에서 4개념의 지식 함정 5개를 봤어요. 잘못된 사실들. 이번엔 4개념을 배우는 본인의 학습 자세 함정 5개를 짚고 가요. 미리 보시면 본인이 빠질 때 빨리 알아챌 수 있어요.

첫 번째 학습 함정, 4개를 한 시간에 다 외우려고 한다. 한 시간에 프로세스·스레드·파일·시스템 콜 4개가 다 나왔어요. 본인이 "이걸 다 외워야지" 끙끙대세요. 안심하세요. **한 시간에 다 외우는 게 목표가 아니에요.** 한 시간에 4개의 한 줄 정의 + 호텔 비유 한 명씩 머리에 들어가면 충분해요. 정확한 디테일은 두 해 동안 매번 다시 만나요. 프로세스는 Ch047 Docker에서 한 단계 깊게, 스레드는 Ch068 nginx에서 깊게, 파일은 Ch035 DB에서 깊게, 시스템 콜은 Ch091 성능 튜닝에서 깊게. 5층 빌딩 짓는 거예요. 첫 시간엔 1층만 짓는 거고요. 1층이 너무 두꺼우면 2층 못 올려요. 첫 시간엔 가볍게, 두 해 동안 무거워지게.

두 번째 학습 함정, 프로세스와 스레드를 같은 거라고 본다. 본인이 두 단어가 비슷해서 헷갈리세요. "둘 다 실행 단위 아닌가" 단순화. 안심하세요, 한 줄 차이만 잡으면 끝이에요. **"같은 방을 쓰느냐 다른 방을 쓰느냐."** 프로세스 = 다른 방(메모리 격리), 스레드 = 같은 방(메모리 공유). 이 한 줄 차이가 두 해 코스의 모든 동시성 문제의 뿌리예요. race condition이 왜 스레드에서만 일어나는가? 같은 방을 쓰니까. 프로세스 격리가 왜 중요한가? 다른 방이라서 한 명 죽어도 다른 명 안 죽으니까. **방** 한 글자가 핵심이에요. 본인 머리에 두 단어 비교될 때마다 "이건 같은 방이야 다른 방이야?" 한 번 자문하세요. 30초 만에 풀려요.

세 번째 학습 함정, 시스템 콜을 진짜로 다 외우려고 한다. 본인이 §15의 8개(open·read·write·close·fork·exec·socket·mmap)를 다 외우려고 노력하세요. 안심하세요. **두 개만 외우세요. read와 write.** 나머지 6개는 평생 검색이에요. read·write 두 개가 모든 시스템 콜의 90%예요. 디스크에서 읽고 쓰고, 네트워크에서 읽고 쓰고, 키보드에서 읽고 쓰고. 다 read와 write의 변형이에요. 두 개를 손에 잡으면 8개가 자연스럽게 친근해져요. 외운다는 건 머리에 박는 게 아니라 자주 만난다는 뜻이에요. 두 해 동안 read·write를 매주 한 번씩만 만나도 평생 자기 거예요.

네 번째 학습 함정, fd가 정수라는 사실에 놀란다. 본인이 fd 0·1·2를 보고 "어, 이게 그냥 숫자야?" 충격받으세요. 안심하세요, 함정의 정체부터 풀어드릴게요. fd는 진짜 그냥 정수예요. OS가 본인 프로세스에게 "너가 연 그 파일은 3번이야" 라고 정수 한 개 알려 주는 거. 그래서 빠르고 효율적이에요. 이름으로 매번 찾기보다 번호로 한 번에 잡기. **단순한 게 강한 거예요.** 50년 전 두 사람이 만든 fd 메커니즘이 지금도 그대로 쓰여요. 단순함이 영원함이에요. 본인이 두 해 동안 만나는 모든 좋은 도구의 비밀이에요. 복잡해 보이는 단어 뒤에 단순한 한 줄 정의가 있어요. fd는 그냥 정수, 프로세스는 그냥 실행 중인 프로그램, 시스템 콜은 그냥 부탁 한 마디. 단순함을 두려워 마세요.

다섯 번째 학습 함정, 가장 큰 함정이에요. **"OS 4개념은 백엔드/시스템 엔지니어 영역"이라고 단정한다.** 본인이 프론트엔드로 진로 잡으셨다면 "프로세스·스레드 같은 거 안 봐도 되겠지" 생각하세요. 안심하세요. 진실은 다른 모양이에요. **본인이 짜는 React 코드도 결국 프로세스 한 개 안에서 도는 자바스크립트 스레드 한 명이에요.** 브라우저가 본인 React 앱 띄울 때 새 프로세스(또는 새 탭) 만들고, 그 안에서 자바스크립트 메인 스레드 한 명이 본인 코드를 실행해요. setTimeout, async, Web Worker 같은 기능이 다 OS 4개념의 자바스크립트 버전이에요. 본인이 React 디버깅할 때 "이 페이지가 왜 멈췄지" 진단의 첫 줄이 "메인 스레드가 무거운 일에 막혔나" 예요. 스레드 모르면 그 진단이 안 돼요. **OS 4개념은 모든 개발자의 공통 어휘예요.** 본인이 어느 직무로 가도 두 해 후에 자산으로 살아 있어요.

다섯 가지 학습 함정, 다 들으셨어요. 외우진 마세요. 그저 본인이 빠졌을 때 한 번씩 떠오르시면 돼요. 다 외우려 말기, 방 한 글자 차이, read·write 두 개, fd는 정수, 4개념은 모든 직무. 다섯 함정을 미리 알아둔 본인이 두 해 동안 한 박자 빠르게 손이 움직여요. 첫 시간의 진짜 보험이에요.

## 27. 마무리 — H3 환경 점검으로

자, 2교시 끝났어요. 본인 머리에 5명이 있어요. 부품 4명(CPU·RAM·디스크·네트워크)에 매니저(OS) 한 명, 그리고 매니저의 4가지 손길(프로세스·스레드·파일·syscall). 총 5+4 = 9개 단어. 두 해 코스의 첫 9개 단단한 마디예요.

다음 시간 H3에서는 본인 맥의 OS 정보를 직접 캐 봐요. uname·sw_vers·sysctl·hostname 같은 명령으로. Ch001 H3 에서 sw_vers 했던 거 기억나시죠. 비슷한 패턴인데 OS 쪽 정보로 더 깊이. 본인 맥의 OS 신분증을 한 장 더 만들어요.

그 전에 한 번 멈추시고 ps aux | head -20 한 번 쳐 보세요. STAT 컬럼 보세요. R·S·T·Z·I 중 어느 게 가장 많은가. 보통 S 가 압도적. 본인 매니저의 손님들이 거의 다 자고 있어요. 진짜로요. 1분 멈추고 한 번 보세요. H3로 가요.

본인 잘 따라오셨어요. 두 시간이 끝났어요. 8시간 중 2시간 25%. 본인 스스로에게 작은 박수 한 번. 짝짝짝.

오늘 가장 큰 한 줄이에요. **"프로세스가 스레드를 가지고, 스레드가 파일을 시스템 콜로 다룬다."** 이 한 줄을 외워두세요. 면접에서 OS 관련 질문 받으면 이 한 줄에서 시작하시면 답이 술술 나와요. 면접관이 한 번 멈춰요. "어 이 사람 4개념을 한 줄로 묶었네." 그 한순간이 본인의 합격을 결정해요. 짧은 한 줄이 큰 일을 해요. 두 시간의 진짜 산출물.

그리고 4개 단어 중 본인이 가장 헷갈리는 거 한 개만 머리 속에서 꼽아 주세요. 그 한 개가 본인의 약점이에요. H3·H4·H5에서 그 단어가 다시 등장할 때 한 번씩 더 멈춰서 보세요. 첫 만남에서 약했던 단어가 다섯 번째 만남에서 본인의 무기가 돼요.

본인의 컴퓨터에 지금 200~400개의 손님이 살고 있고, 그 손님들이 4가지 손길을 통해 일을 하고 있다는 사실. 본인이 평소에 그냥 쓰던 컴퓨터가 한 큰 호텔이라는 사실. 그 호텔의 진짜 그림을 본 사람과 안 본 사람의 차이가 본인의 두 해 후를 결정해요. 본인이 그 그림을 보신 거예요. 잘하셨어요. 두 시간 끝까지 잘 들어 주셔서 진짜 고마워요. 잠시 쉬다가 H3에서 만나요.

## 👨‍💻 개발자 노트 (참고 — 비개발자는 그냥 넘기셔도 됩니다)

> - "프로세스가 스레드를 가지고, 스레드가 파일을 시스템 콜로 다룬다"는 한 줄 모델은 본 챕터의 척추 멘트. Ch 047 Docker(컨테이너 = 프로세스 격리 + 파일시스템 격리), Ch 068 nginx(이벤트 루프 = 한 프로세스 + 단일 스레드 + 비동기 syscall), Ch 091 성능 튜닝(strace/perf로 syscall 분석)에서 같은 한 줄로 회수됩니다.
> - fork()+exec() 분리는 유닉스의 진짜 우아한 발명품. Windows의 CreateProcess()는 한 함수가 다 함. 분리의 장점은 (1) 셸 리다이렉션이 자식 exec 전에 가능, (2) fork 후 즉시 실행하지 않고 자원 설정 가능, (3) 프로그램 시작 비용을 두 단계로 amortize. Ch 047 Docker가 fork 의 isolation 측면을 namespaces/cgroups로 한 단계 발전시킨 기술.
> - 프로세스 상태 5가지(R·S·T·Z·I)는 BSD ps의 STAT 컬럼. Linux의 D(uninterruptible sleep)는 별도 — D 상태의 프로세스는 kill -9도 안 들음. H6에서 "D 상태로 멈춘 프로세스" 응급 처치로 회수.
> - PID 1의 책임 변천사 — 1970년대 init(System V)·1990년대 BSD init·2010년대 systemd·2010년대 macOS launchd. systemd 논쟁(2014~)은 "유닉스 철학(작은 도구) vs 통합 시스템(systemd)" 의 대립. 본 챕터에서는 PID 1의 일반 개념만 노출.
> - file descriptor 0·1·2는 POSIX 표준. macOS·Linux 공통. Windows는 stdin·stdout·stderr가 정수가 아니라 HANDLE. WSL2 사용 시 자동으로 fd 모델로 동작.
> - race condition 예시(counter +1 두 번)는 lost update 패턴. 해결법으로 mutex(POSIX `pthread_mutex_t`), atomic 연산(C11 `_Atomic`), 락프리 자료구조(CAS), 액터 모델(Erlang/Akka), 단일 스레드 모델(Node.js·Redis) 등 다양. Ch 087 알고리즘 챕터의 동시성 시간에 정식 학습.
> - syscall 비용 0.1~1µs는 시스템 콜 진입/이탈 자체 비용. 실제 작업(read 1바이트 vs 4096바이트) 시간은 별도. Linux의 io_uring(2019~)은 syscall 진입을 비동기화하여 이 비용을 거의 0으로 만든 신기술. Ch 091에서 깊이 다룸.
> - lsof는 macOS/Linux 공통. Windows는 Process Explorer(Sysinternals) 또는 `handle.exe`. 본 챕터는 macOS·Linux 가족만 다루고 Windows는 H1 §7에서 "WSL로 우회" 안내.
> - §10 race condition 카운터 예시는 Tanenbaum _Modern Operating Systems_ 4판 §2.3.1과 _The Little Book of Semaphores_ (Allen Downey)의 표준 lost update. 두 해 후 면접에서도 같은 예시로 자주 등장.
> - §20 비용 표(프로세스 1~10ms, 스레드 10~100µs, syscall 0.1~1µs)는 Linux 5.x/macOS 14 기준 대략. 정확한 측정은 `lmbench`/`perf bench`. Ch 091 성능 시간에 정량 측정.
> - §26 다섯째 함정(프론트엔드 OS 무관 미신)은 React Fiber·V8 isolate·Web Worker·Service Worker가 모두 OS 4개념의 자바스크립트 버전이라는 사실 회수. Ch 050 React 챕터에서 메인 스레드 = single threaded JS 엔진의 한계와 Web Worker로의 우회를 OS의 멀티 스레드 모델과 1:1 대응시켜 학습.
