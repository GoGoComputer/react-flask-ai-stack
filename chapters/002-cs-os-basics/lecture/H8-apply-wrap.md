## 1. 들어가며 — 7교시 회수와 8교시 약속

자, 마지막 시간입니다. 진짜로 마지막. Ch002의 여덟 번째이자 마지막 시간이에요. 지난 7교시에서 우리는 굉장히 깊은 데까지 내려갔죠. 커널 공간과 사용자 공간이 왜 갈라져 있는지, ring 0과 ring 3의 의미가 무엇인지, 시스템 콜이라는 다리를 건널 때 CPU 안에서 어떤 일곱 단계가 일어나는지를 봤습니다. RAX 레지스터에 번호 넣고, MSR_LSTAR에 적힌 주소로 점프하고, 권한 비트가 3에서 0으로 바뀌고, 일을 마치면 sysret으로 돌아오는 그 7단계요. 그리고 페이지 폴트, 컨텍스트 스위치, strace와 dtruss로 실제 시스템 콜을 눈으로 확인해 봤고, 마지막에는 Meltdown과 KPTI까지 갔다가 vDSO와 io_uring으로 마무리했습니다. 머리가 좀 무거우셨을 거예요. 괜찮습니다. 7교시는 두 번째 코스의 시작에서 다시 만날 내용이에요. 지금은 "아, 커널은 진짜 다른 세계구나"라는 감각만 남기시면 충분합니다. 자, 그래서 8교시는 약속드린 대로 도구를 손에 쥐어 드리는 시간입니다. 응급실 키트를 만들 거예요. emergency.sh라는 이름의, 새벽 3시에 알람이 울려도 졸린 눈으로 한 줄만 치면 자동으로 다섯 종류 사고를 점검하고 최소한의 응급조치까지 해 주는, 100줄짜리 셸 스크립트요. 그리고 Ch002 여덟 시간을 한 장의 그림으로 정리합니다. 다섯 원리, 12회수지도 갱신, macOS와 Linux 변환표 확장, 흔한 오해 다섯 개, FAQ 다섯 개를 거쳐, 마지막에 Ch003 — 파일 시스템 — 으로 다리를 놓고 마칩니다. Ch002의 메시지는 한 문장으로 줄이면 이거예요. 운영체제는 호텔 매니저고, 우리는 그 매니저와 대화하는 법을 8시간 동안 배웠다. 오늘은 그 대화를 손에 쥐는 도구로 만드는 시간입니다.

## 2. Ch002 8시간 큰 그림 회수

가장 먼저 8시간 전체를 한 장으로 펼쳐 봅시다. 1교시 오리엔테이션에서 운영체제를 호텔 매니저에 비유했죠. 손님이 체크인하면 방을 배정하고, 시끄러운 손님은 다른 층으로 옮기고, 화재 경보가 울리면 매뉴얼대로 대피시키는 사람. 그게 OS입니다. 그리고 매니저가 관리하는 자원은 네 가지였어요. CPU, 메모리, 디스크, 네트워크. 요리사, 도마, 냉장고, 배달앱이라는 비유로 8시간 내내 따라다닌 그 네 손길이죠. 2교시는 개념 정비였어요. 프로세스가 호텔 방이고, 스레드가 그 방 안에서 일하는 사람이고, 파일이 창고에 맡긴 짐이고, fd는 짐 번호표라는 비유 사전을 만들었습니다. 3교시는 환경 점검이었습니다. uname, sw_vers, sysctl, hostname, id, env, PATH, ulimit, system_profiler, dmesg, uptime — 내가 지금 어떤 호텔 안에 있는지 자기 자리를 확인하는 시간이었죠. Apple Silicon 위에서 일하는 사람이라면 sysctl machdep.cpu.brand_string도 외워 두시고, .zshrc에 myinfo 함수를 박아 두는 작은 의식도 함께 했습니다. 4교시는 명령어 카탈로그였습니다. ps와 ps의 다섯 옵션, top과 htop, kill과 일곱 가지 신호, killall, pkill, pgrep, jobs, bg, fg, nohup, disown, wait. 일곱 시나리오와 다섯 가지 위험을 함께 다뤘죠. 5교시는 그 명령어들을 한데 묶은 procmon.sh를 직접 짰습니다. 셔뱅, set 가족 세 종, trap, getopts, OS 분기, 다섯 함수(summary, top_cpu, top_mem, zombies, user_procs), local 변수, while 루프, main "$@", bash -x, cron, shellcheck. 셸 스크립트를 글이 아니라 살아 있는 도구로 받아들이는 시간이었습니다. 6교시는 그 도구를 들고 응급처치 다섯 사고를 분류했어요. 좀비, OOM, 디스크 풀, 네트워크 일곱 다리, CPU 100%. 그리고 Jetsam, dmesg, log show, journalctl, lsof, ss, renice라는 진단 도구를 익히고, emergency.sh의 골격을 짜고, 7단계 절차를 외웠습니다. 7교시는 그 절차의 안쪽 — 커널 공간 — 으로 한 번 들어갔다 왔습니다. 자, 이렇게 8시간이 한 줄로 이어집니다. 비유 → 사전 → 자기 위치 → 명령어 → 첫 도구 → 응급실 → 커널 → 그리고 오늘, 도구의 완성. 이 흐름이 머리에 그려지면 Ch002는 다 가져가신 겁니다. 한 시간 한 시간이 따로 떠다니는 게 아니라, 호텔 매니저의 일을 점점 손에 쥐는 8단계 계단이었어요.

## 3. emergency.sh 완성판 — 100줄짜리 응급실 키트

이제 본론입니다. 6교시에 짠 골격을 오늘 완성합니다. 머리부터 보여드릴게요.

```bash
#!/usr/bin/env bash
# emergency.sh — 5사고 응급실 키트
# 사용법: ./emergency.sh [--dry-run] [--case zombie|oom|disk|net|cpu|all]
set -Eeuo pipefail
IFS=$'\n\t'

LOG="/tmp/emergency-$(date +%Y%m%d-%H%M%S).log"
DRY_RUN=0
CASE="all"
OS="$(uname)"

trap 'echo "[ERR] line $LINENO: $BASH_COMMAND" | tee -a "$LOG"' ERR
log() { echo "[$(date +%H:%M:%S)] $*" | tee -a "$LOG"; }
run() { if [ "$DRY_RUN" -eq 1 ]; then log "DRY: $*"; else log "RUN: $*"; eval "$@" 2>&1 | tee -a "$LOG"; fi; }
```

set -Eeuo pipefail은 5교시에서 외운 set 가족 세 종 + ERR 트랩 상속 옵션입니다. IFS을 줄바꿈+탭으로 바꿔서 공백 들어간 파일명에서 깨지지 않게 해 두고, LOG 파일을 시각으로 한 개 만들어 두고, DRY_RUN과 CASE 변수를 기본값으로 둡니다. trap으로 에러가 어느 줄에서 났는지 자동으로 로그에 박고, log 함수는 시간 찍힌 표준 로그, run 함수는 dry-run 분기까지 처리하는 만능 실행기입니다. 이 8줄이 키트의 뼈대예요. 다음은 옵션 파싱과 다섯 사고 함수입니다.

```bash
while [ $# -gt 0 ]; do
  case "$1" in
    --dry-run) DRY_RUN=1; shift;;
    --case)    CASE="$2"; shift 2;;
    -h|--help) sed -n '2,5p' "$0"; exit 0;;
    *) log "unknown: $1"; exit 2;;
  esac
done

case_zombie() {
  log "=== 좀비 점검 ==="
  local zs; zs="$(ps -A -o stat,ppid,pid,comm | awk '$1 ~ /^Z/{print $2}' | sort -u)"
  if [ -z "$zs" ]; then log "좀비 없음"; return 0; fi
  for ppid in $zs; do
    log "좀비 부모 PID=$ppid → SIGCHLD 권유"
    run "kill -CHLD $ppid || true"
  done
}

case_oom() {
  log "=== 메모리 폭주 점검 ==="
  if [ "$OS" = "Darwin" ]; then
    run "vm_stat | head -5"
    run "ps -A -o pid,rss,comm -m | head -6"
  else
    run "free -h"
    run "ps -eo pid,rss,comm --sort=-rss | head -6"
    run "dmesg -T | grep -i 'killed process' | tail -3 || true"
  fi
}
```

옵션 파싱은 4교시에서 본 getopts 대신 손으로 짠 case 루프예요. --dry-run이면 DRY_RUN=1, --case 값은 두 칸 시프트, -h는 파일 자기 머리 2~5줄을 sed로 출력하는 작은 트릭입니다. 좀비 함수는 6교시에 외운 ps -A -o stat에서 첫 글자가 Z인 행의 부모 PID를 모아서 SIGCHLD를 보내요. 죽이는 게 아니라 "자식 좀 거두라"고 두드리는 겁니다. OOM 함수는 OS 분기로 macOS면 vm_stat과 RSS 정렬, Linux면 free + ps + dmesg에서 OOM 킬러 흔적을 봅니다. 이게 6교시에 약속한 5사고 중 두 개고, 나머지 셋은 디스크, 네트워크, CPU예요.

```bash
case_disk() {
  log "=== 디스크 풀 점검 ==="
  run "df -h | awk 'NR==1 || \$5+0 >= 80'"
  run "du -sh /var/log/* 2>/dev/null | sort -h | tail -5 || true"
  if [ "$DRY_RUN" -eq 0 ]; then
    log "주의: 자동 삭제 없음 — 사람 눈으로 확인 후 rm"
  fi
}

case_net() {
  log "=== 네트워크 7다리 점검 ==="
  run "ping -c 2 -W 2 1.1.1.1 || true"
  run "ping -c 2 -W 2 google.com || true"
  if [ "$OS" = "Darwin" ]; then
    run "netstat -rn | head -5"
    run "lsof -nP -iTCP -sTCP:LISTEN | head -10"
  else
    run "ss -tlnp | head -10"
  fi
}

case_cpu() {
  log "=== CPU 100% 점검 ==="
  if [ "$OS" = "Darwin" ]; then
    run "ps -A -o pid,%cpu,comm -r | head -6"
  else
    run "ps -eo pid,%cpu,comm --sort=-%cpu | head -6"
  fi
  log "처리 가이드: 1순위 renice, 2순위 SIGTERM, 마지막에 SIGKILL"
}

main() {
  log "OS=$OS DRY_RUN=$DRY_RUN CASE=$CASE LOG=$LOG"
  case "$CASE" in
    zombie) case_zombie;;
    oom)    case_oom;;
    disk)   case_disk;;
    net)    case_net;;
    cpu)    case_cpu;;
    all)    case_zombie; case_oom; case_disk; case_net; case_cpu;;
    *)      log "unknown case: $CASE"; exit 2;;
  esac
  log "리포트: $LOG"
}

main "$@"
```

자, 이게 100줄 안쪽으로 끝납니다. 디스크 함수는 df로 사용률 80% 넘는 파티션만 골라 보여 주고, /var/log 밑 큰 파일을 정렬해 줍니다. 자동으로 지우지는 않아요. "사람 눈으로 보고 rm 하세요"라고 로그에 박아 둡니다. 응급실에서 가장 위험한 게 자동 삭제예요. 네트워크 함수는 6교시 일곱 다리 중 핵심 — DNS, 라우팅, LISTEN — 만 빠르게 훑습니다. ping은 -W 2로 2초 안에 안 오면 포기하게 막아 두고. CPU 함수는 정렬한 상위 6개를 보여 주면서 처리 가이드 한 줄을 같이 찍습니다. main에서는 시작 한 줄 + case 분기 + 마지막 리포트 경로 한 줄로 끝나요. 이게 핵심입니다. 응급 스크립트는 길지 않아야 합니다. 길면 새벽 3시에 못 읽어요.

이 스크립트를 어디에 둘까요. 저는 ~/bin/emergency.sh를 권합니다. PATH에 ~/bin이 있으면 어디서든 emergency.sh로 부를 수 있죠. .zshrc에 alias er='emergency.sh --dry-run' 같은 두 글자 별칭을 박아 두시면 더 좋습니다. 그리고 cron이나 launchd로 새벽 3시에 자동 실행하는 줄도 6교시에서 본 그 줄을 그대로 쓰시면 됩니다. 0 3 * * * /Users/you/bin/emergency.sh >> /tmp/emergency-cron.log 2>&1. 한 가지 약속만 드릴게요. 이 스크립트는 제가 드린 그대로 쓰지 마세요. 본인 환경에 맞게 한 줄씩 고치세요. 디스크 임계를 80에서 70으로 내리시거나, 네트워크에서 자기 회사 게이트웨이 ping을 추가하시거나. 그렇게 본인 손때가 묻은 emergency.sh가 진짜 응급실 키트입니다. 남이 짠 키트는 새벽에 안 펴져요.

한 가지 더 짚고 갈 게 있습니다. 이 스크립트는 일부러 의존성을 최소로 했어요. bash와 ps와 awk와 sed만 있으면 어디서든 돕니다. python이나 jq나 docker가 깔려 있을 거라고 가정하지 않았어요. 응급실 키트는 의존성이 적을수록 좋습니다. 망가진 머신에서 처음 만나는 건 종종 "패키지가 깨져 있다"는 상황이고, 그때 jq를 깔겠다고 apt를 두드리다가 시간이 다 가요. 그래서 emergency.sh는 macOS 기본 설치 + Linux minimal 설치만으로도 끝까지 돕니다. 그게 이 스크립트의 두 번째 약속이에요. 그리고 한 가지 운영 팁 — 출력의 마지막 줄을 항상 LOG 파일 경로로 끝내세요. 새벽에 알람을 보고 켰을 때 화면에 무조건 첫 줄과 끝 줄만 봅니다. 끝 줄에 "리포트: /tmp/emergency-..."가 있으면 less로 그 파일을 열어 차분하게 다시 보면 돼요. 첫 줄에 OS와 CASE와 LOG 경로를 한 번에 찍는 것도 같은 이유입니다. 응급실에서는 "읽기 시작점"과 "읽기 끝점"이 분명해야 해요. 가운데가 길어도 처음과 끝만 분명하면 한밤중에도 읽힙니다.

## 4. procmon.sh + emergency.sh — 진단과 치료의 짝

5교시 procmon.sh와 오늘 만든 emergency.sh, 이 둘은 한 짝입니다. procmon은 진단이에요. 평상시에 매일 한 번 돌리면서 "지금 호텔이 평소와 같은가"를 보는 도구. emergency는 치료입니다. 알람이 울렸을 때 다섯 사고 중 어떤 게 터졌는지 빠르게 분류하는 도구. 의사로 비유하면 procmon은 정기 검진, emergency는 응급실인 거죠. 운영의 기본은 이 둘을 매일 같이 돌리는 거예요. 저는 cron에 두 줄을 박아 둡니다. 0 9 * * * procmon.sh > /tmp/proc-$(date +\%F).log 와 0 3 * * * emergency.sh --dry-run > /tmp/em-$(date +\%F).log. 아침 9시는 사람이 출근해서 보는 시각이고, 새벽 3시는 사람이 자는 동안 검사하는 시각이에요. 이 두 줄로 호텔의 24시간이 거의 다 커버됩니다. 추가로 권하고 싶은 건 두 스크립트의 출력을 한 디렉토리 — 가령 ~/ops/logs — 에 모으고, 일주일에 한 번 ls -la로 슬쩍 훑는 작은 의식이에요. 평소 분량과 다르면 그게 신호입니다. 오늘 로그가 평소의 10배라면 뭔가 일어난 거고, 평소의 절반이라면 cron이 안 돈 거예요. 양쪽 다 의미가 있어요. 한 가지 더, 두 스크립트는 의도적으로 분리되어 있습니다. procmon에 emergency 코드를 넣지 마세요. 진단과 치료를 섞으면 보기 위해 돌렸다가 사고를 치는 일이 생깁니다. 이건 운영의 오래된 격언이에요. read와 write를 한 함수에 두지 말라는 말과 같은 결입니다.

추가로 두 도구의 출력 포맷에 대해 잠깐 얘기할게요. procmon.sh는 사람이 매일 9시에 한 번 보는 도구라서 컬러를 살짝 써도 됩니다. echo -e로 헤더만 노랗게 칠하시면 정기 검진 느낌이 살아요. 반대로 emergency.sh는 절대 컬러를 쓰지 마세요. 새벽에 알람으로 SMS나 슬랙으로 출력이 전송될 가능성이 있는데, ANSI 이스케이프가 그대로 깨져서 화면에 이상한 기호가 보이면 새벽 3시에 진짜 화납니다. 응급실 도구의 출력은 plain text가 원칙이에요. 또 하나, 두 스크립트 모두 종료 코드를 신경 써 주세요. 정상 종료 0, 점검 결과 이상 발견 1, 명령어 자체가 에러 2 — 이 세 가지로 분리해 두면 cron이나 launchd가 자동으로 재시도할지 말지를 판단할 수 있습니다. 모니터링 시스템에 붙이실 때도 종료 코드가 가장 가벼운 신호예요. exit 0/1/2 한 줄이 100줄짜리 검사 로직보다 가치가 클 때가 많습니다. 마지막으로 — 두 스크립트의 첫 줄은 항상 같이 갱신하세요. 셔뱅 아래에 버전 한 줄을 박아 두면, 6개월 뒤에 본인이 "내가 어떤 버전 쓰고 있지?" 라고 물을 때 답이 나옵니다. git log도 좋지만, 도구 그 자체에 버전이 박혀 있는 게 운영의 위생입니다. 그리고 두 스크립트 모두 README.md를 옆에 두세요. 한 페이지짜리 README — 이름, 사용법, 다섯 케이스 설명, 종료 코드 의미, 의존성 — 가 6개월 뒤의 본인을 살립니다. 코드가 아니라 README가 본인을 구해요.

## 5. Ch002 다섯 가지 원리

Ch002 여덟 시간을 한 줄씩 다섯 원리로 줄여 봅시다. 외워 두시면 Ch003 이후 어디서 만나든 다시 떠올라요.

원리 하나, **자원에는 한도가 있다.** CPU 코어 수, RAM 용량, 디스크 GB, 네트워크 대역폭. 무한이 아니에요. ulimit, free, df, ifstat가 그 한도를 보는 창입니다. 한도를 잊으면 OOM, 디스크 풀, 큐 폭주가 옵니다.

원리 둘, **추상화는 실수에서 시작된다.** 파일이 진짜로 디스크 위에 줄지어 있는 게 아니라, OS가 만들어 준 환상이라는 사실. 메모리 주소도 가상 주소예요. 프로세스도 호텔 방이라는 환상이고요. 이 환상이 깨질 때 — strace로 들여다볼 때, mmap이 EBUSY 낼 때 — OS가 보입니다.

원리 셋, **프로세스는 격리된다.** 한 방에서 일이 잘못돼도 옆방이 안 죽어요. 이게 ring 0/3, 가상 메모리, namespace의 존재 이유입니다. Ch047 Docker는 이 격리를 더 두껍게 한 것이고, Ch091 Lambda는 프로세스 단위 과금이에요.

원리 넷, **권한은 분리된다.** root와 일반 유저, 사용자 공간과 커널 공간, IAM의 역할 분리. 이 다섯 글자 — UID, GID, ring, capability, IAM role — 가 다 권한 분리의 다른 이름이에요. 권한을 합치는 건 항상 사고의 시작입니다.

원리 다섯, **시스템 콜은 좁은 다리다.** 사용자 공간과 커널 공간 사이의 유일한 정문. 모든 read, write, open, fork, exec이 이 다리를 건넙니다. 다리가 좁으니 비용이 들어요. 캐싱과 io_uring과 vDSO는 이 다리를 덜 건너거나 한 번에 많이 건너려는 노력의 다른 이름입니다.

이 다섯 줄을 노트 첫 페이지에 적어 두세요. Ch003부터 Ch120까지, 어느 시간이든 한 번은 이 다섯 중 하나로 환원됩니다. 진짜로요.

## 6. 12회수지도 — Ch002가 어디서 다시 만나나

Ch002의 내용은 Ch003에서 잊히는 게 아니라 살아 돌아옵니다. 코스 전체에서 최소 열두 번 다시 만나요. 정리해 드릴게요. **Ch035 PostgreSQL fsync** — 5교시 syscall 비용표가 거기서 다시 나옵니다. fsync 한 번이 왜 비싼지가 ring 0 진입 비용이거든요. **Ch047 Docker** — 프로세스 격리와 namespace를 다시 만납니다. 호텔 객실 비유가 더 두껍게 돌아와요. **Ch062 Redis maxmemory** — RAM 한도와 OOM 처리를 다시 봅니다. Jetsam과 oom-killer 이야기가 두 번째 무대에 올라요. **Ch068 nginx worker_processes** — CPU 코어와 프로세스/스레드 모델이 다시 등장합니다. **Ch074 zero-copy/sendfile** — 시스템 콜 다리를 한 번 더 줄이는 기술이에요. **Ch087 알고리즘 캐시 친화성** — RAM이 도마라는 비유가 L1/L2 캐시까지 내려가요. **Ch091 perf와 flame graph** — strace의 한 단계 위인 perf로 다시 돌아옵니다. **Ch097 RAG 검색 latency** — read syscall 비용이 검색 응답에 어떻게 누적되는지가 보여요. **Ch108 EC2 인스턴스 타입 선택** — 코어 수, RAM, 디스크 IOPS, 네트워크 대역 — Ch002 4자원이 그대로 청구서가 됩니다. **Ch113 CloudFront/S3** — 디스크가 냉장고라는 비유가 글로벌 캐시로 확장돼요. **Ch116 X-Ray** — 분산 추적이 결국 syscall 단위 이벤트의 모임이라는 걸 보게 됩니다. **Ch119 Bedrock 비용** — 토큰 단위 청구가 자원 한도와 다르지 않다는 깨달음이 옵니다. 열두 자리 다 적어 두실 필요 없어요. 다만 "Ch002는 다시 온다"는 사실만 기억해 두시면, Ch035 fsync 챕터에서 갑자기 ring 0 이야기가 나와도 당황하지 않으십니다. "아, 5교시에 본 그 좁은 다리구나" 하시면 돼요.

그리고 한 가지 — 이 다섯 원리는 OS 안에서만 통하는 게 아니에요. 데이터베이스에서도 같습니다. 자원 한도(connection pool), 추상화(SQL이 디스크 위 B-tree의 환상), 격리(트랜잭션 isolation level), 권한 분리(role/grant), 좁은 다리(prepared statement). 다섯 글자가 그대로 살아 있어요. 분산 시스템에서도 같습니다. 자원 한도(rate limit), 추상화(서비스 메시), 격리(circuit breaker), 권한 분리(IAM), 좁은 다리(API gateway). 그래서 Ch002의 다섯 원리는 OS만의 것이 아니라 "시스템"이라는 것 자체의 다섯 원리에 가까워요. 한 번 익혀 두면 평생 갑니다. Ch003부터 Ch120까지 어느 시간이든 "이건 다섯 원리 중 어느 것일까"를 한 번 묻고 들어가시면, 새 단어가 나와도 자리 잡을 데가 머리에 미리 마련돼 있어요. 학습이라는 건 단어를 외우는 게 아니라 자리를 미리 만드는 일이에요. 그 자리가 바로 다섯 원리입니다. 그리고 다섯 원리는 면접에서도 무기가 됩니다. "운영체제에서 가장 중요한 원리가 뭐라고 생각하세요"라는 질문에 한 단어로 답하는 사람과, 다섯 원리를 한 줄씩 풀어내는 사람은 평가가 다릅니다. 외운 답이 아니라 자기 머릿속 지도에서 꺼내는 답이라는 게 보이거든요. 강의 끝에 다섯 원리를 일부러 따로 정리하는 이유가 그거예요. 면접용이 아니라, 본인 머릿속 지도의 좌표축을 만드는 일입니다.

## 7. macOS ↔ Linux 변환표(OS 버전 확장)

3교시·4교시에 쓴 표를 OS 차원으로 확장합니다. 강의가 macOS 위주였지만, 실무에서 만나는 건 Linux가 더 많기 때문에 한 번 정리하고 갑니다.

| 일 | macOS | Linux |
|---|---|---|
| OS 이름·버전 | sw_vers / uname -a | cat /etc/os-release / uname -a |
| CPU 정보 | sysctl -a \| grep machdep.cpu | cat /proc/cpuinfo / lscpu |
| 메모리 | vm_stat / top -l 1 | free -h / cat /proc/meminfo |
| 디스크 | df -h / diskutil list | df -h / lsblk |
| 프로세스 트리 | ps -A -o pid,ppid,comm | ps -ef / pstree |
| 메모리 정렬 | ps -A -o pid,rss,comm -m | ps -eo pid,rss,comm --sort=-rss |
| CPU 정렬 | ps -A -o pid,%cpu,comm -r | ps -eo pid,%cpu,comm --sort=-%cpu |
| OOM 흔적 | log show --predicate 'eventMessage contains "Jetsam"' | dmesg \| grep -i 'killed process' |
| 시스템 콜 추적 | sudo dtruss -p PID | strace -p PID |
| 열린 파일 | lsof -p PID | lsof -p PID / ls /proc/PID/fd |
| LISTEN 포트 | lsof -nP -iTCP -sTCP:LISTEN | ss -tlnp |
| 라우팅 | netstat -rn | ip route |
| 인터페이스 | ifconfig | ip addr |
| 커널 로그 | log show / log stream | dmesg / journalctl -k |
| 서비스 관리 | launchctl list | systemctl list-units |
| 환경변수 표시 | env / printenv | env / printenv |
| 자원 한도 | ulimit -a | ulimit -a / /etc/security/limits.conf |
| 시각·업타임 | uptime / w | uptime / w |
| 패키지 | brew list | apt list --installed / dnf list installed |

이 표를 한 장 인쇄해서 옆에 두세요. macOS에서 Linux 서버에 ssh 들어갔을 때 손이 헷갈리지 않습니다. 같은 일을 두 OS가 다른 단어로 부르는 것뿐이에요. 호텔 매니저는 같은데 매뉴얼 번역본이 다른 거죠.

한 가지 덧붙이면, 클라우드 환경 — Ch108~Ch120 — 에서는 변환표에 세 번째 열이 더 붙습니다. AWS CloudWatch, GCP Cloud Monitoring, Azure Monitor 같은 매니지드 도구들이 ps와 top과 lsof의 자리를 차지해요. 하지만 안쪽에서 도는 건 같은 OS입니다. CloudWatch가 보여주는 CPU 그래프는 결국 ps 출력의 시계열이고, 디스크 IOPS는 iostat의 시계열이에요. 매니지드라고 해서 OS가 사라진 게 아니라, 한 층 위에서 같은 숫자를 더 예쁘게 그려 주는 것뿐입니다. 그래서 Ch002의 명령어들은 클라우드에 가도 죽지 않아요. EC2 인스턴스에 ssh 들어가서 ps -ef를 치는 일은 Ch108에서도, Ch119에서도 합니다. 매니지드 콘솔과 ssh 터미널, 두 창을 같이 띄워 두는 게 진짜 운영입니다. 한쪽만 보면 절반만 봅니다. 그리고 컨테이너 안에서는 또 한 번 변환이 일어나요. docker exec 안의 ps는 호스트의 ps와 다른 PID를 보여 줍니다. PID 1이 본인 컨테이너의 진입점이에요. 이건 namespace 격리 때문이고 Ch047에서 깊게 다룹니다. 지금은 "호텔이 호텔 안에 있을 수 있다"는 정도만 머리에 두세요. 객실 안에 또 객실이 있는 거예요. 비유가 점점 두꺼워지지만, 다섯 원리는 그대로 통합니다.

## 8. 흔한 오해 다섯 가지

**오해 1: "프로세스를 죽이려면 무조건 kill -9다."** 아니에요. 1순위는 SIGTERM(15), 2순위가 SIGKILL(9)입니다. -9는 정리할 기회를 안 주기 때문에 임시 파일이 남고 락이 풀리지 않습니다. 4교시에서 외운 7신호 중 9는 마지막 카드예요.

**오해 2: "메모리 사용량이 100%면 OOM이다."** 아니에요. Linux는 캐시를 free에 안 잡으면서도 필요할 때 회수합니다. free -h의 available 칼럼이 실제 가용량이에요. macOS의 "Memory Pressure"도 같은 결입니다. 사용량이 아니라 압력을 봐야 합니다.

**오해 3: "셸 스크립트는 코드가 아니다."** 5교시에서 본 그대로, 셔뱅·set·trap·getopts·함수·local·main "$@"는 정식 코드예요. shellcheck를 안 돌리는 셸 스크립트는 lint 안 하는 파이썬과 같습니다.

**오해 4: "커널 공간은 보안 때문에 만들었다."** 일부만 맞아요. 격리·자원중재·공정성·하드웨어 추상화도 큰 이유입니다. ring 0/3은 보안 이전에 운영 안정성의 도구예요.

**오해 5: "시스템 콜은 자주 부를수록 좋다(직접 제어니까)."** 거꾸로예요. 다리 비용이 있어서 가능한 한 한 번에 많이 처리해야 합니다. read를 1바이트씩 천만 번 부르는 코드가 있다면 그건 거의 항상 잘못 짠 코드입니다. io_uring과 sendfile이 존재하는 이유가 여기 있어요.

## 9. FAQ 다섯 개

**Q1. emergency.sh를 root로 돌려야 하나요?** 기본은 일반 유저로도 돌아가게 짰습니다. 하지만 dmesg, lsof -i 일부, 다른 유저 프로세스 kill은 root가 필요해요. 운영 서버에서는 sudo 없이 돌리는 모드와 sudo 있는 모드 두 개를 alias로 분리하시는 게 좋습니다. er(드라이런·권한 없이), erx(실행·sudo) 정도로요. 그리고 sudoers 파일에 emergency.sh 한 줄만 NOPASSWD로 박아 두는 방법도 있어요. 비밀번호 없이 sudo로 이 한 스크립트만 돌게 만드는 거죠. 보안팀과 합의가 필요하지만, 새벽 3시 자동 응급 대응에는 유효한 패턴입니다.

**Q2. macOS에서 Linux 서버 운영 연습은 어떻게 하나요?** 두 가지 권합니다. (1) UTM이나 OrbStack으로 로컬에 Ubuntu VM을 띄우세요. M1/M2/M3에서 잘 됩니다. (2) Ch108에서 EC2 t4g.nano 한 대(월 3천 원대)를 띄워서 진짜 리눅스에서 같은 명령을 한 번씩 쳐 보세요. 변환표가 머리에 박힙니다. 추가로 권하는 건 multipass라는 캐노니컬 공식 도구예요. brew install multipass 한 줄로 Ubuntu VM이 30초 안에 뜹니다. 학습용으로는 OrbStack이 가장 가볍고 빠릅니다. 하루 한 번 multipass shell 또는 orb 들어가서 ps -ef 한 줄 치는 의식이 한 달이면 손에 박혀요.

**Q3. procmon.sh와 emergency.sh를 통합하면 안 되나요?** 안 합니다. 진단과 치료를 한 스크립트에 넣으면 "보러 들어갔다가 사고 친다"가 일어나요. 이건 5교시 마지막에서도 했던 약속입니다. 한 도구는 한 가지 일만. 그리고 두 도구를 굳이 한 자리에 두고 싶다면 ops/ 디렉토리 하나 안에 두 파일로 나란히 두세요. ops/procmon.sh, ops/emergency.sh, ops/README.md. 세 파일이 한 폴더에 있으면 그게 본인의 운영 키트입니다. 한 파일에 합치는 것보다 한 폴더에 모으는 게 훨씬 강해요.

**Q4. Ch002 끝났는데 자신감이 없어요.** 정상입니다. 8시간으로 OS 전체를 가져갈 수는 없어요. 두 해 코스 중 두 번째 마디일 뿐입니다. Ch003부터 Ch120까지 가는 동안 Ch002 내용은 평균 한 챕터에 한 번씩 다시 등장합니다. 그때마다 "아, 봤던 거다"가 쌓이면서 진짜 자신감이 생겨요. 지금은 emergency.sh 한 개를 손에 쥔 사람이 됐다는 사실만 챙기시면 됩니다. 그리고 자신감의 척도를 바꿔 보세요. "OS를 다 안다"가 아니라 "ps 출력에서 좀비 하나는 알아본다", "df -h에서 80% 넘는 줄 하나는 골라낸다", "kill -9 대신 -15부터 쓴다" — 이 세 가지가 되시면 Ch002는 충분히 가져가신 거예요. 자신감은 전체를 아는 데서 오는 게 아니라, 하나를 확실히 손에 쥐는 데서 옵니다.

**Q5. 다음 챕터 예습은 뭘 하나요?** ls -la / 쳐 보세요. 그리고 stat ~/.zshrc, df -h, mount(또는 macOS는 mount -t apfs) 쳐 보세요. 출력에서 모르는 단어 — inode, 블록, 마운트 포인트 — 를 메모해 두시면, Ch003 H1 첫 5분에 그 단어들이 다 풀립니다. 추가로 du -sh ~/* 도 한 번 쳐 보세요. 본인 홈 디렉토리에서 어떤 폴더가 가장 무거운지가 한 줄로 보입니다. Downloads가 1등인 분이 많을 거예요. 그 한 줄을 보는 순간 "디스크는 냉장고고, 안 쓰는 음식이 자리만 차지한다"는 비유가 머리에 박힙니다. 예습은 길지 않아도 됩니다. 다섯 줄짜리 명령으로 충분해요.

FAQ에 들어가지 못한 짧은 메모도 두세 개 남길게요. 첫째, **launchd 사용자**는 cron 대신 ~/Library/LaunchAgents 밑에 plist를 두는 걸 권합니다. macOS는 cron이 점점 더 권장되지 않아서 새 머신에서는 launchctl load가 표준이에요. plist는 XML이라 처음엔 낯설지만, RunAtLoad와 StartCalendarInterval과 StandardOutPath 세 키만 알면 절반은 끝납니다. 둘째, **WSL2에서 보고 계신 분**은 Linux 명령이 거의 그대로 돕니다. 다만 dmesg가 비어 보일 수 있는데 wsl 종료 후 재실행하면 나와요. 셋째, **컨테이너 안에서**는 ps가 호스트 프로세스를 못 봅니다. 그게 namespace 격리예요. Ch047에서 다시 다룹니다. 넷째, **Apple Silicon이나 Graviton 같은 ARM64**와 x86_64는 ps·top 출력이 거의 같지만 perf 같은 깊은 도구는 명령어가 다를 수 있어요. Ch091에서 다룹니다. 다섯째, **회사 정책으로 sudo가 막힌 분**은 dry-run 모드만 cron에 박으세요. 그것만으로도 평소 분량 추적이 됩니다. 권한 없이도 할 수 있는 운영이 의외로 많아요. 그리고 마지막으로 — 회사 보안팀과 친해지세요. emergency.sh를 운영 서버에 두려면 보안팀 검토가 필요한 회사도 많습니다. 100줄짜리 짧은 스크립트라는 게 검토 통과의 가장 큰 무기예요. 1000줄짜리 도구는 검토에서 막혀요. 짧음 자체가 운영의 자산입니다.

## 10. Ch003 다리 — 파일 시스템(inode·블록·디렉토리 트리·마운트)

자, 다음 챕터로 다리를 놓을 시간이에요. Ch003은 파일 시스템입니다. 한 줄로 요약하면 "디스크라는 냉장고를 OS가 어떻게 정리하는가"예요. Ch002에서 디스크는 냉장고라고만 비유했죠. Ch003에서는 그 냉장고 안의 선반과 라벨과 봉투의 이름을 배웁니다. 핵심 단어는 네 개. **inode** — 파일 한 개의 신분증(크기, 권한, 시간, 블록 주소들이 다 여기 적혀 있어요. 이름은 안 들어 있습니다. 이름은 디렉토리 안에). **블록** — 디스크의 최소 저장 단위. 보통 4KB. 1바이트짜리 파일도 블록 한 개를 잡아먹어요. 그래서 작은 파일 백만 개가 큰 파일 한 개보다 디스크를 더 먹는 일이 생깁니다. **디렉토리 트리** — 파일 시스템의 골격. /에서 시작해서 /Users, /etc, /var로 가지를 칩니다. 윈도우의 C 드라이브, D 드라이브와 달리 유닉스는 root 하나에서 모든 디스크가 가지로 합쳐져요. **마운트** — 새 디스크나 USB나 네트워크 드라이브를 트리 어딘가에 붙이는 행위. /Volumes/USB가 마운트 포인트예요. mount 명령어가 그걸 합니다. Ch003 8시간에서는 이 네 단어를 ls, stat, du, df, ln, find, mount, fsck, lsof로 손에 쥐고, 마지막에 "왜 git은 .git이라는 디렉토리에 모든 걸 넣는가"까지 갑니다. 비유 사전도 늘어요. 디렉토리는 호텔의 층, 파일은 방에 둔 짐, inode는 짐의 송장, 블록은 송장에 적힌 칸. 마운트는 옆 호텔과 복도를 잇는 행위. 이 다리가 머리에 그려지면 Ch003은 절반 이상 가져가신 겁니다. 그리고 Ch003이 끝나면 Ch004 — 네트워크 — 로 갑니다. 그땐 배달앱 비유가 7계층 OSI까지 확장돼요. 욕심내지 마시고 한 챕터씩 가시면 됩니다. 한 가지 약속을 미리 드리면, Ch003 H1에서 우리는 ls -la 한 줄의 의미를 30분 동안 풀어낼 거예요. 권한 문자 10칸, 링크 카운트, 소유자, 그룹, 크기, 시각, 이름. 이 일곱 칸이 사실 inode와 디렉토리의 모든 비밀을 담고 있거든요. ls 한 줄을 진짜로 읽을 줄 알게 되면 파일 시스템의 절반이 풀립니다.

## 11. 본인 페이스 가이드 + 8시간 약속 회수

8시간 약속을 회수합니다. 1교시 시작에서 제가 드린 약속은 세 개였어요. 하나, **OS를 호텔 매니저로 한 번에 그릴 수 있게 만들겠다.** 회수했습니다. 네 자원, 다섯 사고, 7단계 응급 절차, 다섯 원리가 한 그림이에요. 둘, **명령어를 외우게 하지 않고 손에 쥐게 하겠다.** 회수했습니다. ps와 7신호와 lsof와 ss를 procmon.sh와 emergency.sh라는 두 도구로 묶었어요. 셋, **Ch002 끝나면 새벽 3시 알람에 졸린 눈으로도 1차 대응이 가능한 사람이 되게 하겠다.** 회수했습니다. emergency.sh --dry-run --case oom 한 줄이면 5초 안에 답이 나와요. 자, 본인 페이스에 대해서. 8시간을 4주에 걸쳐 보신 분도 있을 거고, 8주에 걸쳐 보신 분도 있을 겁니다. 둘 다 정상입니다. 평균 두 해 코스에서 사람들이 Ch002에 쓰는 시간은 6주 ± 2주예요. 빠르다고 자랑할 일도 아니고, 느리다고 자책할 일도 아닙니다. 더 중요한 건 8주가 지난 시점에 손에 뭐가 남아 있느냐예요. 4주에 끝낸 사람이 손에 아무것도 없으면, 8주에 emergency.sh 하나 들고 나온 사람보다 못 갑니다. 다만 두 가지만 약속해 주세요. 첫째, **emergency.sh를 본인 노트북에 ~/bin/ 밑에 두고 권한을 주는 것**. 이건 오늘 안에 해 주세요. 둘째, **다음 7일 동안 매일 한 번 procmon.sh를 돌리는 것**. 일주일이면 본인 머신의 평소 분량이 손에 들어옵니다. 이 두 약속이 Ch002의 진짜 졸업장이에요. 책장에 책을 꽂는 게 졸업이 아닙니다. 손이 그 책의 명령어를 누르는 게 졸업입니다. 그리고 마지막 권유 — Ch003 시작 전에 이틀 정도 휴식을 두세요. 머리가 너무 빡빡한 채로 다음 챕터에 들어가면 첫 시간에서 30분도 못 가서 "아, 이거 봤던 거 같은데"가 흐려집니다. 충분한 간격은 학습의 일부입니다. 잠과 산책과 멍 때리기가 학습의 절반이에요. 진심입니다.

## 12. 한 줄 정리 + 두 해 코스에서 우리 위치

한 줄 정리. **운영체제는 호텔 매니저고, 우리는 8시간 동안 그 매니저와 다섯 가지 원리로 대화하는 두 도구(procmon·emergency)를 손에 쥐었다.** 두 해 코스 우리 위치는 16/960 = 1.67%입니다. 1교시 시작 때 0.83%였으니 정확히 두 배가 됐어요. 한 챕터씩 1%, 두 챕터에 2%, 다섯 챕터에 4%. 이 속도면 100주 즈음에 코스가 끝납니다. 멀어 보이지만 매주 1%씩 자라는 건 복리예요. 1년 뒤엔 다른 사람이 됩니다. 그리고 한 가지 더 — Ch001과 Ch002, 두 챕터를 끝낸 시점에서 우리는 이미 컴퓨터의 두 큰 기둥을 봤어요. 하드웨어(Ch001)와 운영체제(Ch002). 다음 Ch003은 그 두 기둥 위에 얹히는 첫 번째 추상화 — 파일 시스템 — 입니다. 그 위에 Ch004 네트워크가 얹히고, 그 위에 Ch005 데이터베이스가 얹혀요. 한 층씩 쌓이는 건축물이라고 생각하시면 됩니다. 1층, 2층은 끝났습니다. 3층으로 올라갈 시간이에요. 그리고 마지막 한 마디 — 오늘 배운 emergency.sh를 GitHub에 올려 두세요. 본인 dotfiles 레포에 두셔도 좋고, 따로 ops-toolkit 같은 레포를 파셔도 좋습니다. 이게 두 해 코스의 첫 번째 포트폴리오가 됩니다. 100줄짜리 셸 스크립트지만 "운영을 아는 사람"이라는 신호예요. 면접관이 README 한 페이지만 봐도 압니다. 자, Ch002는 여기서 닫습니다. 다음 시간 Ch003 H1 — 파일 시스템 오리엔테이션 — 에서 만나요. 그동안 ls -la /와 stat ~/.zshrc 두 줄만 쳐 두세요. 그게 Ch003의 첫 단어들이에요. 8시간 따라와 주신 분, 정말 수고 많으셨습니다.
