## 1. 4교시 회수 + 5교시 약속

자, 다섯 번째 시간이에요. 본인 8시간 약속의 5/8. 62.5%. 절반 넘었어요. 진짜 잘 견뎌 주셨어요. 잠깐 박수. 짝짝짝.

지난 한 시간(H4) 에 본인이 명령 18개를 손에 익혔어요. ps · top · kill · jobs · nohup · wait · `$$` 등등. 호텔 매니저(OS) 의 도구들. 한 명령씩 한 시간 손에. 머리가 좀 무거우셨을 거예요. 18개를 한 시간에 다 외우는 건 불가능. 외울 필요도 없어요.

그래서 5교시. **18개 명령을 묶어서 한 개의 셸 스크립트로 만드는 시간.** 이름은 `procmon.sh`. process monitor 의 줄임말. 본인 호텔의 진단 보고서를 한 줄 명령으로 뽑아 주는 도구. 본인이 두 해 후 매일 한 번씩 돌릴 도구. 60분 동안 100~150줄짜리 스크립트 한 개를 손에서 빚어요.

오늘 약속 한 줄. **"5교시 끝나면 본인 ~/myscripts/procmon.sh 가 동작한다."** 그리고 그 파일이 본인 손으로 만든 두 번째 셸 스크립트. 첫 번째는 Ch001 H5 의 sysinfo.sh. 두 번째가 오늘. 두 해 후 끝날 때 본인 ~/myscripts 에는 50~100개의 .sh 가 쌓여 있어요. 오늘이 두 번째. 본인 도구 가족이 한 명씩 늘어나는 중.

가요.

## 2. 셸 스크립트가 뭐였더라 — Ch001 H5 회수

Ch001 H5 에서 만든 sysinfo.sh 한 번 떠올려 봐요. 본인이 만든 첫 .sh 파일. 한 번 cat 해 봐요.

> ▶ **같이 쳐보기** — 두 주 전 만든 본인 첫 스크립트 다시 보기
>
> ```bash
> cat ~/myscripts/sysinfo.sh    # 본인 첫 스크립트
> ```

핵심 5가지가 있었어요. 셔뱅(첫 줄), set 가족(엄격 모드), 함수, main 호출, 변수. 오늘 procmon.sh 에서 정확히 같은 5가지를 다시 써요. 그리고 4가지를 추가. **trap, 옵션 파싱, OS 분기, while 루프.** 9가지 패턴이 본인 머리에.

**셸 스크립트의 진짜 정체.** 본인이 터미널에서 한 줄씩 치는 명령을 한 파일에 모아 둔 것. 그 파일에 실행 권한(`chmod +x`) 을 주면, 본인이 그 파일 이름만 쳐도 그 안의 명령들이 순서대로 실행돼요. 그게 다예요. 마법이 아니에요. 그냥 명령을 묶어둔 메모장.

근데 이 단순한 메모장이 두 해 후 본인의 진짜 무기예요. CI/CD 파이프라인, 배포 스크립트, 백업 스크립트, 모니터링 — 다 .sh 파일이에요. 본인이 두 해 후 매일 100번 호출하는 게 .sh 파일. 그래서 오늘 한 시간이 진짜 큰 의미.

## 3. procmon.sh 의 5가지 일 — 한 장의 그림

먼저 큰 그림. procmon.sh 가 무슨 일을 하느냐.

```
procmon.sh 한 번 호출
├── 1. 호텔 요약  — Processes/Load/CPU 한 줄씩
├── 2. CPU 상위  — 가장 CPU 먹는 5개
├── 3. 메모리 상위 — 가장 메모리 먹는 5개
├── 4. 좀비 사냥  — Z 상태인 프로세스 목록
└── 5. 본인 프로세스 — 본인 사용자가 띄운 것들
```

다섯 가지 일. 5분 안에 끝나는 진단. 본인이 매일 아침 한 번 돌리면 본인 시스템의 건강이 한 화면. 두 해 후 EC2 에 ssh 들어가서 첫 한 줄로 칠 명령.

옵션 세 가지도 둬요.

- `-n N` 상위 N 개 (기본 5)
- `-i SEC` watch 모드의 갱신 간격 (기본 3초)
- `-w` watch 모드 (한 번이 아니라 반복)

이 세 옵션으로 본인이 procmon.sh 를 다양한 상황에서 써요. 평소엔 `procmon.sh` 한 번. 사고 났을 땐 `procmon.sh -w -i 1` 로 1초마다 갱신. 두 해 후 새벽 세 시에 진짜 자주 쓸 패턴.

가요. 한 줄씩.

## 4. 첫 줄 — 셔뱅(shebang)

본인 procmon.sh 의 첫 줄. **무조건** 이거예요.

```bash
#!/usr/bin/env bash
```

`#!/usr/bin/env bash` — 셔뱅(shebang). #+! 두 글자. 1980년 BSD 에서 등장. 47년 된 첫 줄.

이 줄의 의미. **"이 파일을 실행할 때 /usr/bin/env 가 가리키는 bash 인터프리터로 실행하라."** OS 가 본인 .sh 파일을 만나면 첫 두 바이트(`#!`) 를 보고 "아 이건 인터프리터로 돌릴 파일이구나" 알아요. 그리고 첫 줄의 나머지에 적힌 인터프리터를 띄워서 본인 파일을 그 입력으로 던져요.

**왜 `/usr/bin/env bash` 이고 그냥 `/bin/bash` 가 아니냐.** macOS 의 /bin/bash 는 진짜 오래된 bash 3.2 (2007년). brew 로 새 bash (5.x) 를 깔아도 /bin/bash 는 안 바뀌어요. /usr/bin/env bash 는 본인 PATH 에서 bash 를 찾으니까 brew 의 새 bash 를 만나요. 더 안전.

추가 한 가지. macOS 의 기본 셸은 zsh 인데, 셸 스크립트는 보통 **bash** 로 짜요. 왜냐하면 더 표준적이고, Linux 어느 서버에 가져가도 동작해요. zsh 스크립트는 zsh 가 깔린 곳에서만. bash 가 더 안전한 선택. 셔뱅에 `bash` 라고 명시.

비유 하나. 셔뱅이 호텔 입구의 안내판이에요. "이 호텔의 매니저는 bash 입니다" 라는 한 줄. 본인 .sh 파일의 입구에 그 안내판이 걸려 있어요. 손님(OS 의 exec 시스템 콜 — H2 에서 외운 그 exec) 이 들어올 때 이 안내판을 먼저 봐요. 그리고 "아, bash 매니저에게 가야 하는구나" 알게 돼요. 만약 안내판이 없으면 손님은 본인 셸의 디폴트 매니저(macOS 면 sh, Linux 면 sh) 에게 가요. 그 디폴트 매니저가 본인이 의도한 매니저랑 다르면 본인 스크립트가 이상하게 돌아요. 그래서 무조건 첫 줄에 안내판. 첫 두 글자 #! 의 진짜 의미.

그리고 한 가지 작은 사실. 셔뱅 첫 줄은 OS 가 직접 읽어요. bash 가 읽는 게 아니라 OS 커널이 읽어요. 본인이 ./procmon.sh 라고 치면 OS 가 그 파일의 첫 두 바이트를 검사하고, 그 자리에 #! 가 있으면 첫 줄을 분석해서 인터프리터를 띄워요. 이 메커니즘이 1980년 BSD 에서 만들어진 거예요. 거의 50년 된 OS 의 작은 약속. 본인이 매일 의지하는 그 약속.

## 5. set 가족 — 엄격 모드 3종

두 번째 줄 (보통 빈 줄 한 칸 두고 셋째 줄). **set** 가족.

```bash
set -euo pipefail
```

세 가지 옵션을 한 줄에. 한 글자씩 풀어 봐요.

**`-e` (errexit).** 어느 명령이든 0 이 아닌 종료 코드를 반환하면 즉시 스크립트 종료. 즉, **실패하면 멈춤.** 디폴트 bash 는 명령이 실패해도 다음 줄 계속 실행해요. 그게 진짜 큰 함정. `-e` 를 켜야 본인 스크립트가 정직.

**`-u` (nounset).** 정의 안 된 변수 쓰면 에러. 디폴트는 빈 문자열. 빈 문자열이 더 위험한 이유 — `rm -rf $UNDEFINED/` 가 `rm -rf /` 가 돼요. 본인 시스템이 한 줄로 죽어요. `-u` 가 그걸 막아 줘요.

**`-o pipefail`.** 파이프(`|`) 의 종료 코드는 마지막 명령의 코드. `false | true` 는 0 (성공) 이에요. `pipefail` 을 켜면 파이프 안의 어느 명령이든 실패하면 전체가 실패. 더 정직.

이 세 옵션을 합친 `set -euo pipefail` 은 **셸 스크립트의 안전벨트**. 본인 .sh 파일의 거의 무조건 두 번째 줄. 두 해 후 본인이 만들 50~100개 .sh 의 99% 가 이 줄을 갖고 있을 거예요.

추가 한 가지. **`-x` (xtrace).** 실행하는 모든 명령을 화면에 출력. 디버깅용. 평소엔 안 켜요. 디버깅할 때만 `bash -x procmon.sh` 처럼 호출 시점에 켜요.

set -e 의 진짜 무서운 점 한 번 더. 본인이 set -e 없이 짠 50줄 .sh 가 한 번 돌면 어떻게 되냐. 첫 줄이 실패해도 둘째 줄 실행, 둘째 줄이 실패해도 셋째 줄 실행. 본인이 그 .sh 의 마지막 줄에 "echo 성공" 을 넣어 두면 5개 줄이 다 실패해도 마지막에 "성공" 이 출력. 본인이 그걸 보고 "음, 잘 돌았네" 하고 넘어가요. 다음 날 본인 시스템에 사고가 났을 때 그 .sh 가 사실 5번 실패했다는 걸 깨달아요. set -e 한 줄이 그 사고를 막아요. 본인 .sh 의 안전벨트라고 한 이유.

set -u 의 사례 한 번 더. 본인이 .sh 에 `BACKUP_DIR=$HOME/backup` 한 줄. 그리고 다른 자리에 `rm -rf $BAKUP_DIR/*` (오타). 디폴트 bash 면 BAKUP_DIR 이 빈 문자열이라 `rm -rf /*` 실행 — 본인 시스템 통째로 죽어요. set -u 가 켜져 있으면 BAKUP_DIR 정의 안 됨 에러. 본인 시스템이 살아남아요. 한 글자 오타가 본인 디스크 통째로 날릴 수 있는 게 셸의 본질. set -u 는 그 함정의 99% 를 막아요. 무조건 켜요.

pipefail 의 사례. `curl https://api.example.com | jq .data | grep error`. 본인이 이 한 줄 결과를 if 로 검사해요. curl 이 네트워크 끊겨서 빈 출력, jq 도 빈 출력, grep 도 빈 출력 — 그래도 grep 의 종료 코드는 1 (매치 없음). pipefail 없으면 마지막 grep 의 1 만 보고. pipefail 켜져 있으면 curl 의 실패도 보여요. 진짜 정직한 진단.

## 6. trap — Ctrl+C 깨끗하게

본인 procmon.sh 가 watch 모드로 돌고 있어요. 본인이 Ctrl+C 누르면 어떻게 될까. 그냥 죽어요. 임시 파일이 남아 있으면 청소 안 됨.

**trap.** 신호를 받았을 때 실행할 코드를 등록.

```bash
cleanup() {
  echo "정리 중..."
  rm -f /tmp/procmon.$$.tmp
}
trap cleanup EXIT INT TERM
```

`trap 함수명 신호목록`. 위 한 줄은 — "EXIT(정상 종료), INT(Ctrl+C), TERM(kill) 어느 것이든 받으면 cleanup 함수 실행."

`$$` 는 본인 스크립트의 PID. 임시 파일 이름에 PID 를 박으면 같은 스크립트 두 개가 동시에 돌아도 충돌 안 해요. 두 해 후 본인이 매일 쓰는 패턴.

본인 procmon.sh 에 임시 파일 안 써도 trap 한 줄은 둬요. 두 해 후 임시 파일 추가하는 순간 trap 도 같이 동작. 미리 골격.

```bash
#!/usr/bin/env bash
set -euo pipefail

TMPFILE="/tmp/procmon.$$.tmp"
cleanup() {
  rm -f "$TMPFILE"
}
trap cleanup EXIT INT TERM
```

여기까지 7줄. 본인 procmon.sh 의 첫 7줄. 이 7줄이 본인의 두 해 후 모든 .sh 의 골격. 외워두세요.

trap 의 비유 한 번. 본인 호텔 매니저가 사무실 벽에 "퇴근할 때 무조건 불 끄고 가세요" 라는 메모를 붙여 뒀어요. 매니저가 정상 퇴근하든(EXIT), 불난 도중에 도망가든(INT — Ctrl+C), 화재 알람으로 급히 대피하든(TERM — 외부 kill), 어느 경우든 "사무실 불 끄기" 는 실행. 그게 trap 의 약속이에요. 본인 .sh 가 어떻게 끝나든 정리는 한다. 신뢰할 수 있는 도구의 진짜 모습. 본인이 trap 한 줄 잘 둬 두면 두 해 후 본인 .sh 가 진짜 어떤 상황에서도 깨끗하게 동작해요. 지저분한 .sh 와 깨끗한 .sh 의 차이가 trap 한 줄. 한 줄로 본인 .sh 가 프로 수준이 돼요. 한 줄도 빼먹지 마세요.

여러 신호에 서로 다른 동작을 두는 패턴도 있어요. 정상 종료엔 한 메시지, Ctrl+C 엔 다른 메시지. 두 trap 을 따로 두면 돼요. 오늘 procmon.sh 는 한 줄짜리만. 두 줄짜리 분기는 두 해 후 자동화 챕터에서 자세히 봐요. 오늘은 한 패턴만 손에.

## 7. 변수 + 옵션 파싱

이제 옵션 세 개. `-n` `-i` `-w`.

먼저 기본값.

```bash
TOP_N=5
INTERVAL=3
WATCH=0
```

그리고 `getopts` 로 파싱.

```bash
while getopts ":n:i:wh" opt; do
  case $opt in
    n) TOP_N="$OPTARG" ;;
    i) INTERVAL="$OPTARG" ;;
    w) WATCH=1 ;;
    h) usage; exit 0 ;;
    \?) echo "Unknown option: -$OPTARG" >&2; usage; exit 1 ;;
    :) echo "Option -$OPTARG requires an argument" >&2; exit 1 ;;
  esac
done
shift $((OPTIND - 1))
```

`getopts` 의 5 가지 부분.

1. **`":n:i:wh"`** — 옵션 명세. 콜론 한 개가 첫 글자면 "에러를 직접 처리" 라는 신호. 글자 뒤에 콜론은 "그 옵션은 인자 필요" (예: `-n 10`).
2. **`opt`** — 파싱된 옵션 글자가 들어가는 변수.
3. **`$OPTARG`** — 옵션 인자 (예: `-n 10` 의 10).
4. **`\?`** — 알 수 없는 옵션.
5. **`:`** — 인자 빠진 옵션.

마지막 `shift $((OPTIND - 1))` 는 파싱된 옵션을 `$@` 에서 제거. 남은 인자를 처리하기 위해.

`usage` 함수도 둬요.

```bash
usage() {
  cat <<EOF
사용법: procmon.sh [-n N] [-i SEC] [-w]
  -n N    상위 N개 (기본 5)
  -i SEC  watch 갱신 간격 (기본 3)
  -w      watch 모드 (반복)
  -h      도움말
EOF
}
```

`<<EOF ... EOF` 는 **here document**. 여러 줄 문자열을 한 번에. 본인이 매일 쓰는 패턴.

## 8. OS 분기 — macOS vs Linux

H4 의 변환표에서 본 차이. `top -l 1` (macOS) vs `top -b -n 1` (Linux). 본인 procmon.sh 가 두 OS 에서 다 동작하려면 분기가 필요.

```bash
OS="$(uname -s)"
case "$OS" in
  Darwin)
    TOP_ONCE="top -l 1"
    PS_HEADER="USER PID %CPU %MEM COMMAND"
    ;;
  Linux)
    TOP_ONCE="top -b -n 1"
    PS_HEADER="USER PID %CPU %MEM COMMAND"
    ;;
  *)
    echo "지원 안 되는 OS: $OS" >&2
    exit 1
    ;;
esac
```

`uname -s` 는 H3 에서 본 명령. `Darwin` 또는 `Linux` 또는 `FreeBSD` 등. case 로 분기.

이 패턴이 본인 .sh 가 진짜 다양한 시스템에서 동작하게 하는 비결. macOS 에서 짠 스크립트가 EC2(Amazon Linux) 에서도 그대로. 두 해 후 본인이 매일 쓰는 호환성 패턴.

uname 의 다른 출력도 미리 되세요. **FreeBSD**, **OpenBSD**, **CYGWIN_NT-10.0** (Windows + Cygwin), **MINGW64_NT-10.0** (Windows + Git Bash), **MSYS_NT-10.0**. 웰 세상 다양해요. 하지만 두 해 코스에서 본인이 만나는 건 macOS 와 Linux 둘. 그 둘만 처리하는 게 오늘 procmon.sh 의 최점. 다른 OS 는 *) 에서 안전하게 거부.

그리고 한 가지 더. case 의 각 가지는 `;;` 로 끝나요. 더블 세미콜론 두 개. 다른 언어의 break 와 비슷한 역할. 한 세미콜론만 두면 그 분기 끝나고 다음으로 고자 안 떨어져요. 이게 술 스크립트의 작은 함정. 세미콜론 두 개 무조건 두 개.

## 9. 함수 1 — show_summary

함수의 첫 번째. 호텔 요약 5줄.

```bash
show_summary() {
  echo "=== 호텔 요약 ($(date '+%H:%M:%S')) ==="
  echo "사용자:   $(whoami)"
  echo "호스트:   $(hostname)"
  echo "OS:       $(uname -srm)"
  echo "Uptime:   $(uptime | sed 's/.*up //; s/, *load.*//')"

  echo "프로세스: $(ps -e | wc -l | tr -d ' ') 개"
}
```

여기서 `OS_MEM_CMD` 라는 변수를 OS 분기에서 미리 세팅해 둬요.

```bash
case "$OS" in

    OS_MEM_CMD="vm_stat | head -5 | tr -d '\n' | sed 's/ \{2,\}/ /g'"
    ;;
  Linux)

    ;;
esac
```

근데 이 한 줄이 너무 복잡해요. 더 간단히.

```bash
mem_info() {
  case "$OS" in
    Darwin) echo "$(sysctl -n hw.memsize | awk '{printf "%.1fG", $1/1024/1024/1024}')" ;;

## 10. 함수 2 — show_top_cpu

CPU 상위 N 개.

```bash
show_top_cpu() {
  echo ""
  echo "=== CPU 상위 $TOP_N ==="
  ps -eo pid,user,%cpu,%mem,comm | sort -k3 -nr | head -n $((TOP_N + 1))
}
```

`ps -eo pid,user,%cpu,%mem,comm` — H4 의 ps -o 패턴. 5개 열만. `sort -k3 -nr` — 3번째 열(%cpu) 을 숫자(`-n`) 역순(`-r`) 으로 정렬. `head -n $((TOP_N + 1))` — 헤더 1줄 + 본문 N 줄.

`$((TOP_N + 1))` 은 산술 평가. bash 의 작은 계산기. 본인이 매일 쓸 패턴.

## 11. 함수 3 — show_top_mem

메모리 상위 N 개. CPU 와 거의 같음. 정렬 키만 바뀜.

```bash
show_top_mem() {
  echo ""
  echo "=== 메모리 상위 $TOP_N ==="
  ps -eo pid,user,%cpu,%mem,rss,comm | sort -k4 -nr | head -n $((TOP_N + 1))
}
```

`-k4` 는 4번째 열(%mem). `rss` 도 추가해서 진짜 RAM 차지 양도 보여줘요.

여기서 한 가지 디자인 결정. show_top_cpu 와 show_top_mem 이 거의 같은데 따로 둘까, 한 함수로 합칠까. 두 해 후 본인이 만나는 트레이드오프.

**함치기.**
```bash
show_top() {
  local key=$1 label=$2
  echo ""
  echo "=== $label 상위 $TOP_N ==="
  ps -eo pid,user,%cpu,%mem,comm | sort -k$key -nr | head -n $((TOP_N + 1))
}
# 호출: show_top 3 "CPU"
# 호출: show_top 4 "메모리"
```

`local key=$1` — 함수 안에서만 사는 지역 변수. `$1` 은 첫 번째 인자.

본인이 두 해 후 만들 .sh 에서 자주 만나는 결정. 짧으면 따로, 길어지면 합치기. 오늘 procmon.sh 는 짧아서 따로 두어요. 가독성 우선.

한 가지 더. 함수 이름의 축차 궁자. 쿨래스타일 설명으로는 show_zombies 같이 **동사 항목으로 끝서 함수가 뭐를 하는지** 한 줄로 도는 이름이 좋아요. show_summary, show_top_cpu, show_top_mem, show_zombies, show_user_procs. 다섯 개 함수의 이름이 일관되게. 두 해 후 본인 코드를 다른 사람이 읽을 때 이름 일관성이 가장 큰 친절.

그리고 sort 의 몇 가지 옵션도 더 들어 두세요. `-k3` 는 3번 열 기준. `-n` 은 숫자. `-r` 은 역순. `-h` (human) 는 "5G" 같은 접미사 숨자자도 올바로 정렬. 두 해 후 `df -h | sort -hk3 -r` 같이 쓸 때 진짜 유용. 워래 이름은 *human-numeric*. sort 의 장명.

## 12. 함수 4 — show_zombies

좀비 사냥.

```bash
show_zombies() {
  echo ""
  echo "=== 좀비 프로세스 ==="
  local zombies
  zombies="$(ps -eo pid,ppid,stat,comm | awk '$3 ~ /^Z/ {print}')"
  if [ -z "$zombies" ]; then
    echo "(없음 — 호텔 깨끗 ✅)"
  else
    echo "PID  PPID STAT COMMAND"
    echo "$zombies"
    echo ""
    echo "⚠️  좀비 발견 — 부모 PPID 처리 필요 (H6 참고)"
  fi
}
```

`awk '$3 ~ /^Z/'` — 3번째 열이 Z 로 시작하는 줄만. H2 에서 외운 5상태(R·S·T·Z·I) 의 Z. 정규표현식 한 줄.

`if [ -z "$zombies" ]` — `-z` 는 빈 문자열인지. 좀비가 없으면 깨끗 표시. 있으면 빨간 경고. 본인 시스템 상태가 한 줄로.

여기 `local` 키워드. 함수 안에서만 살게. 함수 끝나면 사라짐. 함수 밖의 `zombies` 변수가 있어도 안 건드려요. **함수 안의 변수는 거의 무조건 `local`.** 두 해 후 본인이 절대 까먹으면 안 되는 한 줄.

## 14. main 한 줄 + while 루프

다섯 함수를 묶는 main.

```bash
main_once() {
  show_summary
  show_top_cpu
  show_top_mem
  show_zombies
  show_user_procs
}

main() {
  if [ "$WATCH" -eq 1 ]; then
    while true; do
      clear
      main_once
      echo ""
      echo "(${INTERVAL}초 후 갱신... Ctrl+C 로 종료)"
      sleep "$INTERVAL"
    done
  else
    main_once
  fi
}

main "$@"
```

`main_once` 는 한 번만 실행. `main` 은 watch 모드 분기. `while true` 무한 루프 + `clear` + `sleep INTERVAL`.

`main "$@"` — main 함수에 모든 인자 전달. `"$@"` 는 모든 인자 (원형 보존).

마지막 줄이 진짜 중요. 본인 .sh 의 마지막 줄은 거의 무조건 `main "$@"`. 함수 정의만 하고 main 호출 안 하면 아무 일도 안 일어나요. 흔한 실수.
한 가지 더 설명이 필요해요. 본인이 함수를 정의만 하고 호출을 안 하면 본인 .sh 는 그냥 함수의 정의만 메모리에 올리고 종료돼요. 아무 일도 안 일어나요. 그리고 본인이 그걸 모르고 "제 .sh 가 안 돌아요!" 하며 디버깅을 하고 있을 거예요. 두 해 동안 본인이 한 번은 이 실수를 해요. 모든 사람이 한 번용 사고이에요. main "$@" 한 줄. 쟊지 마세요.

`main "$@"` 의 따옴표 도 진짜 중요해요. `"$@"` 와 `$@` 의 차이. 따옴표 있으면 인자 하나하나가 공백이 있어도 따로따로 보존. "file with space.txt" 같은 인자가 한 개로 올바르게 전달. 따옴표 없으면 공백으면 세 개 인자로 쪼개져요. 따옴표 필수.

## 16. 디버깅 — bash -x 한 번

본인이 짠 스크립트가 안 돌면.

```bash
bash -x ~/myscripts/procmon.sh -n 3
```

`-x` 는 xtrace. 실행하는 모든 줄을 `+` 표시와 함께 출력. 본인이 어디서 멈췄는지 한 줄씩 보여줘요.
본인이 두 해 후 디버깅할 때 시간의 절반은 스크립트가 의도한 대로 안 돌아가는 경우. 그때 bash -x 가 첫 도구. 본인의 디버깅 외칩. 다른 방법은 해당 자리에 echo 추가. printf 디버깅 이라 부르고 50년 전부터 쓰이는 기법. 세계에서 가장 오래된 디버깅 방법.

추가 디버깅 한 가지. **echo + exit.** 의심되는 자리에 한 줄.

```bash
echo "DEBUG: TOP_N=$TOP_N OS=$OS"

```

이 한 줄이 본인이 진짜 자주 쓰는 디버깅 패턴. printf 디버깅의 셰 버전. 두 해 후도 매일.

## 17. cron 으로 매 5분 자동 (보너스)

본인 procmon.sh 를 자동으로 매 5분 돌리고 싶다면.

```bash
crontab -e        # 본인 cron 편집
# 아래 줄 추가:
*/5 * * * * /Users/mo/myscripts/procmon.sh >> /tmp/procmon.log 2>&1
```

cron 의 5필드. **분 시 일 월 요일.** `*/5` 는 5분마다. 그 뒤 4개 `*` 는 모든 시·일·월·요일.

크론 출력은 화면에 못 가니까 `>> /tmp/procmon.log 2>&1` 로 파일에 쌓아요. 본인이 다음 날 아침 그 파일 보면 24시간 동안 본인 시스템의 부하 추세가 한 화면.

>
> ```bash

## 18. FAQ 5

**Q1. set -e 가 켜져 있는데 한 명령이 실패해도 계속 가게 하려면?**
A. 그 명령 뒤에 `|| true` 를 붙여요. `command_that_might_fail || true`. 또는 `if command; then ... fi` 로 명시적으로. set -e 가 if 안에선 안 동작. 매일 쓰는 패턴. 그리고 한 가지 더. `set +e` 로 일시적으로 끌 수도 있어요. 몇 줄 실패 허용 구간 테두르고 다시 `set -e`. 도구가 풍부.

**Q2. shell-check 라는 도구 들었는데?**
A. shellcheck. 셰 스크립트의 lint 도구. `brew install shellcheck` 로 받고 `shellcheck procmon.sh` 한 번 돌려요. 본인이 못 본 함정 5~10개 잡아 줘요. 두 해 후 본인이 짤 모든 .sh 에 무조건 한 번. VS Code 확장도 있어요. 저장할 때마다 자동 검사. shellcheck 가 창은 함정의 종류 — 따옴표 빠진 변수(SC2086), 함수 안의 local 빠진 것(SC2155), `[ ]` vs `[[ ]]` 의 차이, 그리고 그 외 100종 이상의 실수 종목. 본인이 모르는 함정을 shellcheck 는 이미 알아요. 무조건 하나 깔아요.

**Q3. local 변수 안 쓰면 무슨 일?**
A. 전역 변수처럼 동작. 함수 밖의 같은 이름 변수를 덼어써요. 큰 스크립트에서 진짜 무서운 버그 원인. 함수 안의 모든 변수는 local. 무조건. 한 번 덼어쓰고 다음 날 디버깅하는 데 5시간 씬 수도 있어요. 자이제트 이쉬우. local 붙이는 한 글자 습관이 본인을 구해요.

**Q4. `"$@"` 와 `$*` 차이?**
A. `"$@"` 는 인자를 따로따로 (각 인자 그대로). `$*` 는 한 문자열로 합침. 본인이 99% 쓰는 건 `"$@"`. 따옴표 포함이 핵심. 따옴표 빼면 공백으로 단어 분리되어 의미 망함. 본인이 쓴 모든 .sh 의 main 은 main "$@". 한 글자의 고목. 이 한 글자가 두 해 후 "제 이름의 인수가 이상하게 올라와요!" 하는 버그의 원인. 따옴표 두 개 무조건.

**Q5. 셸 스크립트 vs Python 스크립트, 언제 어느 거?**
A. 한 줄로 — **셸 스크립트는 명령을 묶을 때, Python 은 데이터를 다룰 때.** 본인이 ps·grep·awk 같은 명령을 순서대로 호출 = 셸. JSON 파싱·HTTP 호출·데이터 가공 = Python. 본인이 셸 스크립트가 50줄 넘어가고 if 가 5개 넘어가면 Python 으로 옮길 신호. 그 경계가 본인 손에 익어요.
하나 더 추가. **셰 스크립트는 이식성이 진짜 큰 장점.** Python 스크립트는 Python 이 깔려 있어야 돌아요. 셰 스크립트는 모든 Linux · macOS · BSD 에 기본 깔려 있는 bash 로 돌아요. 두 해 후 본인이 EC2 AMI 처음 띄워서 Python 안 깔렸을 때, 그때에도 .sh 는 돌아요. 철들 장면. **처음 띄운 서버에서 첫 명령은 무조건 .sh.** Python 은 그 이후.

## 20. 흔한 실수 다섯 가지 + 안심 멘트 — 셸 스크립트 학습 편

본인이 오늘 100줄짜리 첫 셸 스크립트를 짜셨어요. 자주 빠지는 함정 다섯을 짚고 가요.

첫 번째 함정, 한 줄에 너무 많이 욱여 넣는다. 본인이 멋 부리려고 명령 다섯 개를 한 줄에 파이프로 다. 안심하세요. **명확한 다섯 줄이 짧은 한 줄을 이겨요.** Ch001 H5에서 약속한 그 원칙. 6개월 후 본인이 본인 코드 다시 보면 한 줄짜리는 못 읽어요. 다섯 줄짜리가 살아남아요. 코드는 컴퓨터에게가 아니라 미래 본인에게 친절하게.

두 번째 함정, set -e 안 쓴다. 본인이 셸 스크립트 첫 줄에 `#!/bin/bash` 만 쓰고 시작. 명령이 실패해도 다음 줄로 진행. 가짜 성공으로 사고 남. 안심하세요. **세 줄 표준으로 시작하세요.** `#!/usr/bin/env bash` + `set -euo pipefail` + 빈 줄. 이 세 줄이 본인 모든 .sh의 안전벨트. 첫 줄을 외워두세요. 두 해 동안 매번 이 세 줄로 시작.

세 번째 함정, 변수에 따옴표 안 붙인다. 본인이 `$FILE` 라고 쓰는데 파일 이름에 공백 들어 있으면 셸이 두 단어로 봄. 사고 남. 안심하세요, 함정 풀이부터. **변수는 항상 큰따옴표로.** `"$FILE"`. 30번 중 29번은 안 써도 동작하지만 1번이 사고. 그 1번이 새벽 3시 알람이에요. 매번 큰따옴표가 본인의 안전벨트. 30개 따옴표가 1번 사고를 막아요.

네 번째 함정, 에러 메시지를 stderr로 안 보낸다. 본인이 에러도 그냥 echo로 표준 출력에. 그러면 정상 출력과 에러가 섞여서 파이프로 못 분리. 안심하세요. **에러는 `>&2`로.** `echo "ERROR: 파일 없음" >&2`. 두 해 후 본인 코드를 자동화 시스템(cron, CI/CD)이 호출할 때, 정상 출력은 로그에, 에러는 알림으로. 두 채널 분리가 운영의 ABC. 첫날부터 습관 들이세요.

다섯 번째 함정, 가장 큰 함정. **하드코딩 절대 금지.** 본인이 `/Users/mo/myscripts` 같은 본인 경로를 코드에 직접. 다른 사람이 못 써요. 안심하세요. **`$HOME`·`$USER`·`$PWD` 같은 환경 변수 쓰세요.** 또는 `$(dirname "$0")` 으로 스크립트 위치 자동 잡기. 본인 스크립트가 본인 노트북에서만 동작하면 안 돼요. 본인이 새 컴퓨터 사도 동작해야 해요. 두 해 후 클라우드 서버에서도 동작해야 해요. 환경 변수가 그 휴대성의 진짜 비결. Twelve-Factor App 세 번째 원칙(Config in Environment)의 첫 단계.

다섯 함정 미리 알아둔 본인이 두 해 동안 한 박자 빠르게 손이 움직여요.

여섯 번째 함정 보너스. **본인 손으로 짜지 않고 ChatGPT/Claude한테만 시킨다.** 본인이 셸 스크립트 짤 때 매번 AI에 부탁만 하세요. 안심하세요. 처음엔 AI 도움 받아도 좋아요. 다만 **AI가 짜준 코드를 본인이 한 줄씩 다 이해하고 가셔야 해요.** 안 그러면 사고 났을 때 본인이 못 풀어요. 일곱 번째 함정 보너스. **너무 빨리 production에 올린다.** 본인이 procmon.sh 짜자마자 cron에 등록해서 매 5분 자동 실행. 안심하세요, 함정 풀이부터. **로컬 테스트 100번 → cron 1번이 표준.** 셸 스크립트는 한 번 사고 나면 시스템 전체에 영향을 미칠 수 있어요. 본인 procmon.sh가 무한 루프에 빠져서 CPU 100% 잡고 있는데 cron은 5분마다 또 실행. 시스템이 본인 스크립트로 가득 차요. 그래서 항상 (1) 로컬에서 100번 실행, (2) `bash -x` 로 한 줄씩 검증, (3) 로그 파일 확인, (4) 그 후에야 cron. 네 단계가 안전벨트예요. 한 단계라도 빼면 사고가 본인을 찾아와요.

여덟 번째 함정 보너스. **에러 핸들링을 잊는다.** 본인이 명령이 실패할 가능성을 안 생각해요. `cp file1 file2` 가 디스크 풀로 실패했는데 다음 줄에서 file2를 사용. 사고. 안심하세요. **명령 다음에 `if [ $? -ne 0 ]; then` 또는 `||` 로 처리.** 또는 `set -e`로 자동 종료. 셸 스크립트는 영원히 잘 돌 수 없어요. 언젠가 한 번은 실패해요. 그 실패를 어떻게 다룰지 미리 정하는 게 본인 스크립트의 진짜 안전성. 두 해 후 본인 production 코드의 50%가 에러 핸들링이에요. 첫 챕터부터 습관 들이세요.

여덟 함정 다 들으셨어요. 외우진 마세요. 그저 본인이 빠졌을 때 한 번씩 떠오르시면 돼요. 한 줄 욱여넣기·set -e 안 쓰기·따옴표 안 붙이기·에러 stderr 안 보내기·하드코딩·AI 의존·너무 빨리 production·에러 핸들링 누락. 여덟 함정을 미리 알아둔 본인이 셸 스크립트의 진짜 안전한 작가가 돼요.

## 👨‍💻 개발자 노트 (참고 — 비개발자는 그냥 넘기셔도 됩니다)

> - `#!/usr/bin/env bash` 셔뱅 패턴은 PATH의 첫 bash를 사용 — `/bin/bash` 직접 지정보다 호환성 우월. macOS의 /bin/bash는 3.2 (2007년) 고정, brew로 설치한 5.x는 PATH에 있음. env 패턴이 신 버전 자동 선택.
> - `set -euo pipefail` 4부분: -e (실패 시 즉시 종료), -u (미정의 변수 에러), -o pipefail (파이프 안 명령 실패도 잡음). 시니어 셸 스크립트의 표준 첫 줄. 본 챕터는 학습 부담을 위해 -e 만 우선 노출.
> - `trap 'cleanup' EXIT INT TERM` 패턴은 신호 처리의 표준. 임시 파일 정리·DB 연결 종료·로그 마무리 등에 필수. 두 해 후 production 셸 스크립트의 90%가 trap 한 줄 이상 포함.
> - getopts vs getopt 차이 — POSIX getopts는 셸 빌트인, 짧은 옵션만(-h). GNU getopt는 외부 명령, 긴 옵션(--help) 지원. macOS 기본은 BSD getopt(긴 옵션 미지원). 본 챕터는 getopts로 일관.
> - `local` 키워드는 함수 안 변수 스코프 제한. POSIX 표준 아니지만 bash/zsh 공통. 함수 작성의 기본 위생. fish 셸은 `set -l`로 다름.
> - while 루프의 무한 루프 + sleep 패턴은 cron 대안. cron은 분 단위 최소, 더 빠르면 셸 while + sleep. systemd timer가 더 표준. Ch 091 운영 시간에 다룸.
> - `main "$@"` 패턴은 셸 스크립트의 진입점 표준. C/Go/Java의 main()과 사상 동일. 함수 정의를 모두 위에 두고 마지막 한 줄만 main 호출 — 가독성 + 테스트 가능성 ↑.
> - cron 표현식 (분 시 일 월 요일)은 1970년대 유닉스부터 표준. macOS는 launchd plist가 대체, Linux는 cron + systemd timer 병존. 본 챕터는 portability 위해 cron 노출.
