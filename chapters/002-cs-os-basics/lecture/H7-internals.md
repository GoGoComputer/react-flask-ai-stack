## 1. 6교시 회수 + 7교시 약속

자, 일곱 번째 시간이에요. 본인 8시간 약속의 7/8. 87.5%. 진짜 마지막 한 발 직전. 본인 진심으로 잘 견뎌 주셨어요. 짝짝짝.

지난 6교시(H6) 에 본인이 다섯 사고 다섯 처방을 손에 넣었어요. 좀비·OOM·디스크 풀·네트워크·CPU 100%. 의사의 손. 그리고 한 가지 큰 원리를 만났죠 — **자원 한도**. 다섯 사고가 다섯 자원의 한도를 넘은 모습. 호텔 매니저(OS) 의 다섯 자원 관리.

오늘 7교시. **그 매니저가 왜 그렇게 강한지 — 그의 진짜 비밀 방.** 본인 OS 의 가장 깊은 곳. **kernel space** 와 **user space** 의 두 공간. H1·H2 에서 잠깐 본 두 공간을 오늘 진짜 깊이 들어가요. 한 시간 동안.

오늘 약속 한 줄. **"7교시 끝나면 본인이 syscall 한 번 칠 때 CPU 안에서 무슨 일이 일어나는지 7단계로 설명할 수 있다."** 그게 약속. 본인이 매일 치는 ls·cat·vim — 이 모든 명령이 결국 syscall 을 호출하고, syscall 이 호출되면 CPU 가 ring 3 에서 ring 0 으로 모드 전환하고, kernel space 에 들어가서 일을 하고, 다시 ring 3 으로 돌아와요. 그 7단계가 본인 머리에 박혀요.

왜 이게 중요하냐. 두 가지.

첫째, **성능.** 시스템 콜은 비용이 있어요. 함수 호출은 0.001 마이크로초, 시스템 콜은 0.1~1 마이크로초. 100~1000 배 비싸요. 본인이 두 해 후 진짜 빠른 코드를 짜려면 syscall 을 줄여야 해요. 줄이는 법은 syscall 의 진짜 동작을 알 때만 가능.

둘째, **보안.** 본인 코드가 user space 에서만 돌고, kernel space 의 진짜 위험한 일(파일 삭제·메모리 접근·하드웨어 제어) 은 kernel 만 할 수 있어요. 그래서 본인 앱이 망해도 OS 는 안 망해요. 그 보안 모델의 진짜 모습.

가요. 깊이.

## 2. 두 공간 — 왜 OS 가 둘로 나뉘었나

먼저 큰 그림 한 줄. **OS 는 두 공간이에요.** 모든 메모리·CPU 가 두 모드로 나뉘어 있어요.

**user space (사용자 공간).**
- 본인이 짠 모든 코드가 사는 자리.
- 본인 브라우저·VS Code·Python·Node 다 여기.
- CPU 모드: ring 3 (인텔), EL0 (ARM).
- 메모리: 각 프로세스마다 자기만의 가상 주소 공간.
- 권한: 제한적. 직접 하드웨어 못 만짐. 다른 프로세스 메모리 못 봄.

**kernel space (커널 공간).**
- OS 자체의 코드가 사는 자리.
- macOS 의 XNU 커널·Linux 의 vmlinux 커널 다 여기.
- CPU 모드: ring 0 (인텔), EL1 (ARM).
- 메모리: 모든 물리 메모리에 직접 접근 가능.
- 권한: 무제한. 하드웨어 직접 제어. 모든 프로세스 메모리 다 봄.

**왜 둘로 나눴냐.** 한 줄. **버그가 시스템 전체를 죽이지 않게.** 본인 브라우저가 크래시 나도 macOS 는 멀쩡하잖아요? 그게 두 공간 분리 덕분이에요. 브라우저가 user space 에서만 살아서, 그 공간 안에서 무슨 짓을 해도 kernel 은 안 다쳐요. macOS 가 죽으려면 kernel 자체가 버그여야 해요. 그래서 macOS 가 진짜 잘 안 죽어요.

만약 모든 코드가 한 공간에 있으면? 1980년대 MS-DOS 가 그랬어요. 한 앱이 잘못 짜이면 시스템 전체 다운. "응답 없음" 알림 자체가 없었어요. 그냥 컴퓨터 멈춤. 본인 부모님 세대가 이걸 매일 겪었어요. 두 공간 분리는 1990년대 NT(Windows) 와 BSD/Linux 가 본격 도입. 그 이후 컴퓨터가 진짜 안정됐어요.

**역사 한 줄.** 두 공간 모델의 아버지는 1965년 Multics. 50년 전 IBM 메인프레임에서 시작. 그 모델을 1969년 Unix 가 채택. 그 후 50년 동안 본인이 매일 의지하는 그 두 공간 모델이 안 바뀌었어요. 안 바뀐 이유 — **너무 잘 동작해요.**

## 3. 호텔 비유 확장 — 1층 vs 지하 금고

본인 호텔(OS) 비유를 한 번 더 깊이.

**1층 (user space).** 손님 방들이 있는 자리. 손님(앱) 들이 자유롭게 다닐 수 있어요. 호텔 식당, 라운지, 객실. 본인이 매일 보는 자리. 다만 손님끼리는 서로 방에 못 들어가요. 보안.

**지하 금고 (kernel space).** 호텔 매니저만 들어가는 자리. 호텔의 진짜 비밀이 여기 있어요. 손님 데이터, 결제 정보, 마스터 키, 화재경보 시스템. 손님은 절대 못 들어가요.

**손님이 매니저에게 부탁하는 방법.** 손님이 1층 로비의 인터폰을 들어요. "방 키 부탁합니다", "체크아웃 부탁합니다", "택시 불러 주세요". 매니저가 인터폰 받고 지하에서 일 처리. 그 인터폰이 정확히 **syscall**.

손님은 절대 지하로 못 가요. 왜? 마스터 키 하나로 모든 방을 열 수 있으니까. 한 명의 잘못이 호텔 전체 위험. 그래서 인터폰만 가능. 인터폰은 매니저가 통제. 손님은 부탁만, 결정은 매니저.

**이 비유가 왜 깊냐.** 본인 코드가 파일을 열려고 하면 — 본인 코드는 직접 디스크를 못 만져요. open() 함수를 호출. open() 은 CPU 명령어로 syscall 을 발동. 커널이 디스크에 가서 파일을 열고, 파일 디스크립터(fd) 를 본인 코드에 돌려줘요. 본인은 fd 만 받아요. 디스크는 절대 본인이 직접 못 만짐. 인터폰만.

이 모델이 두 해 후 본인이 만들 모든 코드의 진짜 작동 방식. 본인 브라우저 한 번 클릭이 100 번의 syscall. 본인 SQL 쿼리 한 번이 50 번의 syscall. 본인 ChatGPT 호출 한 번이 30 번의 syscall + 100 번의 네트워크 호출. **모든 게 syscall 위에서 돌아가요.**

## 4. CPU 모드 — ring 0 vs ring 3 (인텔 4링)

CPU 자체가 두 공간 분리를 도와줘요. 인텔 x86 CPU 는 **4개의 보호 링** 을 가지고 있어요.

```
       ring 0 — 커널 (가장 안)
        ring 1 — (안 씀, 거의)
         ring 2 — (안 씀, 거의)
          ring 3 — 사용자 코드 (가장 밖)
```

**ring 0** — 모든 명령 실행 가능. 모든 메모리 접근. 모든 하드웨어 제어.
**ring 3** — 일반 명령만. 자기 가상 메모리만. 하드웨어 못 만짐.

ring 1·2 는 1985년 인텔이 만들었는데 거의 안 써요. 마이크로커널 OS 가 한 번 시도. 지금은 ring 0 + ring 3 두 모드만.

**ARM CPU.** Apple Silicon 의 진짜 모습. ARM 은 **EL** (Exception Level) 이라고 불러요. EL0 (사용자), EL1 (커널), EL2 (하이퍼바이저), EL3 (보안). 본인 M1 맥은 EL0·EL1 에 코드, EL2 는 가상화(Docker Desktop) 가 씀, EL3 는 Apple 의 Secure Enclave (지문·페이스ID 보안 칩).

**CPU 모드 전환의 진짜 비밀.** ring 3 → ring 0 으로 가는 길은 딱 세 가지.

1. **시스템 콜** (정상 부탁) — `syscall` 명령어 또는 `int 0x80`.
2. **인터럽트** (외부 사건) — 키보드 누름·네트워크 패킷 도착.
3. **예외** (비정상) — 0 으로 나누기·페이지 폴트.

이 세 가지 외에는 절대 ring 0 에 못 들어가요. 그게 보안의 진짜 비결. 세 길 모두 OS 가 미리 등록한 "들어오는 자리" (interrupt vector) 로만. 본인 코드가 마음대로 ring 0 에 점프 못 해요. CPU 가 막아요.

본인 맥의 CPU 가 매초 본인 코드(ring 3) 와 커널(ring 0) 사이를 수만 번 왔다 갔다 해요. 본인 눈에는 안 보이는 그 무도회가 매초 일어나요.

## 5. 메모리 분할 — 가상 주소 공간의 두 영역

본인 프로세스 한 개의 메모리 모습.

```
0xFFFFFFFFFFFFFFFF  ┌────────────────┐
                    │ kernel space   │  ring 0 만 접근
                    │ (커널 코드·데이터) │
0xFFFF800000000000  ├────────────────┤
                    │       ↑        │
                    │   안 씀 영역     │
                    │       ↓        │
0x00007FFFFFFFFFFF  ├────────────────┤
                    │ user space     │  ring 3 도 접근
                    │  - 스택         │
                    │  - 힙           │
                    │  - 데이터        │
                    │  - 코드         │
0x0000000000000000  └────────────────┘
```

x86_64 의 진짜 메모리 지도. 64-bit 주소 공간 중 **상위 절반은 커널, 하위 절반은 사용자.** 모든 프로세스가 같은 커널 영역을 공유해요. 사용자 영역은 프로세스마다 따로.

**왜 같은 주소 공간에 둘 다 둬요?** syscall 이 빨라요. 만약 커널이 다른 주소 공간에 있으면 syscall 마다 주소 공간 전환 — 진짜 비싸요. 같은 공간에 두면 ring 0 모드만 켜면 됨. 한 명령으로.

다만 한 가지 보안 문제. 2018년 **Meltdown** 취약점이 발견됐어요. 사용자 코드가 트릭으로 커널 메모리를 엿보는 공격. 그래서 **KPTI** (Kernel Page Table Isolation) 라는 패치가 나왔어요. 커널과 사용자를 다른 주소 공간으로 분리. 보안은 올라갔는데 syscall 비용이 5~30% 늘어났어요. 두 해 후 본인이 들을 단어 — Meltdown·Spectre.

**본인 맥에서 한 번 봐요.**
```bash
sysctl vm.shared_region_size       # macOS 공유 영역
sysctl hw.memsize                  # 전체 RAM
sysctl vm.page_size                # 페이지 크기 (4096 또는 16384)
```

Apple Silicon 은 **16K** 페이지. Intel/AMD 는 4K 페이지. 페이지가 클수록 TLB(주소 변환 캐시) 효율 ↑, 메모리 낭비 ↑. 두 해 후 본인이 만들 고성능 앱에서 한 번씩 만나는 트레이드오프.

## 6. syscall 의 진짜 메커니즘 7단계 (H2 회수 + 깊이)

자, 이제 핵심. 본인이 `open("/etc/hosts", O_RDONLY)` 한 줄 호출했을 때 CPU 안에서 일어나는 7단계.

**1단계 — 사용자 코드.** 본인 C 코드의 `open()` 함수 호출. 이건 **glibc** (또는 macOS 의 libsystem) 의 wrapper 함수. 진짜 syscall 은 아님.

**2단계 — wrapper 가 syscall 번호 준비.** `open` 의 syscall 번호는 macOS x86_64 에서 5, ARM64 에서 5. 그 번호를 CPU 레지스터(RAX 또는 X16) 에 넣어요. 인자들도 다른 레지스터(RDI·RSI·RDX 또는 X0·X1·X2) 에.

**3단계 — `syscall` 명령어 발동.** CPU 가 그 한 명령어를 만나면 **즉시 ring 0 으로 모드 전환.** 동시에 미리 등록된 syscall 진입점(MSR_LSTAR 레지스터의 주소) 으로 점프. 이게 **trap**.

**4단계 — 커널이 받음.** 커널의 syscall handler 가 RAX 의 번호를 보고 그에 맞는 함수를 찾음. open 이면 sys_open() 호출.

**5단계 — 커널 작업.** sys_open() 이 진짜 일을 함. VFS (Virtual File System) 에 물어 inode 찾기, 파일 디스크립터 테이블에 fd 등록, 권한 검사, 파일 위치 0 으로 설정.

**6단계 — 결과 반환.** sys_open() 이 fd 번호(예: 3) 를 RAX 에 넣음. 또는 에러면 음수 (-1, -2 등).

**7단계 — `sysret` 명령어로 ring 3 복귀.** CPU 모드 다시 ring 3. 본인 코드가 RAX 의 값을 받아 변수에 저장. `int fd = open(...)` 의 fd 가 그 RAX.

**7단계 전체가 0.1~1 마이크로초.** 본인 머리로 1초 동안 1000~10000번 가능. 컴퓨터는 1초 동안 100만~1000만 번 가능. 빠르긴 한데 함수 호출보다 100~1000배 느려요.

**왜 그렇게 비싸냐.**
- ring 모드 전환 비용 (CPU 캐시·TLB 일부 무효)
- 레지스터 저장·복원
- 보안 검사 (인자 유효성, 권한)
- KPTI 적용 시 페이지 테이블 전환
- 캐시 미스 (커널 코드는 다른 주소)

이 비용을 줄이는 게 두 해 후 본인이 진짜 빠른 코드 짤 때 핵심. **syscall 을 묶기.** read 1000번 대신 read 1번에 1000 바이트. 100배 빠름.

## 7. trap·인터럽트·예외 — 세 가지 경계 통과 사건

ring 3 → ring 0 의 세 길을 한 번 더 정리.

**trap (synchronous, 의도적).**
- syscall 명령어
- 디버깅 트랩 (int 3, ptrace)
- 본인이 부탁한 일.

**interrupt (asynchronous, 외부).**
- 키보드·마우스 입력
- 네트워크 패킷
- 타이머 틱 (10ms 마다)
- 디스크 I/O 완료
- 외부에서 본인을 부름.

**exception (synchronous, 비정상).**
- 0 으로 나누기 (#DE)
- 페이지 폴트 (#PF) — 메모리 접근했는데 페이지가 없음
- 일반 보호 위반 (#GP) — 권한 없는 메모리 접근
- 본인 코드의 버그.

세 가지 모두 CPU 가 자동으로 ring 0 으로 전환하고, 미리 등록된 핸들러로 점프. **interrupt vector table** 의 256 자리 중 한 곳.

**페이지 폴트의 진짜 비밀.** 본인이 메모리에 접근하면 가상 주소 → 물리 주소 변환. 그 변환표(페이지 테이블) 에 그 주소가 없으면 페이지 폴트. 커널이 그때 디스크에서 읽어와 메모리에 올리고, 페이지 테이블 업데이트, 다시 본인 코드로. 본인은 모르는 사이에 일어나요.

**타이머 인터럽트가 멀티태스킹의 비결.** CPU 의 하드웨어 타이머가 보통 100~1000Hz 로 인터럽트 발생. 매번 커널이 깨어나서 "지금 어떤 프로세스 차례지?" 결정. 그게 **스케줄러** (Ch 062 에서 깊이). 본인 맥이 100개 앱을 동시에 돌리는 비결이 매초 100~1000번 일어나는 그 인터럽트.

## 8. context switch — 호텔 방을 바꾸는 비용

CPU 가 한 프로세스에서 다른 프로세스로 넘어갈 때 일어나는 일.

```
프로세스 A 실행 중
↓
타이머 인터럽트 (또는 syscall)
↓ ring 0 진입
스케줄러: "다음은 B 차례"
↓
A 의 모든 레지스터 저장 (16~32개)
A 의 페이지 테이블 포인터 저장
↓
B 의 레지스터 복원
B 의 페이지 테이블 포인터 적재
TLB flush (또는 부분 무효)
↓ ring 3 복귀
프로세스 B 실행
```

**한 번의 context switch 비용.** 1~10 마이크로초. syscall 보다 훨씬 비싸요. 왜? 페이지 테이블 전환 + TLB 무효화 + CPU 캐시 일부 무효.

**본인 맥에서 한 초에 몇 번?**
```bash
# Linux
vmstat 1 5
# cs 열이 초당 컨텍스트 스위치
```
보통 1000~50000 번/초. 본인 맥에 100개 프로세스가 있으면 매 프로세스가 1초에 10~500번 차례 받음. 본인이 매끄러운 마우스 움직임을 보는 비결.

**비용을 줄이는 방법.**
- 프로세스 수 줄이기 (프로세스 → 스레드 또는 비동기)
- CPU 친화도 (affinity) 설정 — 한 프로세스를 한 CPU 에 고정. 캐시 효율 ↑
- 스레드 풀 — 매번 새 프로세스 fork 안 함

두 해 후 본인이 만들 고성능 서버에서 만나는 핵심. Node·Go·Rust 의 비동기 모델이 이걸 줄이는 길. 본인이 두 해 후 chap 062 (성능) 에서 깊이.

## 9. strace · dtruss — syscall 엿보기 도구

본인이 두 공간의 경계를 직접 보는 도구.

**Linux: strace.**
```bash
strace ls                          # 모든 syscall
strace -e open ls                  # open 만
strace -c ls                       # 카운트 요약
strace -p <PID>                    # 돌고 있는 프로세스 attach
strace -f -p <PID>                 # 자식까지
strace -o trace.log ls             # 파일로
```

**macOS: dtruss.**
```bash
sudo dtruss ls                     # 모든 syscall (sudo 필요)
sudo dtruss -t open ls             # open 만
sudo dtruss -c ls                  # 카운트
sudo dtruss -p <PID>               # attach
```

dtruss 는 macOS 의 DTrace 위에서 동작. SIP (System Integrity Protection) 가 켜져 있으면 시스템 바이너리(/usr/bin/ls 같은) 는 못 봐요. 본인이 직접 짠 바이너리는 OK.

**왜 이게 진짜 강력한 도구냐.** 본인 앱이 느릴 때 — strace 로 보면 어떤 syscall 이 비싼지 한눈에. read·write 가 너무 많으면 → 버퍼링 부족. open·close 가 반복되면 → fd 재사용 안 함. socket 호출이 많으면 → 네트워크 문제.

두 해 후 본인이 신입에서 주니어로 갈 때 — 동료가 "왜 이 앱이 느려요?" 물으면 본인이 strace 한 줄로 보여주면 본인이 그 자리에서 영웅. 정말이에요.

## 10. 진짜 시연 — `dtruss -t open ls` 한 줄

본인이 지금 한 번 쳐 봐요. 본인 맥에서.

```bash
sudo dtruss -t open_nocancel ls / 2>&1 | head -30
```

(SIP 때문에 ls 직접은 못 볼 수 있음. 그러면 본인이 짠 작은 C 프로그램 또는 python:)

```bash
# python 으로 대안
sudo dtruss python3 -c "open('/etc/hosts').read()" 2>&1 | grep open | head
```

본인이 보는 출력:

```
open("/etc/hosts\0", 0x0, 0x0)        = 3 0
open("/usr/lib/libSystem.B.dylib\0", ...) = 3 0
open("/dev/null\0", ...) = 3 0
...
```

**해석.** 본인이 친 한 줄 ls 가 — 실제로 수십 번의 open 을 부름. /etc/hosts 한 번 외에도 libSystem 같은 라이브러리들, 설정 파일들. 본인 코드가 아닌 OS 가 알아서 부르는 것들.

이게 두 공간 분리의 진짜 모습. 본인 ls 는 user space 에서 살고, 진짜 파일 열기는 모두 kernel 에 부탁. dtruss 가 그 부탁들을 본인에게 보여줌.

**본인이 직접 시연:**
```bash
# 셸에서 진짜로 쳐 보기
echo '#include <stdio.h>
int main(){FILE*f=fopen("/etc/hosts","r");char b[100];fread(b,1,99,f);fclose(f);return 0;}' > /tmp/t.c
clang /tmp/t.c -o /tmp/t
sudo dtruss /tmp/t 2>&1 | head -20
```

본인 5줄 C 프로그램이 약 10~30 번의 syscall 을 부름. 본인 fopen 한 번 + fread 한 번 + fclose 한 번 = 3번 본인 코드 호출이 — 실제로는 그 안에 추가 syscall 들. 본인이 두 공간의 경계를 진짜 본 첫 순간.

## 11. 시스템 콜 비용 — 0.1~1μs 의 진짜 의미

숫자 한 번 더 깊이.

| 동작 | 비용 |
|---|---|
| 함수 호출 | 0.001 μs (1 ns) |
| L1 캐시 접근 | 0.001 μs |
| L2 캐시 접근 | 0.005 μs |
| L3 캐시 접근 | 0.02 μs |
| RAM 접근 | 0.1 μs |
| **시스템 콜** | **0.1~1 μs** |
| 컨텍스트 스위치 | 1~10 μs |
| SSD 랜덤 읽기 | 100 μs |
| HDD 랜덤 읽기 | 10000 μs |
| 네트워크 (같은 데이터센터) | 500 μs |
| 인터넷 1000km | 30000 μs |

**시스템 콜이 RAM 접근과 비슷한 비용.** 본인 머리에 박아 두세요. **함수 호출의 100~1000배.**

본인이 두 해 후 만들 코드의 진짜 시나리오:

**나쁜 코드:**
```python
for i in range(1000000):
    f.write(str(i))   # 매번 syscall (잠재적)
```
→ 100만 번 syscall = 100ms ~ 1초.

**좋은 코드:**
```python
buf = []
for i in range(1000000):
    buf.append(str(i))
f.write(''.join(buf))   # syscall 1번
```
→ 1번 syscall = 0.001초. **1000배 빠름.**

이게 **버퍼링.** 본인 stdio (printf·fwrite) 가 자동으로 해 주는 일. 그래서 printf 가 빠른 거예요. 직접 write syscall 부르면 진짜 느려요.

두 해 후 본인이 만나는 단어 — **buffered I/O vs unbuffered.** 본인이 fwrite 쓰면 버퍼 ON, write 직접 쓰면 OFF. 본인이 로그 라이브러리 만들 때 — 100ms 마다 flush 하면 100배 빠르고 안전.

## 12. 왜 user space 가 kernel 을 직접 못 부르나 (보안 5층)

본인이 ring 0 에 직접 못 들어가는 이유 5겹.

**1층 — CPU 하드웨어.** ring 3 에서 ring 0 명령(예: HLT, CLI) 실행하면 CPU 가 즉시 General Protection Fault. 본인 코드 죽음.

**2층 — 페이지 테이블 권한 비트.** 모든 메모리 페이지에 "user/supervisor" 비트. supervisor 페이지에 user 코드 접근 = page fault.

**3층 — syscall 진입점만 등록됨.** ring 0 으로 가는 길은 OS 가 미리 등록한 한 자리(MSR_LSTAR) 만. 그 외 점프 = trap.

**4층 — syscall 인자 검증.** 커널이 본인 인자(포인터·크기) 를 다 검증. 잘못된 포인터 보내면 EFAULT 에러.

**5층 — 권한 검사.** 파일 권한·UID·capability. root 가 아니면 /etc/passwd 못 씀.

이 5층이 본인 컴퓨터를 안전하게 지켜요. 한 층만 뚫어도 진짜 위험. 그래서 OS 보안 패치가 매달 나와요. 한 층의 작은 구멍이 큰 사고가 되니까.

**진짜 사례.** 2014년 **Shellshock** — bash 의 환경 변수 처리 버그. user space 코드 한 줄로 임의 명령 실행. 5층 중 5층(권한 검사) 을 우회. 결국 인터넷 서버 수백만 대가 위험. 2017년 **WannaCry** — Windows SMB 의 syscall 처리 버그. ransomware 가 전 세계에 퍼졌어요. 두 공간 모델은 강하지만 — **OS 자체에 버그가 있으면 전부 무너져요.**

## 13. mmap — 두 공간 사이의 다리

두 공간 사이의 데이터를 **복사 없이** 공유하는 마법. **mmap (memory map).**

**일반 read 의 문제.**
```
디스크 → 커널 버퍼 → 사용자 버퍼
```
2번 복사. 메모리 2배 사용.

**mmap 의 진짜 모습.**
```
디스크 → 커널 버퍼 ← 사용자 가상 주소가 가리킴
```
복사 없음. 같은 메모리를 두 공간이 같이 봄.

```c
int fd = open("big.bin", O_RDONLY);
void *p = mmap(NULL, size, PROT_READ, MAP_SHARED, fd, 0);
// p 가 파일 내용을 가리킴. read 안 해도 됨.
```

**왜 빠르냐.** read 의 복사 비용 + syscall 비용이 사라짐. 첫 접근에만 페이지 폴트 발생, 커널이 디스크에서 읽어옴. 그 후로는 일반 메모리 접근.

**본인이 두 해 후 만나는 자리.**
- DB 의 페이지 캐시 (PostgreSQL·SQLite)
- ML 모델 로딩 (HuggingFace 의 safetensors)
- Redis 의 영속화
- nginx 의 정적 파일 서빙

**Ch 087 알고리즘** 챕터의 큰 데이터 처리에서 한 번 더 만나요. **Ch 062 성능** 에서 mmap vs read 벤치마크. 두 해 후 본인이 진짜 큰 데이터 다룰 때 mmap 한 줄이 본인을 영웅으로.

## 14. eBPF — 두 공간을 연결하는 신기술 (보너스)

요즘 핫한 단어. **eBPF** (extended Berkeley Packet Filter).

**한 줄.** user space 의 프로그램을 kernel space 안에서 안전하게 실행하는 기술.

옛날엔 kernel 에 코드를 넣으려면 커널 모듈 작성 → 컴파일 → insmod. 위험. 한 번 잘못하면 시스템 다운. eBPF 는 작은 sandbox 에서 검증된 코드만 커널에서 실행. 안전.

**eBPF 가 가능하게 한 것:**
- 커널 안의 패킷 처리 (Cilium, Calico)
- 커널 안의 프로파일러 (bcc tools, bpftrace)
- 커널 안의 보안 정책 (Falco, Tetragon)
- 시스템 옵저버빌리티 (Pixie)

본인이 두 해 후 SRE 또는 DevOps 자리에 가면 eBPF 매일 만나요. 2020년대의 진짜 큰 변화. **Ch 116 옵저버빌리티** 에서 X-Ray 와 함께 eBPF 한 번 더.

본인이 지금 알 한 줄 — **두 공간 모델의 50년 벽을 안전하게 뚫는 방법이 나왔다.** 미래의 OS 는 이 방향으로 진화 중.

## 15. macOS XNU vs Linux — 두 커널의 가족 나무

H1 의 가족 나무 회수. 두 커널의 진짜 차이.

**Linux 커널.** 1991년 리누스가 시작. **모놀리식.** 모든 게 한 큰 커널 안. 파일 시스템·네트워크·드라이버. 빠르지만 큼.

**macOS 의 XNU.** XNU = X is Not Unix. 두 부분.
- **Mach** (마이크로커널) — CMU 카네기 멜런 1985년. IPC·메모리·스케줄링.
- **BSD 레이어** — FreeBSD 의 시스템 콜·파일 시스템·네트워크.
- **I/O Kit** — 드라이버.

**하이브리드.** 마이크로커널 + 모놀리식의 좋은 점만. 1996년 NeXT 의 Steve Jobs 가 Apple 에 돌아오면서 가져왔어요. 본인 macOS 는 진짜로 1985년 Mach + 1990년대 BSD + 2000년대 Apple 의 합작.

**Windows NT.** 1993년 Dave Cutler. **하이브리드.** Mach 영향 받았는데 Microsoft 만의 길. NT 4.0 부터 GDI(그래픽) 를 커널 안에 넣어 빠르게 함. 보안 vs 성능 트레이드오프.

본인이 두 해 후 만날 OS 들 — Linux (서버), macOS (개발 노트북), Windows (게임/회사), Android (Linux 기반), iOS (XNU 기반). 다섯 OS 가 두 공간 모델 위에 다 올라가요. 한 모델 알면 다섯 다 알 수 있음. 그게 OS 학습의 진짜 가치.

## 16. 마이크로커널 vs 모놀리식 — 30년 논쟁

**모놀리식.** 모든 OS 기능이 ring 0 에 한 큰 덩어리. 빠름. 단점 — 한 버그가 시스템 전체. 예: Linux.

**마이크로커널.** ring 0 에는 최소한(IPC·스케줄링·메모리) 만. 파일 시스템·네트워크·드라이버는 모두 ring 3 에서 따로 프로세스로. 안전. 느림. 예: Mach, MINIX, QNX, Hurd.

**Tanenbaum vs Linus 1992.** 인터넷의 전설적 논쟁. 운영체제 교과서 저자 Tanenbaum (마이크로커널 옹호) 이 "Linux 는 시대에 뒤떨어졌다" 비판. 리누스가 답글 — "이론과 실제는 달라요. 빠른 게 중요해요." 결론은 — **둘 다 살아남았어요.** Linux (모놀리식) 가 실용에서, 마이크로커널이 임베디드·자동차·방위에서.

**현대의 흐름.** 하이브리드. macOS XNU·Windows NT·Google Fuchsia (Zircon 마이크로커널). 좋은 점만 골라요. Linux 도 점점 모듈화. 30년 논쟁의 결과 — **실제로는 가운데가 답.**

이게 본인이 두 해 후 시스템 디자인 면접에서 만나는 단골 질문. "마이크로커널 vs 모놀리식 의 차이?" 본인이 두 줄로 답하면 면접관 눈빛 변함. 약속.

## 17. FAQ 5

**Q1. ring 0 코드는 본인이 짤 일이 있나요?**
A. 신입 5년 안엔 거의 없음. 드라이버·커널 모듈·BPF 정도. 두 해 후 본인 자리에서는 ring 3 만. 다만 ring 0 의 동작을 알면 ring 3 코드가 진짜 빨라져요.

**Q2. syscall 을 줄이는 일반 패턴 한 줄?**
A. **버퍼링·배치·캐싱.** 한 번 부를 일을 묶기. read 1000번 → read 1번 큰 버퍼. log 매줄 flush → 100ms 마다 flush. DB 쿼리 100개 → 1개 batch 쿼리.

**Q3. macOS 에 strace 가 왜 없어요?**
A. macOS 가 DTrace 를 채택. dtruss 가 그 위에서 strace 와 비슷한 일. 다만 SIP 때문에 시스템 바이너리는 못 봐요.

**Q4. eBPF 는 macOS 에 있나요?**
A. 없어요. Linux 전용. 비슷한 게 macOS 의 DTrace + Endpoint Security Framework. macOS 는 Apple 이 자기 길을 가요.

**Q5. 두 공간 분리 없이 OS 를 만들면 어떻게 되나요?**
A. MS-DOS·Windows 9x·고전 Mac OS 가 그랬어요. 한 앱 죽으면 시스템 죽음. 진짜 불편. 1990년대 후반 모두 두 공간 모델로 이동. 그 이후 컴퓨터 안 죽음.

## 18. 두 해 후 본인이 만나는 두 공간 — 깊이의 진짜 가치

본인이 두 해 후 만날 다섯 자리.

**1. 성능 튜닝.** 본인 백엔드가 느릴 때. strace 한 줄로 어떤 syscall 이 많은지. 90% 가 read 면 → 버퍼링. 90% 가 socket 호출이면 → keep-alive·connection pool. 한 시간에 결론.

**2. 보안 사고 분석.** 본인 서버가 해킹당했을 때. 어떤 syscall 이 비정상이었는지. auditd 로그·eBPF 트레이스. 두 공간 모델을 알면 어디서 뚫렸는지 한 번에.

**3. 컨테이너 운영.** Docker/Kubernetes (Ch 047) 가 두 공간 모델의 진짜 응용. namespace·cgroup 모두 ring 0 의 isolation. 본인이 두 공간 알면 컨테이너의 진짜 모습이 보여요.

**4. 시스템 면접.** "syscall 한 번의 비용?", "context switch 의 비용?", "user/kernel space 차이?". 본인이 7단계로 답하면 면접관이 멈춤.

**5. 진짜 큰 시스템.** 본인이 두 해 후 만들 ML 추론 서버·트레이딩 봇·실시간 게임 서버. 모두 syscall 비용을 줄여야 살아남는 자리. 본인의 두 공간 이해가 본인 코드를 100배 빠르게.

**진짜 한 줄.** 두 공간 모델은 50년 묵은 모델인데 — **두 해 후에도 그대로.** 본인이 오늘 한 시간 들이면 본인의 다음 30년이 편해요. 진짜.

## 19. 12회수 지도 — H7 의 자리

이 챕터의 큰 그림에서 H7 의 자리.

```
H1 → 호텔 매니저 (정체성)
H2 → 4개념 (프로세스·스레드·파일·syscall)
H3 → uname/sysctl (집 점검)
H4 → ps/top/kill (도구)
H5 → procmon.sh (자동화)
H6 → 응급처치 (사고)
H7 → kernel vs user space (깊이) ← 지금
H8 → emergency.sh + 회고 + Ch003 다리
```

H1 이 정체성, H2 가 어휘, H3·H4 가 도구, H5 가 자동화, H6 가 응급, **H7 이 OS 의 진짜 깊이.** H2 의 syscall 한 단어를 30분 동안 풀었어요. 이게 OS 학습의 가장 깊은 자리.

**12회수 지도** (H1 의 그 지도) 에서 H7 의 자리:
1. 호텔 매니저 ← H1
2. 유닉스 가족 나무 ← H1
3. 4개념 ← H2
4. 디바이스 = 파일 ← H2
5. uname/sysctl ← H3
6. ulimit ← H3
7. ps/top ← H4
8. kill ← H4
9. 자동화(셸) ← H5
10. 응급처치 ← H6
11. **두 공간 ← H7 (지금)**
12. 회고 ← H8

11번. 12 중 11번. **H8 한 시간만 남았어요.**

## 20. 마무리 — H8 적용+회고로

자, 7교시 끝. **본인 진짜 잘 견뎠어요.** 한 시간 동안 ring 0·ring 3·syscall 7단계·페이지 폴트·context switch·mmap·eBPF·XNU·Tanenbaum-Linus 논쟁까지. 이게 컴퓨터 사이언스 학과 OS 수업 한 학기 분량.

**오늘 약속 회수.** 본인 약속 — "syscall 한 번 칠 때 7단계로 설명." 본인 한 번 머리에서 그려 봐요.

1. 본인 코드 → open() 호출
2. wrapper 가 RAX 에 syscall 번호
3. `syscall` 명령어 → ring 0 진입 (trap)
4. 커널 syscall handler → sys_open()
5. 커널이 진짜 일 (VFS·fd 등록·권한)
6. 결과 RAX 에 반환
7. `sysret` → ring 3 복귀, 본인 코드가 결과 받음

7개. 손가락 7개. **본인이 외운 거예요. 진짜.**

**한 줄 더.** 본인이 두 해 후 면접관 앞에서 이 7단계를 손가락으로 짚으면 — 면접관이 멈춰서 본인 얼굴 봄. "이 사람 OS 진짜 아네." 한 시간이 본인 인생을 바꿔요.

다음 8교시. **마지막 시간.** **emergency.sh 완성** + **Ch002 전체 회고** + **12회수 지도 갱신** + **Ch003 (메모리·캐시) 다리.** 본인이 H1 의 호텔 매니저로 돌아와서, 한 챕터 8시간을 한 줄로 정리하고, 다음 챕터 인사. 거의 다 왔어요.

본인 진짜 잘했어요. 7/8. 한 시간만 더. 가요.

추신 하나. 본인이 오늘 외울 한 단어 — **trap.** 사용자에서 커널로 가는 그 한 순간의 이름. 이 단어 하나가 OS 의 비밀의 문을 여는 열쇠. 본인이 두 해 후 동료가 "trap" 한 단어 말하면 본인이 그 자리에서 "ring 0 진입 메커니즘" 한 줄 답해요. 그 순간 본인이 동료 위로 올라가요. 진짜.

추신 둘. ring 0 vs ring 3 의 정치 비유 한 번. ring 3 = 시민. ring 0 = 정부. 시민은 도로 교통법(syscall) 을 통해 정부 서비스 받음. 직접 국방부 서버 못 만짐. 정부만 그 권한. 시민이 정부에 부탁만 함. 호텔 비유와 같은 구조. 본인이 매일 사는 정치 모델이 본인 컴퓨터 안에도 그대로.

추신 셋. 페이지 폴트의 진짜 무서운 이야기. 본인 메모리가 부족하면 커널이 일부 메모리를 디스크(swap) 로 옮겨요. 본인 코드가 그 주소 접근 → 페이지 폴트 → 커널이 디스크에서 읽어옴 → 0.0001ms 가 100ms 로. **10만 배 느려져요.** 본인 맥이 갑자기 버벅대면 십중팔구 swap 발생. **활성 메모리 모니터** 의 swap used 가 그 신호. 0 이면 OK, 수백 MB 면 위험.

추신 넷. 본인이 좋아하는 게임이 빠른 비결. AAA 게임은 syscall 을 진짜 적게 부름. 게임 시작 시 모든 자산을 mmap 으로 메모리에 올리고, 그 후 syscall 거의 없음. 60fps = 16.6ms 마다 한 프레임. syscall 한 번이 0.1~1μs 면 16ms 안에 16000~160000 번 가능. 게임은 그 한도 안에서 살아요.

추신 다섯. **본인이 짠 모든 코드는 user space 에서만 살아요.** 영원히. 본인이 그 위에서 모든 일을 할 수 있어요. 백엔드·프론트엔드·ML·게임. 모두 ring 3 의 자리. ring 0 은 OS 가 알아서. 본인은 부탁만. 그 분리가 본인을 자유롭게 함. 본인이 짠 한 줄이 시스템 전체를 죽일 일 없음. 본인이 안전하게 실험할 수 있음. **두 공간 모델은 본인의 자유의 비결.**
